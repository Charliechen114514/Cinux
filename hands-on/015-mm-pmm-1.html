015-1 E820 内存映射解析与位图基础 ​

导语 ​

到 tag 014 为止，我们的内核已经能往屏幕打字、能收到键盘输入了。但如果你想问一个最基本的问题——"这台机器到底有多少物理内存可用？"——内核是一问三不知的。从这一章开始，我们要给内核装上"眼睛"，让它看清楚物理内存的布局，然后用一个位图分配器把这些内存管理起来。

这是整个内存管理子系统（PMM -> VMM -> 堆分配器）的第一步——物理内存管理器。没有 PMM，后面所有的页表构建、进程地址空间、slab 分配器都是空中楼阁。就像盖房子一样——PMM 是地基，VMM 是框架，堆分配器是内装修。地基没打好，上面再漂亮也是白搭。

这一章我们聚焦两件事：第一，从 BootInfo 中提取 BIOS E820 内存映射，经过过滤和对齐后得到"真正可用的物理内存区域列表"；第二，设计位图的核心辅助函数——set、clear、test 和 find_first_free——它们是后面所有分配操作的地基。

知识前置：你需要了解 BootInfo 结构体（tag 004 中 bootloader 填写、内核通过物理地址 0x7000 读取的那个 824 字节的结构体），以及基本的位运算操作（移位、掩码、ctz）。我们在 tag 006 中已经在 mini kernel 里实现过一个简易 PMM，这次是它的"大内核升级版"——接口更完整、扫描更快、测试更完善。如果你已经忘了 tag 006 的内容也不用担心，这一章会从头讲起，只是遇到重叠的部分会加快节奏。

概念精讲 ​

这一章涉及的概念并不复杂，但它们组合在一起形成了一套完整的内存管理基础设施。我们先从最宏观的视角看一遍整个 PMM 的数据流：BIOS 通过 E820 告诉我们物理内存的布局 -> parse_memory_map 过滤和对齐后输出干净的可用区域列表 -> 基于最高地址计算位图大小 -> 位图通过 set/clear/test 操作跟踪每一页的状态 -> bm_find_first_free 用 64 位加速扫描快速找到空闲页。理解了这个数据流，后面每一步的具体实现就只是填充细节了。

E820 内存映射：BIOS 给你的"内存地图" ​

在 x86 PC 启动的实模式阶段，BIOS 提供了一组 INT 15h 中断服务来查询物理内存布局。其中最常用的是 E820 功能（AH=0xE820），它返回一个内存区域列表，每个区域包含四个字段：基地址、长度、类型、ACPI 属性。类型值是我们最关心的——type=1 表示"可用 RAM"，type=2 表示"保留"（比如 BIOS ROM、MMIO 区域），type=3 和 type=4 是 ACPI 相关的。Cinux 的 bootloader 在实模式阶段已经调用过 E820，把结果填入了 BootInfo 结构体的 mmap[] 数组（最多 32 条记录，每条 24 字节）。

这里有一个关键的认识：E820 返回的列表既不保证有序，也不保证不重叠。BIOS 的实现质量参差不齐，有些 BIOS 会返回重复的区域或者顺序混乱的条目。我们的解析函数必须能处理这些情况——好在"无序"对位图操作来说不是问题，因为我们只是逐个标记每个区域的可用性，顺序无所谓。而"重叠"的情况在 QEMU 和主流 BIOS 中基本不会出现，所以我们暂时不做合并处理。

这个 E820 接口最早出现在 1996 年左右的 BIOS 规范中，目的是替代之前只能报告总内存大小的旧接口（INT 15h AH=0x88）。旧接口只能告诉你"机器有 64MB 内存"，但不知道这些内存分布在哪里、中间有没有 MMIO 洞。E820 的出现让操作系统能精确地知道物理地址空间中的每一处"凹凸不平"——哪些是真正的 RAM，哪些是 PCI MMIO 窗口，哪些是 ACPI 回收区域，哪些是永久保留的区域。对于现代操作系统来说，这是启动阶段最关键的数据之一，没有它你根本无法安全地使用物理内存。

低 1MB 过滤的必要性 ​

E820 可能会报告 0x00000-0x9FFFF 这段"传统低内存"为 type=1（可用），但这段空间实际上充满了各种固件数据结构：中断向量表（IVT，0x0000-0x03FF）、BIOS 数据区（BDA，0x0400-0x04FF）、BootInfo 结构体本身（0x7000）、E820 缓冲区（0x5000）、VESA 信息区（0x6400），以及 0xA0000-0xFFFFF 的视频内存和 BIOS ROM。这些布局是 PC 兼容机的固件约定，并非 Intel 架构规范的一部分——OSDev Wiki 的 "Memory Map (x86)" 页面有完整的梳理。第一兆字节在 PC 架构中是"神圣不可侵犯"的，充满了约定俗成的固定用途区域。

所以 Cinux 设定了一条硬规则：LOW_MEM_BOUNDARY = 0x100000（1MB）。所有起始地址低于 1MB 的可用区域，要么被整体跳过（如果整个区域都在 1MB 以下），要么被截断到 1MB（如果跨越了 1MB 边界）。这个策略虽然保守——我们可能白白浪费掉传统低内存中确实空闲的某些部分——但换来了绝对的安全。在一个教学 OS 里，这一点点浪费完全值得。

4KB 对齐与页粒度 ​

Intel 的 4 级分页机制（long mode）使用 4KB 作为最小的页大小，页表项中的物理基地址必须是 4KB 对齐的（低 12 位必须是零，因为它们被用作标志位）。Intel SDM Vol.3A Section 4.5 明确规定了这一点。这意味着我们的 PMM 也必须以 4KB 为最小分配粒度——不可能分配"半个页"或者"不对齐的 4096 字节"。

所以 parse_memory_map 的最后一步是把每个可用区域的基地址向上对齐到 4KB 边界、长度向下对齐到 4KB 边界。如果对齐后长度不足一个页（4096 字节），这个区域就被丢弃——不到 4KB 的碎片对页分配器来说没有意义。

对齐操作的核心公式是 (base + PAGE_SIZE - 1) & ~(PAGE_SIZE - 1)——先用加法把 base "推"到下一个对齐边界（如果它本来就不对齐的话），然后用 AND 操作截断低 12 位。长度的处理类似：先减去对齐带来的偏移量，再用 AND 操作向下截断。这两个操作合在一起保证了输出的区域一定是 4KB 对齐的、长度是 4KB 的整数倍。

位图：一位一页的紧凑记录 ​

位图分配器的核心思想极其简单：物理内存中的每一页（4KB）对应位图中的一个 bit。bit 值为 1 表示该页已使用，值为 0 表示空闲。对于一个有 128MB 物理内存的系统，总共有 32768 个页，位图只需要 32768 / 8 = 4096 字节 = 1 个页。即使物理内存大到 4GB，位图也才 128KB——比链表或者 buddy 系统的元数据开销小得多。

位图的另一个好处是查询操作是 O(1) 的——给定一个物理地址，你可以立刻算出对应的 bit 索引，然后用一次数组访问加一次位运算就知道这页是忙是闲。分配操作（找第一个空闲页）是 O(N) 的，需要线性扫描位图——但通过 64 位批量扫描加速（后面会讲），实际上开销可以接受。

为什么没有选更"高级"的方案呢？比如 xv6 用的链表（free list），每个空闲页的前几个字节存 next 指针，零额外内存开销，分配释放都是 O(1)。听起来很完美，但它有一个致命的缺点——无法回答"页 X 是否已分配？"。如果你想检查某个物理页是不是被 double-free 了，你必须遍历整个链表。而位图方案天生就能 O(1) 地查询任意页的状态。再比如 Linux 和 SerenityOS 用的 buddy 分配器，支持 O(log N) 的连续分配，但实现复杂度远高于位图——SerenityOS 的 PhysicalZone 有上千行代码，而我们的整个 PMM 只有 267 行。对于一个教学 OS 来说，简洁性比高效性更有价值。

动手实现 ​

Step 1: 定义常量和 MemoryRegion 结构体 ​

目标: 建立 PMM 所需的基础常量和可用内存区域的数据结构。

设计思路: 我们需要三组常量。第一组是页大小——4KB（4096 字节），这是 x86 分页的最小粒度，也是位图中每一个 bit 代表的物理内存大小。第二组是低内存边界——1MB（0x100000），所有低于这个地址的物理内存都不过 PMM。第三组是内核虚拟地址偏移——KERNEL_VMA = 0xFFFFFFFF80000000ULL，Cinux 是一个 higher-half 内核，虚拟地址和物理地址之间有一个固定的偏移量，后面计算位图的物理位置时会用到。

然后我们定义一个 MemoryRegion 结构体来表示一个可用内存区域——只有两个 uint64_t 字段：base（基地址）和 length（长度）。这个结构体是 parse_memory_map 的输出类型。在 parse_memory_map 内部，我们会把 E820 原始数据转换成一个 MemoryRegion 数组，后续的 init 流程直接基于这个数组工作，不再需要回看 E820 原始条目。

实现约束: 常量定义在 pmm.cpp 文件的匿名命名空间外部但仍在 cinux::mm 命名空间内，用 constexpr uint64_t 声明。MemoryRegion 结构体定义在 pmm.hpp 头文件中，放在 cinux::mm 命名空间里。不要把常量放进头文件——它们是实现细节，不需要暴露给 PMM 的使用者。虽然 constexpr 变量在 C++ 中是内联链接的，放进头文件也不会导致 ODR 违规，但从 API 设计的角度来说，kernel/mm/pmm.hpp 的 include 者不需要知道 KERNEL_VMA 是多少。

踩坑预警: KERNEL_VMA 的值一定要和你的链接脚本（kernel.ld）中设定的虚拟地址基址一致。如果你在链接脚本里改了 VMA 但忘了同步这里，后面所有涉及虚拟-物理地址转换的计算都会出错——而且这种错误在初始化阶段不会 crash（位图写到的只是"错误的物理内存位置"），但在分配出页面给页表使用后会引发莫名其妙的 page fault。

这种"静默错误"是内核开发中最令人头疼的问题之一。不像用户态程序出错了会 segfault，内核出错后可能继续运行一小段时间，然后在完全无关的地方 crash——因为错误的位图位置可能碰巧指向了一块可写的内存区域，位图操作看起来"正常工作"了，但实际上你在破坏别的数据结构。如果你遇到"明明 PMM 输出看起来正常，但后续页表映射总是出错"的情况，第一个检查的就是 KERNEL_VMA 常量和链接脚本是否一致。

验证: 这一步只涉及常量和类型定义，编译通过即可。你可以在后续步骤的实现中引用这些常量来验证拼写是否正确。一个常见的手误是 0xFFFFFFFF80000000ULL 少写了一个 F 或者多写了一个 0——这种错误编译器不会报任何警告，但运行时所有地址计算都会出错，排查起来非常痛苦。

Step 2: 实现 parse_memory_map 解析函数 ​

目标: 从 BootInfo 的 E820 原始数据中提取、过滤、对齐出可用内存区域列表。

设计思路: 这个函数接收三个参数——BootInfo 引用（数据源）、MemoryRegion 数组指针（输出）、数组最大容量。返回值是实际写入的区域数量。注意这个函数不依赖任何 PMM 内部状态——它是一个纯函数。把它设计成自由函数而不是 PMM 类的成员方法是有意为之的：host 端单元测试可以独立测试 E820 解析逻辑，而不需要构造一个完整的 PMM 对象。这种"函数和数据分离"的设计在内核开发中很常见——数据结构尽量简单（POD 类型），操作函数尽量无状态（纯函数），这样既方便测试也方便推理。

函数的逻辑可以拆成四道关卡。第一道关卡：类型过滤。遍历 BootInfo 中的 mmap[] 数组，只有 type == 1（可用 RAM）的条目才放行，其他全部跳过。第二道关卡：低内存过滤。如果区域基地址低于 1MB，分两种情况处理——如果区域完全在 1MB 以下，整条跳过；如果跨越了 1MB 边界（基地址 < 1MB 但基地址 + 长度 > 1MB），把基地址截断到 1MB，同时相应缩短长度。第三道关卡：4KB 对齐。把基地址向上对齐到 4KB 边界（使用 (base + 4095) & ~4095 的经典公式），长度先减去对齐带来的偏移量，再向下截断到 4KB 边界（length & ~4095）。第四道关卡：大小检查。如果对齐后的长度不足一个页（< 4096 字节），丢弃这个区域。

这四道关卡是顺序执行的，每一条 E820 条目依次经过四道检查，通过后才被写入输出数组。如果你仔细看，会发现这个过程隐含了一个假设：对齐操作不会让区域"反向增长"——也就是说对齐后基地址增大、长度减小是正确的。但如果 E820 返回了一个基地址不对齐但长度恰好等于 PAGE_SIZE 的条目，对齐后基地址变大、长度变成零，就会被第四道关卡正确丢弃。

实现约束: 函数签名是 uint32_t parse_memory_map(const BootInfo& info, MemoryRegion* regions, uint32_t max_regions)。放在 cinux::mm 命名空间中，声明在 pmm.hpp 里，实现在 pmm.cpp 中。注意 info.mmap_count 是 BootInfo 中记录的有效 E820 条目数（不超过 32），循环用双重约束 i < info.mmap_count && count < max_regions 来防止越界——既不越界读取 E820 条目，也不越界写入输出数组。

踩坑预警: 截断跨 1MB 区域时，长度的计算公式要格外小心。正确的做法是 length -= (LOW_MEM_BOUNDARY - base)，也就是先算"低 1MB 部分占了多少"，然后从总长度里减去这部分。千万不要直接写成 length = length - LOW_MEM_BOUNDARY + base——虽然数学上等价，但如果减法溢出（比如 base 接近 1MB 时 LOW_MEM_BOUNDARY - base 很小但 length - LOW_MEM_BOUNDARY 可能下溢），行为就不一样了。另外，对齐公式中 ~(PAGE_SIZE - 1) 的运算符优先级比减法低，所以必须写成 & ~(PAGE_SIZE - 1) 而不是 & ~PAGE_SIZE - 1——后者会先对 PAGE_SIZE 取反再减 1，结果完全错误。这种运算符优先级的 bug 是位操作中最常见的陷阱之一，而且编译器不会给出任何警告。

验证: 我们稍后会写 host 端单元测试来验证 parse_memory_map 的各种边界情况。现在可以先在脑海中构造一些测试用例来确认逻辑：一个 type=2 的条目应该被跳过；一个 0x00000-0x100000 的 type=1 条目应该被整体跳过；一个 0x80000-0x280000 的 type=1 条目应该被截断为 0x100000-0x280000（长度 0x180000）。这种"先构思测试、再写代码"的习惯在内核开发中特别有用——因为内核代码的调试成本远高于用户态代码，提前想清楚边界情况可以省下大量调试时间。

Step 3: 实现位图辅助函数 ​

目标: 编写位图的四个核心操作——set（标记为已用）、clear（标记为空闲）、test（查询状态）、find_first_free（查找第一个空闲页）。

设计思路: 位图本质上是一个 uint8_t 数组，每个字节包含 8 个 bit，每个 bit 代表一个物理页。给定一个页索引 idx，它所在的字节是 bm[idx / 8]，在该字节中的位偏移是 idx % 8。

bm_set 的操作是把对应 bit 置 1：bm[idx/8] |= (1 << (idx % 8))。bm_clear 是把对应 bit 清零：bm[idx/8] &= ~(1 << (idx % 8))。bm_test 是检查对应 bit 是否为 1：(bm[idx/8] & (1 << (idx % 8))) != 0。这三个操作都是 O(1) 的单次数组访问加一次位运算。

这里有一个经常被忽略的 C++ 细节：我们在位运算表达式中使用了 static_cast<uint8_t> 进行显式类型转换。这是因为 C++ 的整型提升规则会把 1U << (idx % 8) 自动提升为 int 或 unsigned int，OR/AND 操作也在 int 宽度上进行。虽然赋值回 uint8_t 时会隐式截断，但显式 cast 让意图更清晰，也避免某些编译器发出有符号/无符号混合运算的警告。在内核开发中，消除编译器警告是基本素养——你不想在几百条警告中漏掉一个真正的 bug。

bm_find_first_free 是分配操作的核心——线性扫描位图，找到第一个值为 0 的 bit。朴素的实现是逐字节扫描，找到非 0xFF 的字节后再逐 bit 检查。但我们可以做得更快：把位图数组重新解释为 uint64_t 数组，每次检查 64 个 bit。如果一个 uint64_t 值等于 ~0ULL（全 1），说明这 64 页全部已用，直接跳过；如果不等于全 1，说明其中有空闲页，用 __builtin_ctzll(~value) 直接定位到最低位的那个 0——~value 把 0 变成 1、1 变成 0，然后 ctzll（count trailing zeros）告诉我们最低位的 1 在哪个位置，也就是原来第一个 0 的位置。这相当于一次操作检查了 64 页，速度提升最多 64 倍。

这里有一个必须反复强调的安全前提：__builtin_ctzll 在输入为 0 时是未定义行为。但由于我们在调用它之前已经检查了 bm64[i] != ~0ULL，所以 ~bm64[i] 保证非零。这个守卫条件绝对不能删掉，否则在位图全满时触发 UB，而且这种 bug 不会 crash，只是返回错误的结果，特别难排查。

当然，位图的总大小不一定是 8 字节的整数倍。所以我们还需要一个尾部循环来处理 qword 扫描覆盖不到的剩余字节——逐字节逐 bit 检查，逻辑和朴素版本一样。

值得一提的是，这个 64 位加速技巧在操作系统开发中非常常见。Linux 的 bootmem 分配器在扫描空闲页时也使用了类似的方法（只不过它把位图声明为 unsigned long 数组，利用 ffz — find first zero — 来定位空闲 bit）。SerenityOS 甚至更进一步，使用了 SIMD 指令（SSE2 的 _mm_cmpeq_epi8）一次比较 16 字节（128 页）。对于我们的教学 OS 来说，64 位扫描已经足够好了——代码简洁、性能可接受，而且不引入 SIMD 编程的复杂性。

实现约束: 四个函数都放在 pmm.cpp 的匿名命名空间中（内部链接，不暴露给外部）。bm_find_first_free 的返回类型是 int64_t——找到时返回页索引（非负），找不到时返回 -1。参数需要传入 highest_page（最高页号，用于边界检查）和 bitmap_size（位图字节数，用于计算 qword 数和尾部循环范围）。位运算中使用 static_cast<uint8_t> 确保不会因为整型提升引入意外行为。

踩坑预警: __builtin_ctzll 在输入为 0 时是未定义行为。但由于我们在调用它之前已经检查了 bm64[i] != ~0ULL，所以 ~bm64[i] 保证非零——如果位图中 64 个 bit 全是 1，那 bm64[i] == ~0ULL，条件不满足就不会走到 ctzll 调用。但如果你不小心把条件写反了或者删掉了这个检查，就会在位图全满时触发 UB——而且这种 bug 在大多数情况下不会 crash，只是返回一个错误的结果，特别难排查。另外，reinterpret_cast<const uint64_t*>(bm) 要求位图缓冲区的起始地址是 8 字节对齐的——我们的位图是放在页边界上的，页大小 4096 是 8 的倍数，所以天然满足。

验证: 位图辅助函数的验证可以通过 host 端单元测试完成。你可以构造一个小的位图（比如 128 个 bit = 16 字节），全部初始化为 0xFF，然后 clear 某一个 bit，调用 bm_find_first_free 检查返回的索引是否正确。如果返回了正确的索引，说明 set/clear/test/find_first_free 四个函数的基本逻辑是对的。更全面的测试在 Step 5 中编写。另外值得测试的一个边界情况是：位图全满时 bm_find_first_free 是否正确返回 -1，以及位图的最后一个 qword 之后还有剩余字节时尾部扫描是否正常工作。

Step 4: 声明 PMM 类和全局实例 ​

目标: 在头文件中定义 PMM 类的完整接口。

设计思路: PMM 类封装了物理内存管理的全部状态和操作。公开接口包括六个方法：init（从 BootInfo 初始化）、alloc_page（分配单页）、free_page（释放单页）、alloc_pages（分配连续多页）、free_pages（释放连续多页）、free_page_count 和 total_page_count（查询统计信息）。私有成员包括：位图指针（uint8_t*）、总页数、空闲页数、最高页号、位图字节数，以及两个内部辅助方法 mark_region_used 和 mark_region_free。

类的设计有几个值得说的决策。第一，所有状态都是实例成员而不是 static——这样便于测试（可以创建多个 PMM 实例），而且如果将来要支持 NUMA 架构的 per-node PMM 也很容易扩展。在实际的生产级内核中，NUMA 系统的每个 node 都有独立的物理内存管理器，实例化设计让这种扩展成为可能。第二，位图指针用 uint8_t* 而不是 uint64_t*——虽然 64 位扫描时需要 cast，但基础操作（set/clear/test）以字节为单位更直观，也不容易出现对齐问题。第三，free_page_count 和 total_page_count 是 const 方法——纯查询，不修改状态，这为后续加锁提供了便利（const 方法通常只需要读锁）。

全局实例 g_pmm 声明为 extern PMM g_pmm，在 pmm.cpp 中定义。整个内核只有一个 PMM 实例，其他模块通过 cinux::mm::g_pmm 访问。这种"单例模式"在内核开发中很常见——虽然全局状态在原则上是不好的，但在内核中，物理内存管理器确实应该只有一个（除非你做 NUMA 感知的设计，那也是每个 node 一个实例）。

实现约束: 类定义在 kernel/mm/pmm.hpp 中，命名空间 cinux::mm。成员变量使用 C++11 的类内初始化（bitmap_{}、total_pages_{} 等），这样默认构造的对象所有指针为 nullptr、所有整数为 0，不需要额外的构造函数。头文件需要 include boot/boot_info.h（因为 init 方法接收 BootInfo 参数）和 <stdint.h>。

这里有一个 C++ 编程的细节值得展开说一说。PMM 类没有自定义构造函数，它完全依赖类内初始化的默认值。这意味着全局实例 g_pmm 在静态初始化阶段就被正确地置为零状态了——不需要调用任何构造函数代码，因为零初始化是编译器在加载 ELF 时直接完成的（BSS 段天然被清零）。这避免了 C++ 著名的"static initialization order fiasco"——如果 PMM 的构造函数依赖另一个全局对象的构造结果，而那个对象还没被构造，就会出问题。零初始化策略完美规避了这个陷阱。

踩坑预警: 头文件中千万不能定义 KERNEL_VMA 等内部常量——它们是实现细节，应该只出现在 pmm.cpp 中。如果把它们放进头文件，那么每个 include 了 pmm.hpp 的编译单元都会有一份常量定义，虽然 constexpr 变量在 C++ 中是内联链接的不会导致 ODR 违规，但概念上这些常量不应该暴露给 PMM 的使用者。另外，g_pmm 是一个全局对象——它的构造函数在内核的静态初始化阶段运行。由于我们的 PMM 类没有自定义构造函数（依赖类内初始化），所以不存在构造顺序问题。但如果将来你要给 PMM 添加有副作用的构造函数，需要特别小心内核中全局对象的初始化顺序——这是 C++ 著名的"static initialization order fiasco"。

验证: 头文件写完后，在 kernel/main.cpp 中 include kernel/mm/pmm.hpp，然后尝试调用 cinux::mm::g_pmm.init(*boot_info)——如果编译通过，说明类接口定义正确。此时运行会 crash（因为 init 还没实现），但至少验证了类型兼容性。这种"先写接口后写实现"的开发方式在内核开发中很常见——先把类型和声明固定下来，让编译器帮你检查类型错误，然后再填充实现细节。

验证: 头文件写完后，在 kernel/main.cpp 中 include kernel/mm/pmm.hpp，然后尝试调用 cinux::mm::g_pmm.init(*boot_info)——如果编译通过，说明类接口定义正确。此时运行会 crash（因为 init 还没实现），但至少验证了类型兼容性。

Step 5: 编写 host 端 parse_memory_map 单元测试 ​

目标: 在 host 端验证 E820 解析逻辑的正确性，覆盖各种边界情况。

设计思路: 测试文件是 test/unit/test_pmm.cpp，它在 host 端（Linux 用户空间）编译运行，不依赖任何内核代码。为了让测试能编译，我们需要在测试文件中重新实现 parse_memory_map 函数的逻辑（或者直接 include 源文件——但 Cinux 选择重新实现，因为内核代码有很多不可移植的依赖）。

这里有必要解释一下为什么选择"重实现"而不是"直接 include 内核源码"。内核的 pmm.cpp include 了 kernel/lib/kprintf.hpp，还引用了链接器符号（__kernel_end、__kernel_stack_top）——这些东西在 host 端根本不存在。虽然可以通过 mock 和桩函数来解决，但为了一次性的单元测试搞一套 mock 框架未免太重了。相反，parse_memory_map 的逻辑是纯算术的——不涉及任何硬件操作，把它复制到测试文件里只需要改一下类型名。当然，这种做法有一个风险：如果内核代码改了但忘了同步测试，测试就会测"错误的代码"。但在实践中，这种函数一旦写好就很少改动，所以风险可控。

测试用例需要覆盖以下场景。第一个：类型过滤——构造三条 E820 条目（type 分别为 1、2、3），验证只有 type=1 的条目被保留。第二个：低内存过滤——构造一个完全在 1MB 以下的 type=1 条目，验证它被整体跳过。第三个：跨边界截断——构造一个从 0x80000 到 0x280000 的 type=1 条目（跨越 1MB 边界），验证截断后基地址变为 0x100000、长度变为 0x180000。第四个：4KB 对齐——构造一个基地址为 0x100123、长度为 0x2000 的条目，验证基地址向上对齐到 0x101000，长度被缩减到 0x1000（只有一个页）。第五个：过小区域丢弃——构造一个基地址不对齐、长度在对齐后不足一页的条目，验证它被丢弃。

实现约束: 测试使用 Cinux 自带的 test_framework.h 测试框架，用 TEST("name") { ... } 宏定义测试用例。BootInfo 和 MemoryMapEntry 类型从 boot/boot_info.h include 得到。构造测试数据时用初始化列表设置 E820 条目的 base、length、type、acpi 字段。测试文件开头必须 #define TEST_FRAMEWORK_IMPL 来拉入测试框架的实现——这是一个 single-header library 的经典用法。

踩坑预警: 在 host 端测试中，BootInfo 结构体是用 = {} 初始化的（零初始化），所以 mmap_count 默认为 0。别忘了手动设置 info.mmap_count——如果你忘了，循环根本不会执行，所有测试都会"意外通过"（输出 0 个区域，而你期望的也是 0 或者某个特定值，容易产生假阳性）。这个坑特别阴险，因为测试结果是"正确"的，但原因完全不对。另外，BootInfo 的 mmap 数组有 32 个槽位，静态初始化后全是零——所以即使你只设置了前几条，后面的条目也不会干扰（因为 type=0 不是 type=1，会被过滤掉）。

验证: 编写测试后，构建 host 端测试并运行：

cmake --build build && cd build && ctest -L pmm --output-on-failure
1

parse_memory_map 相关的 5 个测试用例应该全部通过。如果有失败，仔细检查失败信息中期望值和实际值的差异——大多数情况下是对齐公式或者截断逻辑的小错误。特别要注意的是十六进制和十进制的混淆：测试数据中的地址值应该用 0x 前缀写成十六进制，如果你不小心写了十进制，截断逻辑的结果会完全出乎意料。

构建与运行 ​

这一章的代码变更涉及两个新文件（kernel/mm/pmm.hpp 和 kernel/mm/pmm.cpp）以及 host 端测试（test/unit/test_pmm.cpp）。在 kernel/CMakeLists.txt 的 big_kernel_common 库中添加 mm/pmm.cpp，在 test/CMakeLists.txt 中添加 test_pmm 测试目标。构建后运行 host 测试验证 parse_memory_map 的逻辑：

cmake --build build && cd build && ctest -L pmm --output-on-failure
1

此时内核可以编译但还不应该运行——因为 PMM::init 的实现要等下一章才完成。如果你现在就在 QEMU 中启动内核，它会在 g_pmm.init() 调用处 crash 或者产生未定义行为。

这一章我们同时引入了 host 端测试框架的完整用法。test_pmm 测试文件使用了 Cinux 自带的 test_framework.h，通过 #define TEST_FRAMEWORK_IMPL 拉入实现（这是一个 single-header library 的模式），然后用 TEST("name") { ... } 宏定义测试用例，用 ASSERT_EQ/ASSERT_NE 等宏做断言。编译后在 test/CMakeLists.txt 中注册为独立的可执行文件，用 ctest 运行。这种"每个模块一个测试文件"的组织方式确保了测试之间完全隔离——一个模块的测试失败不会影响其他模块的测试运行。

host 端测试和 QEMU 集成测试构成了"双轨"策略的基础。Host 端测试的优势是迭代速度快（不需要 QEMU），而且可以方便地构造极端场景。QEMU 测试的优势是使用真实的硬件数据，不存在 host mock 可能引入的偏差。

调试技巧 ​

parse_memory_map 返回 0 个区域: 最常见的原因是 info.mmap_count 没有被正确设置。在 kernel_main 中，boot_info 是从物理地址 0x7000 读取的——用 GDB 或 kprintf 打印 boot_info->mmap_count，看它是否为正数。如果是 0，说明 bootloader 的 E820 调用出了问题。你也可以遍历打印所有 E820 条目的 base、length、type，确认数据格式是否正确。QEMU 的默认配置通常会返回 2-3 个可用区域，所以如果你只看到 0 个或者十几个，肯定是哪里不对。

host 测试中截断结果不对: 构造测试数据时检查 base 和 length 的十六进制值。一个常见的失误是十进制和十六进制搞混——比如你本意是写 0x100000（1MB），但写成了 100000（十进制，约 977KB），截断逻辑就会给出意想不到的结果。一个实用的技巧是在测试用例的注释中写出你期望的十六进制结果，方便对照检查。

位图辅助函数的 64 位扫描返回错误索引: 检查 qword_count 的计算——应该是 bitmap_size / sizeof(uint64_t)，而不是 bitmap_size / 64。前者是"位图字节数除以 8"，得到 qword 的个数；后者是"位图字节数除以 64"，数值差了 8 倍。这是一个典型的"bit 还是 byte"混淆问题，在位操作相关的代码中特别容易出现。另一个调试手段是打印位图的前几个字节，确认 set/clear 操作是否写入了正确的位置。

编译报错 undefined reference to parse_memory_map: 这意味着你的 pmm.cpp 没有被正确加入构建系统。检查 kernel/CMakeLists.txt 中 big_kernel_common 库的源文件列表——mm/pmm.cpp 必须在里面。如果你忘了加这行，链接器在处理 g_pmm.init() 的调用时会找不到 PMM 类的所有方法实现，报一连串的 undefined reference 错误。一个快速排查的方法是用 nm build/libbig_kernel_common.a | grep PMM 看看 PMM 的符号是否在库文件中。

本章小结 ​

这一章我们搭建了 PMM 的基础设施——"眼睛"和"手指"。parse_memory_map 是内核看清物理内存布局的眼睛，它通过四道过滤关卡（类型过滤、低内存过滤、4KB 对齐、大小检查）把 E820 的原始数据转换成干净的可用内存区域列表。位图辅助函数是内核管理内存的手指，bm_set/bm_clear/bm_test 提供了 O(1) 的精确操作，bm_find_first_free 利用 64 位加速扫描在位图中快速定位空闲页。PMM 类的接口设计为实例化而非静态，为未来的扩展和测试留下了空间。

有了这些基础设施，下一章就可以组装完整的初始化流程了——位图放在物理内存的哪个位置、如何保护内核占用的区域、如何处理 higher-half 地址转换。那些才是 PMM 真正容易翻车的地方，所以请确保这一章的所有代码都能编译通过、测试全部绿色后再继续。

概念关键点
E820 内存映射BIOS INT 15h 返回的内存区域列表，type=1 为可用 RAM
低 1MB 过滤LOW_MEM_BOUNDARY = 0x100000，跳过或截断低于 1MB 的区域
4KB 页对齐基地址向上对齐，长度向下截断，不足一页则丢弃
MemoryRegion解析后的可用内存区域，只有 base 和 length 两个字段
位图模型1 bit = 1 页（4KB），1 = 已用，0 = 空闲
bm_set/clear/testO(1) 位操作，字节索引 + 位偏移
bm_find_first_free64 位批量扫描 + ctzll，尾部逐字节兜底
PMM 类接口init/alloc_page/free_page/alloc_pages/free_pages + 统计查询
Host 端单元测试重实现 parse_memory_map 算法，5 个测试覆盖边界情况
