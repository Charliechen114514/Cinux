031-3: GUI 集成 — 让 Shell 跑在终端窗口里 ​

导语 ​

前两章我们分别搭好了管道 IPC 和终端组件——管道能在两个进程之间传递字节流，终端能接收字节流并渲染成屏幕上的文字。现在到了最激动人心的部分：把所有东西串起来，让 shell 进程真正跑在一个 GUI 窗口里。这涉及到进程创建（fork + execve）、管道连接（stdin/stdout 重定向）、系统调用路由重构（让 read/write 能走管道）、FDTable 的扩展、以及 GUI tick 回调中的终端轮询。当这一切都完成后，你将看到 Cinux 的 GUI 桌面上出现一个真正可交互的终端窗口——shell 提示符闪烁着等待你的命令，就像一个真正的桌面操作系统一样。

完成这一章后，当你在 QEMU 中启动 Cinux GUI，点击桌面上的 Shell 图标，一个终端窗口会弹出来，shell 的提示符出现在窗口里，你可以用键盘输入命令，shell 的执行结果也会实时显示在终端上。

前置要求：理解 tag 029-023 中的 fork/execve 机制（子进程创建、地址空间设置、用户态跳转）、tag 030 中的窗口管理器（WindowManager、composite、事件处理）、以及本章前两篇的 pipe 和 terminal 机制。

概念精讲 ​

Shell 终端架构 ​

整个数据流形成了一个完整的闭环：用户敲键盘 -> 键盘中断 -> WindowManager 分发给焦点窗口 -> Terminal::on_key -> stdin 管道（try_write）-> shell 进程的 fd 0（sys_read 从管道读取）-> shell 处理输入 -> shell 输出到 fd 1（sys_write 写入管道）-> stdout 管道 -> Terminal::poll_output（try_read）-> Terminal::write -> 字符缓冲区 -> render_to_canvas -> Canvas -> WindowManager::composite -> 屏幕显示。从键盘到屏幕，每一步都是我们自己实现的。

这里面有两对管道，方向各不相同。stdin 管道的方向是"终端 -> shell"：终端的 on_key 往里面写（try_write），shell 的 sys_read 从里面读。stdout 管道的方向是"shell -> 终端"：shell 的 sys_write 往里面写，终端的 poll_output 从里面读。每个管道内部的读写端是通过 PipeReadOps/PipeWriteOps 适配到 VFS Inode 的，所以 shell 进程只知道自己读写 fd 0 和 fd 1，完全不知道底层是管道还是真实的设备文件——这就是 VFS 抽象层的威力。

fork + execve + pipe 三件套 ​

这是 Unix 创建子进程并建立管道通信的经典模式。流程是：先创建管道（得到读写两个 fd），然后 fork 出子进程，在子进程中把管道的读写端安装到 fd 0 和 fd 1（通过 FDTable::set），最后 execve 加载新程序。新程序继承已设置好的 fd 0 和 fd 1，读 fd 0 就是读管道，写 fd 1 就是写管道。整个过程在 POSIX 标准里有明确的定义，几乎所有 Unix shell 都用这种方式启动子命令。

在 Cinux 的实现中有一些特殊之处。gui_worker（创建终端的父进程）是一个内核线程，没有自己的地址空间和 FDTable。所以 fork 出来的子进程需要额外创建一个 AddressSpace 对象（否则 execve 无法映射 ELF 段）和一个独立的 FDTable（不能继承父进程的，因为父进程的 FDTable 是内核初始化时设置的全局表），然后手动把管道的 inode 安装到 fd 0 和 fd 1。

sys_read/sys_write 路由重构 ​

在引入管道之前，sys_read 对 fd=0 直接读键盘缓冲区，sys_write 对 fd=1 直接调用 kprintf 输出到串口。这种硬编码的方式在只有串口终端的时候没问题，但现在 shell 进程的 fd 0 和 fd 1 指向管道 inode，我们需要让系统调用走到管道路径。

重构的核心是增加了一个"VFS 优先"的路由层——在处理具体的 fd 编号之前，先查当前进程的 FDTable。如果 fd 对应的位置有一个 File 条目，而且这个 File 的 inode 有有效的 InodeOps，就直接调用 ops->read 或 ops->write。管道的 PipeReadOps/PipeWriteOps 会在这一步被命中。如果 FDTable 里没有对应条目，才回退到传统的硬编码路径。

这个设计的精妙之处在于对用户态完全透明——Shell 进程调用 sys_read(0, buf, 256) 时，它不知道 fd 0 背后是管道还是键盘缓冲区，路由由内核根据 FDTable 的内容自动决定。其他没有管道 fd 的进程（比如内核线程）在 FDTable 中找不到对应条目，VFS 路径不命中，自然回退到传统的键盘缓冲区和 kprintf 输出。

动手实现 ​

Step 1: 扩展 FDTable 支持 set 操作 ​

目标: 在 FDTable 中添加 set(fd, file) 方法，允许在指定的 fd 槽位安装一个 File 对象。

设计思路: 原来的 alloc 方法只从 fd=3 开始搜索空闲槽位（fd 0/1/2 保留给 stdin/stdout/stderr），不支持在 fd 0/1/2 这些保留位安装文件。但 shell 终端的初始化需要把管道 inode 安装到 fd 0（stdin）和 fd 1（stdout），所以我们需要一个 set 方法来做强制覆盖。这个方法在未来的 dup2 系统调用中也会用到——dup2(old_fd, new_fd) 的语义就是把 new_fd 重定向到 old_fd 指向的同一个 File。

实现约束: set 方法需要验证 fd 范围有效（0 到 FD_TABLE_SIZE-1），然后在自旋锁保护下替换指定槽位的 File 指针。调用方负责释放之前槽位上的 File（如果有的话）——在 shell 子进程的场景中这不是问题，因为子进程的 FDTable 是刚创建的，所有槽位都是 nullptr。set 返回 true 表示成功，false 表示 fd 越界。

验证: 创建 FDTable，使用 set 分别在 fd 0 和 fd 1 安装 File 对象，然后通过 get 验证安装成功。验证 fd 0 和 fd 1 的 File 的 flags（一个是 RDONLY，一个是 WRONLY）和 inode 指针正确。验证 fd 3 仍然是 nullptr（不受 set 影响）。

Step 2: 重构 sys_read 和 sys_write 的路由逻辑 ​

目标: 修改 sys_read 和 sys_write，让它们优先检查 FDTable 中的 VFS 条目，有则走 InodeOps 路径，否则回退到传统的硬编码路径。

设计思路: 两个系统调用的修改模式相同——在处理具体的 fd 编号之前，先通过 current_fd_table() 获取当前进程的 FDTable，然后 get(fd) 查看是否有 File 条目。如果有且其 inode->ops 非空，就调用 ops->read 或 ops->write，传入 inode、offset、buffer 和 count。返回值大于 0 时更新 file->offset。这个 VFS 路径不关心 fd 编号——fd 0 可以是管道也可以是键盘，完全取决于 FDTable 里装的是什么。

实现约束: 地址验证（canonical address check）保留在函数最前面，VFS 路径和传统路径都需要安全的用户态缓冲区。VFS 路径在获取 File 后需要检查三个条件：file 非 null、inode 非 null、ops 非 null——三个条件缺一不可，因为 FDTable 的 get 可能返回 nullptr（fd 未分配），File 的 inode 可能为 nullptr，inode 的 ops 也可能为 nullptr。offset 的修改需要用 file 的 offset_lock_ 保护（自旋锁的 RAII guard），防止并发访问。

验证: 对有管道 fd 的进程调用 sys_read(fd=0) 应该从管道读取数据、sys_write(fd=1) 应该写入管道。对没有管道 fd 的进程（FDTable 中 fd 0/1 为 nullptr）应该走传统路径——fd=0 读键盘缓冲区、fd=1 走 kprintf 输出到串口。

Step 3: 实现 create_shell_terminal 启动流程 ​

目标: 编写 create_shell_terminal 函数，创建 Terminal 窗口、设置管道、fork 子进程、execve shell。这是整个 tag 中最复杂的函数，涉及进程管理、内存管理、文件系统、GUI 四个子系统的协调。

设计思路: 整个流程可以分为三阶段。第一阶段是准备阶段：创建 Terminal 对象（计算窗口位置和大小、设置字体）、创建两个 Pipe、为每个管道创建对应的 InodeOps 适配器和 Inode、把管道指针绑定到 Terminal 上（set_stdin_pipe/set_stdout_pipe）。第二阶段是 fork 阶段：调用 fork 创建子进程，返回值分为三种情况——大于 0 是父进程、等于 0 是子进程、小于 0 是失败需要清理。第三阶段分两条路径——父进程记录 shell PID 并将 Terminal 窗口加入 WindowManager；子进程创建新的 AddressSpace 和 FDTable、安装管道 fd、调用 execve 加载 /bin/sh、映射用户栈、跳转到用户态。

实现约束: 子进程路径中，在 fork 之后要先 cli（关中断）——fork 后子进程的中断状态不确定，在设置新地址空间之前必须关闭中断防止 PIT 在不完整的 CR3 上触发。然后创建新的 AddressSpace——父进程是内核线程没有用户态地址空间，子进程需要从零创建才能让 execve 映射 ELF 段。接着创建独立的 FDTable 并用 set 把管道 inode 安装到 fd 0 和 fd 1。execve 之后需要手动映射用户栈（从 USER_STACK_TOP 向下分配 USER_STACK_PAGES 个物理页），设置页表属性为 PRESENT | WRITABLE | USER。

踩坑预警: 这是最容易出 bug 的地方，几个常见坑点每一个都会让系统直接 triple fault 重启。第一，忘记在子进程创建 AddressSpace 会导致 execve 时没有页表可用。第二，忘记设置独立的 FDTable 会导致子进程和父进程共享文件描述符，管道读写混乱。第三，也是最隐蔽的——在 SYSRETQ 之前调用 sti 会导致不可预测的 page fault。SYSRETQ 指令会从 R11 恢复 RFLAGS（包含 IF=1），中断在 SYSRETQ 执行后自动启用。如果在此之前手动 sti，会打开一个窗口期——此时 CR3 已经是子进程的页表，但 identity mapping 可能还没建立好，PIT 在这个窗口期触发的话 ISR 试图访问的地址可能在子进程页表中没有映射，直接 page fault。这个 bug 非常隐蔽因为症状取决于 PIT 中断的时序，不是每次都触发。

验证: QEMU 启动后，GUI 应该显示桌面和 Shell 图标。点击 Shell 图标后应该弹出一个标题为 "Shell #1" 的终端窗口，窗口内显示 shell 提示符。键入 ls / 或 echo hello 等命令后 shell 应该执行并显示结果。再次点击 Shell 图标应该弹出第二个终端窗口（"Shell #2"），两个 shell 互不干扰。

Step 4: 修改 PIT tick 回调轮询终端输出 ​

目标: 在 gui_tick_callback 中添加对所有终端窗口的 poll_output 调用，让 shell 输出实时显示在终端窗口内。

设计思路: 之前的 tick 回调只处理鼠标事件和窗口合成。现在我们需要在每次 tick 中增加一步——遍历 WindowManager 中的所有窗口，检查是否是 Terminal 类型（通过 is_terminal() 虚方法判断），如果是就依次调用 poll_output（从 stdout 管道读取 shell 输出）和 render_to_canvas（渲染到 Canvas）。注意我们轮询的是所有终端窗口，不仅仅是获得焦点的那个——这样才能支持多个 shell 会话同时更新各自的显示。poll_output 内部使用 try_read（非阻塞），所以即使某个终端没有数据也不会阻塞 tick 回调。

实现约束: 使用 WindowManager 的 window_count() 和 window_at(i) 遍历窗口。is_terminal() 是 Window 的虚方法——Terminal 类重写返回 true，普通 Window 返回 false。这避免了不安全的 dynamic_cast 或 typeid 检查。poll_output 在 stdout_pipe 为 nullptr 时直接返回（管道未设置），不需要额外的 null check。整个轮询逻辑应该放在事件处理之后、composite 之前，这样每次 composite 都能看到最新的终端内容。

验证: 打开多个终端窗口（多次点击 Shell 图标），每个终端应该独立运行各自的 shell，输出互不干扰。在一个终端中输入命令，只有那个终端的内容会更新，其他终端保持不变。

Step 5: 处理终端窗口关闭时的资源清理 ​

目标: Terminal 析构时关闭管道端点并 reap shell 子进程，防止资源泄漏和僵尸进程。

设计思路: 当用户点击终端窗口的关闭按钮时，WindowManager 删除窗口对象，触发 Terminal 的析构函数。析构函数需要做两件事。第一是关闭管道端点：stdin 关闭写端（通知 shell stdin EOF——shell 的 sys_read 会在管道空时返回 0），stdout 关闭读端（shell 的 sys_write 会因为管道读端关闭而返回 -1——shell 通常会把这当作致命错误退出）。第二是 reap shell 子进程：循环调用 waitpid 收集 shell 的退出状态。waitpid 不能无限等待（shell 可能正在执行长命令），所以我们尝试有限次数（1000 次）。

验证: 打开终端后点击关闭按钮，shell 子进程应该被正确终止。串口日志中应该看到 [TERM] Reaped shell pid=... status=... 的消息。验证进程表中没有留下僵尸 shell 进程。

调试技巧 ​

终端窗口弹出但没有 shell 提示符：检查 fork 的返回值——如果 fork 失败，终端窗口会创建但没有 shell 进程。查看串口日志中是否有 [GUI] fork() failed 消息。另一个可能是 execve 失败——检查 /bin/sh 是否在 ramdisk 中存在，以及 ELF 加载是否成功。串口日志会显示 execve 的错误码。

Shell 执行命令但输出不显示在终端上：检查 stdout 管道的连接。sys_write 在 fd=1 时需要找到 VFS 条目才能走管道路径——如果子进程的 FDTable 没有正确设置，sys_write 会走 kprintf 路径输出到串口而不是管道。在串口日志中能看到 shell 输出就说明管道没接上——检查子进程路径中 FDTable::set(1, ...) 是否被正确调用。

输入一个字符后系统卡死：很可能是管道死锁。检查 stdin 管道的 try_write 是否正确——如果 stdin 管道的写端在终端这边但 on_key 调用的是阻塞 write 而不是 try_write，在 PIT 上下文中调用阻塞操作就会死锁。另一个可能是 stdout 管道的 poll_output 没有被调用（tick 回调中缺少终端轮询逻辑），导致 stdout 管道满后 shell 的 sys_write 阻塞，而 poll_output 永远不被调用来消费数据。
