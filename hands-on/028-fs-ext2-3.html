028-3 Hands-on: 文件读取与目录遍历 ​

导语 ​

上一章我们实现了 mount()——内核从磁盘读出 Superblock、验证 magic、加载 BGDT、读取根 inode，串口输出了一堆漂亮的参数信息。但挂载只是手段，不是目的。用户真正需要的是读写文件——在 shell 里 cat /hello.txt 看到文件内容、ls /etc 看到目录列表。这就要求我们实现 ext2 的文件读取和目录遍历。

这一章的核心任务是两件事：第一，给定一个 inode，如何通过它的 i_block 数组找到文件数据所在的磁盘块，包括直接块和单间接块两种情况；第二，给定一个目录 inode，如何遍历它的数据块中的目录项链表，找到指定名称的子项。完成这两件事之后，我们再把它们组合起来实现路径解析——从根目录出发，逐级分解 "etc/passwd" 这样的路径，最终找到目标文件的 inode。完成本章后，你可以在串口测试中通过 lookup("hello.txt") 找到文件的 VFS Inode，读取它的内容。

知识前置：028-1（ext2 磁盘布局，特别是 inode 和目录项结构）、028-2（mount 和 read_block）。

概念精讲 ​

文件读取的块指针解析 ​

当一个文件的 inode 被加载到内存后，i_block 数组里存放的就是文件数据所在块的编号。对于小文件（不超过 12 个块，即 12KB），所有数据块的编号都直接存放在 i_block[0] 到 i_block[11] 中，读取第 N 个块只需要读 i_block[N] 指向的磁盘块——这就是直接块。

当文件超过 12 个块时，就需要用到间接块了。i_block[12] 存的是一个"间接块"的编号——这个间接块本身不包含文件数据，而是包含一个块指针数组。对于 1024 字节块，这个数组有 256 个元素（1024 / 4 字节），每个元素是一个 uint32_t 的块号。也就是说，间接块提供了额外的 256 个块指针，能让文件最大扩展到 12 + 256 = 268 个块（约 268KB）。

读取间接块指向的数据块的流程是：先用 read_block 把间接块本身读进来，然后从 DMA buffer 中按索引取出目标块号，再对这个块号调用 read_block 读入真正的数据。这个过程涉及两次磁盘读取——第一次读间接块（指针表），第二次读数据块（实际内容）。在我们的单 DMA buffer 设计下，这两次读取之间必须把第一次的数据拷贝出来（或者至少把目标块号记下来），因为第二次 read_block 会覆盖 buffer。

我们可以用一个具体的数字来感受间接块的能力边界。假设块大小 1024 字节、inode_size 128 字节。12 个直接块提供 12KB 数据。单间接块（i_block[12]）指向的指针数组有 256 个元素，提供额外的 256KB。双间接块（i_block[13]）指向的指针数组中每个元素又指向一个指针数组，提供 256256=65536 个块，即 64MB。三间接块（i_block[14]）再扩展一层，提供 256256*256=16777216 个块，即 16GB。当然，这是理论上的最大值——实际受限于 inode 的 i_size 字段（32 位，最大 4GB）和文件系统本身的块数量。对于 Cinux 来说，我们只实现了直接块加单间接块，最大支持约 268KB 的文件，这对于教学 OS 的 Shell 脚本和配置文件完全够用了。

读取过程中的偏移量计算 ​

文件读取操作不是按整块来的——用户可能从文件的任意偏移量开始读，也可能只读几个字节。所以 read 函数需要处理"跨块"的情况：首先计算起始偏移对应的文件块号（offset / block_size）和块内偏移（offset % block_size），然后确定本次读取的长度（取"块剩余空间"和"剩余要读的长度"的较小值）。如果文件总大小不大，一次 read 可能只需要读一个块的一部分；如果文件很大或者偏移跨块，就需要循环读取多个块。

有一点值得特别说明：如果某个块指针为 0（i_block[n] == 0），说明这个"洞"没有被写入过数据——按照 POSIX 语义，读取文件洞应该返回全零字节。我们的实现中，遇到块指针为 0 时不调用 read_block，而是直接把输出缓冲区的对应区域清零。文件洞是 Unix 文件系统中的一个重要特性——它允许程序通过 seek 跳过大量数据后再 write，文件系统不需要为跳过的部分分配实际磁盘块。例如，dd if=/dev/zero of=testfile bs=1M count=1 seek=100 会创建一个看似 101MB 但实际只占 1MB 磁盘空间的文件，中间 100MB 都是文件洞。我们的 ext2 驱动正确处理了这种情况——读取洞区域时返回全零，不会报错。

目录遍历——在链表中查找名称 ​

目录的数据块里存放的是一组 Ext2DirEntry 结构，通过 rec_len 链表式连接。遍历一个目录的方式是：从目录 inode 的第一个数据块开始，在每个块内部从偏移 0 开始，按 rec_len 步进，检查每个条目的 inode 是否非零（非零才是有效条目）以及 name_len 和文件名是否匹配。

这里有三个需要注意的点。第一，步进必须用 rec_len 而不是 name_len + 8——rec_len 包含了对齐填充，而 name_len 只是实际文件名的长度。如果你用 name_len + 8 步进，在某些条目上会对不齐，导致后面的条目全部读乱。第二，如果 rec_len 为 0，说明遇到了损坏的数据或者块末尾——应该立即停止遍历。第三，目录项不能跨越块边界——如果一个块的最后剩余空间不够放下一个新条目，上一个条目的 rec_len 会被扩展到块末尾（这是 mkfs.ext2 和内核的约定），所以我们不需要担心一个条目一半在这个块、一半在下一个块的情况。

我们用一个具体的例子来说明目录遍历。假设根目录的数据块（一个 1024 字节块）内容如下：偏移 0 处是 "." 条目（inode=2, rec_len=12, name_len=1），偏移 12 处是 ".." 条目（inode=2, rec_len=1012, name_len=2）。注意 ".." 的 rec_len 是 1012——它不是 name_len(2)+8=10，而是 12+1012=1024，正好把整个块占满。这是因为 mkfs.ext2 创建文件系统时，根目录只有 "." 和 ".." 两个条目，".." 条目把剩余空间全部"吃掉"了。如果你后来往根目录里添加了 hello.txt，mkfs.ext2 的行为是把 ".." 的 rec_len 从 1012 缩减到 12（因为 ".." 的最小 rec_len 是 8+2=10，向上对齐到 12），然后在偏移 24 处插入 hello.txt 条目，rec_len 为 1000（24+1000=1024，再次吃掉剩余空间）。

路径解析——逐级分解 ​

有了目录遍历的能力之后，路径解析就是水到渠成的事情。给定一个路径比如 "etc/passwd"，解析过程是：从根 inode（inode 2）开始，先在根目录里找 "etc" 这个条目，得到它的 inode 编号；然后确认 "etc" 是一个目录（检查 i_mode 的类型位），再在 "etc" 目录里找 "passwd" 这个条目，得到最终目标的 inode 编号。每一级都是一次 lookup_in_dir 调用——给定目录的 inode 号和组件名称，在目录数据块中搜索匹配的条目。

路径解析有几个边界情况需要处理：前导 '/' 需要跳过（VFS 层会剥离挂载点前缀后传入相对路径）、连续的 '/' 也需要跳过、空路径返回根 inode。如果中间某一级不是目录（比如 "etc/passwd/not-a-dir" 中 passwd 是普通文件），解析应该返回失败。

路径解析的性能特征值得提一下。对于每一级路径组件，我们需要至少一次 read_disk_inode（读目录的 inode）加上若干次 read_block（遍历目录的数据块）。对于 "etc/passwd" 这样的两级路径，至少需要 4 次磁盘 I/O（读根目录 inode、读根目录数据块、读 etc 目录 inode、读 etc 目录数据块）。在真实系统中，这些 I/O 会被 Linux 的 dentry cache（目录项缓存）大幅加速——但我们没有 dentry cache，每次 lookup 都要走完整的磁盘读取路径。不过对于教学 OS 来说这完全可以接受——Shell 命令的响应速度在 QEMU 中已经足够快了。

另外一个有趣的边界情况是符号链接。ext2 支持 symlink 类型的 inode（i_mode 高 4 位为 0xA000），但 Cinux 目前没有实现符号链接解析。如果路径中的某个组件是符号链接，我们的 lookup 会检查它的类型不是 Directory 后直接返回 nullptr——这在语义上不太正确（正确的行为是先读取 symlink 目标再继续解析），但对于教学 OS 来说可以暂时不处理。

动手实现 ​

Step 1: 实现 read_disk_inode ​

目标: 给定 inode 编号（1-based），从磁盘读出完整的 128 字节 inode 数据。

设计思路: 使用上一章讲的寻址公式——先算 group，再从 BGDT 取 inode_table 起始块号，再算组内索引和块内偏移，最后 read_block 并从 DMA buffer 中拷贝 128 字节。这个函数是后续所有 inode 操作的基础——文件读取、目录遍历、路径解析都需要先拿到 inode 数据。

这个函数的实现大约需要以下步骤：首先检查 ino 是否为 0（无效），然后计算 group = (ino-1) / inodes_per_group_，检查 group 是否越界。接着从 bgdt_[group] 中取出 bg_inode_table（inode 表的起始块号），计算 index_in_group = (ino-1) % inodes_per_group_，算出 byte_offset = index * inode_size_，进而得到 target_block 和 within_block_offset。最后调用 read_block(target_block)，从 DMA buffer 的 within_block_offset 处 memcpy 128 字节到输出参数中。

这里有一个非常重要的安全检查：在 memcpy 之前，必须确认 within_block_offset + sizeof(Ext2Inode) 不超过 block_size_。在 1024 字节块和 128 字节 inode 的情况下，一个块最多放 8 个 inode——最后一个 inode 的 within_block_offset 是 896，加上 128 等于 1024，刚好等于 block_size，所以不会越界。但如果将来 inode_size 增大（比如 256 字节的扩展 inode），一个块只能放 4 个 inode，最后一个的偏移是 768，加上 256 等于 1024，仍然不会越界。但如果 inode_size 不是 block_size 的整数倍（理论上不太可能但值得防御），就会出现跨越块边界的情况——此时简单的一次 read_block 就不够了，需要读两个块并拼接。我们的代码通过检查越界并拒绝处理来避免这个问题。

实现约束: ino 为 0 时直接返回 false（inode 0 不存在）。需要做越界检查——group 不能超过 group_count_。还需要做块边界检查——如果 inode 数据跨越了块边界（理论上在 128 字节 inode + 1024 字节块的情况下不会发生，但安全起见应该检查），返回 false 并打印警告。

踩坑预警: memcpy 的源地址是 DMA buffer 的虚拟地址加上块内偏移，拷贝长度是 sizeof(Ext2Inode)（128 字节）。如果你忘了加块内偏移，每次读到的都是块开头的那个 inode——对于 inode 2 这种"每组第一个 inode"恰好能工作，但对于其他 inode 会读到错误的数据。这个 bug 非常隐蔽，只有在读取非首个 inode 时才会暴露。

验证: 编译通过。可以在 mount 成功后加一个临时的测试——手动调用 read_disk_inode(2, ...) 读取根目录 inode，然后打印 i_size 和 i_mode，确认和 dumpe2fs 的输出一致。然后尝试读取一个非首个 inode（比如 inode 11 或 12），确认块内偏移的计算正确。对于 inode 11（通常是 lost+found 目录），你应该看到 i_mode 高 4 位为 0x4000（目录类型）。

验证: 编译通过。

Step 2: 实现 populate_vfs_inode ​

目标: 实现 populate_vfs_inode 辅助函数，把磁盘 inode 的信息转换到 VFS Inode 中。

设计思路: 这个函数从 Ext2CachedInode 中取出 disk_inode 字段，根据 i_mode 的类型位（掩码 0xF000）判断文件类型——0x8000 是普通文件（InodeType::Regular，ops 指向 ext2_file_ops），0x4000 是目录（InodeType::Directory，ops 指向 ext2_dir_ops），其他类型标记为 Unknown 且 ops 设为 nullptr。tag 028 只设置了五个字段：ino（inode 编号）、size（文件大小，取自磁盘 inode 的 i_size）、type（文件类型）、ops（InodeOps 函数指针表）和 fs_private（回指指针，指向 cache entry 本身）。mode、nlink、blocks、uid、gid 等字段不在此处设置，它们会在后续 tag 的 stat 功能中按需添加。

实现约束: i_mode 的低 12 位是权限位（如 0x1ED = rwxr-xr-x），高 4 位是类型（0x4 = 目录，0x8 = 普通文件）。我们用 EXT2_S_IFMT（0xF000）掩码提取类型部分，然后和 EXT2_S_IFREG（0x8000）或 EXT2_S_IFDIR（0x4000）比较。如果都不是（比如符号链接 0xA000 或设备文件），设为 Unknown。

验证: 编译通过。

Step 3: 实现 ext2_file_read ​

目标: 给定 VFS Inode、偏移量和长度，从 ext2 文件中读取数据到用户缓冲区。

设计思路: 先通过 inode 的 fs_private 指针拿到 Ext2CachedInode，从中获取磁盘 inode。然后计算文件大小限制（不能超过 i_size）。进入循环：每次迭代处理一个块——计算当前偏移对应的文件块号（(offset+total_read)/block_size）和块内偏移（(offset+total_read)%block_size），然后确定本次迭代的读取长度（取"块剩余空间"和"剩余要读的长度"的较小值）。判断文件块号是直接块（< 12）还是间接块（12 到 12+block_size/4-1），拿到磁盘块号后 read_block 读入数据，从 DMA buffer 拷贝到用户缓冲区。循环直到读满请求数量或到达文件末尾。

实现约束: 间接块的处理需要先 read_block 把间接块读进来，然后从 DMA buffer 中按索引取出目标块号，再 read_block 读入真正的数据。这里有一个微妙之处：两次 read_block 之间，第一次读到的间接块数据会被第二次覆盖——但没关系，因为我们已经把目标块号提取出来了。还有一个重要的细节是 EOF 处理——如果 offset 已经大于等于 i_size，直接返回 0（EOF），不应该再尝试读取任何数据。

踩坑预警: 当文件大小刚好是块大小的整数倍时（比如 1024 字节的文件，i_size=1024），offset=1024 已经是 EOF 了。但 i_block[0] 指向的块确实有数据（1024 字节），而 i_block[1] 可能是 0（文件洞）或非零（预分配的块）。读函数必须严格以 i_size 为边界——不能因为 i_block[n] 非零就读取超过 i_size 的数据。

验证: 编译通过。可以在 mount 成功后加一个临时测试：手动构造一个 ext2_file_read 调用，传入根 inode 的某个文件子 inode 的信息（比如 hello.txt 的 inode），offset=0，缓冲区 256 字节，打印返回的字节数和缓冲区内容。

Step 4: 实现 ext2_dir_readdir ​

目标: 给定目录 Inode 和索引号，返回第 N 个有效目录项的文件名。

设计思路: 索引 0 和 1 分别硬编码返回 "." 和 ".."（这是 VFS 层的约定，因为每个目录都必然有这两个条目，不需要每次都从磁盘读）。索引 >= 2 时，遍历目录的所有数据块，跳过 inode=0 的空洞条目以及 "." 和 ".." 这两个硬编码条目，计数有效条目直到找到第 (index-2) 个，拷贝文件名到输出缓冲区。

实现约束: 遍历时每个块都通过 read_block 读取，遍历完一个块后进入下一个。块数量由目录的 i_size 和 block_size 计算。返回值语义：1 表示成功读到一项、0 表示目录已读完（没有更多条目）、-1 表示错误。名称拷贝时要注意 name_max 的限制——如果文件名长度超过 name_max-1，需要截断并在末尾加 '\0'。

踩坑预警: 如果目录数据块很多（大目录可能超过 12 个块），当前实现只遍历直接块（i_block[0..11]），不处理间接块。对于我们的测试镜像来说这完全够用——一个目录很难有那么多条目。如果你以后需要支持更大的目录，需要在 readdir 中加入间接块遍历。另外，readdir 的 index 参数是 0-based 的目录项索引，不是字节偏移——调用方通常用 File 结构体的 offset 字段当索引用。

踩坑预警: 如果目录数据块很多（大目录可能超过 12 个块），当前实现只遍历直接块（i_block[0..11]），不处理间接块。对于我们的测试镜像来说这完全够用——一个目录很难有那么多条目。如果你以后需要支持更大的目录，需要在 readdir 中加入间接块遍历。

验证: 编译通过。

Step 5: 实现 lookup_in_dir ​

目标: 给定目录的 inode 号和组件名称，在目录数据块中搜索匹配的条目，返回匹配条目的 inode 编号。

设计思路: 先 read_disk_inode 获取目录的磁盘 inode，然后遍历它的每个数据块。在每个块内部，按 rec_len 步进扫描所有条目，对每个有效条目（inode != 0）比较 name_len 和文件名。找到匹配则返回 entry->inode，遍历完没找到则返回 0。这个函数的设计是"纯查找"——它不关心目录项的 file_type，只做名称匹配。

实现约束: 名称比较不能用 strcmp——目录项的文件名不是 null-terminated 的，长度由 name_len 决定。必须先比较 name_len 是否和目标长度一致，如果一致再逐字节比较 name_len 个字符。这是一个常见的优化：如果长度不同，直接跳过，不需要逐字节比较。同时，inode=0 的条目是空洞，也应该直接跳过。

踩坑预警: 名称比较时不仅要比较 name_len 个字符，还要确保 name_len 和目标名称长度完全一致。如果你只比较了目标名称长度的字符而忽略了 name_len 的长度检查，可能会出现误匹配——比如查找 "etc" 时误匹配到 "etc2"（如果 etc2 的 name_len 是 4，而你只比较了 3 个字符）。

验证: 编译通过。

Step 6: 实现 lookup（路径解析） ​

目标: 给定相对路径，从根 inode 开始逐级解析，返回最终目标的 VFS Inode 指针。

设计思路: 空路径或 "/" 返回根 inode。否则跳过前导 '/'，进入循环：提取下一个路径组件（到下一个 '/' 或字符串末尾），用 lookup_in_dir 在当前目录中查找，得到子 inode 号。如果还有更多组件，检查子 inode 是否为目录，然后继续。循环结束后，用 get_cached_inode 返回最终 inode 的 VFS 指针。

实现约束: lookup_in_dir 返回 0 表示未找到，此时 lookup 返回 nullptr。中间组件不是目录时也返回 nullptr。路径中的连续 '/' 需要跳过（不要把空组件传给 lookup_in_dir）。lookup 的参数是相对路径——VFS 层在调用前已经剥离了挂载点前缀，比如 VFS 挂载在 "/" 时，用户调用 sys_open("/hello.txt")，VFS 先匹配 "/" 挂载点，然后把 "hello.txt"（去掉前导 '/'）传给 ext2.lookup。

踩坑预警: lookup_in_dir 需要一个完整的目录 inode 来遍历——如果你传入了错误的 inode 号（比如把文件 inode 当目录 inode 用），遍历数据块时会读出非目录格式的数据（文件内容被当成目录项解析），大概率会导致 rec_len 不合法、pos 越界、甚至死循环。在 lookup 的循环中，每次进入下一级之前必须检查当前 found_ino 对应的 inode 是否为目录类型——这是防止这种错误的关键防线。

验证: 编译并在 QEMU 中运行。此时还没有 Shell 集成（那是下一章的事），但可以在内核初始化阶段手动调用 ext2.lookup("hello.txt") 并打印结果。如果 lookup 返回非 nullptr 的 Inode 指针，打印它的 ino、size、type 字段，确认和你在宿主机上用 dumpe2fs 看到的一致：

Step 7: 实现 inode cache ​

目标: 实现 get_cached_inode，避免对同一个 inode 的重复磁盘读取。

设计思路: 先线性扫描 cache 查找已有的条目。没找到则在空闲槽位中分配一个，read_disk_inode 填充。如果 cache 满了，用简单的驱逐策略——基于 ino 做取模选择一个非 slot-0 的槽位覆盖。slot 0 永远留给根 inode（root_inode_ 直接引用它），所以驱逐的目标范围是 slot 1 到 slot 63。

这种固定大小的 cache 在教学 OS 中是合理的——64 个条目足以缓存一个小型文件系统中最常用的 inode。Linux 内核使用的 inode cache 要复杂得多：它是一个全局哈希表加上 LRU 链表，能动态扩展大小。但我们不需要那么复杂——一个简单的线性扫描 + 驱逐已经能消除大部分重复磁盘读取，而代码量控制在可控范围内。

实现约束: cache 大小 64 条目。populate_vfs_inode 负责把磁盘 inode 转换成 VFS inode——根据 i_mode 的类型位设置 InodeType 和 ops 指针（Regular 设为 &ext2_file_ops，Directory 设为 &ext2_dir_ops），设置 fs_private 指向 cached entry 本身。这个 fs_private 指针是 InodeOps 回调函数获取磁盘 inode 的关键——通过它，ext2_file_read 可以从 VFS Inode 反向找到磁盘上的块指针信息。tag 028 的 InodeOps 是静态函数指针表，回调通过全局指针 g_ext2_instance 访问 Ext2 对象。

踩坑预警: populate_vfs_inode 中设置 fs_private 时，指针应该指向 cache entry 中的 vfs_inode 所属的 Ext2CachedInode 结构体，而不是指向 vfs_inode 本身。如果搞混了，ext2_file_read 中 static_cast<const Ext2CachedInode*>(inode->fs_private) 得到的将是错误的地址，后续对 disk_inode 的访问会读到垃圾数据。

验证: 编译通过。可以在 get_cached_inode 之后打印一下 VFS Inode 的字段值（ino、size、type），确认 populate_vfs_inode 的转换结果正确。

验证: 编译通过。

调试技巧 ​

lookup 返回 nullptr 但文件确实存在: 最常见的原因是名称比较逻辑有误。在 lookup_in_dir 里加打印：遍历到每个条目时打印 inode 号、name_len 和文件名（用 %.*s 格式化 name_len 个字符），看看扫描过程是否正确。特别注意 name_len 为 0 的条目——这种条目应该跳过。

read 返回全零: 检查文件 inode 的 i_block 数组是否全为 0——如果是，说明 read_disk_inode 读到了错误的位置。用上一章的 hexdump 方法确认 inode 数据在镜像中的位置和内容是否正确。如果 i_block[0] 非 0 但 read_block 之后 DMA buffer 中全是 0，说明那个块号指向的是镜像中未使用的区域——检查块号是否超出了镜像大小。

read 返回的字节数少于预期: 可能是间接块解析的问题——文件的实际大小大于 12 个块，但间接块（i_block[12]）为 0 或指向了错误的块。打印 file_block 和 disk_block 的对应关系来排查。

间接块读取后数据错乱: 确认你在 read_block(间接块) 之后、从 DMA buffer 中取出目标块号之后、再调用 read_block(数据块)。两次 read_block 之间不能依赖 DMA buffer 中的旧数据——第二次 read_block 会覆盖它。

readdir 只返回 "." 和 "..": 确认目录的数据块确实被读取了——在 readdir 的循环里加打印，看看遍历了几个块、每个块里有几个条目。如果块数是 0，说明目录 inode 的 i_size 可能不正确。另一个可能的原因是你忘了跳过硬编码的 "." 和 ".."——如果不跳过，readdir 的计数逻辑会把它们算进去，导致后续实际条目的索引偏移。

路径解析中间级别失败: 如果 "etc/passwd" 解析失败但 "etc" 单独能找到，问题可能出在中间级别检查上。在 lookup 循环中加入打印：每级组件的名称、lookup_in_dir 返回的 inode 号、以及该 inode 的类型。如果 lookup_in_dir 返回了正确的 inode 号但类型检查失败了，说明你读取的 inode 的 i_mode 不对——可能是 read_disk_inode 的偏移计算有误。另一个可能的原因是，etc 目录确实存在但它的 i_mode 高 4 位不是 0x4000——如果你的 inode cache 有问题导致读到了脏数据，类型检查就会失败。

lookup_in_dir 死循环: 如果你在遍历目录项时发现 pos 不增长，说明某个条目的 rec_len 为 0——我们的代码在 rec_len == 0 时 break，所以这种情况不会发生。但如果 rec_len 的值异常小（比如 1 或 2，小于最小的 8 字节头），pos 会增长但下一个条目的解析会出错。加一个 if (entry->rec_len < EXT2_DIR_ENTRY_HDR_SIZE) break 的安全检查可以防止这种情况。

inode cache 导致数据不一致: 如果你对同一个文件连续调用两次 lookup，第二次应该命中 cache。但如果两次返回的 Inode 的 size 不同，说明 cache 条目被意外覆盖了——检查驱逐逻辑是否不小心驱逐了不该驱逐的条目。特别是在 cache 满了的情况下，驱逐策略选择的 slot 可能正好是你正在使用的那个。

lookup 对根目录 "/" 返回 nullptr: 检查 lookup 函数中对空路径和 "/" 的判断逻辑。空字符串（path[0]=='\0'）和 "/"（path[0]=='/' && path[1]=='\0'）都应该直接返回 &root_inode_。如果这个判断在 VFS 层剥离前缀之前就被调用了，传入的路径可能是 "/" 而不是空字符串——两种情况都要处理。

本章小结 ​

概念要点
直接块读取i_block[0..11] 直接指向数据块
单间接块i_block[12] 指向指针数组，需要两次 read_block
文件洞块指针为 0 时返回全零字节
目录遍历按 rec_len 步进，inode=0 跳过，不能跨块边界
名称比较非 null-terminated，逐字节比较 name_len 个字符
路径解析逐级分解，lookup_in_dir + 中间类型检查
inode cache64 条目，slot 0 固定给根 inode，简单驱逐策略
populate_vfs_inode根据 i_mode 类型位设置 InodeType 和 ops，fs_private 指向 cache entry
间接块容量直接块 12KB + 单间接 256KB，Cinux 不支持双/三间接
目录项步进必须用 rec_len 而非 name_len+8，防止对齐错误
