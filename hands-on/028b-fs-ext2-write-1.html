028b-1 Hands-on: ext2 块与 inode 分配器 ​

导语 ​

上一个 tag 我们把 ext2 的只读路径全部走通了——mount 读 Superblock，lookup 逐级解析路径，read 从块指针读数据，readdir 遍历目录项链表。但说实话，一个只能读不能写的文件系统只能算半成品。如果 Cinux 永远不能创建文件、不能写入数据、不能删除东西，那 Shell 里就只能跑 ls 和 cat，什么有意义的事情都做不了。所以我们现在要把 ext2 的另一半——写操作——补上来，而写操作的第一步，也是最基础的一步，就是搞明白怎么从磁盘上分配和释放资源。

ext2 的磁盘空间管理说白了就两件事：块（block）的分配与释放、inode 的分配与释放。块是数据存放的容器，inode 是元数据的容器。两者都通过位图（bitmap）来管理——每个块组有自己的 block bitmap 和 inode bitmap，bitmap 中每个 bit 对应一个块或一个 inode，0 表示空闲，1 表示已占用。分配就是找 0 置 1，释放就是找 0（之前清掉那个 1）。听起来简单得令人发指对吧？但魔鬼全在细节里——inode 编号从 1 开始、位图操作后要回写 Superblock 和 BGDT 的空闲计数器、间接块的分配要同时管理间接块本身和它指向的数据块。这些细节如果搞错一个，轻则数据丢失，重则整个文件系统结构崩溃。

这一章我们先聚焦最底层的分配器——把 alloc_block / free_block 和 alloc_inode / free_inode 四个函数搞清楚，把元数据回写的 write_superblock / write_bgdt / write_disk_inode / write_block 搞利索。完成之后，我们就有了构建 create / mkdir / unlink 等高层操作的地基。

知识前置：028-1 到 028-4（ext2 只读驱动的完整实现），需要理解 block bitmap / inode bitmap 的概念、Superblock 和 BGDT 中空闲计数器的位置、AHCI DMA buffer 的读写机制。

概念精讲 ​

Block Bitmap —— 块的"占位表" ​

每个块组有一个 block bitmap，占用整整一个块的空间。对于 1024 字节块大小来说，一个 bitmap block 有 1024 * 8 = 8192 个 bit，也就是说它能管理 8192 个块。如果每个块 1024 字节，那一个块组最大就是 8MB——这正好是 ext2 中 blocks_per_group 的标准值（8192）。

bitmap 中第 0 个 bit 对应组内的第 0 个块，第 1 个 bit 对应第 1 个块，以此类推。但这里有一个很容易被忽略的细节：bitmap 中的 bit 索引是组内局部索引，而块号是全局索引。从局部索引转换到全局块号的公式是 global_block = group * blocks_per_group + first_data_block + local_index。first_data_block 对于 1024 字节块来说是 1（因为 block 0 是引导块），对于更大的块则是 0。

分配的时候，我们从第 0 组开始扫描，找到第一个 bg_free_blocks_count 不为零的组，读它的 block bitmap，然后线性扫描每个字节——如果某个字节不是 0xFF（即不是全满），就逐 bit 检查找到第一个 0 bit。找到之后，把这个 bit 置 1，把 bitmap block 写回磁盘，同时递减 Superblock 的 s_free_blocks_count 和该组 BGDT 的 bg_free_blocks_count，最后把 Superblock 和 BGDT 都写回磁盘。

你可能会问：为什么不用更高效的分配策略？比如 Linux 内核用的是 goal-based 分配加 reservation window，用 rbtree 管理预分配窗口——那个复杂度大约是 balloc.c 五百多行。Cinux 是教学内核，线性扫描足够了，代码量少一个数量级，而且对于我们 8MB 的测试镜像来说性能差异可以忽略不计。SerenityOS 也用的简单 bitmap 扫描，说明这个策略在教学和轻量级系统中是标准做法。

Inode Bitmap —— inode 的"占位表" ​

inode bitmap 和 block bitmap 的原理完全一样，只不过管的是 inode 而不是数据块。每个块组有一个 inode bitmap，bit 0 对应组内第 0 个 inode。

这里有一个需要特别注意的地方：inode 编号从 1 开始，不是从 0 开始。inode 0 被保留用作"无效"标记。这意味着从全局 inode 号到 bitmap bit 的映射是 local_index = (ino - 1) % inodes_per_group。所有计算都必须先减 1——忘了减 1 的话，inode 2 会被映射到 group 0 的第 2 个 bit 而不是第 1 个 bit，所有后续操作都会偏移一格，然后你会花一整个下午追查为什么 cat 读出来的数据全是错的。

另一个关键差异是 inode 分配返回的是 inode 号（1-based），而块分配返回的是块号（0-based）。这种不对称性是 ext2 规范的产物——inode 号从 V7 Unix 开始就从 1 开始了，而块号从 0 开始是硬件的传统。

元数据回写 —— 保证磁盘状态一致 ​

每次分配或释放操作都会修改三类元数据：bitmap 本身、Superblock 的空闲计数器、BGDT 的空闲计数器。Cinux 采用同步写入策略——修改完之后立即把所有受影响的元数据都写回磁盘。这意味着：

write_block 把 DMA buffer 中的内容写入指定块号对应的磁盘位置。这是最底层的写入原语，通过 AHCI 的 DMA 写入完成。

write_superblock 把内存中的 sb_ 结构体拷贝到 DMA buffer，然后通过 AHCI 写入字节偏移 1024 的位置。注意 Superblock 不是在 block 0——它在字节 1024，对于 1024 字节块来说等于 block 1。

write_bgdt 需要做 read-modify-write：先读出 BGDT 所在的块，把修改后的描述符拷贝到正确位置，再整块写回。因为一个 BGDT 块可能包含多个组的描述符，我们不能只写一个描述符而覆盖掉同一块里其他组的数据。

write_disk_inode 也是 read-modify-write：读出包含目标 inode 的那个块，把新的 inode 数据覆盖进去，再写回去。这里要确保 inode 不跨越块边界——我们的实现里有对应的边界检查。

Cinux 没有 write-back cache，每次操作都同步写入磁盘。这在性能上不是最优的——Linux 有 buffer_head 子系统做延迟写入和脏页回写——但好处是数据即时落盘，不会因为内核崩溃而丢失未写入的数据。对于教学内核来说，简单可靠比性能更重要。

动手实现 ​

Step 1: 实现 write_block ​

目标: 提供与 read_block 对称的写入原语，把 DMA buffer 中的数据通过 AHCI 写入指定块号。

设计思路: write_block 接收一个块号参数，计算对应的 LBA（block_num * sectors_per_block），然后调用 AHCI 驱动的 write 方法把 DMA buffer 的物理地址作为数据源写入磁盘。调用方需要先在 DMA buffer 中准备好要写入的数据，然后调用 write_block 触发实际的磁盘写入。

实现约束: write_block 内部会调用 ensure_dma_buffer 确保缓冲区已分配。AHCI write 需要物理地址而非虚拟地址，所以传入 dma_buf_phys_。每次 write_block 只写一个块大小的数据。

验证: 编译通过，确保链接时 AHCI::write 符号能正确解析：

bash
cd build && cmake .. && make big_kernel_common 2>&1 | tail -5
1

Step 2: 实现 write_superblock ​

目标: 把内存中缓存的 Superblock 写回磁盘偏移 1024 处。

设计思路: 将 sb_ 结构体拷贝到 DMA buffer，然后直接通过 LBA 写入——Superblock 的 LBA 起始是 2（字节 1024 / 512），长度是 2 个扇区（1024 字节 / 512）。不需要通过 read_block/write_block 的块号接口，而是直接调用 AHCI 的 sector 级别写入，因为 Superblock 的位置是固定偏移。

验证: 编译通过。这一步暂时没有独立的运行验证——后续分配操作会间接调用它。

Step 3: 实现 write_bgdt ​

目标: 把修改后的某个组描述符写回 BGDT 所在的磁盘块。

设计思路: BGDT 的起始块号取决于块大小——1024 字节块时从 block 2 开始，更大的块从 block 1 开始。一个 BGDT 块可以容纳 block_size / 32 个描述符。所以第 N 个组描述符在 BGDT 的第 N / entries_per_block 个块中，组内偏移是第 N % entries_per_block 个描述符。用 read-modify-write 策略：读出整块、替换目标描述符、写回整块。

验证: 编译通过。

Step 4: 实现 alloc_block ​

目标: 扫描所有块组的 block bitmap，找到第一个空闲块，标记为已用并返回全局块号。

设计思路: 外层循环遍历所有 block group，先检查 bg_free_blocks_count 是否为零——如果为零就跳过这个组（快速路径）。然后读 bitmap block，逐字节扫描，遇到非 0xFF 字节就逐 bit 找第一个 0。找到后置 1，写回 bitmap，递减 s_free_blocks_count 和 bg_free_blocks_count，调用 write_superblock 和 write_bgdt 回写计数器，最后计算全局块号返回。如果所有组都满了，返回 0 表示失败。

实现约束: 最后一个组的 bitmap 可能不满——比如镜像总共有 8192 块但最后一组可能只有几百块，所以需要用 s_blocks_count 来裁剪扫描范围。

验证: 编译通过后，可以在测试用例中验证分配行为：

bash
cd build && cmake .. && make big_kernel_common
# 后续 Step 会通过 QEMU 实际验证

2

Step 5: 实现 free_block ​

目标: 给定一个全局块号，计算它所属的组，清除 bitmap 中的对应 bit，递增空闲计数器。

设计思路: 从块号到组号的映射是 group = (block_num - first_data_block) / blocks_per_group，组内偏移是 local = (block_num - first_data_block) % blocks_per_group。读 bitmap，清除对应 bit，写回，递增计数器，回写元数据。

验证: 编译通过。

Step 6: 实现 alloc_inode ​

目标: 与 alloc_block 对称，扫描 inode bitmap，找到空闲 inode 并标记为已用。

设计思路: 遍历所有 block group，检查 bg_free_inodes_count，读 inode bitmap，找第一个 0 bit。找到后置 1，写回 bitmap，计算全局 inode 号 group * inodes_per_group + local_index + 1（注意 +1 因为 inode 从 1 开始编号）。递减 s_free_inodes_count 和 bg_free_inodes_count，回写 Superblock 和 BGDT。

踩坑预警: 千万别忘了 inode 号的 +1。如果你返回的是 group * inodes_per_group + local_index（从 0 开始），那 inode 1 就变成了 0，而 inode 0 是"无效"标记，所有用这个返回值的后续操作都会出错。这个 bug 的症状非常诡异——lookup_in_dir 能找到文件名但返回的 inode 号是 0，然后 read_disk_inode(0) 直接失败。

验证: 编译通过。

Step 7: 实现 free_inode ​

目标: 给定 inode 号（1-based），清除 bitmap 中的 bit，递增空闲计数器。

设计思路: group = (ino - 1) / inodes_per_group，local_index = (ino - 1) % inodes_per_group。读 inode bitmap，清除 bit，写回，递增计数器，回写元数据。

验证: 编译通过。

踩坑预警——常见错误清单 ​

Superblock 写回偏移错误: write_superblock 必须写到字节偏移 1024，不是块 0。如果你把 Superblock 写到了 block 0 的起始位置（字节 0），那会覆盖引导块，虽然引导块我们不用，但 Superblock 本身的位置就错了，下次 mount 时读到的是全零数据，magic 检查直接失败。

inode 号的 -1 和 +1: 所有从 inode 号计算 group 和 local_index 的地方都要先减 1，所有从 local_index 计算全局 inode 号的地方都要加 1。block 则不需要——block 号从 0 开始，直接算就行。这种不对称是 ext2 规范的"遗产"，我们必须接受。

BGDT 的 read-modify-write: write_bgdt 不能直接 write_block 覆盖整个 BGDT 块——因为一个块里可能有多个组的描述符，直接写会丢失其他组的描述符。必须先 read_block 读出整块，替换目标描述符，再 write_block 写回。write_disk_inode 也是同样的道理。

DMA buffer 单次使用限制: 我们的 DMA buffer 只有一个块的大小。如果你在 alloc_block 中读完了 bitmap，然后在计算全局块号之前又调用了 read_block（比如读别的什么数据），bitmap 数据就被覆盖了。所以所有对 DMA buffer 的操作必须在同一次 read_block 的"有效期"内完成——读完、修改、写回，一气呵成，中间不能插入其他 read_block 调用。

最后一个组的边界处理: 最后一个 block group 的 blocks_in_group 可能小于 blocks_per_group——因为磁盘总块数不一定恰好是 blocks_per_group 的整数倍。alloc_block 中必须用 min(blocks_per_group, s_blocks_count - first_block) 来裁剪扫描范围，否则会读到 bitmap 中超出有效范围的 bit，可能把本来空闲的块误判为已占用（如果那些 bit 碰巧是 1）。

本章小结 ​

概念要点
alloc_block线性扫描 block bitmap，找 0 bit 置 1，返回全局块号
free_block计算组号和组内偏移，清除 bit，递增空闲计数
alloc_inode线性扫描 inode bitmap，找 0 bit 置 1，返回 1-based inode 号
free_inode计算 group=(ino-1)/inodes_per_group，清除 bit，递增计数
write_superblock把 sb_ 写到字节偏移 1024，LBA=2, 2 sectors
write_bgdtread-modify-write 策略，先读整块再替换描述符再写回
write_disk_inoderead-modify-write，定位 inode 所在块再覆盖再写回
DMA buffer 限制单块大小，操作必须原子性完成，中间不能插入其他 read_block
