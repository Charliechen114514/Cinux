002-3 · 切换到保护模式 ​

导语 ​

前两篇我们把 GDT 定义好了、linker script 修正了、debugcon 配置到位了——整个舞台已经搭好，只差最后也是最关键的一幕：让 CPU 真正从实模式切换到保护模式。这一步涉及一系列必须严格按序执行的操作：关中断、清 DS、加载 GDTR、设置 CR0 的 PE 位、执行远跳转，然后在 32 位环境下初始化段寄存器和栈。

整个过程容不得半点差错——任何一步的顺序或操作码出问题，等待你的都是经典的三重故障（Triple Fault），QEMU 窗口直接关闭，连报错的机会都不给你。

本篇我们就逐步拆解保护模式切换的每一步，搞清楚为什么每一步必须存在、为什么在这个位置、如果省略或调换会发生什么。所有代码改动集中在 boot/stage2.S 的 _start 函数末尾，从 VESA 初始化完成之后开始。

动手实现 ​

Step 4: 切换到保护模式 ​

目标: 在 stage2.S 的 _start 函数中，VESA 初始化和内核加载完成后，依次执行：关中断 -> DS 清零 -> lgdt -> CR0.PE 置 1 -> 远跳转到 .code32 入口点。

设计思路: 让我们一步步解释为什么每一步都不可少，以及它们之间的依赖关系。Intel SDM Volume 3A Section 10.9.1 给出了严格的切换步骤，顺序不可调换，任何一步缺失都可能导致不可预测的 CPU 行为。

首先是 cli 关中断。进入保护模式后，BIOS 的中断向量表（IVT）就失效了，而我们的 IDT 还没建立。如果这时候发生任何中断（包括定时器中断），CPU 会尝试查一个不存在的 IDT，直接触发 double fault，然后 triple fault，QEMU 重启。

所以在切换开始前必须把中断关掉。注意 cli 在 stage2 一开始就执行过一次（在 _start 入口处），后来 sti 重新开了中断给 VESA 的 BIOS 调用和内核加载用。现在所有实模式操作全部结束，是时候再次关掉了。

然后是把 DS 清零。这一点特别容易被忽略，但查起来让人血压拉满。在实模式下，所有内存访问的物理地址都是 DS * 16 + offset。当我们执行 lgdt gdt_ptr 时，CPU 读 gdt_ptr 的物理地址就是 DS * 16 + gdt_ptr。

如果 DS 不为 0（比如还是之前 VESA 调用后残留的某个值），CPU 就会从错误的位置读取 GDT 指针，加载一堆垃圾到 GDTR 里。这个问题特别难定位，因为 lgdt 本身不会报错——它只是默默地把数据搬进 GDTR，错误要等到后续第一次段访问时才会暴露。

我们在调试时确实踩到了这个坑，花了大半个小时才定位到是 DS 没清零导致 lgdt 读到了错误地址。

接下来是 lgdt gdt_ptr，这条指令从内存中读取 6 字节的 GDT 指针结构（2 字节 limit + 4 字节 base），加载到 GDTR 寄存器。Intel SDM Volume 3A Section 2.4.1 对此有详细说明。

注意 lgdt 本身不校验 GDT 内容的合法性，它只是做了一次内存拷贝。如果 GDT 里的描述符是乱写的，lgdt 不会报错，错误会在后续 ljmp 尝试加载 CS 时才暴露——CPU 查到无效描述符直接触发 #GP。

然后是设置 CR0 的 PE 位：把 CR0 读到 EAX，把最低位（bit 0）或上 1，再写回 CR0。这一步是"通知"CPU 进入保护模式，但此时 CPU 流水线里还缓存着按 16 位实模式取的指令，CS 寄存器还指向实模式的段。

所以 PE 位设完之后，必须马上做一个远跳转。注意这里不能直接对 CR0 做 or 操作，必须先读出来、修改、再写回去——这是因为 CR0 不能作为 ALU 指令的操作数，必须通过通用寄存器中转。这是 x86 ISA 的规定。

远跳转是整个流程中最关键的一步。远跳转的操作数同时指定了新的代码段选择子（0x08）和目标偏移地址（pm_entry），CPU 在执行这个跳转时会把 CS 刷新为新选择子指向的段描述符，同时清空流水线中的预取指令队列，从新的 CS:EIP 开始以 32 位模式重新取指令。

如果不做这个跳转，后续的 32 位指令会被当作 16 位来解码，几乎必定执行非法指令然后崩溃。

关于远跳转的编码，有一个好消息：在 .code16 模式下，GAS 会自动把远跳转指令编码为正确的 16 位远跳转格式（操作码 EA + 2 字节偏移 + 2 字节选择子）。因为 stage2 的代码全部在 0x8000-0x10000 范围内，16 位偏移足够表示。你不需要手动拼机器码，汇编器会帮你处理好。

踩坑预警: 如果你在 lgdt 之前忘记把 DS 清零，表面上不会报任何错误，但后续远跳转时 CPU 会从错误的 GDT 条目里取段描述符，大概率触发 General Protection Fault 或者 triple fault。

另一个常见问题是有人在远跳转的目标地址上手动减去了加载偏移（比如写 pm_entry - 0x8000），这是不必要的——因为 linker script 已经把 base 设成了 0x8000，链接器算出来的 pm_entry 地址本身就是正确的物理地址，再减一次就跳到错误的内存去了。

bash
gdb build/boot/stage2
(gdb) target remote :1234
(gdb) break *pm_entry
(gdb) continue
(gdb) info registers cr0

Step 5: 保护模式段寄存器初始化与验证输出 ​

目标: 在 .code32 的 pm_entry 入口点中，把 DS、ES、FS、GS、SS 全部加载为数据段选择子 0x10，设置栈指针到 0x90000，然后向端口 0xE9 输出字母 P。

设计思路: 远跳转完成后，CS 已经被自动刷新为代码段选择子 0x08，但其他段寄存器（DS、ES、FS、GS、SS）还残留着实模式的值。

在保护模式下，这些段寄存器也要通过段选择子来寻址，如果不更新，任何内存访问都会因为使用无效的实模式段值而触发异常。所以进入保护模式的第一件事就是统一加载所有数据段寄存器为 0x10（数据段选择子，即 GDT 中索引 2 的数据段描述符）。

栈段的设置也不能遗漏。SS 必须指向有效的数据段描述符，栈指针我们设在 0x90000。

这个地址的选择有几个原因叠在一起：第一，它不会和 stage2 的代码区域（0x8000-0x10000）冲突；第二，它远离 VESA 缓冲区（0x6000-0x64FF）和 E820 内存映射（0x5000-0x5FFF）；第三，从 0x90000 向下增长到 0x80000 有整整 64KB 的空间，对 Bootloader 阶段来说绰绰有余。

在 Flat Memory Model 下，SS 对应的数据段 Base 是 0x00000000，所以 0x90000 就是实际的物理栈顶地址。在实模式阶段，我们用的是 SS:SP = 0x9000:0xFFFE，对应物理地址 0x9000 * 16 + 0xFFFE = 0x9FFFE。进入保护模式后，段寄存器的语义变了——SS 不再存储段地址，而是存储段选择子，所以我们重新把 SS 设成 0x10，然后直接设 ESP 为 0x90000。

最后是验证输出。进入保护模式后，BIOS 中断全部失效，我们不能再用 INT 10h 打印字符了。QEMU 提供了一个非常方便的调试接口叫 debugcon：向 I/O 端口 0xE9 写入一个字节，QEMU 就会把这个字节输出到 debugcon 设备。

我们向它写入 0x50（字母 P 的 ASCII 码），如果一切顺利，debug.log 里就会看到这个 P。这是保护模式下最简单的输出手段，不需要初始化 UART、不需要配置串口，一个简单的输出指令搞定。

验证输出完成后，stage2 的保护模式阶段在这个 tag 里就结束了。我们还没有实现后续的 long mode 切换（那是 tag 003 的事），所以代码进入停机循环：cli 加 hlt 加无条件跳转死循环，确保 CPU 不会跑飞。

调试技巧 ​

GDB 断点调试 ​

在保护模式切换这种底层代码里，GDB 是你最可靠的朋友。启动 QEMU 的调试模式：

bash
cd build
make run-debug
1
2

然后在另一个终端连上 GDB：

bash
gdb build/boot/stage2
(gdb) target remote :1234
1
2

注意这里要用 ELF 文件（build/boot/stage2）而不是 bin 文件（build/boot/stage2.bin）。ELF 文件包含符号表和调试信息，GDB 才能正确识别函数名和行号；bin 文件只是裸二进制，没有任何元数据，GDB 拿到它就像看一堆字节，断点和符号全都对不上。

在 GDB 中可以设置几个关键断点：在 _start 处确认 Stage2 入口正常，在远跳转指令处确认 lgdt 已经执行完毕，在 pm_entry 处确认保护模式入口被正确跳转到。到达 pm_entry 后，用 info registers 查看 CR0 的 PE 位是否为 1，段寄存器是否指向正确的选择子。

QEMU 中断日志 ​

如果 CPU 在某个时刻 triple fault 了（QEMU 窗口直接关闭或者重启），用 -d int 参数启动 QEMU 可以看到所有中断和异常的记录：

bash
qemu-system-x86_64 -drive format=raw,file=build/cinux.img -d int -D qemu_int.log -serial stdio -debugcon stdio -no-reboot
1

然后在 qemu_int.log 里搜索 exception 或 fault 关键字，看看是哪种异常、发生在什么地址。最常见的三种：General Protection Fault 通常是段选择子或描述符有问题，Page Fault 在这个阶段不太可能（还没开分页），Double/Triple Fault 说明前一个异常的处理又触发了异常。

debugcon 面包屑调试 ​

在整个 bootloader 开发过程中，debugcon（端口 0xE9）是最轻量的调试输出方式。它不需要初始化任何硬件，不需要配置波特率，只需要向端口 0xE9 写一个字节即可。

你可以在代码的任何可疑位置插入一个输出操作，输出不同的字符来标记执行到了哪里——比如在 lgdt 前后各输出一个字符，在 CR0 写入后输出一个字符，在远跳转后输出一个字符。通过观察哪些字符出现了、哪些没出现，就能快速定位崩溃发生在哪一步。这比 GDB 单步调试快得多，尤其是你只是想知道"有没有执行到这里"的时候。

常见 Triple Fault 排查 ​

这里总结几种最常见的 triple fault 原因和对应的排查方向，按出现频率排序。

lgdt 读到了错误的 GDT 地址。 最常见的坑，原因是 lgdt 前 DS 没有清零。实模式下 lgdt 用 DS * 16 + offset 计算地址，DS 不为 0 就会读到错误位置。排查方法：在 lgdt 前后各加一个 debugcon 输出，如果前一个字符出现了后一个没有，大概率是 lgdt 导致的三重故障。

GDT 描述符定义错误。 Access Byte 写错一位、Limit 不够大、Base 地址错误，这些都会导致 ljmp 时 CPU 查到无效描述符触发 #GP。排查方法：用 objdump -s -j .gdt build/boot/stage2 检查编译后的 .gdt section，逐字节对照 Intel SDM Vol.3A Section 3.4.5 的描述符格式。

linker script base 地址不匹配。 stage2 被加载到 0x8000，但 linker script 的 location counter 设成了 0x0，导致所有绝对地址引用偏移 0x8000。排查方法：用 readelf -s build/boot/stage2 | grep gdt 查看 gdt 符号的链接地址，如果比预期小 0x8000，就是 base 没设对。

省略了 far jump。 CR0.PE 设完之后直接写 32 位代码，没有远跳转刷新 CS 的隐藏缓存。CPU 还在用 16 位模式解码，32 位编码的指令被解析成乱码。排查方法：检查 CR0 写入后是否紧跟远跳转指令，中间不能有任何其他指令。

本章小结 ​

到这里，我们已经完成了 bootloader 开发中最关键的里程碑之一：把 CPU 从 16 位实模式成功推进到 32 位保护模式。

CR0.PEbit 0 = 1Protection Enable，开启保护模式
远跳转ljmp $0x08, $pm_entry刷新 CS 和流水线，真正进入 32 位
debugcon 端口0xE9QEMU 调试输出，向端口写入即可
栈指针0x90000保护模式下的 ESP 初始值

下一篇（tag 003）我们将继续往上走——从 32 位保护模式进入 64 位长模式。长模式的切换需要设置页表、启用 PAE 和 EFER.LME，流程比保护模式切换更长也更复杂，但底层的思路是一样的：按顺序做、每步验证、不要跳步。
