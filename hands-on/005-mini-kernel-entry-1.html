005-1 动手篇：串口驱动与 I/O 端口编程 ​

本章完成后的可见效果：QEMU 串口终端（stdout）输出由 kprintf 产生的格式化字符串，证明内核可以通过 UART 驱动向外部世界"说话"了

前置要求：已完成 004_boot_load_mini_kernel_C，理解 Bootloader 到 Mini Kernel 的跳转过程

导语 ​

上一章我们在 Bootloader 的末尾打出那个 H 字符然后一记 jmp *%rax 跳了过去，内核终于开始跑自己的代码了。但说实话，此刻的内核依然是一个彻头彻尾的黑盒子——它在跑，我们看不到任何输出，不知道它跑到了哪里，不知道变量是什么值，不知道有没有出错。对于内核开发来说，这种"失明"状态是致命的，一旦出 bug 就只能对着 QEMU 的重启干瞪眼。

所以这一章我们要给内核装上第一件正经的武器：串口驱动。串口是内核开发中最基础也最可靠的调试输出手段，不需要显卡驱动、不需要帧缓冲、不需要字体渲染，只要几条 I/O 端口指令就能把字符送出内核。完成本篇后，我们的内核就拥有了向外部世界输出文字的能力——虽然只有文字，但对于内核开发者来说，这已经是最强大的 debugging 工具了。

本篇要完成的事情分三层。最底层是 x86 I/O 端口机制的封装——inb/outb 这两个内联函数，它们是所有端口 I/O 的基础。中间层是 UART 8250/16550 串口驱动，负责把一个字节变成串口线上的一帧数据。最上层是 Serial 类，提供 putc/puts 这样人类友好的接口，顺带处理一下 \n 到 \r\n 的转换——这一步不做的话你的终端输出会全部挤在一行上，非常刺激。

概念精讲 ​

x86 I/O 端口机制：CPU 和外设的"专用电话线" ​

x86 架构给 CPU 和外设之间准备了两套通信机制：内存映射 I/O（MMIO）和端口映射 I/O（PMIO）。串口用的是后者——它通过两条专用指令 in 和 out 来完成数据交换，而不是像 MMIO 那样读写某个内存地址。你可以把 PMIO 想象成 CPU 和外设之间的一根专用电话线：每根线有一个编号（端口号），CPU 拿起某根线说"读"或者"写"，对应的外设就响应。

这个机制的实现很直接。out 指令把寄存器里的值写到指定的端口，in 指令从指定端口读一个值到寄存器。端口号可以是 8 位立即数（0x00 到 0xFF），也可以放在 DX 寄存器里（支持 16 位端口号，0x0000 到 0xFFFF）。数据宽度可以是字节（8 位）、字（16 位）或双字（32 位），对应的指令后缀分别是 b、w、l。

Intel SDM Vol. 2A 第 3 章对 IN/OUT 指令有详细说明——IN 在 ~page 3-370，OUT 在 ~page 3-510。关键点在于，这两条指令在保护模式下受到 IOPL（I/O Privilege Level）和 TSS I/O 位图的控制，但在我们的内核里运行在 ring 0，所以不存在权限问题。

在 AT&T 汇编语法下，inb 的操作数顺序是"从端口读到寄存器"，写成 inb %port, %reg；outb 是"从寄存器写到端口"，写成 outb %reg, %port。注意 AT&T 语法里源操作数在前面、目的操作数在后面，和 Intel 语法是反的——这个细节搞反了就是读到一个垃圾值然后写到错误的端口，非常难查。

UART 8250/16550：串口背后的那块芯片 ​

UART（Universal Asynchronous Receiver/Transmitter）是串口背后的硬件控制器。PC 上用的 UART 芯片从早期的 8250 演进到后来的 16550A（增加了 FIFO 缓冲），但编程接口完全兼容。在 QEMU 里模拟的也是 16550A。

PC 上最多有四个串口，各有固定的 I/O 端口基地址。COM1 是 0x3F8，COM2 是 0x2F8，COM3 是 0x3E8，COM4 是 0x2E8。我们用的是 COM1，因为它是 QEMU 默认启用并且通过 -serial stdio 参数重定向到终端的那个。

UART 的寄存器通过基地址加偏移来访问。我们关心的寄存器并不多：

偏移 +0 处，读的时候是 RBR（接收缓冲寄存器），写的时候是 THR（发送保持寄存器）。这是同一个端口地址，但读写行为不同——读就是拿数据，写就是发数据。

偏移 +1 是 IER（中断使能寄存器）。我们用轮询模式，不需要中断，所以把它设成 0 就行了。

偏移 +2 写的时候是 FCR（FIFO 控制寄存器）。值设成 0xC7 的含义是：启用 FIFO、清空发送和接收缓冲区、设置 14 字节的中断触发阈值。

偏移 +3 是 LCR（线路控制寄存器）。值 0x03 代表"8N1"——8 位数据位、无奇偶校验、1 位停止位。这是串口通信最标准的配置。

偏移 +5 是 LSR（线路状态寄存器），只读。它的第 5 位叫 THRE（Transmit Holding Register Empty），当这个位为 1 时表示发送缓冲区空了，可以写入下一个字节。我们在发送字符之前必须轮询这个位——不等待的话，如果上一个字节还没发完你就往 THR 里塞新数据，上一个字节就丢了。

初始化序列的步骤是这样的：先禁用中断（IER 写 0），然后设置 8N1 数据格式（LCR 写 0x03），接着启用 FIFO 并清空缓冲区（FCR 写 0xC7），最后设置 MCR（Modem 控制寄存器）为 0x03 表示 RTS + DTR 信号有效。至于波特率——QEMU 默认就是 115200，而且模拟环境里波特率对数据正确性没有影响（虚拟串口不存在信号失真），所以我们跳过了 DLAB 分频器的设置。在真实硬件上你需要根据目标波特率计算分频系数，但那是以后的事。

参考来源：OSDev Wiki Serial Ports（https://wiki.osdev.org/Serial_Ports）对 UART 寄存器映射和初始化流程有完整的说明。

Intel SDM Vol. 3A Chapter 18 对 I/O 保护机制有详细说明——IOPL 字段在 EFLAGS 中控制 IN/OUT 指令的特权级别，TSS 中的 I/O 位图可以精确到每个端口的访问控制。不过由于我们的内核运行在 Ring 0，所有 I/O 端口操作都是允许的，所以暂时不需要关心这些保护机制。

QEMU 的两条输出通道：串口 vs debugcon ​

在继续之前，我们需要厘清 Cinux 项目里两个不同的调试输出通道，因为后面会同时用到它们。

第一个是串口（COM1，端口 0x3F8）。这是标准的硬件接口，真实 PC 上也有，通过 QEMU 的 -serial stdio 参数重定向到宿主机的标准输出。数据经过完整的 UART 流程：等待 THRE -> 写 THR -> QEMU 捕获 -> 显示到终端。优点是兼容真实硬件，缺点是每次输出都有轮询开销。

第二个是 debugcon（端口 0xE9）。这是 QEMU 的私有调试设备，不存在于真实硬件上。往端口 0xE9 写一个字节，QEMU 就直接把它输出到 debugcon 日志文件（通过 -debugcon file:debug.log 配置）。不需要初始化、不需要等待、不需要任何握手——单条 outb 指令就完事。缺点是只有 QEMU 支持，而且输出到文件而不是终端。

在之前几章里我们一直用 debugcon 输出 O、P、L、H 这些标记字符来追踪 Bootloader 的执行进度。从本篇开始，我们要加一条新的通道——串口。串口是内核正式的输出通道，kprintf 的输出走串口；kdebugf 的输出走 debugcon。两条通道互为备份，串口看不到的时候去看 debugcon 日志，往往能找到线索。

在实际的内核开发中，这种"双通道"策略非常实用——串口输出面向开发者日常查看（实时、直观），debugcon 输出面向自动化测试和事后分析（持久化、不影响串口交互）。两个通道的格式化引擎是同一套代码（下一篇会讲的 vkprintf_impl 模板函数），只是输出后端不同。

好了，概念铺垫得差不多了，下面我们正式开始动手。

动手实现 ​

Step 1: 创建 I/O 端口封装层 ​

目标：在 kernel/mini/driver/io.h 中实现 inb 和 outb 两个内联函数，封装 x86 的 IN/OUT 指令。

设计思路：我们选择用 C++ inline 函数而不是 C 宏来封装 I/O 端口操作。宏虽然也能干这活，但它不做类型检查、参数可能被多次求值、而且不好调试。内联函数有完整的类型信息，编译器会在调用点展开，性能和宏完全一样，但安全得多。

这两个函数的核心是 GCC 内联汇编。inb 的内联汇编模板把 inb 指令的输出约束到 AL 寄存器（通过 "=a" 约束），输入端口号约束为 "Nd"——意思是可以是 8 位立即数也可以是 DX 寄存器（x86 的 I/O 指令只接受这两种形式）。outb 反过来，输入值通过 "a" 约束放到 AL 寄存器，端口号同样是 "Nd" 约束。

两个函数都给汇编语句加了 volatile 修饰符。这一点非常重要：I/O 操作是副作用操作，编译器不知道 outb 会改变外部硬件状态，如果它觉得这个函数的返回值没被使用就可能把整个调用优化掉。volatile 明确告诉编译器"不要删、不要重排、老老实实生成这条指令"。

实现约束：把这两个函数放在一个命名空间下（比如 io），避免和系统头文件里的同名函数冲突。头文件只需要包含 <stdint.h>，不依赖任何其他东西。这两个函数既不依赖 Serial 类也不被 Serial 类以外的代码使用（暂时），所以放在 driver/ 目录下作为驱动层的最底层基石。

踩坑预警：千万别把 inb 汇编模板里的操作数顺序搞反了。AT&T 语法下 inb %1, %0 的意思是"从 %1（端口）读到 %0（输出变量）"，如果你写成 inb %0, %1，编译不会报错（约束都对得上），但运行时读到的值是垃圾，因为端口号和输出变量的角色互换了。另外，"Nd" 约束里的大写 N 表示"8 位立即数"，小写 n 没有这个含义——写成小写 n 的话编译器可能把端口号放到任意寄存器里，而 x86 的 in/out 指令不接受除 DX 以外的寄存器。

验证：这一步只有头文件，没有独立的可执行代码。编译通过即可——确保 kernel/mini/driver/io.h 存在且被其他驱动文件 include 后编译无报错。

Step 2: 实现 Serial 类——UART 初始化 ​

目标：在 kernel/mini/driver/serial.h 和 serial.cpp 中实现串口驱动类，首先是 init() 方法完成 UART 初始化。

设计思路：Serial 类需要持有串口的基地址（COM1 = 0x3F8），通过构造函数传入。init() 方法按照标准的 UART 初始化序列设置各个寄存器：禁用中断 -> 设置 8N1 -> 启用 FIFO -> 设置 MCR。这个序列是 OSDev Wiki 和所有 UART 文档都推荐的标准流程。

类的设计上，我们把寄存器偏移量定义为枚举常量（比如 THR 偏移 0、IER 偏移 1、FCR 偏移 2、LCR 偏移 3、LSR 偏移 5），LSR 的 THRE 位也定义为常量（值 0x20，即第 5 位）。这样代码里写的是 base_port + LSR 和 value & TX_READY，而不是一堆裸的魔数 base_port + 5 和 0x20。

实现约束：init() 内部每一步都是调用 io::outb() 写入对应的寄存器偏移。基地址加上偏移就是目标端口地址——io::outb(0x3F8 + 0, value) 写 THR，io::outb(0x3F8 + 1, value) 写 IER，以此类推。不要直接在代码里硬编码 0x3F8，而是用成员变量 base_port 加上偏移枚举来计算。

踩坑预警：关于波特率的问题——很多人照着教程设置了 DLAB（Divisor Latch Access Bit，LCR 的第 7 位）和分频系数，然后在 QEMU 里发现设不设都一样。这是因为 QEMU 的虚拟串口不受物理时钟限制，115200 的默认波特率在模拟环境下始终成立。但如果你的代码将来要跑在真实硬件上，就必须正确设置 DLAB 和分频系数。所以一种稳妥的做法是：在 QEMU 开发阶段先跳过波特率设置（反正不影响功能），等上真机测试前再补上。

验证：这一步暂时还没有可见的输出效果，需要等 Step 3 的 putc() 实现后一起验证。但你可以先检查 serial.h 是否被正确 include 到项目中——在 main.cpp 中 #include "driver/serial.h"，编译通过说明头文件路径配置正确。

Step 3: 实现 Serial 类——字符和字符串输出 ​

目标：实现 Serial::putc() 和 Serial::puts() 方法，前者发送单个字符，后者发送整个字符串。

设计思路：putc() 的逻辑很简单：轮询 LSR 的 THRE 位，等它变成 1（表示发送缓冲区空了），然后往 THR 写入目标字符。轮询循环里加了一条 __asm__ volatile("pause") 指令——这是 x86 的 hint 指令，告诉 CPU"我在等一个可能很慢的事件"，CPU 会降低功耗并改善超线程环境下的性能。在 QEMU 里其实区别不大，但这是一个好习惯。

puts() 遍历字符串的每个字符，逐个调用 putc()。但有一个关键细节：遇到 \n（换行符）时，必须先发送一个 \r（回车符），再发送 \n。原因是串口终端（以及 QEMU 重定向到的标准输出）对换行的处理方式和 VGA 文本模式不同——串口终端收到 \n 只是光标下移一行，但不回到行首；只有收到 \r\n 组合才会回到行首再下移。如果你不处理这个转换，终端里所有输出都会呈阶梯状偏移，就像打字机不回车直接换行一样——看起来非常混乱。另外，puts() 需要处理传入空指针的情况，直接 return 即可。

实现约束：putc() 中轮询条件的写法需要正确——先读 LSR，然后检查 THRE 位是否为 1。如果写成 while (io::inb(base_port + LSR) & THRE) 就错了——THRE 为 1 才表示"可以发送"，所以应该是"等待 THRE 不为 1 的时候继续等"，即 while (!(io::inb(base_port + LSR) & THRE))。

踩坑预警：\r\n 的问题真的会坑到很多人。在调试时如果看到输出像楼梯一样一行比一行缩进多，多半就是忘了在 \n 前面加 \r。而且这个问题在 debugcon 通道里不会出现（debugcon 直接输出到文件，不做终端控制），所以如果你看 debugcon 正常但串口输出阶梯状，第一个嫌疑就是回车符。

验证：现在可以验证了。在 main.cpp 的 mini_kernel_main() 里调用 Serial 的初始化和 puts，然后构建运行。具体来说：你需要获取一个 Serial 对象（通过全局单例或者直接构造），调用其 init() 方法，然后调用 puts("Hello from serial!\n")。构建并运行 QEMU 后，串口终端（标准输出）应该出现这行文字。如果看到了阶梯状的偏移输出，说明 \r\n 转换没做好。

Step 4: 全局 Serial 单例与 debugcon 输出函数 ​

目标：提供一个全局可用的 Serial 对象，以及一个直接写 debugcon 端口的字符输出函数。

设计思路：C++ 的全局构造函数机制（.init_array 段）允许我们在 main() 之前执行初始化代码。我们利用这个机制创建一个全局 Serial 对象，在全局构造函数里自动调用 init()——这样内核的任何代码在任何时刻都可以直接使用这个已经初始化好的串口，不需要显式调用初始化。

具体做法是定义一个返回 Serial 引用的函数（比如 get_initial_serial()），函数内部持有一个静态的 Serial 实例。由于内核的 .init_array 机制已经在之前的 tag 里实现了（boot.S 里的 _init_global_ctors 调用），全局构造函数会在 mini_kernel_main() 被调用之前执行完毕，所以 Serial 的初始化一定在第一次使用之前完成。

同时我们还要实现一个 debugcon_putc() 函数，它只需要一条 outb 指令把字符写到端口 0xE9。这个函数将来要被 kdebugf 使用。

实现约束：debugcon_putc() 不需要等待任何状态位——debugcon 不是真正的 UART，QEMU 直接把写入的字节捕获到日志文件，没有任何缓冲或流控。所以这个函数就是一条裸的 outb。

踩坑预警：全局构造函数的执行依赖于链接脚本里 .init_array 段的正确设置，以及 boot.S 里 _init_global_ctors 的正确调用。如果这两步有任何一个出了问题（比如下一节要讲的链接脚本 LMA 计算错误），全局构造函数数组里存放的就是垃圾指针（0xFFFFFFFF 之类），一调用就 Triple Fault。所以如果你的内核在进入 mini_kernel_main() 之前就崩溃了，除了检查 boot.S 的逻辑，还要检查 .init_array 的内容——用 readelf -l kernel.elf 查看 LMA 是否正确。

另外需要注意的是，如果你的 boot.S 中没有正确处理 __boot_info_ptr（把它保存在 BSS 清零之前），全局构造函数中如果有人访问了依赖 BootInfo 的数据，也会出问题。不过我们的 Serial 不依赖 BootInfo，所以这一点在这里不影响。

验证：在 main.cpp 中通过全局获取函数拿到 Serial 对象，调用 puts 输出一段文字。同时用 debugcon_putc 输出几个字符到 debugcon。构建运行后，串口终端应该看到 puts 的输出，debug.log 文件里应该看到 debugcon 的输出。注意，如果你在 mini_kernel_main 之前就尝试使用串口（比如在全局构造函数里），这也是安全的——因为全局 Serial 对象的构造函数会先于其他全局构造函数执行（只要它在链接顺序上排在前面）。

调试技巧 ​

区分串口和 debugcon 的问题 ​

当你发现"没有输出"的时候，第一步是判断是串口的问题还是内核根本没跑起来。查看 build/debug.log——如果里面有 Bootloader 的标记字符（OPL 加上高半核跳转标记），说明内核入口代码至少执行到了 boot.S；如果 debugcon 日志里出现了你在 C++ 代码中通过 debugcon_putc 输出的字符，说明 C++ 运行时（包括全局构造函数）已经正常工作了。此时如果串口依然没有输出，问题就锁定在 Serial 驱动本身——多半是初始化序列有遗漏，或者 LSR 轮询卡住了。

单步跟踪 Serial 初始化 ​

如果串口初始化看起来没有效果，可以在 GDB 里单步跟踪。在 Serial::init() 设置断点，观察每一步 outb 之后对应端口的状态。虽然 GDB 不能直接读 UART 的内部寄存器，但你可以检查 base_port 的值是否是 0x3F8、io::outb 被调用时的参数是否正确。

QEMU 的串口模拟细节 ​

QEMU 对串口的模拟有一个值得注意的特点：它的虚拟 UART 在任何时刻都是"就绪"的——THRE 位永远为 1。这意味着在 QEMU 环境下，putc() 里的轮询循环实际上永远不会真的等，第一次检查就通过了。但在真实硬件上，THRE 位的轮询是必须的，因为物理 UART 发送一个字节需要时间（115200 波特率下大约 87 微秒），连续写入而不等待会导致数据丢失。所以在 QEMU 里"能跑"不代表在真机上也没问题——这一点在代码审查时要注意。

关于 io.h 中额外的 I/O 函数 ​

实际的 io.h 中除了 inb/outb 之外，还定义了 inw/outw（16 位）和 inl/outl（32 位）函数。它们的实现模式完全相同——唯一的区别是约束的寄存器大小和汇编指令后缀（w 对应 AX，l 对应 EAX）。当前我们只用到了 inb/outb（8 位），但后续章节在配置 isa-debug-exit 设备时会用到 outl，所以理解它们的封装方式是有价值的。如果感兴趣可以提前看看 io.h 中 16 位和 32 位版本的定义，它们和 8 位版本是对称的。

用 readelf 验证构建产物 ​

构建完成后可以用 readelf -l build/kernel/mini/mini_kernel 查看 ELF 的 Program Headers。你应该能看到至少两个 LOAD 段：一个只读的（.text + .rodata）和一个可写的（.data + .init_array + .bss）。VirtAddr 列应该显示 0xFFFFFFFF8002xxxx 的高半核地址，PhysAddr 列应该显示 0x000000000002xxxx 的物理地址。如果 PhysAddr 不是从 0x20000 开始的，说明链接脚本的 LMA 计算有问题——这个问题我们在第三篇会详细讲到。

概念要点
x86 I/O 端口 (PMIO)通过 IN/OUT 指令访问，端口号 0x0000-0xFFFF，受 IOPL 和 TSS 位图控制
inb/outb 内联函数GCC 内联汇编封装，volatile 防止优化，"Nd" 约束匹配 x86 端口操作数
UART 8250/16550 寄存器THR(+0W) 发送、IER(+1) 中断、FCR(+2W) FIFO、LCR(+3) 线路、LSR(+5R) 状态
串口初始化序列IER=0 -> LCR=0x03(8N1) -> FCR=0xC7 -> MCR=0x03
THRE 轮询发送前等待 LSR bit5=1，QEMU 下始终就绪但真机必须等待
