026-fs-ramdisk-2: 构建系统集成——二进制嵌入与链接 ​

导语 ​

上一篇我们把 ustar 格式的"图纸"画好了——头部结构体、常量定义、八进制转换函数都有了。但这些只是纸上谈兵， 真正的挑战在于：怎么把一个 tar 归档文件"塞进"内核的二进制镜像里，让内核启动后能通过指针直接访问里面的 数据？

这不是一件 trivial 的事情。内核是 ELF 格式的可执行文件，tar 是一个普通的数据文件——两者完全不是一个 世界的东西。我们需要一座桥梁，把 tar 文件转换成内核可以链接的 ELF 目标文件。这篇我们做的就是搭这座桥： 写一个 shell 脚本做二进制转换，修改 CMake 构建系统把转换后的目标文件链接进内核，再修改链接脚本给 initrd 数据安排一个专门的位置。

完成本篇后，内核镜像里就会包含完整的 initrd 数据，通过链接器符号可以访问它的起始地址、结束地址和大小。

前置知识：上一篇的 ustar 格式定义。需要对 CMake 和链接脚本有基本了解。

概念精讲 ​

objcopy 的二进制嵌入魔法 ​

GNU binutils 的 objcopy 工具有一个不太为人知但极其实用的功能：-I binary 输入格式。当你告诉 objcopy 输入是一个 "binary" 格式的文件时，它会做一件非常聪明的事——把整个文件内容包装成一个 ELF 目标文件的 .data 段，同时自动生成三个符号：_binary_路径_start、_binary_路径_end 和 _binary_路径_size，分别 指向数据的起始地址、结束地址和大小信息。

这个功能在嵌入式开发中很常见——把图片、字体、音频等资源编译进固件。但在 OS 开发中同样好用——我们可以 把任何二进制数据（文件系统镜像、初始 RAM 磁盘）编译进内核。Cinux 用它来嵌入 initrd.tar 归档。

在 C++ 代码中，这些符号通过 extern "C" 声明为 const uint8_t 数组来访问。数组名在 C/C++ 中天然代表 首元素的地址，所以 _binary_initrd_start 的值就是数据的第一个字节所在的地址。end 减 start 就是数据 的总字节数。

符号命名的坑——为什么要两步走 ​

这里有一个大坑：objcopy 生成的符号名来自输入文件的绝对路径，把所有非字母数字字符替换成下划线。比如 输入 /home/user/cinux/kernel/data/initrd.tar，生成的符号会变成类似 _binary__home_user_cinux_kernel_data_initrd_tar_start 的怪物。

这有几个严重问题。第一，符号名又长又丑，在代码中 extern 声明非常难读。第二，不同开发者的源码路径不同 ，符号名也不同——我的机器上叫 binary__home_charlie...，你的机器上叫 binary__home_alice...，代码 根本没法共享。第三，CI 环境的路径也可能不同，导致构建失败。

我们的解决方案是两步走。第一步用 objcopy 的 -I binary 做初始转换，同时用 --rename-section 把默认的 .data 段改名为 .initrd。第二步用 nm 命令提取自动生成的符号名，然后用 objcopy 的 --redefine-sym 把符号重命名为稳定的 _binary_initrd_start、_binary_initrd_end 和 _binary_initrd_size。

链接脚本中的 .initrd 段 ​

有了目标文件，还需要告诉链接器把它放在内核镜像的哪个位置。Cinux 的内核运行在更高的半内存地址空间 （higher-half kernel），虚拟地址比物理地址高 KERNEL_VMA（0xFFFFFFFF80000000）。所以链接脚本里需要 处理 VMA（虚拟地址）和 LMA（加载地址）的映射关系。

我们为 initrd 创建一个专门的 .initrd 段，放在 .init_array 和 .bss 之间，4KB 页对齐。使用 AT(ADDR(.initrd) - KERNEL_VMA) 确保物理地址正确。如果不设 AT()，bootloader 会在错误的物理地址加载 initrd 数据——higher-half 内核的虚拟地址不能直接用作物理加载地址。

.initrd 段的位置必须在 .bss 之前。.bss 段的特殊之处在于它不占用文件空间，只有运行时才需要清零。如果 .initrd 放在 .bss 后面，链接器可能在计算 .bss 大小时出错，或者 .initrd 数据会被 .bss 的清零操作覆盖。

动手实现 ​

Step 1: 编写 embed_binary.sh 脚本 ​

目标: 创建 scripts/embed_binary.sh，将任意二进制文件转换为带稳定符号名的 ELF 目标文件。

设计思路: 脚本接收四个参数：输入文件路径、输出目标文件路径、目标段名、符号前缀。分两步执行：第一步 用 objcopy 的 -I binary 把文件转成 ELF 对象，同时用 --rename-section 把 .data 段改名为我们指定的 段名；第二步用 nm 命令提取自动生成的符号名，再用 objcopy 的 --redefine-sym 重命名为稳定名字。

实现约束: 脚本开头加 set -euo pipefail 保证任何命令失败立即退出。使用 OBJCOPY 环境变量支持交叉 编译工具链（比如 x86_64-elf-objcopy）。段属性设为 CONTENTS,ALLOC,LOAD,READONLY,DATA——initrd 数据 在运行时只读。

第一步的 objcopy 命令需要指定输出格式为 elf64-x86-64，二进制架构为 i386:x86-64。--rename-section 把 .data 段改名为传入的 SECTION 参数（在我们的场景中是 ".initrd"），同时设置段属性。

第二步先用 nm 列出目标文件的符号表，用 grep 找到以 _start、_end、_size 结尾的符号，awk 提取符号名 （第三列）。然后对每个符号执行 --redefine-sym 重命名。

踩坑预警: 如果脚本不加 set -euo pipefail，objcopy 成功但 nm 提取失败时脚本会继续执行，后续的 objcopy 重定义符号会因为变量为空而出错，但错误信息会非常难懂。另外，OBJCOPY 环境变量默认为系统 objcopy ，交叉编译时需要手动设置。

验证: 手动运行脚本，用 nm 检查输出的目标文件，确认包含 _binary_initrd_start、_binary_initrd_end 和 _binary_initrd_size 三个符号。命令：nm initrd.o | grep initrd。

Step 2: 创建 initrd 归档内容 ​

目标: 准备要嵌入内核的文件，打包成 ustar 格式的 tar 归档。

设计思路: 在 kernel/data/initrd_contents/ 目录下放几个测试文件，然后用系统 tar 命令打包成 kernel/data/initrd.tar。tar 默认创建的就是 ustar 格式（在 Linux 上实际上是 GNU tar 格式，但头部 包含 "ustar" magic）。

我们的测试文件包括 hello.txt（包含问候语 "Hello from Cinux!"）、readme.txt（说明文件 "This is a readme file."）和 etc/passwd（模拟最小化系统目录结构，内容为 "root❌0:0"）。这些文件涵盖了普通文件 和目录结构两种场景。

实现约束: 打包时使用相对路径，确保 tar 内的文件名不以 / 开头。使用 tar cf initrd.tar -C initrd_contents . 命令。文件内容保持简短——initrd 在内核中占用内存，不需要放大文件。

踩坑预警: 不同系统上 tar 的默认格式可能不同。macOS 的 bsdtar 和 Linux 的 GNU tar 行为有差异。确保 tar 归档中每个头部的 magic 字段包含 "ustar"。可以用 tar -tvf initrd.tar 检查归档内容是否正确。

验证: 用 tar -tvf kernel/data/initrd.tar 列出归档内容，确认能看到三个文件及其大小。你还应该看到 一个 "./" 条目（根目录）和一个 "etc/" 条目。

Step 3: 修改 CMake 构建系统 ​

目标: 在 kernel/CMakeLists.txt 中添加自定义命令，将 initrd.tar 编译为可链接的目标文件，并加入 内核的链接依赖。

设计思路: 使用 CMake 的 add_custom_command 定义转换规则：输入是 initrd.tar 和 embed_binary.sh， 输出是 initrd.o。用 add_custom_target 创建一个虚拟目标 initrd_obj 来管理依赖关系。然后把这个目标 文件加入 big_kernel 和 big_kernel_test 两个可执行目标的链接列表中。

实现约束: 输出文件路径设为构建目录下的 kernel/initrd.o（不是源码目录）。DEPENDS 同时依赖 tar 文件和脚本文件——修改任何一个都会触发重新构建。COMMENT 提供构建日志中的可读描述。

CMake 中需要定义三个变量：INITRD_TAR（源码目录中的 tar 文件路径）、INITRD_OBJ（构建目录中的目标 文件路径）、EMBED_SCRIPT（脚本路径）。add_custom_command 的 OUTPUT 指向 INITRD_OBJ，COMMAND 调用 bash 运行脚本。

add_custom_target 创建 initrd_obj 虚拟目标，依赖于 INITRD_OBJ。然后 add_dependencies 让 big_kernel 和 big_kernel_test 都依赖 initrd_obj。target_sources 把 INITRD_OBJ 加入链接列表。

踩坑预警: CMake 的 add_custom_command 的 OUTPUT 路径必须和后续引用完全一致，否则 CMake 找不到 依赖关系，可能不会在链接前执行转换命令。建议用变量统一管理路径，避免手写不一致。

验证: 运行构建命令，观察构建日志中出现 "Converting initrd.tar -> initrd.o (embedded ramdisk)" 的消息，且不报错。

Step 4: 修改链接脚本 ​

目标: 在 kernel/linker.ld 中添加 .initrd 段，为嵌入的 initrd 数据分配内核地址空间中的位置。

设计思路: 新段放在 .init_array（全局构造函数）之后、.bss（未初始化数据）之前。使用 4KB 页对齐 （ALIGN(4096)），因为 initrd 数据可能比较大，对齐到页边界便于后续内存管理。

段的通配符模式是 *(.initrd)，匹配所有目标文件中名为 .initrd 的段。由于只有 initrd.o 包含这个段名 （由 embed_binary.sh 的 --rename-section 设置），所以恰好只会把 initrd 数据放进来。

实现约束: 使用 AT(ADDR(.initrd) - KERNEL_VMA) 设置加载地址（LMA）。这个表达式把虚拟地址减去 KERNEL_VMA 偏移量，得到正确的物理地址。不带 AT 的话，链接器会假设 LMA 等于 VMA，在 higher-half 内核 中这会导致 bootloader 加载数据到错误的物理地址。

踩坑预警: .initrd 段的位置很重要——它必须在 .bss 之前。如果放在后面，.bss 的清零操作可能覆盖 initrd 数据。

验证: 构建内核后，用 objdump -h big_kernel | grep initrd 查看 .initrd 段的地址和大小。用 nm big_kernel | grep initrd 确认 _binary_initrd_start 和 _binary_initrd_end 符号存在。

构建与运行 ​

完整的构建流程：

bash
cd build
cmake ..
cmake --build . --target big_kernel
1
2
3

构建日志中应该能看到 "Converting initrd.tar -> initrd.o" 和链接步骤（无错误）。

用以下命令验证符号存在：

bash
nm build/kernel/big/big_kernel | grep initrd
1

预期输出包含 _binary_initrd_end 和 _binary_initrd_start 两个符号，它们的地址差值等于 initrd.tar

两者的差值恰好应该是 10240（即 initrd.tar 的文件大小）。如果差值不对，说明链接时目标文件有问题——用 readelf -S initrd.o 检查目标文件的段大小。 的大小（10240 字节）。

调试技巧 ​

如果构建时报 "undefined reference to _binary_initrd_start"，说明目标文件没有被正确链接。检查 CMakeLists.txt 中的 target_sources 是否包含了 INITRD_OBJ，以及 add_dependencies 是否依赖了 initrd_obj。

如果 .initrd 段没有出现在最终的 ELF 文件中，检查链接脚本的模式匹配 *(.initrd) 是否正确。 embed_binary.sh 中 --rename-section 的段名必须和链接脚本中的段名一致。

如果构建成功但运行时访问 initrd 数据时内核崩溃，很可能是 LMA 设置错误——bootloader 把数据加载到了 错误的物理地址。检查链接脚本中 AT() 表达式是否正确计算了物理地址偏移。

本章小结 ​

概念要点
objcopy -I binary把任意文件转换成可链接的 ELF 目标文件
符号重命名objcopy 根据路径生成不稳定符号名，需手动重定义
embed_binary.sh两步转换：binary 到 ELF 加符号重命名
CMake custom commandadd_custom_command 定义转换规则
.initrd 段链接脚本中专门的位置，页对齐，正确设置 LMA
AT() 表达式higher-half 内核中正确映射 VMA 到 LMA

另外，你还可以用 readelf 工具更详细地检查 initrd.o 的段信息：

bash
readelf -S build/kernel/initrd.o
1

这会显示所有段的名称、类型、地址、偏移和大小。你应该能看到一个名为 .initrd 的段，大小恰好等于 initrd.tar 的字节数。段的类型应该是 PROGBITS，标志包含 ALLOC（可分配到内存）。

如果你修改了 initrd_contents 目录下的文件内容或新增了文件，记得重新运行 tar 打包和内核构建。CMake 的 DEPENDS 声明会自动检测 initrd.tar 文件的变化并重新运行 embed_binary.sh，但如果你只是改了源文件 而忘了重新打包 tar，CMake 不会帮你——它只关心 tar 文件本身的修改时间。

到这里，构建系统的集成就完成了。下一篇我们将实现 Ramdisk 驱动，利用这些链接器符号和构建产物，在内核 启动时解析 initrd 归档并列出所有文件。
