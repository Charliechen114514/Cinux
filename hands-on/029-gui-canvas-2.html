基础图元绘制：矩形填充、描边、Bresenham 直线与位图 ​

标签：Bresenham, draw_rect, draw_line, draw_bitmap, blit, 2D primitives 前置：029 Canvas 架构

导语 ​

上一章我们搭好了 Canvas 的双缓冲架构——初始化、draw_pixel 和 flip。但一个只能画单个像素的画布显然不够用，本章我们要给它加上真正的图元绘制能力：矩形填充、矩形描边、Bresenham 直线算法、像素位图渲染，以及画布间的 blit 操作。这些都是 2D 图形系统的基本功——从 Linux 的 libdrm 到 SerenityOS 的 LibGfx Painter 类，每个图形库都要实现这些基本原语。

完成本章后，我们的 Canvas 将具备完整的 2D 绘制能力，可以在屏幕上画出任意形状的图形组合。

概念精讲 ​

矩形绘制：填充 vs 描边 ​

矩形是最基本的 2D 图元，也是最常用的。Canvas 提供两种矩形操作：draw_rect（实心填充）和 draw_rect_outline（1 像素宽的边框描边）。

填充矩形的实现很直观——双重循环遍历行和列，对每个像素调用像素索引计算。但有一个优化点：我们不需要通过 draw_pixel 来写（它会做边界检查），而是直接在循环中计算索引并写入 back buffer。这比逐像素调用快得多，因为省去了重复的函数调用开销和边界判断。

描边矩形稍微复杂一些——它不是简单地画四条线，而是分别画上边、下边、左边、右边四条 1 像素宽的线段。需要特别注意角落像素的处理（不能重复写入），以及宽度和高度为 0 的退化情况。我们的实现中，上下两条边通过水平循环 draw_pixel 画，左右两条边通过垂直循环 draw_pixel 画。

Bresenham 直线算法 ​

直线绘制是计算机图形学中最经典的算法之一。Bresenham 算法的核心思想是：只用整数加法和比较来决定下一个像素的位置，完全避免浮点运算和乘法。这在一个没有 FPU 支持的内核环境里尤为重要——虽然 x86_64 有 SSE/XMM 寄存器，但内核代码通常不使用浮点指令。

算法的输入是两个端点 (x0, y0) 和 (x1, y1)，输出是一系列像素坐标。核心逻辑是维护一个误差值 err，每次迭代根据 err 的符号决定在主方向和副方向上各走一步还是只在主方向上走一步。需要注意的是，直线可能走向任意方向（8 个象限），所以 dx 和 dy 都要取绝对值，步进方向用 step_x 和 step_y 的正负号来表示。

与 SerenityOS 的 Painter 类对比：SerenityOS 的 draw_line 支持抗锯齿（anti-aliasing），使用的是 Wu 算法或类似方法。Cinux 目前只实现了基本的 Bresenham，输出的是锯齿状的像素阶梯——对于一个教学内核来说完全够用，后续如果想改善视觉效果可以加抗锯齿。

位图渲染与色彩键透明 ​

draw_bitmap 方法用于渲染一个像素数组到 Canvas 上。这个方法接收一个 uint32_t* 像素数组（row-major 排列），把每个非零像素写入 back buffer。零值像素（0x00000000）被视为完全透明，跳过不写。这是一种最简单的「色彩键」（color key）透明机制——选一个特定颜色值作为透明色。

这种透明方案虽然粗糙（无法实现半透明效果），但在内核级别的图形系统中非常实用。它不需要混合运算，性能开销极低，而且对后续的图标渲染、光标绘制等场景完全够用。

Blit 操作：画布间像素拷贝 ​

blit 方法实现从一个 Canvas 拷贝矩形区域到另一个 Canvas。这是窗口系统的核心操作——每个窗口在自己的离屏 Canvas 上绘制内容，然后窗口管理器通过 blit 把窗口内容合成到屏幕 Canvas 上。

blit 的一个关键设计是目标坐标使用有符号整数（int32_t）。为什么？因为窗口可能被拖动到屏幕边缘之外——比如窗口顶部被拖出屏幕顶部，此时目标 y 坐标是负数。blit 需要正确处理这种「部分可见」的情况：跳过目标 y 坐标为负的行，调整源区域的起始位置来补偿偏移量。

动手实现 ​

Step 1: 实现矩形填充 draw_rect ​

目标: 实现实心矩形填充，直接在 back buffer 上批量写入像素。

设计思路: 双重循环遍历行和列。外层循环从 y 到 y+h，内层循环从 x 到 x+w。循环条件同时检查边界（row < height_, col < width_），避免越界写入。每行的起始索引通过 row * pixels_per_row 计算，然后加上列偏移。

实现约束: 直接操作 back_buf_ 数组，不调用 draw_pixel。使用 pitch_ / 4 作为行步进（pixels_per_row），不是 width_。

踩坑预警: 如果你的矩形坐标 (x, y) 加上宽高 (w, h) 超过了 Canvas 的范围，循环的边界检查必须正确——row < y + h && row < height_，不能只检查 row < height_，否则当 y + h > height_ 时会溢出。同理 col 的检查。

验证: 画一个红色矩形在 (10, 10) 位置，大小 50x30，flip 后检查 framebuffer 上 (10, 10) 和 (35, 25) 的像素值是否为 0x00FF0000。

Step 2: 实现矩形描边 draw_rect_outline ​

目标: 实现 1 像素宽的矩形边框。

设计思路: 分别画四条边——上边、下边、左边、右边。上下边是水平方向的像素行，用循环从 x 到 x+w 调用 draw_pixel。左右边是垂直方向的像素列，从 y 到 y+h 调用 draw_pixel。

实现约束: 底边 y 坐标是 y + h - 1（不是 y + h），右边 x 坐标是 x + w - 1（不是 x + w）。需要检查 h > 0 和 w > 0 来避免下溢（y + h - 1 在 h=0 时会下溢为 uint32_t 最大值）。

验证: 画一个蓝色描边矩形在 (50, 50) 位置，大小 20x10，flip 后检查四角和内部像素。四角应该是蓝色，内部 (51, 51) 应该保持背景色。

Step 3: 实现 Bresenham 直线 draw_line ​

目标: 实现任意方向的直线绘制。

设计思路: 经典的 Bresenham 算法。首先计算 dx 和 dy 的绝对值以及步进方向。然后初始化 err = dx - dy，进入循环。每次迭代画当前像素，然后计算 2*err，根据其与 -dy 和 dx 的关系决定 x 和/或 y 方向的步进。当到达终点 (x1, y1) 时退出循环。

实现约束: 所有计算使用 int32_t（有符号整数），因为 dx 和 dy 可能为负。但最终的像素坐标是 uint32_t，所以步进时需要 static_cast<uint32_t>(static_cast<int32_t>(cx) + step_x) 这种看起来奇怪但正确的写法——先转成有符号做加减，再转回无符号。

踩坑预警: 最容易出错的地方是循环终止条件。必须是 cx == x1 && cy == y1，不能只检查 cx == x1（水平线时 cy 永远不变，但如果是近水平线 cy 可能还没到终点就因为 cx 先到了而退出）。

验证: 画一条从 (0, 0) 到 (30, 30) 的黄色对角线，flip 后检查起点 (0, 0) 和终点 (30, 30) 的像素值。

Step 4: 实现位图渲染 draw_bitmap ​

目标: 把像素数组渲染到 Canvas 上，零值像素视为透明。

设计思路: 双重循环遍历位图的行和列。对于每个像素，从源数组读取颜色值，如果是 0x00000000 则跳过，否则写入 back buffer 的对应位置。行和列都有边界裁剪——超过 Canvas 范围的像素直接不画。

实现约束: 源数组的索引是 row * w + col（w 是位图宽度，不是 pitch）。目标 Canvas 的索引是 (y + row) * pixels_per_row + (x + col)（使用 pitch 计算）。

验证: 构造一个小的 4x4 像素数组（中间两个像素有颜色，周围是 0），渲染到 Canvas 上，检查只有中间两个像素被写入。

Step 5: 实现 blit 画布间拷贝 ​

目标: 从源 Canvas 拷贝矩形区域到当前 Canvas。

设计思路: 遍历 h 行。对于每一行：检查目标行号（dst_y + row）是否在合法范围内——负数行跳过，超过高度则 break。然后处理列方向：如果 dst_x 为负数，需要从源区域跳过相应列数（col_skip = -dst_x），调整有效起始位置。逐像素拷贝，边界检查两端。

实现约束: dst_x 和 dst_y 是 int32_t（有符号），用于支持窗口部分在屏幕外的情况。源区域用 uint32_t 索引。两个 Canvas 可能有不同的 pitch。

踩坑预警: 当 dst_x 为负数时，不能简单地从源的第 |dst_x| 列开始拷贝——你需要同时调整源起始列 sx 和拷贝列数。col_count = w - col_skip，如果 col_count <= 0 说明整个 blit 区域都在屏幕外，直接跳过。

验证: 创建一个小的离屏 Canvas，画一个绿色矩形，然后 blit 到屏幕 Canvas 上。检查屏幕上对应区域的像素值。

Step 6: 实现 GUI Demo 综合测试 ​

目标: 编写一个综合测试，在屏幕上画 10 个随机颜色的矩形和 "Cinux GUI" 标题文字，作为视觉里程碑验证。

设计思路: 这个测试用 LCG（线性同余伪随机数生成器）生成确定性随机数——每次运行结果完全相同，方便调试。先生成深色背景 (0x001A1A2E)，然后循环 10 次生成随机位置、大小和颜色的矩形。最后在顶部居中位置画白色文字。每个矩形的颜色使用 0x40-0xFF 范围的 RGB 分量，保证颜色足够亮、不会和深色背景混在一起。

实现约束: LCG 的初始种子为 12345，乘数 1103515245，增量 12345。next_color() 生成可见颜色——每个通道至少 0x40（64/255），最大 0xFF（255/255）。文字居中的计算是 (canvas_width - text_pixel_width) / 2，其中 text_pixel_width = num_chars * font_width。

踩坑预警: 随机矩形的位置需要保证不超出 Canvas 范围——rng.next() % (canvas.width() - 100) 确保矩形至少距离右边 100 像素。如果不用这个偏移，矩形可能部分超出 Canvas，虽然 draw_rect 有边界检查不会崩溃，但视觉上矩形会被裁剪，影响验证效果。

验证: 运行后 QEMU 窗口显示深色背景 + 10 个彩色矩形 + 白色 "Cinux GUI" 文字。串口输出 [BIG] GUI demo rendered to framebuffer.。

Step 7: 设计 Canvas 的 CMake 编译条件 ​

目标: 配置 CMakeLists.txt，使 Canvas 和 GUI 相关代码只在启用 CINUX_GUI 时编译。

设计思路: 在顶层 CMakeLists.txt 中添加 option(CINUX_GUI "Enable GUI subsystem" OFF)。当启用时，添加编译定义 -DCINUX_GUI，并包含 kernel/gui/CMakeLists.txt（GUI 子系统的源文件列表）。Canvas.cpp 和 GUI init 代码通过 #ifdef CINUX_GUI 条件编译，在非 GUI 模式下不会编译也不会链接。

实现约束: kernel/drivers/canvas.cpp 应该只在 CINUX_GUI 启用时加入编译列表。kernel/test/test_canvas.cpp 中的 GUI 测试也用 #ifdef CINUX_GUI 包裹。PIT 的 tick callback 部分同样用这个宏控制。

验证: 分别用 -DCINUX_GUI=ON 和 -DCINUX_GUI=OFF 编译，确认两种模式都能正常编译通过。OFF 模式下运行不应该出现任何 GUI 相关的串口输出。

调试技巧 ​

直线画歪或缺失像素：检查 Bresenham 的 err 更新逻辑——应该是 err -= dy 和 err += dx，不是 err -= dx 和 err += dy。另外确认 step_x 和 step_y 的正负号与 dx/dy 的方向一致。

blit 后画面位置不对：打印 dst_x、dst_y、col_skip 的值到串口，确认坐标计算是否正确。特别是 dst_x 为负数时的逻辑——这是最容易出 bug 的地方。

draw_bitmap 透明度不生效：检查你的色彩键判断是否是 color == 0x00000000，不是 color == 0（虽然这两个值相同，但意图不同，用前者更清晰）。确认源像素数组的排列是 row-major 的。

性能考虑 ​

图元绘制的性能在 GUI 系统中非常重要——尤其是合成阶段，窗口管理器需要每帧把所有窗口 blit 到屏幕上。对于 1024x768 的 Canvas，以下是一些关键的性能数据点：

一个 100x50 的矩形填充需要 5000 次数组写入，在 1GHz CPU 上大约需要 5-10 微秒。一条 1000 像素的对角线大约需要 1000 次 draw_pixel 调用（每次有边界检查和索引计算），约 2-5 微秒。一帧完整的 flip（3MB 拷贝）大约需要 1-2 毫秒。所以以 100 Hz PIT 频率刷新（10ms 间隔），flip 占用约 10-20% 的帧时间，留给绘制操作还有 8-9ms，足够绘制几十个窗口和大量图元。
