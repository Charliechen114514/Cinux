031-1: Pipe IPC 机制 — 从环形缓冲区到系统调用 ​

导语 ​

到目前为止，我们的内核已经有了一个能跑 shell 的用户态环境，也有了一个基本的窗口管理器，但这两个世界是完全隔离的——shell 在串口终端里输出文字，GUI 在屏幕上画窗口，彼此没有交集。接下来我们要做的事情就是建立一条通道，让 shell 进程的输出能够流到 GUI 终端窗口里，而用户的键盘输入也能从终端窗口流回 shell 进程。这条通道就是 pipe——Unix 世界里最经典、最简洁的进程间通信方式。

完成这一章后，我们将在内核中拥有一套完整的 pipe IPC 机制，包括内核态的环形缓冲区数据结构、VFS 层的适配器、以及一个 pipe() 系统调用，用户态程序可以通过它创建一对读写文件描述符。整个数据流的骨架将是：键盘输入 -> stdin 管道 -> shell -> stdout 管道 -> 终端显示，环环相扣，每一环都是我们自己手搓的。

前置要求：你需要已经理解前一个 tag（030）中的 VFS InodeOps 接口和文件描述符表机制，因为管道的 VFS 适配直接依赖这些基础设施。

概念精讲 ​

管道（Pipe）是什么 ​

如果你在 Linux 终端里敲过 ls | grep foo，那你已经用过管道了。竖线 | 就是让前一个命令的输出变成后一个命令的输入——ls 的 stdout 连到 grep 的 stdin，中间这条通道就是 pipe。在内核层面，这靠的是一对文件描述符——一个只读端、一个只写端——中间连着一个内核缓冲区。写入端往缓冲区里塞字节，读取端从缓冲区里取字节，先进先出，就像一根水管一样。

POSIX 标准规定 pipe() 系统调用返回两个文件描述符：pipefd[0] 是读端，pipefd[1] 是写端。读端只能读不能写，写端只能写不能读。这个单向约束非常关键——如果你试图在读端写入数据，内核会返回错误。如果写端关闭了，读端会收到 EOF（read 返回 0）；如果读端关闭了，写端的 write 会返回 EPIPE 错误。这种语义在 Unix 世界里已经运行了五十多年，稳定可靠。

OSDev Wiki 对管道的定义很简洁：管道是一个单向字节流，通常用环形缓冲区实现，缓冲区位于内核空间，大小约 4KB。如果缓冲区满了（消费者太慢），生产者进程会被内核挂起等待。当消费者读取数据腾出空间后，内核唤醒生产者继续写入。

环形缓冲区 ​

管道的底层存储通常是一个环形缓冲区（ring buffer，也叫 circular buffer）。你可以把它想象成一个首尾相连的数组：我们维护三个变量——head（下一个要读取的位置）、tail（下一个要写入的位置）、count（当前缓冲区中有多少字节）。写入时数据放到 tail 位置然后 tail 前进，读取时从 head 位置取然后 head 前进。当 tail 走到数组末尾时，它回绕到数组开头——这就是"环形"的含义。

       v             v
[_][_][_][A][B][C][D][_][_][_]
              count = 4

这个设计的核心优势在于：不需要移动数据，只需要移动指针。不过有一点需要特别注意——当写入的数据跨越数组末尾时（tail 到了数组右端但还没写完），我们需要分两段写入：先写到数组末尾，再回绕到数组开头继续写。读取同理。这就是所谓的"两段拷贝"（two-segment copy），是环形缓冲区实现中最容易出 bug 的地方——如果你在计算 second 段长度时差了一个 off-by-one，数据就会错位。

阻塞语义 ​

当缓冲区满了（写端无法继续写入）或者空了（读端无法继续读取）时怎么办？在 Unix 系统里，进程会被阻塞（sleep），等到另一端腾出空间或写入新数据后被唤醒。但我们目前还没有实现基于调度器的阻塞机制，所以 Cinux 采用了一种折中方案——spin-wait：在一个循环里反复检查条件，同时调用 hlt 指令让 CPU 暂时进入低功耗状态，让出总线给其他硬件操作。当然我们不会无限等下去，最多等 PIPE_SPIN_WAIT_ITERS（一百万次）就放弃。

另外我们还提供了非阻塞版本的 try_read 和 try_write，它们在缓冲区满或空时立即返回 0，不做任何等待。这个非阻塞版本后面会被终端组件的 poll_output 使用——PIT 中断回调里绝对不能阻塞等待，否则整个系统的时钟都停了。

动手实现 ​

Step 1: 设计并实现 Pipe 数据结构 ​

目标: 创建一个 Pipe 类，封装 4096 字节的环形缓冲区，提供阻塞和非阻塞的读写接口。

设计思路: Pipe 是纯粹的内核态数据结构，不直接暴露给用户态。它的职责就是管理缓冲区和提供线程安全的读写。我们需要一把自旋锁来保护所有可变状态（head、tail、count、读写端开关），以及两个布尔标记（reader_open、writer_open）来跟踪两端的存活状态。整个类禁用拷贝构造和拷贝赋值——管道内含自旋锁和原始缓冲区数组，拷贝语义没有意义。

实现约束: 构造函数初始化 head、tail、count 为 0，两个 open 标记为 true。write 方法要处理缓冲区满时的 spin-wait（释放锁、启用中断、hlt 循环检查）、跨数组末尾的两段写入、以及对端已关闭的检测。read 方法的逻辑是对称的——缓冲区空时 spin-wait 等待数据、两段读取、写端关闭且缓冲区耗尽时返回 0 表示 EOF。try_read 和 try_write 与之类似但不做 spin-wait，直接在获取锁后检查条件、有数据就操作、没有就返回 0。close_reader 和 close_writer 只需要在锁保护下设置对应标记为 false。

踩坑预警: 在 spin-wait 循环里一定要释放锁并启用中断！这是最容易出错的地方。如果持有锁且关中断去做 hlt，PIT 中断处理程序就无法运行，而 PIT 回调恰恰会调用 try_read 来消费管道数据——死锁就发生了。正确的做法是：保存原始中断标志（irq_save），在每次 hlt 之前启用中断（irq_enable），在检查条件之前关闭中断重新获取锁（irq_disable + lock_.acquire），检查完毕后释放锁。这样 PIT 中断有机会在 hlt 期间触发，try_read 能执行并消费数据，缓冲区就不满了。

验证: 编写单元测试验证以下场景——基本读写（写入 N 字节、读取 N 字节、验证内容一致）、缓冲区满时 write 阻塞直到有空间、缓冲区空时 read 阻塞直到有数据、跨数组末尾的两段写入（tail 靠近 4096 时写入大量数据）、对端关闭后 read 返回 EOF（0）和 write 返回错误（-1）、try_read/try_write 在空/满时不阻塞立即返回 0。

Step 2: 实现 VFS 适配器 PipeReadOps 和 PipeWriteOps ​

目标: 创建两个 InodeOps 子类，将 Pipe 的读写接口桥接到 VFS 层。

设计思路: Unix 管道的核心设计是"两个文件描述符共享同一个管道但方向相反"。在 Cinux 的 VFS 架构里，每个打开的文件都关联一个 Inode，Inode 上挂着一组 InodeOps 虚函数。所以我们的策略很自然——创建两套 InodeOps，一套只转发读请求到 Pipe::read，一套只转发写请求到 Pipe::write。这样管道就无缝融入了 VFS 的读写路径，上层的 sys_read/sys_write 不需要知道 fd 底层是管道还是普通文件。

实现约束: PipeReadOps 的 read 方法调用 Pipe::read，write 方法直接返回 -1。PipeWriteOps 的 read 方法返回 -1，write 方法调用 Pipe::write。InodeOps 的 read/write 签名里有一个 offset 参数，但管道是不可寻址的字节流——你不可能对管道做 fseek——这个参数直接忽略即可。两个适配器都需要在构造时保存指向底层 Pipe 的指针，所有方法通过这个指针委托调用。

验证: 通过后续的 sys_pipe 系统调用测试和终端集成测试间接验证。单元测试可以创建 Pipe 和 PipeReadOps，调用 read 方法验证数据流转。

Step 3: 实现 sys_pipe 系统调用 ​

目标: 暴露 pipe() 系统调用给用户态，创建一对读写文件描述符。

设计思路: 系统调用的职责是把内核内部的 Pipe 对象和 VFS InodeOps 适配器组装起来，然后通过 FDTable 暴露给用户态。整个流程可以分解为六步——第一步在内核堆上 new 一个 Pipe 对象；第二步 new PipeReadOps 和 PipeWriteOps，分别绑定到这个 Pipe；第三步创建两个 Inode 对象，分别挂载对应的 ops；第四步在 FDTable 中分配两个 fd（read_fd 和 write_fd）；第五步把 fd 编号写入用户态的 int 数组（pipefd[0] 和 pipefd[1]）；第六步返回 0 表示成功。

实现约束: 系统调用接收的第一个参数是用户态虚拟地址，指向一个两个元素的 int 数组。我们需要先验证这个地址是否合法——非空、非内核空间、符合 x86_64 canonical address 规则。创建失败时需要按逆序清理所有已分配的资源，先 delete 后 new 的对象，先 close 后 alloc 的 fd。fd[0] 必须是读端，fd[1] 必须是写端——这是 POSIX 规范的硬性要求。

踩坑预警: 地址验证要检查 x86_64 的 canonical address 规则：bit 47 必须等于 bits 48-63。bit 47 为 1 表示内核空间，必须拒绝。如果跳过这个检查，恶意的用户程序可以传入一个内核地址让内核往自己的数据结构里写 fd 编号——这是一个典型的权限提升漏洞。另外，FDTable::alloc 从 fd=3 开始分配（0/1/2 保留给 stdin/stdout/stderr），所以 pipe() 返回的 fd 编号通常是 3 和 4，这是正常的。

验证: 编写用户态测试程序调用 pipe()，验证返回值为 0、两个 fd 编号不同且非负、向写端写入数据能从读端读出且内容一致、关闭写端后读端 read 返回 0（EOF）、关闭读端后写端 write 返回 -1。

构建与运行 ​

完成以上三步后，整个 pipe 子系统就可以编译链接进内核了。确保 kernel/ipc/pipe.cpp、kernel/ipc/pipe_ops.cpp、kernel/syscall/sys_pipe.cpp 都被加入了 CMakeLists.txt，同时 syscall 分发表中注册了新的 pipe 系统调用号。Pipe 模块本身不依赖 GUI，即使在没有图形界面的配置下也能工作。

构建命令：

bash
cmake --build build
1

运行 QEMU 后，在内核初始化日志中应该能看到 pipe 相关的初始化信息。如果你写了一个简单的用户态测试程序调用 pipe() 并打印返回的 fd，应该在串口输出中看到类似 pipe returned: fd[0]=3, fd[1]=4 的结果。也可以尝试向写端写入一段文字，然后从读端读取并打印，验证数据传输的正确性。

调试技巧 ​

pipe 写入后读取不到数据：最常见的原因是 read/write 方向搞反了。记住 fd[0] 是读端、fd[1] 是写端，不要搞混。另一个可能是 try_read 在空缓冲区上返回 0 被误判为错误——0 在管道语境里是 EOF，但 try_read 在空缓冲区上也返回 0 来表示"暂时没有数据"，需要结合 writer_alive() 区分到底是真的 EOF 还是暂时没有数据。

死锁：如果在持有 pipe 自旋锁的情况下调用 hlt 且中断被关闭，PIT 回调无法执行，而死锁就发生了。检查 write/read 中的 spin-wait 循环是否正确地释放了锁并启用了中断。一个可靠的判断方法是：如果你在代码中看到 lock_.acquire() 后面跟着 hlt() 但中间没有 lock_.release() 和 irq_enable()，那一定有问题。

内核崩溃在 sys_pipe：检查用户态传入的地址是否合法，特别是 NULL 指针和未映射的页面。sys_pipe 会直接把 fd 编号写入用户态内存，如果地址无效会触发 page fault。串口输出中的崩溃栈信息会告诉你 fault 地址，可以据此判断是用户传入的地址有问题。
