028-2 Hands-on: ext2 挂载与块 I/O ​

导语 ​

上一章我们用 dumpe2fs 和 hexdump 把 ext2 的磁盘布局翻了个底朝天——Superblock、BGDT、Inode、Directory Entry 的字段含义和布局规则都搞清楚了。现在我们要把这些知识变成可以执行的代码：实现 ext2 的 mount() 函数，让内核从 SATA 磁盘上读出 Superblock，验证它的合法性，加载块组描述符表，最终读取根目录的 inode。完成本章后，你会在串口输出里看到 ext2 超级块的关键参数——这意味着内核已经成功理解了磁盘上的文件系统结构。

这一章还涉及一个底层的工程问题：块 I/O。AHCI 驱动（tag 025）使用 DMA 方式读写磁盘，需要一个物理连续的缓冲区。我们会分配一个物理页面作为 DMA buffer，把它映射到内核虚拟地址空间，所有 ext2 的块读取都通过这个单一缓冲区完成。

知识前置：028-1（ext2 磁盘布局）、025（AHCI 驱动）、015/016（PMM/VMM）、027（VFS FileSystem 抽象基类）。

概念精讲 ​

Superblock 的读取——从字节偏移到扇区编号 ​

上一章我们知道 Superblock 在字节偏移 1024 处，占 1024 字节。但 AHCI 驱动只认扇区（LBA），不认"字节偏移"。所以我们需要做一个简单的换算：字节偏移 1024 = LBA 2（每个扇区 512 字节，1024/512=2），读取长度 1024 字节 = 2 个扇区。也就是说，调用 AHCI 的 read 函数时，参数是 LBA=2、count=2——一次读出完整的 Superblock。

这里有一个细节值得注意：我们是用 AHCI 的扇区级接口读的，不是 ext2 的块级接口。原因很简单——在 Superblock 解析之前，我们根本不知道块大小是多少，所以没法用块级抽象。Superblock 的读取是整个挂载流程中唯一一次"硬编码"扇区参数的操作，之后的读取都可以走 read_block。

read_block —— 从块号到物理读取 ​

有了 Superblock 之后，我们知道了块大小（假设是 1024），就可以实现 read_block：给定一个块号，换算成 LBA（块号 * 每块扇区数），然后用 AHCI read 把数据读到 DMA buffer 里。这个函数是 ext2 驱动的 I/O 核心——所有后续的 BGDT 读取、inode 读取、文件数据读取，最终都通过 read_block 完成。

一个值得思考的设计选择是：read_block 只使用一个全局 DMA buffer（一个物理页，4KB）。这意味着每次 read_block 都会覆盖上一次的数据——调用方必须在 read_block 之后、下一次 read_block 之前把数据拷贝走。对于单核内核来说这完全没问题，因为我们没有任何并发。但对于生产级文件系统来说，这就不够了——Linux 使用 buffer_head 子系统管理成百上千个缓存块，而我们的"缓存"就是一个 4KB 的窗口。

这个设计选择有一个直接的后果：在读取间接块时需要特别小心。当我们需要通过 i_block[12] 间接块读取文件数据时，过程是先 read_block(间接块号) 把间接块读进 DMA buffer，从中取出目标数据块号，然后再 read_block(数据块号) 把实际数据读进来。问题在于，第二次 read_block 会覆盖第一次读到的间接块数据——但这没关系，因为我们已经把需要的信息（目标数据块号）提取出来了。只要你不试图在第二次 read_block 之后再去 DMA buffer 里读间接块的数据，就不会出问题。

DMA buffer 的分配与映射 ​

DMA buffer 需要满足两个条件：物理地址连续（因为 AHCI 控制器通过 DMA 直接访问物理内存），以及在内核虚拟地址空间中可访问（因为 CPU 需要读取 DMA buffer 的内容）。分配过程分两步：先用 PMM 的 alloc_page 分配一个 4KB 物理页，然后用 VMM 的 map 函数把这个物理页映射到一个预留给 ext2 DMA 的虚拟地址上。这个虚拟地址在内存布局中是固定预留的（KMEM_EXT2_DMA_BASE），不会和内核的其他区域冲突。

映射完成后，我们得到了一对地址——dma_buf_phys（物理地址，传给 AHCI 的 read/write）和 dma_buf_virt（虚拟地址，CPU 访问数据用）。在整个 ext2 驱动的生命周期中，这对地址不会变。

这里有一个你可能关心的问题：为什么不直接用 kmalloc 分配 DMA buffer？答案是 DMA 的物理连续性要求。kmalloc 虽然能分配内核虚拟地址空间中的内存，但在我们的内核中，kmalloc 底层使用的堆分配器不保证物理连续——它可能把多个不连续的物理页拼接成一个虚拟连续区域。而 AHCI 控制器的 DMA 引擎只认物理地址，它不知道页表的存在。所以我们必须用 PMM 的 alloc_page 直接分配一个物理连续的页，然后手动映射到虚拟地址空间。这个模式和我们在 AHCI 驱动本身的初始化中做的事情完全一样。

挂载流程的完整步骤 ​

mount() 函数做的事情可以归纳为六步，每一步都依赖上一步的结果：第一步，分配并映射 DMA buffer；第二步，用 AHCI 从 LBA 2 读 2 个扇区，获取 Superblock 原始字节；第三步，验证 magic number 为 0xEF53——如果不匹配，直接返回 false，挂载失败；第四步，从 s_log_block_size 计算实际块大小，同时计算每块扇区数、每组 inode 数、组数等派生参数；第五步，从 BGDT 起始块开始逐块读取，把所有块组描述符拷贝到内存中的 bgdt 数组；第六步，读取 inode 2（根目录），把它放入 inode cache 的第 0 号槽位，建立 VFS Inode。

这六步走完之后，Ext2 对象的 mounted_ 标志被设为 true，后续的 lookup 和文件读取操作就可以正常工作了。整个 mount 过程涉及的磁盘 I/O 并不多——一次 Superblock（2 扇区）、一到几次 BGDT（取决于块组数量）、一次根 inode（2 扇区），加起来可能不超过十次扇区读取。在 QEMU 中这些操作几乎是瞬间完成的，但在真实硬件上你会感受到机械硬盘寻道的延迟——这也是 ext2 块组设计想要优化的问题。

Inode 编号的寻址公式 ​

读取根 inode（inode 2）时用到的寻址公式是整个 ext2 驱动最核心的数学之一，后续每次 read_disk_inode 都会复用。给定 inode 编号 ino：首先计算它所属的块组 group = (ino - 1) / inodes_per_group，然后从 BGDT 中取出该组的 bg_inode_table（inode 表的起始块号），再计算组内索引 index = (ino - 1) % inodes_per_group，进而算出字节偏移 byte_offset = index * inode_size，最终得到目标块号 target_block = bg_inode_table + byte_offset / block_size 和块内偏移 within_block_offset = byte_offset % block_size。

整个计算链条中最容易出错的地方是"先减 1"——inode 编号从 1 开始，所有下标计算都要先把 ino 减 1 变成 0-based 索引。如果你忘了这一步，inode 2 会被当成 group 0 的第 2 个 inode（实际应该是第 1 个），所有后续的路径解析都会偏移一格。

我们用一个具体的例子来验证这个公式。假设 ino=2（根目录）、inodes_per_group=2048、inode_size=128、block_size=1024，且 BGDT[0] 中 bg_inode_table=5。那么 group = (2-1)/2048 = 0，index = (2-1)%2048 = 1，byte_offset = 1*128 = 128，target_block = 5 + 128/1024 = 5 + 0 = 5，within_block_offset = 128 % 1024 = 128。也就是说，inode 2 的数据在 block 5 的第 128 字节处——紧跟着 inode 1（在 block 5 的第 0 字节处）。你可以用 hexdump 在宿主机上验证这一点。

再来看一个稍大的例子——假设 ino=13，且 inodes_per_group=2048、inode_size=128、block_size=1024、bg_inode_table=5。那么 group = (13-1)/2048 = 0，index = (13-1)%2048 = 12，byte_offset = 12*128 = 1536，target_block = 5 + 1536/1024 = 5 + 1 = 6，within_block_offset = 1536 % 1024 = 512。所以 inode 13 在 block 6 的第 512 字节处——它已经不在第一个 inode table 块里了，跨到了第二个块。这个例子说明了为什么 inode table 需要占用多个块（对于 1024 字节块，每个块只能放 8 个 inode——1024/128=8），以及为什么我们的公式必须正确处理跨块的情况。

动手实现 ​

Step 1: 定义 ext2 磁盘数据结构头文件 ​

目标: 创建 kernel/fs/ext2_types.hpp，定义 Ext2Superblock、Ext2BlockGroupDescriptor、Ext2Inode、Ext2DirEntry 四个 packed 结构体以及相关常量。

设计思路: 所有的磁盘结构体都用 [[gnu::packed]] 属性标记，因为磁盘上的布局不允许编译器插入填充字节。用 static_assert 确保每个结构体的大小与规范一致——Ext2Superblock 必须是 1024 字节、Ext2BlockGroupDescriptor 32 字节、Ext2Inode 128 字节。常量定义包括 EXT2_SUPER_MAGIC（0xEF53）、EXT2_DIRECT_BLOCKS（12）、EXT2_INODE_CACHE_SIZE（64）、EXT2_SUPERBLOCK_OFFSET（1024）、EXT2_SECTOR_SIZE（512）等。这些常量在后续的 mount 和 read_block 中都会大量使用。

实现约束: 这些结构体只定义磁盘上的布局，不包含任何行为。Ext2CachedInode 是一个辅助结构——把磁盘 inode 和 VFS inode 绑在一起，还有 ino 编号和 in_use 标志。

踩坑预警: Ext2Superblock 的字段非常多，但 Cinux 不需要用到全部字段。不过为了保证结构体大小正确（1024 字节），后面的字段必须全部列出来（包括大量的 reserved/padding 字段）。一个常见错误是少写了末尾的 padding 数组，导致 sizeof 不是 1024。

验证: 编译通过且 static_assert 全部成立：

bash
cd build && cmake .. && make big_kernel_common 2>&1 | grep -E "(error|static_assert)" | head -10
1

Step 2: 定义 Ext2 驱动类 ​

目标: 创建 kernel/fs/ext2.hpp，声明 Ext2 类（继承 FileSystem）。InodeOps 使用 ext2_file_ops 和 ext2_dir_ops 两个静态函数指针表。

设计思路: Ext2 类持有 AHCI 引用和端口号、Superblock 缓存、BGDT 数组、inode cache 数组、DMA buffer 地址对、以及各种派生参数（block_size、sectors_per_block、group_count 等）。InodeOps 使用静态函数指针表（ext2_file_ops 和 ext2_dir_ops），回调函数通过全局指针 g_ext2_instance 访问 Ext2 对象的 read_block、read_disk_inode 等底层方法。

实现约束: BGDT 数组大小上限为 128（EXT2_MAX_GROUPS），足以覆盖约 8GB 的磁盘（128 组 * 每组 8192 块 * 4096 字节/块）。inode cache 固定 64 条目，使用简单的驱逐策略（cache 满时用 ino 做哈希选择淘汰槽位，slot 0 永远留给根 inode）。

验证: 编译通过。

Step 3: 实现 DMA buffer 分配 ​

目标: 在 kernel/fs/ext2.cpp 中实现 ensure_dma_buffer() 方法。

设计思路: 第一次调用时分配物理页、映射到预留虚拟地址、清零。后续调用直接返回 true（dma_ready_ 标志防止重复分配）。如果 PMM 分配失败或 VMM 映射失败，打印错误信息并返回 false。

实现约束: 虚拟地址使用内存布局中预留的 KMEM_EXT2_DMA_BASE。映射标志为 PRESENT | WRITABLE（内核数据，不需要用户态访问）。分配完成后要把整个 buffer 清零——虽然 AHCI read 会覆盖这些字节，但清零可以防止 mount 过程中的调试打印意外输出垃圾数据。

踩坑预警: 确保在内核启动序列中，PMM 和 VMM 已经初始化完毕之后再调用 ext2 的 mount()。如果 PMM 还没初始化就调 alloc_page，你会收获一个非常漂亮的 kernel panic。另外，如果你之前的 tag 中已经在 KMEM_EXT2_DMA_BASE 这个虚拟地址上映射了别的东西（比如调试用的临时映射），ensure_dma_buffer 会尝试重新映射同一个虚拟地址——VMM 的 map 函数需要处理这种情况（要么报错，要么先 unmap 再重新 map）。

验证: 编译通过。在 mount 的 kprintf 中加入 DMA buffer 物理地址和虚拟地址的打印，确认两者都是非零值。

Step 4: 实现 read_block 和 write_block ​

目标: 将块号换算为 LBA，调用 AHCI 的 read/write 方法。

设计思路: LBA = block_num * sectors_per_block_，sector count = sectors_per_block_。read 把数据读到 DMA buffer（物理地址），write 从 DMA buffer（物理地址）写出。调用方通过 dma_buf_virt() 获取虚拟地址来访问数据。

实现约束: 每次 read_block 会覆盖 DMA buffer 中上一次的数据——这是单缓冲区设计的固有行为。如果调用方需要保留之前读到的数据，必须在下一次 read_block 之前拷贝走。read_block 的第一步总是 ensure_dma_buffer()——这是一个幂等操作，buffer 已经分配时直接返回 true，不会有额外开销。

验证: 编译通过。你可以在 read_block 中加入一个调试打印，显示每次读取的 block_num 和换算后的 LBA，方便追踪 I/O 操作。但在正式使用时记得去掉——否则串口输出会非常啰嗦。

Step 5: 实现 mount() 函数 ​

目标: 按照上面描述的六步流程实现完整的挂载逻辑。

设计思路: 先 ensure_dma_buffer，然后读 Superblock（硬编码 LBA 2、2 扇区），验证 magic，计算派生参数，读 BGDT，最后读根 inode。每一步都有对应的串口打印，方便调试。整个 mount 过程是同步的——不存在异步回调或中断驱动的 I/O 等待，read_block 调用返回时数据已经在 DMA buffer 中了（因为 AHCI 的 read 在我们的实现中是同步阻塞的）。

实现约束: BGDT 的起始块号取决于块大小——1024 字节时从 block 2 开始（block 0 是 boot block，block 1 是 Superblock），更大的块从 block 1 开始（Superblock 在 block 0 的偏移 1024 处）。inode_size 在 revision 0 时固定 128 字节，revision >= 1 时取 s_inode_size 字段值。group_count 通过向上取整计算：(s_blocks_count + blocks_per_group - 1) / blocks_per_group。

踩坑预警: 有两个地方特别容易出错。第一，BGDT 可能跨越多个块——如果块组数很多，一个块装不下所有描述符（1024 字节块只能装 32 个描述符），需要循环读取多个块并拼接。虽然我们的 8MB 测试镜像只有 1 个块组，但代码必须处理多块的情况——否则换成更大的磁盘就会挂掉。第二，根 inode 必须放入 cache slot 0——这个槽位被 root_inode_ 引用，后续 lookup("/") 时直接返回这个引用，不再走 cache 查找。如果你把根 inode 放到了别的 slot，lookup("/") 就会返回一个未初始化的 Inode，接下来的操作必定崩溃。

Step 7: 在内核初始化中调用 ext2 mount ​

目标: 在内核的启动序列中，在 AHCI 初始化完成之后，调用 ext2 的 mount() 函数。

设计思路: 在 kernel/main.cpp 或你使用的初始化入口中，找到 AHCI 初始化之后的位置。创建一个全局的 Ext2 对象（或使用 new 在堆上分配），传入 AHCI 引用和端口号 1，然后调用 mount()。如果挂载成功，打印一条确认信息；如果失败，打印警告但不 panic——因为我们还可能有 Ramdisk 作为后备。

实现约束: Ext2 对象的生命周期必须覆盖整个内核运行期间——不能用栈变量。最安全的做法是把它声明为全局静态对象，或者在堆上 new 之后不 delete。Ext2 的构造函数除了初始化成员变量之外不做任何 I/O 操作，真正的磁盘读取都在 mount() 中发生，所以不用担心构造函数在初始化早期被调用的问题——只要 mount() 在 AHCI 就绪之后调用就行。

验证: 在 QEMU 中运行，确认串口输出包含 [EXT2] 相关的挂载信息。如果你看到 [EXT2] Root inode: size=1024 mode=0x41ed，恭喜你——根目录的 inode 已经成功读取了，mode=0x41ed 表示这是一个目录（0x4000）加上 rwxr-xr-x 权限（0x1ed）。

调试技巧 ​

mount() 返回 false，没有 Superblock 输出: 检查 AHCI 驱动是否正确初始化。在 tag 025 中，AHCI 探测和端口初始化必须在 ext2 mount 之前完成。如果你在串口中看到 [EXT2] Failed to read superblock，说明 AHCI read 调用失败了——可能是端口号不对，也可能是 QEMU 的 AHCI 设备没有正确配置。

magic 不匹配（看到 0x0000 或其他值）: 最大的嫌疑是镜像文件路径或 QEMU 的 -drive 参数配置有误。确认 ext2.img 确实是 ext2 格式（用 file ext2.img 查看），并且 QEMU 的 AHCI 设备和端口对应关系正确。

group_count 特别大或特别小: 检查 blocks_per_group 和 blocks_count 的值是否合理。8MB 镜像、1024 字节块大小下，block 数约为 8192，blocks_per_group 为 8192，group_count 应该是 1。如果 group_count 是几百，说明某处读取了错误的字段值。可以在 mount 的 kprintf 输出中对照查看 blocks_count 和 blocks_per_group 的实际数值。

BGDT 读取失败: 确认 BGDT 起始块号计算正确——对于 1024 字节块，它应该在 block 2。用 hexdump -C ext2.img -s 2048 -n 64 验证 block 2 的内容非全零。BGDT 读取失败的另一个常见原因是 sectors_per_block_ 计算有误——如果 block_size 在这一步还没被正确设置（比如你把 s_log_block_size 的值直接当 block_size 用了），sectors_per_block_ 可能是 0，导致 LBA 计算全部错误。

根 inode 读取失败但 BGDT 加载正常: 确认 read_disk_inode 的寻址公式中每一步的值是否正确。在 read_disk_inode 中加入调试打印：输出 group、inode_table_block、index_in_group、target_block、within_block_offset 的值，然后和你在宿主机上用 dumpe2fs + hexdump 手动计算的结果对比。

根 inode 的 mode 不是 0x41ED: 0x41ED 表示"目录 + rwxr-xr-x 权限"。如果你看到 0x8000 或其他值，说明你读到了错误的 inode——很可能是寻址公式中忘了减 1，导致读到了 inode 1（bad blocks inode）而不是 inode 2（根目录）。

本章小结 ​

概念要点
Superblock 读取LBA 2，2 扇区，硬编码扇区参数
magic 验证必须是 0xEF53，否则挂载失败
block_size1024 << s_log_block_size，不是直接值
DMA buffer一个物理页（4KB），PMM 分配 + VMM 映射
read_block块号 * 每块扇区数 = LBA，数据读到 DMA buffer
BGDT 加载从 Superblock 之后的块开始，可能跨多个块
根 inodeinode 2，放入 cache slot 0
ino 寻址group=(ino-1)/ipg, index=(ino-1)%ipg, 偏移=index*inode_size
初始化顺序PMM → VMM → 堆 → AHCI → ext2 mount → vfs_mount_add
全局 Ext2 对象必须是全局静态或堆分配，不能是栈变量
