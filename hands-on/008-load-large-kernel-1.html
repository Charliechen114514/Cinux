008-1 Hands-on: ATA PIO 磁盘驱动与 Freestanding 工具函数 ​

导语 ​

上一章（007）我们把中断系统搞定了，mini kernel 终于能在异常发生时体面地报告错误信息。但说实话，这个 mini kernel 还是个"一次性"的东西——Bootloader 把它从磁盘加载到内存里，它跑完就停在 cli; hlt 的死循环里，什么都干不了。问题出在哪呢？Mini kernel 受限于 Bootloader 的加载能力，只能被放在 0x20000 开始的那块区域里，最大 416KB。这个空间塞不下一个真正有用的内核。

所以我们需要一个"加载器中的加载器"：mini kernel 自己充当一个迷你的 bootloader，从磁盘读取一个更大的 ELF 格式的内核，把它搬到正确的内存位置，然后跳过去执行。这一章我们先搞定第一步——让 mini kernel 能从磁盘读数据。

完成本章后，我们会看到 mini kernel 成功初始化 ATA 磁盘控制器，从磁盘读取 MBR 扇区并验证引导签名 0xAA55，证明磁盘读取管线已经就位。

本章的前置知识是上一章（007）的中断系统，因为 ATA PIO 驱动涉及大量的 I/O 端口轮询操作。如果中断处理出了问题，整个读取流程可能会被意外打断。

概念精讲 ​

为什么不能用 BIOS 中断了？ ​

如果你和我一样是从实模式开始折腾的，你可能会问：读磁盘不是可以用 INT 13h 吗？确实，Bootloader 的 Stage2 就是用 BIOS 中断读磁盘的。但问题在于，我们现在已经处于 long mode 下了——64 位保护模式里根本没有 BIOS 中断可用。所有实模式服务在切换模式的那一刻就已经灰飞烟灭了。

所以我们只能直接和硬件对话。ATA（Advanced Technology Attachment）是 x86 平台上最经典的硬盘接口协议，即使现在的主流 SATA 硬盘在软件层面也保持了 ATA 兼容性。QEMU 模拟的硬盘就是一个标准的 ATA 设备，我们可以通过 I/O 端口直接发送命令来读取扇区。

PIO 是什么？和 DMA 有什么区别？ ​

PIO（Programmed I/O）是 ATA 传输方式中最简单的一种——CPU 主动轮询数据端口，一个字一个字地把数据从磁盘控制器读进来。没有 DMA（Direct Memory Access，直接内存访问），没有中断通知，就是纯粹的"发了命令就等着，数据准备好了就读"。效率当然不高，但在 boot 阶段没有其他进程在运行，CPU 全部时间都是我们的，所以简单可靠才是第一位的。

DMA 的方式是让磁盘控制器直接把数据写到内存里，CPU 只需要等待一个中断通知就能拿到数据。这比 PIO 高效得多，但实现复杂度也高得多——需要配置 DMA 引擎、处理物理地址映射、管理 buffer descriptor。对于 bootloader 级别的磁盘读取来说，这完全是杀鸡用牛刀。

PIO 读取流程（简化版）：

CPU → 发送 READ SECTORS 命令到 I/O 端口 0x1F7
     ↓
CPU → 轮询状态端口 0x1F7，等待 DRQ 位变 1（数据就绪）
     ↓
CPU → 从数据端口 0x1F0 连续读取 256 个 16 位字 = 512 字节 = 1 个扇区
     ↓
重复以上步骤直到所有扇区读完

ATA 控制器的 I/O 端口布局 ​

x86 平台上，主 ATA 控制器使用 0x1F0 到 0x1F7 这组 I/O 端口，外加一个控制端口 0x3F6。每个端口的功能是固定的，不能改也不能猜——这是 ATA 规范里写死的。

理解这些端口非常重要，因为 ATA PIO 编程本质上就是对这组端口进行读写的"仪式"。主控制器基址是 0x1F0，各寄存器的偏移如下：数据寄存器偏移 0（16 位宽，读取扇区数据就从这个端口一个字一个字地取）；错误寄存器偏移 1（只读，出错时读取错误码）；扇区计数寄存器偏移 2（告诉控制器要读几个扇区）；LBA 低/中/高字节分别在偏移 3/4/5（定位要读的扇区位置）；驱动器选择寄存器偏移 6（选择主/从盘以及 LBA 模式）；状态寄存器偏移 7（读操作时返回状态，写操作时接收命令）——这是最重要的端口。

其中 LBA（Logical Block Addressing）是现代磁盘的寻址方式，你可以把它理解为"扇区编号"。LBA 0 就是第一个扇区（MBR），LBA 1 是第二个扇区，以此类推。相比古老的 CHS（柱面-磁头-扇区）寻址，LBA 就是一个简单的线性编号，对软件来说友好得多。

状态寄存器的关键位 ​

状态寄存器（0x1F7 读）有几位是我们必须关注的：BSY（bit 7，值 0x80）表示驱动器忙，这时候其他状态位都无效；DRQ（bit 3，值 0x08）表示数据就绪，可以开始读写了；ERR（bit 0，值 0x01）表示发生了错误，需要读取错误寄存器了解详情；RDY（bit 6，值 0x40）表示驱动器就绪；DF（bit 5，值 0x20）表示驱动器故障。

轮询的顺序通常是：先等 BSY 清零（驱动器不忙了），然后检查 DRQ 是否置位（数据准备好了）。如果 ERR 或 DF 置位，说明出错了。

400ns 延迟的坑 ​

这是一个很容易踩的坑：发送 ATA 命令之后，不能立即读状态寄存器，必须等至少 400ns。原因是磁盘控制器在收到命令后需要一点时间来设置内部状态，如果你过早地读了状态寄存器，可能会读到命令发送之前的老状态。在真实硬件上读一次 I/O 端口大约需要 100ns，所以连续读 4 次控制端口（0x3F6）就能满足 400ns 的时序要求。

这里有一个关键细节：延迟期间必须读控制端口（0x3F6），而不是状态端口（0x1F7）。因为读状态端口有一个副作用——它会清除中断标志位。如果你不小心读错了端口，后续的轮询逻辑可能会出现难以排查的时序问题。

LBA28 与 LBA48 ​

ATA 规范有两种 LBA 寻址方式。LBA28 使用 28 位地址，最多能寻址 2^28 个扇区，也就是 128GB。对于小磁盘来说够用，但现代硬盘轻松超过这个容量。LBA48 使用 48 位地址，寻址空间达到 2^48 个扇区 = 128PB，在未来很长一段时间内都够用了。

在实现上我们做了一个自动判断：如果 LBA 地址超过 28 位范围（0x10000000），或者要读的扇区数超过 256（LBA28 一次最多读 256 个扇区），就自动切换到 LBA48 模式。在 QEMU 的默认磁盘镜像里，LBA28 绰绰有余，但代码里两种模式都实现了，为将来对接更大的虚拟磁盘做准备。

LBA28 的命令码是 0x20（READ SECTORS），LBA48 的命令码是 0x24（READ SECTORS EXT）。两者发送 LBA 地址的方式也不同：LBA28 把 24 位 LBA 分散到三个端口（0x1F3/4/5），高 4 位塞进驱动器选择寄存器；LBA48 则发送两次——先发高 16 位，再发低 16 位。

Freestanding 环境下的 memset 和 memcpy ​

这一章还新增了手写的 memset、memcpy、memmove 三个函数。你可能会觉得奇怪，这些不是标准库里最基础的函数吗？但别忘了我们的内核是 freestanding 环境，编译时带了 -ffreestanding -nostdlib，没有任何标准库可用。而后续的 ELF 加载器在拷贝段数据和清零 BSS 的时候必须用 memcpy 和 memset，所以只能自己写。

这三个函数的实现非常直白——逐字节操作。其中 memmove 比 memcpy 多了一点讲究：它需要处理源地址和目标地址重叠的情况。如果目标在源前面，从前到后复制；如果目标在源后面，从后到前复制——否则会覆盖还没读的数据。虽然在我们当前的加载场景里不太会遇到重叠，但作为一个通用工具函数，做对了总是好的。这三个函数都用 extern "C" 包裹，确保使用 C 语言的链接约定，避免 C++ 的 name mangling。

动手实现 ​

Step 1: 实现 Freestanding 内存工具函数 ​

目标：提供 memset、memcpy、memmove 三个函数，供 ELF 加载器在拷贝段数据和清零 BSS 时使用。

设计思路：我们的内核在 -ffreestanding -nostdlib 下编译，标准库完全不可用。但 ELF 加载器需要对内存区域进行拷贝和清零操作，所以必须自己实现最基本的内存操作函数。用 extern "C" 包裹确保链接时不被 C++ name mangling 干扰。

实现约束：创建一个头文件声明三个函数，使用 extern "C" 块包裹声明以支持 C 和 C++ 混合调用。实现文件中，memset 逐字节填充、memcpy 逐字节拷贝（假设不重叠）、memmove 根据地址关系选择前向或后向拷贝以处理重叠情况。memcpy 的参数使用 __restrict__ 关键字提示编译器源和目标不重叠。

踩坑预警：如果你忘记用 extern "C" 包裹，链接器会报 undefined reference——因为 C++ 编译器会把函数名 mangle 成类似 _Z6memsetPvim 的样子，和 ELF 加载器调用时期望的 C 符号名 memset 对不上。这个坑笔者确实踩过，排查了半天链接错误才发现是 name mangling 的问题。

验证：这一步没有独立的运行验证，它会在后续步骤中被间接使用。编译通过即可确认声明正确。

Step 2: 搭建 ATA PIO 驱动——头文件与常量定义 ​

目标：定义 ATA PIO 驱动需要的所有常量和接口，包括 I/O 端口号、状态位掩码、命令码、驱动器选择值。

设计思路：ATA 规范里每个端口和每个状态位的值都是固定的，不能靠猜。把所有常量定义为命名空间内的 constexpr 值，这样代码可读性好，也不容易手滑写错数字。比如看到 ATA_REG_STATUS 就知道是状态寄存器，比直接写 7 要清楚得多。

实现约束：在一个命名空间下定义所有常量。主控制器基址是 0x1F0，控制端口是 0x3F6。寄存器偏移量（数据 0、扇区计数 2、LBA 低/中/高 3/4/5、驱动器选择 6、状态/命令 7）都要定义。状态位掩码（BSY 0x80、DRQ 0x08、ERR 0x01、RDY 0x40、DF 0x20）、命令码（LBA28 读 0x20、LBA48 读 0x24）、驱动器选择（主盘 0xE0）也需要。公开接口只有两个函数：init() 初始化控制器，read(lba, count, buffer) 读取扇区。

踩坑预警：驱动器选择寄存器的值 0xE0 不是随便选的——它的 bit 5 和 bit 7 必须置 1（这是 ATA 规范要求的），bit 6 是 LBA 模式使能位。如果你把 LBA 位忘了，控制器会按 CHS 模式寻址，读出来的数据完全不对。另外一个容易搞混的地方是：同一个端口 0x1F7，读操作返回状态，写操作接收命令——它们在功能上是两个不同的寄存器，只是碰巧共用一个端口号。

验证：头文件只需要编译通过即可确认常量定义正确。

Step 3: 实现 ATA PIO 驱动——辅助函数 ​

目标：实现底层的 I/O 端口读写辅助函数和状态轮询函数。

设计思路：ATA PIO 驱动的核心就是一堆 inb/outb/inw 的组合，但直接在业务逻辑里写内联汇编太乱了。我们封装几个辅助函数：read_reg 读 8 位寄存器、write_reg 写 8 位寄存器、read_data 用 inw 读 16 位数据端口。inw 必须用内联汇编实现，因为我们的 io 库只提供了 inb 和 outb。

状态轮询方面，需要两个函数：wait_not_busy 轮询直到 BSY 位清零（驱动器不忙了），wait_data_ready 轮询直到 BSY 清零且 DRQ 置位（数据准备好了）。这两个函数都需要超时机制（一个合理的循环上限），否则如果硬件出问题，会永远卡在轮询里。在轮询循环中插入 pause 指令（__asm__ volatile("pause")）提示 CPU 当前在等待，可以降低功耗和总线争用。

delay_400ns 函数读取控制端口 4 次，每次约 100ns，满足 ATA 规范要求的 400ns 延迟。

实现约束：辅助函数放在匿名命名空间里（相当于 static），对外不暴露。read_data 使用 inw 指令读取 16 位值，约束修饰符 "=a"(value) 表示输出到 AX 寄存器，"Nd"(port) 表示输入可以是 8 位立即数或 DX 寄存器。轮询超时设为 10,000,000 次循环，在 QEMU 里足够了。wait_data_ready 在轮询时还要检查 ERR 和 DF 位，如果出错就立即返回 false。

踩坑预警：delay_400ns 必须读控制端口（0x3F6）而不是状态端口（0x1F7）！读状态端口会清除中断标志位，虽然我们用的是轮询模式不依赖中断，但养成好习惯很重要。另外 wait_data_ready 里检查错误的顺序很关键——先检查 ERR 和 DF，再检查 DRQ，否则可能会漏掉错误状态。

验证：这一步没有独立的运行验证，编译通过即可。

Step 4: 实现 ATA PIO 驱动——init() 初始化 ​

目标：实现 ATA 控制器的软件复位和初始化流程。

设计思路：初始化分为四步——软件复位、等待驱动器就绪、选择主盘、验证驱动器存在。软件复位通过控制端口（0x3F6）操作：先写入 0x04（置位 SRST 位进行复位，同时置位 nIEN 位禁用中断），等 400ns，再写入 0x00（取消复位），再等 400ns。然后等待 BSY 位清零，选择主盘（写 0xE0 到驱动器选择寄存器），等 400ns，再等 BSY 清零。最后读一次状态寄存器验证驱动器确实存在（状态不是 0xFF 表示浮空总线），并且 RDY 位置位。

实现约束：函数返回 bool 表示成功或失败。失败时通过 kprintf 输出详细错误信息。初始化成功后设置一个模块级标志变量，后续的 read 函数会检查这个标志。

踩坑预警：软件复位的顺序必须是"置位 SRST → 等待 → 清除 SRST"，不能跳过任何一步。如果状态寄存器读出来是 0xFF，说明总线上没有设备（浮空总线），QEMU 在某些配置下可能会出现这种情况。另外一个容易忽略的问题是：选择驱动器后必须再等一次 400ns，因为驱动器切换需要时间响应。

验证：编译并运行 mini kernel，在串口输出中应该能看到：

[INIT] Initializing ATA controller...
[INIT] ATA controller initialized successfully (status=0xXX).
1
2

其中 status 值应该包含 RDY 位（bit 6 置 1），通常在 QEMU 下是 0x50（RDY + DRQ 清零 + BSY 清零）。

调试技巧 ​

常见 Bug 1: ATA 初始化失败——"drive did not come out of reset" ​

这个通常是 QEMU 配置问题。检查你的磁盘镜像是否存在且格式正确。另外确保 QEMU 使用 -hda 而不是 -drive 的某些特殊配置——有些配置下 QEMU 不会创建传统的 ATA 设备。

排查方法：在 init() 中加更多的 kprintf 输出，看具体是哪一步失败的。

常见 Bug 2: 读到的数据全是 0xFF 或 0x00 ​

这说明你读错了端口，或者 LBA 地址不对。0xFF 通常是浮空总线的特征（没有设备响应），0x00 可能是读到了未写入的磁盘区域。检查：数据端口是否是 0x1F0（不是 0x1F7！），LBA 地址是否正确，扇区计数是否合理。
