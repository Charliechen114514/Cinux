028-1 Read-through: ext2_types.hpp — 磁盘数据结构定义 ​

概览 ​

本文逐行拆解 kernel/fs/ext2_types.hpp，这是 Cinux ext2 驱动的基石。文件里定义了 ext2 文件系统在磁盘上的四大核心数据结构——超级块（Ext2Superblock）、块组描述符（Ext2BlockGroupDescriptor）、索引节点（Ext2Inode）和目录项（Ext2DirEntry），以及一批常量和辅助类型。理解这些结构是理解后续 mount、inode 读写、目录遍历全部流程的前提，因为驱动所做的所有磁盘 I/O 本质上都是在解析和构造这些字节对齐的 packed 结构。

架构图 ​

磁盘布局 (4MB ext2 image, 1K block):
┌──────────┐ Block 0
│  Boot     │  (1024 bytes 引导保留区)
├──────────┤ Block 1
│ Superblock│  ← 偏移 1024, 1024 bytes
├──────────┤ Block 2
│   BGDT    │  ← 块组描述符表
├──────────┤ Block 3+
│ 数据块    │  位图 + inode 表 + 文件数据
│   ...     │  每个 block group 重复布局
└──────────┘

内存中的缓存结构:
Ext2CachedInode ──┬── disk_inode (Ext2Inode, 磁盘副本)
                  ├── vfs_inode  (Inode,    VFS 层视角)
                  ├── ino        (uint32_t, inode 编号)
                  └── in_use     (bool,     槽位占用标记)

这些 static constexpr 常量是我们跟 ext2 规范对齐的锚点。0xEF53 是 ext2 超级块的魔数，mount 的时候验证这个值——如果从磁盘读出来的不是 0xEF53，那就说明要么分区不是 ext2，要么数据损坏了。超级块固定在字节偏移 1024 处，大小固定 1024 字节，无论块大小怎么配置都是如此；而 1K 偏移恰好是 LBA 2 开始的两个扇区。

EXT2_DIRECT_BLOCKS = 12 意味着每个 inode 有 12 个直接块指针，加上一个一级间接、一个二级间接、一个三级间接，总共 15 个 i_block 槽位。在 Cinux 当前的实现里，文件读写只支持到一级间接（第 13 个指针），二级和三级暂时不涉及。EXT2_INODE_CACHE_SIZE = 64 是驱动内存中缓存的 inode 数量上限，选 64 是一个小而实用的折衷——既不会占太多内核内存，又能缓存常用文件。

ext2 的 i_mode 字段高 4 位标识文件类型，低 12 位是权限位。我们用 EXT2_S_IFMT 做掩码提取类型部分，然后跟 EXT2_S_IFREG 或 EXT2_S_IFDIR 比对。这种方式跟 Linux 内核完全一致。之所以只定义了 Regular 和 Directory 两个掩码，是因为 Cinux 目前只关心这两种文件对象——字符设备、符号链接之类的暂不处理。

Ext2FileType 是目录项（Ext2DirEntry）里 file_type 字段的值，用 scoped enum 定义。跟 i_mode 的掩码不同，这是直接存储在目录项里的"快速类型"标记，在 revision 1+ 的 ext2 中存在。定义了全部 8 种类型是为了跟规范对齐，但驱动实际使用的只有 Regular 和 Directory。

超级块是 ext2 的核心元数据容器，被 [[gnu::packed]] 修饰以确保编译器不做任何对齐填充——因为它的内存布局必须精确匹配磁盘上的字节排列。s_log_block_size 是一个对数值，真正的块大小等于 1024 << s_log_block_size，也就是说值为 0 对应 1024 字节、1 对应 2048、2 对应 4096。s_first_data_block 在 1K 块大小时为 1（因为 block 0 被引导扇区占掉了），更大块大小时为 0。

结构体末尾有一个 387 字节的 s_pad 数组，这是为了把整个结构撑到恰好 1024 字节。ext2 规范定义了超级块的完整 1024 字节布局，包括大量我们暂时用不到的字段（日志、配额、加密等），pad 数组吸收了所有剩余空间。static_assert 确保编译期能捕获任何大小偏移。

ext2 把整个分区划分为若干 block group（块组），每个块组有自己的数据块位图、inode 位图和 inode 表。这三个起始块号由 bg_block_bitmap、bg_inode_bitmap、bg_inode_table 指定。tag 028 的只读驱动使用这三个字段来定位块组和 inode 数据——bg_inode_table 在 read_disk_inode 中被频繁使用。后续 tag 实现写入操作时，会通过位图扫描 bg_free_blocks_count 和 bg_free_inodes_count 来分配新块和 inode。bg_used_dirs_count 在 mkdir 和 rmdir 时需要维护，用于 ext2 的 Orlov 目录分配策略。

Ext2Inode — 128 字节的文件描述核心 ​

这是我们打交道最多的结构。每个文件、目录、符号链接在 ext2 中都由一个 inode 描述。i_size 是文件内容的字节长度（目录的 i_size 是其数据块的总量）。i_blocks 比较特殊——它统计的是 512 字节扇区数而不是块数，所以一个 1024 字节的块会让 i_blocks 增加 2。这样做是为了和 POSIX 的 stat.st_blocks 保持兼容。

i_block[15] 数组是 ext2 寻址的核心。前 12 个槽位（i_block[0..11]）是直接块指针，每个指针直接指向一个数据块。第 13 个（i_block[12]）是一级间接指针，指向一个填满 uint32_t 的块，每个 uint32_t 又指向一个数据块。第 14 个是二级间接，第 15 个是三级间接。在 1K 块大小下，这种设计理论上支持的最大文件是 (12 + 256 + 256^2 + 256^3) * 1K ≈ 16 GB，远超我们的使用场景。

目录项是 ext2 里最精巧的设计之一。目录的数据块不像普通文件那样存字节流，而是存一条 Ext2DirEntry 链表。rec_len 既表示当前项的总长度（含 name），又充当跳转到下一个项的步进值。name[] 是 C99 弹性数组，实际长度由 name_len 指定，磁盘上不以 '\0' 结尾。

这种设计有一个非常实用的特性：删除一个目录项时，只需要把它的 rec_len 合并到前一个项即可，不需要移动后续数据。同理，创建新项时可以找一个 rec_len 比其实际需要大的项，把多出来的空间切分给新项。后续 tag 的写操作实现会利用这个特性。

Ext2CachedInode — 磁盘 inode 的内存缓存 ​

cpp
struct Ext2CachedInode {
    Ext2Inode disk_inode;
    Inode     vfs_inode;
    uint32_t  ino;
    bool      in_use;
};

这是驱动内部的缓存条目，把磁盘 inode（Ext2Inode）和 VFS 层的 Inode 绑定在一起。disk_inode 是从磁盘读出的完整副本，后续的 read/write 操作直接访问这里而不是每次都读磁盘。vfs_inode 是 VFS 层看到的统一接口，InodeOps 的回调函数通过 fs_private 指针反过来找到这个缓存条目，从而拿到 disk_inode。ino 记录 inode 编号用于快速查找，in_use 标记这个槽位是否被占用——当缓存满时，驱动会根据简单的哈希取模策略驱逐旧条目。

设计决策 ​

决策：使用 [[gnu::packed]] 而非手动位域 ​

问题: 磁盘数据结构必须精确匹配字节布局，编译器的默认对齐策略会插入填充字节导致偏移错误。 做法: 所有磁盘结构体使用 [[gnu::packed]] 属性，配合 static_assert 校验大小。 备选: 手动计算偏移量并逐字节 memcpy；或者用 #pragma pack(1)。 为什么不选: 手动偏移维护成本太高且容易出错；#pragma pack 是预处理器指令，作用域不如属性精确。[[gnu::packed]] 是 GCC/Clang 的标准扩展，在我们的工具链中完全可靠。 扩展: Linux 内核也使用 __attribute__((packed)) 定义这些结构，我们的做法跟上游一致。

决策：Ext2DirEntry 使用弹性数组成员 ​

问题: 目录项的 name 字段是变长的，C++ 没有标准的变长结构体。 做法: 使用 C99 风格的 char name[] 弹性数组，配合 [[gnu::packed]]。 备选: 把 name 定义为 char name[1]（旧式 C 技巧）然后手动计算偏移。 为什么不选: name[1] 会在 sizeof 里多算一个字节，offsetof 断言会失败。弹性数组是更干净的做法。 扩展: C++ 标准并不正式支持弹性数组成员，但 GCC/Clang 作为扩展支持它，Linux 内核大量使用。

扩展方向 ​

  • 支持 revision 1 的大 inode（s_inode_size > 128），利用额外的纳秒时间戳字段 -- ⭐⭐
  • 实现 HTree 目录索引，加速大目录的查找性能 -- ⭐⭐⭐
  • 添加 extent tree 支持（ext4 风格），取代传统间接块指针以减少大文件的元数据开销 -- ⭐⭐⭐⭐
  • 在 Ext2CachedInode 中加入脏标记，实现延迟写回（writeback）而非每次 write 都立即刷盘 -- ⭐⭐⭐
  • 实现 triple indirect block 支持，将最大单文件尺寸提升到理论极限 -- ⭐⭐
