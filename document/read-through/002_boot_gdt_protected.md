# 002 通读版 · GDT 与进入保护模式

## 章节概览

如果你是从实模式时代走过来的开发者，本章一定会让你感慨万千。我们要完成的是 x86 启动链条中最关键的一次跃迁 —— 从 16 位实模式切换到 32 位保护模式。完成本章后，QEMU 的 debugcon 终端会输出一个孤单的字母 `P`，这个看似简单的字符背后，是 CPU 从 8086 兼容模式到现代保护模式的完整切换过程。

在整个 Cinux OS 的架构中，本章扮演着承上启下的角色：它既是对实模式 bootlaoder 的延续，又是后续启用分页、进入长模式的前置条件。没有这一步，我们会被困在 1MB 地址空间里寸步难行；完成这一步后，4GB 的线性地址空间就向我们敞开了大门。

本章的核心设计决策包括：采用平坦内存模型（base=0, limit=4GB）简化地址转换；使用 `.code32` 显式声明 32 位代码段避免汇编器混淆；利用 QEMU debugcon 端口（0xE9）实现零配置调试输出。与 Linux 0.11 的早期启动代码相比，我们的实现更加精简 —— Linux 那时候还需要处理临时 GDT 的重定位问题，而由于 stage2 被固定加载到 0x8000，我们的 GDT 地址计算可以静态完成。

### 关键设计决策一览

* **平坦内存模型**：代码段和数据段基地址都设为 0，大小为 4GB，线性地址 = 物理地址
* **GDT 三段式结构**：null 描述符（必需）+ 32 位代码段 + 32 位数据段，满足最小可用配置
* **远跳转刷新流水线**：设置 CR0.PE 后立即用 `ljmp` 刷新 CS，避免 CPU 继续用 16 位方式解码指令
* **Debugcon 输出**：利用 QEMU 特有的 0xE9 端口实现零配置调试，比串口初始化更简单

## 关键代码精讲

接下来我们要逐段拆解 `boot/stage2.S` 中与保护模式切换相关的代码。你会发现整个切换过程可以浓缩为几个关键操作：关中断、加载 GDT、设置 CR0、远跳转，每一步都有其存在的理由。

### GDT 定义：段描述符表的构造

GDT 是保护模式的核心数据结构，每个段描述符占 8 字节，定义了段的基地址、大小和访问权限。我们把它放在独立的 `.gdt` 段中，并强制 8 字节对齐 —— 这是 CPU 硬件要求的，对齐错误会导致 `lgdt` 加载失败。

```asm
.section .gdt,"a"
.align 8                          // Align to 8 bytes (2^3)

gdt:
gdt_null:
    .quad 0                        // Null descriptor (required)

gdt_code:
    .word 0xFFFF                   // Limit 15:0
    .word 0x0000                   // Base 15:0 (= 0x8000)
    .byte 0x00                     // Base 23:16
    .byte 0x9A                     // Present, DPL=0, Code, Executable, Readable
    .byte 0xCF                     // Granularity=4KB, 32-bit, Limit 19:16=0xF
    .byte 0x00                     // Base 31:24

gdt_data:
    .word 0xFFFF                   // Limit 15:0
    .word 0x0000                   // Base 15:0 (= 0x8000)
    .byte 0x00                     // Base 23:16
    .byte 0x92                     // Present, DPL=0, Data, Writable
    .byte 0xCF                     // Granularity=4KB, 32-bit, Limit 19:16=0xF
    .byte 0x00                     // Base 31:24

gdt_end:
```

第一个描述符 `gdt_null` 必须全零，这是 Intel 的硬性规定。虽然它没有任何实际作用，但索引 0 永远指向 null 描述符，这方便我们在需要时用 `movw $0, %ax` 加载一个空段选择子来触发异常。

第二个描述符 `gdt_code` 定义了我们的 32 位代码段。重点在于访问权限字节 `0x9A`：bit 7 (P=1) 表示段在内存中，bits 5-6 (DPL=0) 表示内核级特权，bit 4 (S=1) 表示这是普通代码/数据段而非系统段，bits 0-3 (Type=0xA) 表示这是一个可执行、可读的代码段。粒度字节 `0xCF` 则设置了 G=1（4KB 粒度）和 D=1（32 位默认操作数大小），配合 limit 字段可以覆盖完整的 4GB 地址空间。

第三个描述符 `gdt_data` 与代码段类似，但 Type 字段为 `0x2`，表示这是一个可读写的数据段。注意我们把基地址设为 0，这意味着在平坦内存模型下，线性地址直接等于物理地址，后续的分页机制可以在此基础上实现更灵活的地址映射。

### GDT 指针：lgdt 指令的入口

`lgdt` 指令需要一个特殊的 6 字节结构：前 2 字节是 GDT 的长度（limit - 1），后 4 字节是 GDT 的线性基地址。我们用一个简单的结构来表示：

```asm
gdt_ptr:
    .word (gdt_end - gdt - 1)     // Limit = sizeof(GDT) - 1 = 23
    .long gdt                      // Linear address of GDT base
```

这里用了一个小技巧：`gdt_end - gdt - 1` 让链接器自动计算 GDT 的大小，每次我们增删描述符时都不需要手动修改 limit 值。`gdt` 这个符号由链接器解析为 stage2 加载基址（0x8000）加上 GDT 在二进制文件中的偏移量，因此它正好指向 GDT 的物理地址。

### 实模式到保护模式的切换序列

现在我们来到整个切换过程的核心部分。代码位于 `_start` 函数的 VESA 初始化之后，我们将分步拆解每一行的作用：

```asm
    cli                             // 关中断，保护模式下 IDT 还未设置

    // 清除 DS，为 lgdt 准备正确的段基址
    movw $0, %ax
    movw %ax, %ds

    // 1. 加载 GDTR
    lgdt gdt_ptr                    // Load GDTR from absolute address

    // 2. 设置 CR0 的 PE 位
    movl %cr0, %eax                 // Read CR0
    orb $0x1, %al                   // Set PE bit (bit 0)
    movl %eax, %cr0                 // Write CR0

    // 3. 远跳转刷新 CS
    ljmp $0x08, $pm_entry
```

首先执行 `cli` 关中断，这一步非常关键。在实模式下，中断向量表（IVT）位于内存 0x0000 地址，但进入保护模式后中断描述符表（IDT）的格式完全不同，而我们还没设置 IDT。如果这时候发生中断（比如定时器中断或 NMI），CPU 会尝试从 IDT 读取中断处理函数地址，结果必然触发 triple fault，导致 QEMU 直接退出。

接着我们清零 DS 寄存器。这里的陷阱比较隐蔽：`lgdt` 指令在 16 位模式下执行时，CPU 会用 DS * 16 + offset 的方式计算 `gdt_ptr` 的物理地址。如果 DS 还保持着之前的值（比如 0x0800），计算出的地址就会完全错误。将 DS 设为 0 后，物理地址就等于 offset，也就是 `gdt_ptr` 的链接地址，这样 `lgdt` 就能正确加载 GDT 了。

加载 GDT 后，我们通过读取-修改-写入 CR0 寄存器来设置 PE 位（Protection Enable）。`orb $0x1, %al` 只修改最低位，其他位保持不变，这样做比较安全。一旦 PE 位被置 1，CPU 就进入保护模式，但此时 CS 寄存器还保留着实模式的段值，CPU 的指令预取和译码流水线可能还在用 16 位方式处理后续指令。

这就是为什么我们需要远跳转 `ljmp $0x08, $pm_entry`。远跳转会同时加载新的段选择子和偏移量到 CS 和 EIP，从而彻底刷新流水线。`0x08` 是代码段选择子：index=1（指向 gdt_code），TI=0（使用 GDT 而非 LDT），RPL=0（请求特权级 0）。跳转目标 `pm_entry` 是我们 32 位代码段的入口点，它的地址由链接器计算。

### 32 位保护模式入口点

远跳转把我们带入了 `.code32` 声明的 32 位代码段。这里的第一个任务是设置所有数据段寄存器：

```asm
.code32
pm_entry:
    // 4. 设置所有数据段寄存器
    movw $0x10, %ax                 // Data selector value
    movw %ax, %ds
    movw %ax, %es
    movw %ax, %fs
    movw %ax, %gs
    movw %ax, %ss

    // 5. 设置新的栈指针
    movl $0x90000, %esp             // New stack in protected mode

    // 6. 调试输出：'P' 表示 Protected Mode
    movb $0x50, %al                 // 'P'
    outb %al, $0xE9                 // Debugcon output

    // 7. 无限停机
    cli
.pm_halt:
    hlt
    jmp .pm_halt
```

`0x10` 是数据段选择子：index=2（指向 gdt_data），TI=0，RPL=0。我们将这个值写入 DS、ES、FS、GS、SS 所有数据段寄存器，确保它们都指向同一个平坦数据段。注意 SS 也必须更新，虽然栈的物理位置没变，但在保护模式下它必须通过段选择子访问。

栈指针设为 `0x90000`，这个地址在实模式阶段我们用了 `0x9000:0xFFFE`，换算成线性地址正好是 `0x90000 + 0xFFFE`。现在直接用 `0x90000` 作为栈顶，与之前的布局是兼容的。

最关键的一行是 `outb %al, $0xE9`。`0x50` 是字符 `'P'` 的 ASCII 码，`0xE9` 是 QEMU 特有的 debugcon 端口。当 QEMU 用 `-debugcon stdio` 启动时，写入这个端口的字节会直接输出到标准输出，不需要任何初始化。这个单字节输出是我们确认成功进入保护模式的唯一信号 —— 如果一切正常，终端会显示 `P`，然后 CPU 进入 `hlt` 停机循环。

## 设计决策深度分析

### 决策 1：平坦内存模型 vs 多段模型

**问题**：在设计 GDT 时，我们有两种选择：平坦内存模型（base=0, limit=4GB）或多段模型（为代码、数据、栈分别设置不同的基地址）。

**本项目的做法**：我们选择了平坦内存模型，所有段（除 null 段外）的基地址都是 0，大小都是 4GB。这意味着线性地址直接等于物理地址，后续的分页机制可以在不改变段配置的情况下实现虚拟内存。

**备选方案**：早期 Windows 和 OS/2 使用的是多段模型，为不同类型的内存区域（代码、数据、栈、TSS 等）分配不同的段选择子。这种设计在硬件层面提供了一定的隔离保护，但增加了地址计算的复杂度。

**为什么不选备选方案**：现代 x86-64 操作系统（包括 Linux 和 Windows）几乎都采用了平坦模型 + 分页的组合。段机制的保护功能被分页和特权级机制取代，64 位长模式下甚至强制要求基地址为 0。我们的选择与主流实践保持一致，也为后续过渡到长模式扫清了障碍。

**如果要扩展/改进**：如果你想实现更细粒度的内存隔离，可以在后续引入分页机制时设置不同的页表属性。比如内核代码段映射为只执行，数据段映射为不可执行，这样可以在硬件层面防止代码注入攻击。

### 决策 2：在 stage2 中切换 vs 在独立内核中切换

**问题**：保护模式的切换可以在 stage2 bootloader 中完成，也可以延迟到真正的内核代码中。

**本项目的做法**：我们在 stage2 中就完成了到保护模式的切换，然后再进入后续的内核初始化。这样做的好处是 stage2 后续可以调用 32 位代码，为启用分页和加载 ELF 内核做准备。

**备选方案**：另一种设计是让 stage2 完全运行在实模式，只在把控制权交给内核前的最后一刻才切换到保护模式。GRUB 的早期版本采用了类似策略。

**为什么不选备选方案**：实模式的编程限制太多（1MB 地址空间、16 位寄存器、BIOS 调用依赖），在 stage2 中尽早切换到保护模式，我们可以用更现代的汇编语法和 C 代码完成复杂的初始化工作。此外，VESA 模式设置需要在保护模式后进行，以便正确访问线性帧缓冲区。

**如果要扩展/改进**：当前的实现假设 stage2 被加载到固定的 0x8000 地址。如果你希望支持动态加载地址，需要实现位置无关代码（PIC）或在切换前重定位 GDT。

### 决策 3：远跳转 vs 近跳转+段加载

**问题**：在设置 CR0.PE 后，我们需要刷新 CS 寄存器。可以选择远跳转（`ljmp`），也可以用近跳转配合手动加载 CS。

**本项目的做法**：我们使用 `ljmp $0x08, $pm_entry` 远跳转指令，同时加载新的段选择子和偏移量。这是 Intel 手册推荐的切换模式方式，能够确保 CPU 流水线被正确刷新。

**备选方案**：理论上可以先近跳转到 32 位代码，然后用 `movw $0x08, %cs` 尝试加载 CS。但实际上 CS 不能用 `mov` 指令直接加载，必须通过远跳转、远调用、中断返回等特殊指令修改。

**为什么不选备选方案**：这不是一个真正的选择 —— 硬件不支持直接加载 CS。我们只能用远跳转或远调用，而远跳转更简洁，不需要考虑返回地址的问题。

**如果要扩展/改进**：如果你需要在保护模式下实现任务切换，可以利用 TSS（任务状态段）和任务门机制。那时候 `ljmp` 可以跳转到 TSS 选择子，触发硬件任务切换。但对于单核单任务的内核，当前的设计已经足够。

## 常见变体与扩展方向

下面列出几个你可以尝试的扩展实验，按照难度排序：

1. **⭐ 添加更多 GDT 描述符**：在 `gdt_data` 后添加用户态代码段（DPL=3）和数据段，修改选择子值并测试不同特权级的访问控制。这个扩展可以帮助你理解特权级检查机制。

2. **⭐⭐ 实现 TSS 描述符**：添加一个任务状态段描述符，并在保护模式下用 `ltr` 指令加载任务寄存器。TSS 在后续的中断处理和任务切换中会用到，提前准备可以减少后续的改动量。

3. **⭐⭐⭐ 动态 GDT 重定位**：让 stage2 能够处理被加载到任意地址的情况，通过计算当前代码段地址来推导 GDT 的实际位置。这需要实现一些位置无关的代码技巧，对于理解链接器的工作原理很有帮助。

4. **⭐⭐⭐ 用 C 代码设置保护模式**：尝试把 GDT 定义和切换逻辑移到 C 文件中，用内联汇编或 `.byte` 数组构造 GDT，然后用 C 函数执行切换。这个实验能让你体会到混合编程的便利性和陷阱。

5. **⭐⭐⭐ 添加 GDB 调试支持**：利用 QEMU 的 `-s -S` 参数启动 GDB 服务器，编写 `.gdbinit` 脚本自动设置断点并检查 CR0、GDTR 等关键寄存器的值。这对于深入理解硬件状态变化非常有帮助。
