# 028-2 Read-through: ext2 mount 与块 I/O

## 概览

本文拆解 ext2 驱动的初始化链路：构造函数、DMA 缓冲区管理、底层块读写接口，以及最重要的 `mount()` 函数。mount 是整个驱动的入口——它要完成超级块读取和验证、块大小计算、块组描述符表加载、根 inode 就绪这一系列步骤。我们还会覆盖 `ext2.hpp` 中 Ext2 类的声明以及 `Ext2FileOps`/`Ext2DirOps` 两个 InodeOps 子类怎样为文件和目录提供多态分发。

## 架构图

```
Ext2 驱动初始化流程:
  Ext2(ahci, port)          ← 构造函数, 初始化 file_ops_ 和 dir_ops_
       │
       ▼
  mount()
       │
       ├── 1. ensure_dma_buffer()   ← PMM 分配 + VMM 映射一页物理内存
       │
       ├── 2. AHCI read superblock  ← LBA 2, 2 sectors
       │       └── memcpy → sb_
       │
       ├── 3. 验证 magic == 0xEF53
       │       └── 计算 block_size, sectors_per_block, group_count...
       │
       ├── 4. 循环 read_block(bgdt_block + i)  ← 加载 BGDT
       │       └── memcpy → bgdt_[]
       │
       ├── 5. read_disk_inode(2)     ← 读取根目录 inode
       │       └── 填入 inode_cache_[0]
       │
       └── 6. mounted_ = true

InodeOps 回调架构 (C++ 虚基类):
  Ext2FileOps : public InodeOps    ← read(), write(), stat()
  Ext2DirOps  : public InodeOps    ← readdir(), create(), mkdir(), unlink(), stat()
  每个子类持有 Ext2& 引用，无全局指针

块 I/O 数据流:
  read_block(N) ──► lba = N * sectors_per_block_
                   └── ahci_.read(port, lba, sectors, dma_buf_phys_)
                          │
                   DMA 直写到 dma_buf_phys_ 页
                          │
                   CPU 通过 dma_buf_virt_ 访问数据

  write_block(N) ─► CPU 先填 dma_buf_virt_
                   └── ahci_.write(port, lba, sectors, dma_buf_phys_)
```

## 代码精讲

### Ext2FileOps 与 Ext2DirOps —— InodeOps 的两个子类

```cpp
class Ext2FileOps : public InodeOps {
public:
    explicit Ext2FileOps(Ext2& ext2);
    int64_t read(const Inode* inode, uint64_t offset, void* buf, uint64_t count) override;
    int64_t write(Inode* inode, uint64_t offset, const void* buf, uint64_t count) override;
    int64_t stat(const Inode* inode, struct stat* st) override;
private:
    Ext2& ext2_;
};

class Ext2DirOps : public InodeOps {
public:
    explicit Ext2DirOps(Ext2& ext2);
    int64_t readdir(const Inode* inode, uint64_t index, char* name, uint64_t name_max) override;
    Inode*  create(Inode* dir, const char* name, uint32_t namelen) override;
    Inode*  mkdir(Inode* dir, const char* name, uint32_t namelen) override;
    int64_t unlink(Inode* dir, const char* name, uint32_t namelen) override;
    int64_t stat(const Inode* inode, struct stat* st) override;
private:
    Ext2& ext2_;
};
```

Cinux 的 VFS 定义了 `InodeOps` 虚基类，ext2 用两个子类分别处理文件和目录操作。每个子类持有 `Ext2&` 引用——不是全局指针，不是静态函数表，就是纯粹的 C++ 虚基类继承。`Ext2FileOps` 覆盖了 `read`、`write`、`stat` 三个方法，`Ext2DirOps` 覆盖了 `readdir`、`create`、`mkdir`、`unlink`、`stat`。VFS Inode 的 `ops` 字段在 `populate_vfs_inode` 中被设为 `&file_ops_` 或 `&dir_ops_`，之后 VFS 层的调用全部走虚函数分发。这种设计天然支持多磁盘多分区挂载——每个 Ext2 实例有自己独立的 ops 对象，互不干扰。

### Ext2 构造函数

```cpp
Ext2::Ext2(cinux::drivers::ahci::AHCI& ahci, uint8_t port_index)
    : file_ops_(*this), dir_ops_(*this), ahci_(ahci), port_index_(port_index) {}
```

构造函数做的事情很少但很关键：通过初始化列表依次构造 `file_ops_(*this)` 和 `dir_ops_(*this)`，让两个 ops 子类持有对 Ext2 对象的引用；然后保存 AHCI 引用和端口索引。实质性的初始化工作全部推迟到 `mount()`——构造函数不适合做可能失败的 I/O 操作，而 `mount()` 可以返回 bool 告诉调用者是否成功。

注意初始化顺序：`file_ops_` 和 `dir_ops_` 在成员列表中最先出现（对应头文件中的声明顺序），必须先于 `ahci_` 和 `port_index_` 初始化，因为 C++ 成员初始化顺序由声明顺序决定而非初始化列表书写顺序。

### DMA 缓冲区管理

```cpp
bool Ext2::ensure_dma_buffer() {
    if (dma_ready_) {
        return true;
    }

    // Allocate a physical page for DMA
    dma_buf_phys_ = cinux::mm::g_pmm.alloc_page();
    if (dma_buf_phys_ == 0) {
        cinux::lib::kprintf("[EXT2] Failed to allocate DMA page\n");
        return false;
    }

    // Map it into kernel virtual address space
    constexpr uint64_t flags = cinux::arch::FLAG_PRESENT | cinux::arch::FLAG_WRITABLE;
    dma_buf_virt_            = EXT2_DMA_VIRT_BASE;

    if (!cinux::mm::g_vmm.map(dma_buf_virt_, dma_buf_phys_, flags)) {
        cinux::lib::kprintf("[EXT2] Failed to map DMA page\n");
        cinux::mm::g_pmm.free_page(dma_buf_phys_);
        dma_buf_phys_ = 0;
        return false;
    }

    // Zero the buffer
    auto* buf = reinterpret_cast<uint8_t*>(dma_buf_virt_);
    for (uint32_t i = 0; i < cinux::arch::PAGE_SIZE; ++i) {
        buf[i] = 0;
    }

    dma_ready_ = true;
    return true;
}
```

ext2 驱动只有一个 DMA 缓冲区，大小恰好一页（4KB）。分配分两步：先从 PMM 拿到物理页帧号，再通过 VMM 映射到 `EXT2_DMA_VIRT_BASE` 这个固定的内核虚拟地址。映射标志只需要 PRESENT | WRITABLE——不需要用户态访问，也不需要 cache-disable（AHCI DMA 要求缓冲区在物理内存中连续且可 DMA 访问，但不需要特别禁用缓存，因为我们的 AHCI 驱动会处理 cache flush）。

映射失败时回收物理页，避免泄漏。映射完成后整页清零，防止读到残留数据。`dma_ready_` 标志保证幂等性——后续调用 `ensure_dma_buffer()` 时直接返回 true，不做重复分配。

整个驱动只有一个 DMA 缓冲区意味着所有块操作都是串行的。对单核协作式调度的内核来说完全够用，且避免了并发 DMA 的复杂性。

### 块读写接口

```cpp
bool Ext2::read_block(uint32_t block_num) {
    if (!ensure_dma_buffer()) {
        return false;
    }

    // Compute the LBA of the first sector of this block
    uint64_t lba = static_cast<uint64_t>(block_num) * sectors_per_block_;

    // Read all sectors for this block
    bool ok =
        ahci_.read(port_index_, lba, static_cast<uint16_t>(sectors_per_block_), dma_buf_phys_);
    if (!ok) {
        cinux::lib::kprintf("[EXT2] read_block(%u) I/O failed\n", block_num);
    }
    return ok;
}

bool Ext2::write_block(uint32_t block_num) {
    if (!ensure_dma_buffer()) {
        return false;
    }

    // Compute the LBA of the first sector of this block
    uint64_t lba = static_cast<uint64_t>(block_num) * sectors_per_block_;

    // Write all sectors for this block from the DMA buffer to disk
    bool ok =
        ahci_.write(port_index_, lba, static_cast<uint16_t>(sectors_per_block_), dma_buf_phys_);
    if (!ok) {
        cinux::lib::kprintf("[EXT2] write_block(%u) I/O failed\n", block_num);
    }
    return ok;
}
```

两个函数完全对称，都是三步走：先确保 DMA 缓冲区存在，然后用 ext2 块号乘以 `sectors_per_block_` 算出 LBA，最后调用 AHCI 的读写接口。传给 AHCI 的是物理地址 `dma_buf_phys_`——AHCI 控制器通过 DMA 直接读写这块物理内存。`read_block` 结束后 CPU 通过 `dma_buf_virt_` 读数据；`write_block` 之前 CPU 先往 `dma_buf_virt_` 填数据。

`sectors_per_block_` 在 mount 时根据块大小计算：1K 块对应 2 个扇区，4K 块对应 8 个扇区。因为只有一个 DMA 缓冲区，读写操作天然互斥——这在单核内核中不是问题，但如果未来引入多核或异步 I/O 就需要改造为缓冲池。

### mount() —— 六步初始化

```cpp
bool Ext2::mount() {
    cinux::lib::kprintf("[EXT2] Mounting ext2 filesystem on AHCI port %u\n", port_index_);

    // Step 1: Ensure DMA buffer is ready
    if (!ensure_dma_buffer()) {
        return false;
    }
```

第一步，确保 DMA 缓冲区就绪。因为 `ensure_dma_buffer` 是幂等的，如果之前已经分配过就跳过。

```cpp
    // Step 2: Read the superblock (at byte offset 1024 = LBA 2, 2 sectors)
    constexpr uint64_t SB_LBA     = EXT2_SUPERBLOCK_OFFSET / EXT2_SECTOR_SIZE;
    constexpr uint16_t SB_SECTORS = EXT2_SUPERBLOCK_SIZE / EXT2_SECTOR_SIZE;

    if (!ahci_.read(port_index_, SB_LBA, SB_SECTORS, dma_buf_phys_)) {
        cinux::lib::kprintf("[EXT2] Failed to read superblock\n");
        return false;
    }

    // Copy the superblock from the DMA buffer
    auto* dma = reinterpret_cast<const uint8_t*>(dma_buf_virt_);
    memcpy(&sb_, dma, sizeof(Ext2Superblock));
```

第二步，读取超级块。这里直接调用 `ahci_.read` 而不是 `read_block`，因为此时 `sectors_per_block_` 还没初始化——我们还不知道块大小是多少。超级块的位置是固定的：字节偏移 1024，即 LBA 2，长度 1024 字节即 2 个扇区。读完后从 DMA 缓冲区 memcpy 到 `sb_` 成员。

```cpp
    // Step 3: Validate magic number
    if (sb_.s_magic != EXT2_SUPER_MAGIC) {
        cinux::lib::kprintf("[EXT2] Invalid magic: 0x%x (expected 0x%x)\n", sb_.s_magic,
                            EXT2_SUPER_MAGIC);
        return false;
    }
```

第三步，验证魔数。0xEF53 是 ext2 文件系统的身份证——如果读出来的不是这个值，说明要么分区不是 ext2，要么数据损坏了。

```cpp
    // Step 4: Compute filesystem parameters
    block_size_        = 1024U << sb_.s_log_block_size;
    sectors_per_block_ = block_size_ / EXT2_SECTOR_SIZE;
    first_data_block_  = sb_.s_first_data_block;
    inode_size_        = (sb_.s_rev_level == 0) ? EXT2_INODE_SIZE_DEFAULT : sb_.s_inode_size;
    inodes_per_group_  = sb_.s_inodes_per_group;
    blocks_per_group_  = sb_.s_blocks_per_group;

    // Compute group count
    group_count_ = (sb_.s_blocks_count + blocks_per_group_ - 1) / blocks_per_group_;
    if (group_count_ > EXT2_MAX_GROUPS) {
        group_count_ = EXT2_MAX_GROUPS;
    }
```

第四步，从超级块提取文件系统参数。`s_log_block_size` 是对数值：0 对应 1K，1 对应 2K，2 对应 4K。`sectors_per_block_` 算出来后，后续所有的 `read_block`/`write_block` 才能正常工作。inode 大小在 revision 0 固定为 128 字节，高版本从 `s_inode_size` 读取。group_count 向上取整计算，clamp 到 `EXT2_MAX_GROUPS`（128），这个上限覆盖约 8GB 的 4K 块分区。

```cpp
    // Step 5: Read the block group descriptor table
    uint32_t bgdt_block = (block_size_ == 1024) ? 2 : 1;

    uint32_t bgdt_entries       = group_count_;
    uint32_t bgdt_bytes         = bgdt_entries * sizeof(Ext2BlockGroupDescriptor);
    uint32_t bgdt_blocks_needed = (bgdt_bytes + block_size_ - 1) / block_size_;

    for (uint32_t i = 0; i < bgdt_blocks_needed; ++i) {
        if (!read_block(bgdt_block + i)) {
            cinux::lib::kprintf("[EXT2] Failed to read BGDT block %u\n", bgdt_block + i);
            return false;
        }

        auto*    src                   = reinterpret_cast<const uint8_t*>(dma_buf_virt_);
        uint32_t entries_in_this_block = block_size_ / sizeof(Ext2BlockGroupDescriptor);
        uint32_t start_entry           = i * entries_in_this_block;
        uint32_t copy_count            = entries_in_this_block;

        if (start_entry + copy_count > bgdt_entries) {
            copy_count = bgdt_entries - start_entry;
        }

        memcpy(&bgdt_[start_entry], src, copy_count * sizeof(Ext2BlockGroupDescriptor));
    }
```

第五步，加载块组描述符表（BGDT）。起始块号取决于块大小：1K 块从 block 2 开始（block 0 是引导区，block 1 是超级块），更大块从 block 1 开始（超级块嵌在 block 0 的 1024 字节偏移处）。每个描述符 32 字节，一页 4K 块能装 128 个描述符。循环逐块读取，每块 memcpy 到 `bgdt_[]` 对应位置。最后一块可能不满，`copy_count` 做 clamp。

```cpp
    // Step 6: Set up the root directory inode (inode 2 in ext2)
    Ext2Inode root_disk;
    if (!read_disk_inode(2, root_disk)) {
        cinux::lib::kprintf("[EXT2] Failed to read root inode (ino=2)\n");
        return false;
    }

    // Place root inode in cache slot 0
    inode_cache_[0].ino        = 2;
    inode_cache_[0].disk_inode = root_disk;
    inode_cache_[0].in_use     = true;
    populate_vfs_inode(inode_cache_[0]);
    root_inode_ = inode_cache_[0].vfs_inode;

    cinux::lib::kprintf("[EXT2] Root inode: size=%u mode=0x%x\n", root_disk.i_size,
                        root_disk.i_mode);

    mounted_ = true;
    return true;
}
```

第六步，读取根目录 inode。ext2 规范规定 inode 2 永远是根目录（inode 1 是坏块管理器）。把它放到缓存槽位 0，调用 `populate_vfs_inode` 把磁盘 inode 翻译成 VFS Inode。`root_inode_` 单独保存一份副本，`lookup("/")` 直接返回它，不需要走缓存查找。最后设置 `mounted_ = true` 标记挂载成功。

### read_disk_inode —— 定位并读取一个磁盘 inode

```cpp
bool Ext2::read_disk_inode(uint32_t ino, Ext2Inode& out_inode) {
    if (ino == 0) {
        return false;
    }

    // Compute block group
    uint32_t group = (ino - 1) / inodes_per_group_;

    if (group >= group_count_) {
        cinux::lib::kprintf("[EXT2] Inode %u: group %u out of range\n", ino, group);
        return false;
    }

    // Get the inode table start block from the group descriptor
    uint32_t inode_table_block = bgdt_[group].bg_inode_table;

    // Compute the index within the group
    uint32_t index_in_group = (ino - 1) % inodes_per_group_;

    // Compute byte offset within the inode table
    uint64_t byte_offset = static_cast<uint64_t>(index_in_group) * inode_size_;

    // Compute which block of the inode table contains this inode
    uint32_t block_offset        = static_cast<uint32_t>(byte_offset / block_size_);
    uint32_t within_block_offset = static_cast<uint32_t>(byte_offset % block_size_);

    uint32_t target_block = inode_table_block + block_offset;

    // Read the block
    if (!read_block(target_block)) {
        cinux::lib::kprintf("[EXT2] Failed to read inode block %u\n", target_block);
        return false;
    }

    // Extract the inode from the DMA buffer
    auto* block_data = reinterpret_cast<const uint8_t*>(dma_buf_virt_);

    // Safety: ensure we don't read past the block boundary
    if (within_block_offset + sizeof(Ext2Inode) > block_size_) {
        cinux::lib::kprintf("[EXT2] Inode %u crosses block boundary\n", ino);
        return false;
    }

    memcpy(&out_inode, block_data + within_block_offset, sizeof(Ext2Inode));
    return true;
}
```

给定 inode 编号（1-based），定位并读取它的磁盘副本。计算分三步：先确定所在块组 `(ino - 1) / inodes_per_group_`，再算组内索引 `(ino - 1) % inodes_per_group_`，最后把索引转为字节偏移并拆分为块偏移和块内偏移。inode 表起始块号从 BGDT 的 `bg_inode_table` 字段获取。安全检查确保 inode 不会跨越块边界——128 字节的 inode 在 1K 块里最多装 8 个，在 4K 块里最多装 32 个，跨越块边界理论上不会发生，但防御性编程总没坏处。

### 访问器方法

```cpp
uint32_t Ext2::block_size() const {
    return block_size_;
}

bool Ext2::is_mounted() const {
    return mounted_;
}

uint64_t Ext2::dma_buf_virt() const {
    return dma_buf_virt_;
}
```

三个简单的内联访问器。`block_size()` 在 InodeOps 回调中频繁使用——文件读写需要按块切分。`dma_buf_virt()` 让回调能直接访问 DMA 缓冲区的内容。`is_mounted()` 暴露挂载状态给外部查询。它们都很短小，编译器大概率内联。

## 设计决策

### 虚基类 InodeOps + 引用注入，拒绝全局指针

如果用全局指针 `g_ext2_instance` 来让 InodeOps 回调访问 Ext2 对象，单实例时能工作，但一旦需要多分区挂载（比如同时挂载 `/` 和 `/home`）就崩溃了。现在 `Ext2FileOps` 和 `Ext2DirOps` 作为 InodeOps 的子类持有 `Ext2&` 引用，每个 ops 实例天然知道自己属于哪个 Ext2 对象。虚函数分发让 VFS 层完全不需要知道具体文件系统类型——只需要 `inode->ops->read(...)`，C++ 的动态分发自动路由到正确的实现。

### 单 DMA 缓冲区设计

块设备 DMA 需要物理连续的缓冲区，但内核堆分配器可能产生物理不连续的页面。mount 期间预分配一页物理内存并映射到固定虚拟地址，所有块 I/O 共用这个缓冲区。动态分配开销太大，多页缓冲池增加复杂度但在单核内核中没有收益。缺点是所有块操作串行化——未来引入异步 I/O 时需要改为缓冲池。

### 超级块和 BGDT 常驻内存

每次块分配和 inode 分配都需要查询超级块和 BGDT 的空闲计数。如果在需要时才从磁盘读取，I/O 延迟不可接受。mount 时把整个超级块和全部 BGDT 条目读到内存中，修改后按需写回磁盘。4MB 测试镜像只有几个块组，BGDT 才几十字节，常驻内存的开销微乎其微。

## 扩展方向

- 支持多块大小的 DMA 缓冲区（当前 4KB 页面在 4K 块大小时刚好只装一个块）
- 添加超级块校验和（metadata checksum feature），检测静默数据损坏
- 实现 sparse superblock 特性（不在每个块组都存超级块备份），减少元数据开销
- mount 时检查 `s_state` 字段判断文件系统是否需要 fsck
- 实现 journal replay（从 ext2 迁移到 ext3 的前提）
