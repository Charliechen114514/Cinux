# 028-3 Read-through: 文件读取与目录遍历

## 概览

前两篇我们看完了超级块、块组描述符这些"搭台子"的工作，现在进入驱动最核心的数据路径——文件读取与目录遍历。我们要拆解的代码覆盖 inode 缓存管理（`get_cached_inode`、`populate_vfs_inode`）、文件读取回调（`ext2_file_read`）和目录枚举回调（`ext2_dir_readdir`）。这些函数构成了一条从 VFS `Inode` 出发、经过缓存查找、块寻址、最终落到 AHCI 磁盘 I/O 的完整只读通路。

## 架构图

```
读取链路:
  inode->ops->read → ext2_file_read (静态函数)
    ├── g_ext2_instance → Ext2 对象
    ├── fs_private → Ext2CachedInode → disk_inode
    ├── file_block < 12 → i_block[file_block]
    ├── file_block < 12+ptrs → 间接块查表
    ├── disk_block == 0 → 空洞填零
    └── read_block → memcpy

目录遍历链路:
  inode->ops->readdir → ext2_dir_readdir (静态函数)
    ├── g_ext2_instance → Ext2 对象
    ├── index 0/1 → "."/".." 硬编码返回
    ├── index 2+ → 遍历目录数据块
    └── rec_len 步进, inode!=0 计数

缓存查找:
  get_cached_inode(ino)
    ├── 线性扫描命中 → 返回 vfs_inode
    ├── 空闲槽位 → read_disk_inode + populate_vfs_inode
    └── 缓存满 → 驱逐 slot 1-63 (slot 0 固定给根)
```

## 代码精讲

### inode 缓存：get_cached_inode

```cpp
Inode* Ext2::get_cached_inode(uint32_t ino) {
    // 第一阶段：线性扫描，命中直接返回
    for (uint32_t i = 0; i < EXT2_INODE_CACHE_SIZE; ++i) {
        if (inode_cache_[i].in_use && inode_cache_[i].ino == ino)
            return &inode_cache_[i].vfs_inode;
    }
    // 第二阶段：找空闲槽位，从磁盘读取并填充
    for (uint32_t i = 0; i < EXT2_INODE_CACHE_SIZE; ++i) {
        if (!inode_cache_[i].in_use) {
            if (!read_disk_inode(ino, inode_cache_[i].disk_inode)) return nullptr;
            inode_cache_[i].ino = ino;
            inode_cache_[i].in_use = true;
            populate_vfs_inode(inode_cache_[i]);
            return &inode_cache_[i].vfs_inode;
        }
    }
    // 第三阶段：缓存满了，伪随机驱逐
    uint32_t evict = 1 + (ino % (EXT2_INODE_CACHE_SIZE - 1));
    inode_cache_[evict].in_use = false;
    if (!read_disk_inode(ino, inode_cache_[evict].disk_inode)) return nullptr;
    inode_cache_[evict].ino = ino;
    inode_cache_[evict].in_use = true;
    populate_vfs_inode(inode_cache_[evict]);
    return &inode_cache_[evict].vfs_inode;
}
```

固定 64 槽位的 `Ext2CachedInode` 数组，三阶段查找：线性扫描命中、空闲槽位冷加载、都失败就驱逐。驱逐公式 `1 + (ino % 63)` 保证结果落在 1-63，永远踢不到 slot 0——那是根目录的家。

### ext2_file_read —— 分块读取

```cpp
int64_t ext2_file_read(const Inode* inode, uint64_t offset,
                       void* buf, uint64_t count) {
    if (inode == nullptr || inode->fs_private == nullptr || buf == nullptr) {
        return -1;
    }

    auto* cached = static_cast<const Ext2CachedInode*>(inode->fs_private);
    const Ext2Inode& disk = cached->disk_inode;

    if (offset >= disk.i_size) {
        return 0;
    }

    uint64_t available = disk.i_size - offset;
    uint64_t to_read = (count < available) ? count : available;

    if (to_read == 0) {
        return 0;
    }

    Ext2* ext2 = g_ext2_instance;
    if (ext2 == nullptr) {
        return -1;
    }

    uint32_t bs = ext2->block_size();
    uint64_t block_ptrs_per_block = bs / sizeof(uint32_t);

    auto* dst = static_cast<uint8_t*>(buf);
    uint64_t total_read = 0;

    while (total_read < to_read) {
        uint64_t file_block = (offset + total_read) / bs;
        uint64_t block_offset = (offset + total_read) % bs;
        uint64_t chunk = bs - block_offset;
        if (chunk > to_read - total_read) {
            chunk = to_read - total_read;
        }

        uint32_t disk_block = 0;

        if (file_block < EXT2_DIRECT_BLOCKS) {
            disk_block = disk.i_block[file_block];
        } else if (file_block < EXT2_DIRECT_BLOCKS + block_ptrs_per_block) {
            uint32_t indirect_block = disk.i_block[EXT2_INDIRECT_BLOCK];
            if (indirect_block == 0) {
                break;
            }

            if (!ext2->read_block(indirect_block)) {
                break;
            }

            uint32_t idx = static_cast<uint32_t>(
                file_block - EXT2_DIRECT_BLOCKS);
            auto* indirect = reinterpret_cast<uint32_t*>(
                ext2->dma_buf_virt());
            disk_block = indirect[idx];
        } else {
            break;
        }

        if (disk_block == 0) {
            for (uint64_t i = 0; i < chunk; ++i) {
                dst[total_read + i] = 0;
            }
            total_read += chunk;
            continue;
        }

        if (!ext2->read_block(disk_block)) {
            break;
        }

        auto* src = reinterpret_cast<const uint8_t*>(
            ext2->dma_buf_virt()) + block_offset;
        memcpy(dst + total_read, src, chunk);
        total_read += chunk;
    }

    return static_cast<int64_t>(total_read);
}
```

`ext2_file_read` 是一个静态函数（不是类方法），通过 `g_ext2_instance` 全局指针访问 Ext2 对象。它首先通过 `inode->fs_private` 反向拿到 `Ext2CachedInode`，从中获取磁盘 inode 的副本。然后进入 while 循环逐块读取——核心是块寻址逻辑。

块寻址分两种情况。`file_block < 12` 时直接从 `i_block[file_block]` 取物理块号，不需要额外 I/O。`file_block` 在 12 到 12+block_ptrs_per_block 之间时，走单间接块路径：先读间接块本身（`i_block[12]` 指向的块），从中按索引取出目标数据块号，再读数据块。这里有一个微妙之处：两次 `read_block` 之间，第一次读到的间接块数据会被第二次覆盖，但没关系——目标块号已经提前提取到 `disk_block` 局部变量里了。

`disk_block == 0` 代表文件空洞（Sparse file），POSIX 语义要求读取未分配的块返回全零。`offset >= disk.i_size` 保证不越过文件末尾。

### ext2_dir_readdir —— 基于索引的目录枚举

```cpp
int64_t ext2_dir_readdir(const Inode* inode, uint64_t index,
                         char* name, uint64_t name_max) {
    if (inode == nullptr || inode->fs_private == nullptr ||
        name == nullptr || name_max == 0) {
        return -1;
    }

    auto* cached = static_cast<const Ext2CachedInode*>(inode->fs_private);
    const Ext2Inode& disk = cached->disk_inode;

    Ext2* ext2 = g_ext2_instance;
    if (ext2 == nullptr) {
        return -1;
    }

    uint32_t bs = ext2->block_size();

    // Special entries: "." and ".."
    if (index == 0) {
        if (name_max < 2) return -1;
        name[0] = '.';
        name[1] = '\0';
        return 1;
    }
    if (index == 1) {
        if (name_max < 3) return -1;
        name[0] = '.';
        name[1] = '.';
        name[2] = '\0';
        return 1;
    }

    uint64_t target = index - 2;
    uint64_t found = 0;
    uint64_t dir_size = disk.i_size;

    uint32_t total_blocks = static_cast<uint32_t>(
        (dir_size + bs - 1) / bs);
    if (total_blocks > EXT2_DIRECT_BLOCKS) {
        total_blocks = EXT2_DIRECT_BLOCKS;
    }

    for (uint32_t b = 0; b < total_blocks; ++b) {
        uint32_t blk = disk.i_block[b];
        if (blk == 0) continue;

        if (!ext2->read_block(blk)) return -1;

        auto* block_data = reinterpret_cast<const uint8_t*>(
            ext2->dma_buf_virt());
        uint32_t pos = 0;

        while (pos < bs) {
            if (pos + EXT2_DIR_ENTRY_HDR_SIZE > bs) break;

            auto* entry = reinterpret_cast<const Ext2DirEntry*>(
                block_data + pos);

            if (entry->rec_len == 0) break;

            if (entry->inode != 0) {
                if (found == target) {
                    uint32_t copy_len = static_cast<uint32_t>(name_max) - 1;
                    if (entry->name_len < copy_len)
                        copy_len = entry->name_len;

                    for (uint32_t i = 0; i < copy_len; ++i)
                        name[i] = entry->name[i];
                    name[copy_len] = '\0';
                    return 1;
                }
                ++found;
            }

            pos += entry->rec_len;
        }
    }

    return 0;  // no more entries
}
```

`ext2_dir_readdir` 也是静态函数，同样通过 `g_ext2_instance` 访问 Ext2 对象。索引 0 和 1 硬编码返回 "." 和 ".."——这是所有 Unix 目录都有的两个特殊条目，不需要每次都从磁盘读取。从 index 2 开始才真正扫描磁盘：`target = index - 2` 表示要找的磁盘上第 N 个非零 inode 条目。

内层循环按 `rec_len` 步进，遇到 `inode != 0` 的有效条目时递增计数器 `found`，匹配 `target` 时拷贝名称返回 1。`rec_len == 0` 是损坏或末尾的标志，立即 break。返回 0 表示索引越界，调用者知道枚举结束。tag 028 的 readdir 不会跳过 "." 和 ".." 条目——它把磁盘上所有非零 inode 条目都计入计数，这与后续 tag 的行为略有不同，但在只读场景下是正确的。

### ext2_file_write —— 只读驱动的写桩

```cpp
int64_t ext2_file_write(Inode*, uint64_t, const void*, uint64_t) {
    return -1;
}
```

tag 028 是一个纯粹的只读驱动。`ext2_file_write` 只返回 -1，表示"写操作不支持"。这个函数同时出现在 `ext2_file_ops` 和 `ext2_dir_ops` 中——无论是文件还是目录，写操作都会被拒绝。真正的写操作（包括块分配、inode 分配、目录项修改）要等到 tag 028b 才会实现。

## 设计决策

### 静态函数 + 全局指针 vs 虚基类

选择静态函数配合 `g_ext2_instance` 全局指针，而不是 C++ 虚基类继承，有三个原因。一是 `InodeOps` 在 VFS 中被定义为 POD 结构体（函数指针表），不是虚基类——ext2 的回调必须匹配这个签名。二是静态函数零运行时类型信息开销，函数指针直接存在于结构体中。三是实现极其简单——全局指针一行赋值，不需要引入继承体系、构造顺序、虚函数表等复杂度。

代价是只支持单个 ext2 实例。但在 tag 028 的场景下（Cinux 只挂载一个 ext2 分区），这完全足够。tag 028b 会将 `InodeOps` 扩展为虚基类，通过引用注入消除全局状态。

### 文件洞的正确处理

当 `disk_block == 0` 时，`ext2_file_read` 不报错，而是用循环逐字节填零返回。这是因为 ext2 允许文件有稀疏区域——逻辑上存在但物理上没有分配块的区域。POSIX 规定读取未分配的区域应返回零字节，Cinux 的实现正确地遵循了这个语义。

### 间接块读取时的 DMA 缓冲区争用

间接块路径涉及两次 `read_block`：第一次读间接块取指针，第二次读数据块取内容。因为只有一个 DMA 缓冲区，第一次的数据会被第二次覆盖。解决方案是在第一次 `read_block` 后立即把 `disk_block` 提取到局部变量，之后的操作不再依赖 DMA 缓冲区中的间接块数据。这个模式在单缓冲区架构下是标准做法。
