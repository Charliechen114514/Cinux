# 028-4 Read-through: 路径解析与 VFS 集成

## 概览

前几篇我们拆解了 ext2 的 mount 流程、文件读取和目录遍历。现在把最后几块拼图补齐：路径解析（`lookup_in_dir`、`lookup`）以及 ext2 驱动如何挂载到 VFS 命名空间中被 syscall 层透明调用。这些代码把前面实现的"原子操作"（读一个块、遍历一个目录、读一个 inode）串联成一条从用户态 `sys_open("/etc/motd")` 到磁盘扇区数据的完整调用链。

## 架构图

```
路径解析流程:
  VFS: sys_open("/etc/motd")
    └── vfs_resolve → 匹配 "/" 挂载点 → ext2.lookup("etc/motd")
          │
          ├── skip leading '/'
          ├── current_ino = 2 (root)
          │
          ├── lookup_in_dir(2, "etc", 3)
          │     ├── read_disk_inode(2) → 目录 inode
          │     ├── read_block(dir_data_block)
          │     └── 逐条目比较 name_len + 逐字节匹配 → found ino=11
          │
          ├── 检查 ino=11 的 i_mode → EXT2_S_IFDIR ✓
          │
          ├── lookup_in_dir(11, "motd", 4)
          │     ├── read_disk_inode(11) → 目录 inode
          │     ├── read_block(dir_data_block)
          │     └── 逐条目比较 → found ino=15
          │
          └── get_cached_inode(15) → 返回 VFS Inode*

VFS 集成:
  kernel/main.cpp:
    static Ext2 ext2(ahci, 1);  // g_ext2_instance = this
    ext2.mount();                // 读 SB + BGDT + root inode
    vfs_mount_add("/", &ext2);   // 注册到 VFS 挂载表

  调用链:
    sys_open → vfs_resolve → ext2.lookup
    sys_read → inode->ops->read → ext2_file_read
    sys_getdents → inode->ops->readdir → ext2_dir_readdir
```

## 代码精讲

### lookup_in_dir —— 单级目录查找

```cpp
uint32_t Ext2::lookup_in_dir(uint32_t dir_ino, const char* name,
                             uint32_t name_len) {
    Ext2Inode dir_disk;
    if (!read_disk_inode(dir_ino, dir_disk)) {
        return 0;
    }

    uint32_t bs = block_size_;
    uint32_t dir_size = dir_disk.i_size;
    uint32_t total_blocks = (dir_size + bs - 1) / bs;

    if (total_blocks > EXT2_DIRECT_BLOCKS) {
        total_blocks = EXT2_DIRECT_BLOCKS;
    }

    for (uint32_t b = 0; b < total_blocks; ++b) {
        uint32_t blk = dir_disk.i_block[b];
        if (blk == 0) {
            continue;
        }

        if (!read_block(blk)) {
            return 0;
        }

        auto* block_data = reinterpret_cast<const uint8_t*>(dma_buf_virt_);
        uint32_t pos = 0;

        while (pos < bs) {
            if (pos + EXT2_DIR_ENTRY_HDR_SIZE > bs) {
                break;
            }

            auto* entry = reinterpret_cast<const Ext2DirEntry*>(
                block_data + pos);

            if (entry->rec_len == 0) {
                break;
            }

            if (entry->inode != 0 &&
                entry->name_len == name_len) {
                bool match = true;
                for (uint32_t i = 0; i < name_len; ++i) {
                    if (entry->name[i] != name[i]) {
                        match = false;
                        break;
                    }
                }

                if (match) {
                    return entry->inode;
                }
            }

            pos += entry->rec_len;
        }
    }

    return 0;  // not found
}
```

`lookup_in_dir` 是路径解析的原子操作：给定一个目录 inode 号和一个组件名称，扫描目录数据块找匹配条目。先比较 `name_len`——长度不同直接跳过，省去逐字节比较。ext2 文件名不以 null 结尾，长度信息存在 `name_len` 字段中，所以这里的逐字节比较是必要的。

`total_blocks` 限制了只扫描直接块（`i_block[0..11]`）。对于 1K 块大小，这能覆盖最大 12KB 的目录。12KB / 平均条目大小（约 16 字节）= 约 750 个目录项，远超教学 OS 的需求。如果将来目录真的超过 12 个块，需要加入间接块遍历。

### lookup —— 完整路径解析

```cpp
Inode* Ext2::lookup(const char* path) {
    if (path == nullptr) {
        return nullptr;
    }

    if (path[0] == '\0' || (path[0] == '/' && path[1] == '\0')) {
        return &root_inode_;
    }

    if (path[0] == '/') {
        ++path;
    }

    uint32_t current_ino = 2;

    while (path[0] != '\0') {
        uint32_t comp_len = 0;
        while (path[comp_len] != '\0' && path[comp_len] != '/') {
            ++comp_len;
        }

        if (comp_len == 0) {
            ++path;
            continue;
        }

        uint32_t found_ino = lookup_in_dir(current_ino, path, comp_len);
        if (found_ino == 0) {
            return nullptr;
        }

        path += comp_len;
        if (path[0] == '/') {
            ++path;
        }

        if (path[0] != '\0') {
            Ext2Inode check;
            if (!read_disk_inode(found_ino, check)) {
                return nullptr;
            }
            if ((check.i_mode & EXT2_S_IFMT) != EXT2_S_IFDIR) {
                return nullptr;
            }
        }

        current_ino = found_ino;
    }

    return get_cached_inode(current_ino);
}
```

`lookup` 实现经典的 Unix 路径解析：从根 inode (2) 开始，逐级分解路径组件。每找到一级后，如果后面还有更多组件，就读取 inode 的 `i_mode` 验证是目录——这是防止把普通文件当目录用的防线。最终结果通过 `get_cached_inode` 放入缓存并返回。

空路径和 "/" 直接返回 `root_inode_`——这是 mount 时单独保存的副本，不需要走缓存查找。前导 '/' 被跳过是因为 VFS 层在调用 lookup 之前已经剥离了挂载点前缀，但这里做了防御性处理以防调用者传入了绝对路径。

连续的 '/' 被跳过（`comp_len == 0` 时 `++path`），这避免了空组件被传给 `lookup_in_dir`。

### kernel/main.cpp 集成

```cpp
// 在 kernel/main.cpp 的初始化序列中:

// Step 21: AHCI 初始化
static cinux::drivers::ahci::AHCI ahci;
cinux::drivers::pci::PCIDevice ahci_dev;
if (pci.find_ahci(ahci_dev)) {
    ahci.init(ahci_dev);
    // ... AHCI 测试 ...
}

// Step 22: Mount ext2 filesystem from AHCI port 1
cinux::lib::kprintf("[BIG] ===== Milestone 028: ext2 Filesystem =====\n");
static cinux::fs::Ext2 ext2(ahci, 1);
if (!ext2.mount()) {
    cinux::lib::kprintf("[EXT2] mount failed!\n");
}

// Step 27: Register ext2 in VFS mount table
cinux::lib::kprintf("[BIG] ===== Milestone 027: VFS =====\n");
cinux::fs::vfs_mount_init();
cinux::fs::vfs_mount_add("/", &ext2);
cinux::lib::kprintf("[VFS] ext2 mounted at /\n");
```

内核启动时的初始化顺序非常关键。`AHCI` 对象被声明为 `static`——这是因为它的生命周期必须覆盖整个内核运行期间，不能是栈变量。同样，`Ext2` 对象也是 `static`。`ahci` 从局部变量改为 `static` 是 tag 028 引入的一个改动——因为 `Ext2` 构造函数保存了 `ahci` 的引用，如果 `ahci` 是局部变量，出了作用域后引用就悬空了。

之前的 tag 用 Ramdisk 作为根文件系统，tag 028 把它替换为 ext2。挂载调用 `vfs_mount_add("/", &ext2)` 注册到 VFS 挂载表。之后用户态的 `sys_open("/etc/motd")` 会经过 VFS 路径解析：匹配 "/" 挂载点，剥离前缀后调用 `ext2.lookup("etc/motd")`，走我们上面拆解的完整路径解析流程。

### 端到端调用链

以 `cat /hello.txt` 为例，完整调用链是这样的：用户态 Shell 解析命令后调用 `sys_open("/hello.txt")`，触发 syscall 进入内核；内核的 `sys_open` handler 调用 `vfs_resolve("/hello.txt")`，VFS 匹配到 "/" 挂载点上的 Ext2 对象，调用 `ext2.lookup("hello.txt")`；ext2 的 lookup 在根目录（inode 2）的数据块中搜索 "hello.txt" 条目，找到后通过 `get_cached_inode` 返回 VFS Inode 指针；`sys_open` 把这个 Inode 包装成 File 对象放入 FDTable，返回 fd。接着 Shell 循环调用 `sys_read(fd, buf, 256)`，每次 read 最终走到 `ext2_file_read`——通过 `inode->ops->read` 分发到这个静态函数，在函数内部通过 `g_ext2_instance` 访问 Ext2 对象的 `read_block`，从磁盘读出文件数据。最后 Shell 通过 `sys_write(fd=1)` 把内容打印到串口。

## 设计决策

### 路径解析中逐级类型检查

`lookup` 的循环中，每当找到一个子条目但后面还有更多路径组件时，都会读取 inode 的 `i_mode` 验证它是目录。这个额外的 `read_disk_inode` 调用有性能开销，但提供了正确性保障——如果路径中某个中间组件是普通文件（比如 `/etc/motd/not-a-dir`），不带检查的代码会试图把文件数据当目录项解析，大概率导致 `rec_len` 不合法甚至死循环。防御性编程在这里是值得的。

### VFS 挂载替换 Ramdisk

tag 028 用 ext2 替换了 Ramdisk 作为根文件系统。这意味着内核启动时不再需要嵌入 initrd 归档，所有文件都从 AHCI 磁盘上的 ext2 分区读取。好处是文件系统内容可以在构建脚本中动态修改（用 debugfs 操作 ext2 镜像），不需要重新编译内核。同时 ext2 的持久化特性意味着理论上修改可以写入磁盘——但 tag 028 只实现了读取，写入要到 tag 028b。
