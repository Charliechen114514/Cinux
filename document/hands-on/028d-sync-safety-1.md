# 028d-1: 自旋锁——从原子操作到 RAII 守卫

## 导语

到目前为止，我们的内核已经有了完整的进程调度器——多个线程在时钟中断的驱动下轮流执行。但这里有一个我们一直在回避的根本问题：**如果两个线程同时操作同一个数据结构会怎样？** 答案是数据竞争，而数据竞争的结局通常是内核崩溃。更糟糕的是，中断处理程序随时可能打断正在执行一半的操作——比如你正在遍历空闲链表，一个时钟中断进来，调度器切到另一个线程，那个线程也开始遍历同一条链表，两个线程同时修改链表指针，boom，链表结构被彻底破坏。

这一章我们要做的就是给内核装上"互斥锁"——准确地说是自旋锁（Spinlock）。完成这一章后，我们将拥有一套完整的同步原语基建：纯自旋锁、中断安全守卫、独立的禁中断守卫，以及它们对应的 RAII 自动管理机制。

知识前置方面，你需要了解 021 章中的 Mutex 和 Semaphore 概念（它们在内部会用到我们这里实现的 Spinlock），以及基本的 x86 中断机制（IF 标志位、cli/sti 指令）。

## 概念精讲

### 自旋锁的本质

自旋锁的核心思想极其简单：**用一个布尔变量表示锁的状态，获取锁的时候原子地把它设为 true，如果发现已经是 true 了，就反复重试（"自旋"）。**

你可以把它想象成一间只有一个入口的会议室，门口有一个指示灯。灯灭着说明没人，你进门的时候顺手把灯打开；灯亮着说明里面有人，你就在门口等着，一直盯着灯看，灯灭了就立刻冲进去。关键在于"看灯+开灯"这个动作必须是原子的——不能两个人同时看到灯灭着，同时挤进去。

在 x86 上，这个"原子地检查并设置"操作由硬件保证。GCC 提供了 `__atomic_test_and_set` 内建函数，它在底层会编译成带有 LOCK 前缀的指令（或者隐含 LOCK 语义的 XCHG 指令），确保在多核环境下也是原子的。虽然 Cinux 目前是单核设计，但原子操作对于单核上中断驱动的并发同样至关重要——我们不是在防另一个 CPU，而是在防当前 CPU 被中断打断后切换到另一个线程。

### 内存序：为什么不只是 volatile

很多初学者会用 `volatile` 来保护共享变量，但这是一个危险的误解。`volatile` 只是告诉编译器不要优化掉对变量的读写，它完全不保证原子性，也不建立" happens-before"关系。具体来说：

- 线程 A 获取锁、修改数据、释放锁——这三个操作之间必须有明确的顺序保证
- 其他线程在获取同一个锁之后，必须能看到线程 A 做的所有修改
- 这需要**获取语义**（acquire）和**释放语义**（release）来保证

`__ATOMIC_ACQUIRE` 保证：在获取锁之后的读操作不会被重排到获取锁之前。`__ATOMIC_RELEASE` 保证：在释放锁之前的写操作不会被重排到释放锁之后。这两个语义组合在一起，就建立了线程之间的"同步-与"关系——只要两个线程对同一个锁执行 acquire-release，释放者之前的所有写操作对获取者完全可见。

### PAUSE 指令：自旋等待的节能优化

在自旋循环中，CPU 会疯狂地执行 test-and-set 操作。这在 Intel P4 和 Xeon 处理器上有一个严重问题：由于自旋循环的内存访问模式看起来像是"内存顺序违规"，处理器的内存子系统会触发流水线刷新，导致严重的性能损失。PAUSE 指令（x86 上是 `F3 90`，即 REP NOP）就是给处理器的一个提示："我知道我在自旋等待，别触发那些检测。"

在现代 Intel 处理器上，PAUSE 还有一个额外好处：它让出执行资源给超线程 sibling。虽然 Cinux 目前不关心 HT，但加上 PAUSE 是零成本的正确做法。

## 动手实现

### Step 1: Spinlock 核心数据结构

**目标**: 设计一个最轻量的自旋锁结构。

**设计思路**: 我们只需要一个布尔标志位。初始化为 false（未锁定）。获取时原子地设为 true，释放时原子地设为 false。就这么简单。

**实现约束**:
- 布尔成员变量表示锁状态，初始值 false
- `acquire` 方法：循环调用原子 test-and-set，失败时执行 PAUSE 指令
- `release` 方法：原子地清除标志位
- acquire 使用 `__ATOMIC_ACQUIRE` 语义，release 使用 `__ATOMIC_RELEASE` 语义
- 锁不可拷贝、不可移动

**踩坑预警**: 千万不要用普通的赋值操作来释放锁！`locked_ = false` 不是原子操作，编译器可能把它优化成读-改-写序列。必须用 `__atomic_clear` 或等价的原子存储。

**验证**: 暂时还没有可运行的验证——我们需要先完成 RAII 守卫，然后在后续章节中通过测试来验证。

### Step 2: RAII Guard——自动加锁解锁

**目标**: 实现 C++ 风格的 RAII 守卫，让锁的生命周期跟作用域绑定。

**设计思路**: 在内核开发中，手动配对 acquire/release 是 bug 的温床——尤其当函数有多个返回路径时，很容易忘记某个分支没释放锁。RAII（Resource Acquisition Is Initialization）是 C++ 解决这类问题的标准手段：构造函数获取锁，析构函数释放锁，无论控制流怎么离开作用域（正常返回、提前 return、甚至异常——虽然我们内核里禁用异常），锁都一定会被释放。

**实现约束**:
- Guard 类：构造时调用 Spinlock 的 acquire，析构时调用 release
- 删除拷贝构造和拷贝赋值（防止锁被意外复制）
- Spinlock 提供一个 `guard()` 工厂方法返回 Guard 对象，返回类型标记 `[[nodiscard]]` 防止写了 `lock.guard()` 却没保存返回值
- Guard 的生命周期就是临界区的范围

**踩坑预警**: `[[nodiscard]]` 属性非常重要。如果你写了 `lock.guard()` 但没把返回值赋给一个变量，编译器不会报错，但守卫对象是临时对象，在这一行结束时就会被析构——锁立刻就被释放了，你的临界区根本没被保护。加上 `[[nodiscard]]` 后，编译器会对这种用法发出警告。

**验证**: 同样暂时不单独验证，我们在 Step 4 中一起测试。

### Step 3: IrqGuard——中断安全守卫

**目标**: 实现一种组合守卫，在获取自旋锁的同时禁用中断。

**设计思路**: 这是内核开发中最常见的锁使用模式。想象一下：线程 A 正在操作调度器运行队列，持有自旋锁，此时一个时钟中断到来。中断处理程序也试图获取同一个自旋锁来修改运行队列——但锁已经被当前线程持有了，中断处理程序开始自旋。问题是，中断处理程序不可能被调度出去（它在当前线程的上下文中运行），而持有锁的线程又被中断打断了无法继续执行。死锁。

解决方案：在获取自旋锁之前先禁用中断。x86 上这需要两步：保存当前的 RFLAGS 寄存器（其中包含 IF 中断使能标志位），然后执行 `cli` 指令清除 IF。释放时反过来：先释放锁，再用保存的 RFLAGS 执行 `popfq` 恢复中断状态。

这里有一个微妙但关键的细节——`popfq` 恢复的是保存时的 RFLAGS，而不是简单地执行 `sti` 开中断。如果中断在守卫创建之前就已经被禁用了（比如外层已经有一个 IrqGuard），那内层守卫析构时不应该打开中断。`pushfq`/`popfq` 天然支持这种嵌套。

**实现约束**:
- IrqGuard 构造时：保存 RFLAGS 到成员变量，执行 cli，然后获取自旋锁
- IrqGuard 析构时：释放自旋锁，用保存的 RFLAGS 执行 popfq
- 保存 RFLAGS 用 `pushfq; popq %0` 内联汇编，恢复用 `pushq %0; popfq`
- saved_flags 成员的类型是无符号 64 位整数
- Spinlock 提供 `irq_guard()` 工厂方法返回 IrqGuard

**踩坑预警**: 构造和析构的顺序很重要。构造时必须先保存 flags 再 cli 再获取锁；析构时必须先释放锁再恢复 flags。如果析构时先恢复 flags 再释放锁，那在锁还没释放的时候中断就被打开了，中断处理程序可能立刻尝试获取同一个锁——死锁。虽然先释放锁再恢复 flags 之间有一个极小的窗口中断可能到来，但由于锁已经释放了，中断处理程序可以成功获取锁，不会死锁。

**验证**: 暂无单独验证。

### Step 4: InterruptGuard——独立的中断禁用守卫

**目标**: 有时候我们不需要自旋锁，只需要暂时禁用中断（比如保护一个简单的标志位或环形缓冲区）。

**设计思路**: InterruptGuard 是 IrqGuard 的简化版——只做 pushfq/cli/popfq，不涉及自旋锁。它适用于数据只在中断处理程序和普通代码之间共享的场景：比如键盘驱动的环形缓冲区，IRQ1 写入端和 poll 读取端之间不需要锁（单核上同一时刻只有一个在执行），只需要确保 poll 读取时不会被 IRQ1 打断。

**实现约束**:
- 构造时保存 RFLAGS 并执行 cli
- 析构时用保存的 RFLAGS 执行 popfq
- 支持嵌套（天然支持，因为每次 pushfq 保存的都是当前的完整 RFLAGS）
- 不可拷贝

**验证**: 完成以上所有 Step 后，我们可以编写一个简单的测试来验证：

1. 读取当前的 RFLAGS，记录 IF 位的状态
2. 创建一个 InterruptGuard
3. 读取 RFLAGS，确认 IF 位被清除
4. 创建嵌套的 InterruptGuard
5. 读取 RFLAGS，确认 IF 位仍然被清除
6. 销毁内层守卫
7. 读取 RFLAGS，确认 IF 位仍然被清除（因为外层保存的就是清除状态）
8. 销毁外层守卫
9. 读取 RFLAGS，确认 IF 位恢复到初始状态

## 构建与运行

在 CMakeLists.txt 中确保 `kernel/proc/sync.cpp` 被编译到内核中。这个文件在之前的 tag（021）中已经存在了，我们只是给它添加了 IrqGuard 和 InterruptGuard 的实现。

构建命令：

```
cmake --build build --target big_kernel_test
```

运行测试内核：

```
qemu-system-x86_64 -cdrom build/big_kernel_test.iso -serial stdio -display none
```

## 调试技巧

**问题 1: 死锁——系统挂住无输出**
最常见的原因是 IrqGuard 析构顺序错误，或者在持有锁的情况下调用了也会获取同一个锁的函数。排查方法：在 acquire 和 release 中加入 kprintf 调试输出（注意调试输出本身不能在持锁状态下调用，否则会递归获取 kprintf 用到的锁——如果有的话）。

**问题 2: 中断状态被意外修改**
如果发现中断莫名被禁用，检查所有 IrqGuard/InterruptGuard 的配对情况。最容易出错的是函数提前返回时漏掉了守卫的析构——这也是为什么 RAII 比手动 pushf/cli/popf 安全得多。

**问题 3: Host 端编译报错 "pushfq" 不可用**
Host 端测试需要单独的 stub 实现，不能使用 x86 内联汇编。Host 端的 Spinlock 用 `std::atomic` 的 `exchange`/`store` 替代，InterruptGuard 直接做空操作（Host 没有 x86 中断概念）。

## 本章小结

| 概念 | 关键点 |
|------|--------|
| 自旋锁 | 原子 test-and-set + PAUSE 自旋，最底层的互斥原语 |
| `__ATOMIC_ACQUIRE`/`__ATOMIC_RELEASE` | 内存序语义，建立 happens-before 关系 |
| RAII Guard | 构造获取锁、析构释放锁，防止遗漏 |
| IrqGuard | 先关中断再获取锁，防止中断处理程序死锁 |
| InterruptGuard | 独立的禁中断守卫，用于简单的中断安全场景 |
| 嵌套安全 | pushfq/popfq 保存完整 RFLAGS，天然支持嵌套 |
