# 028-4 Hands-on: VFS 集成与 Shell 验证

## 导语

前两章我们把 ext2 驱动的核心功能全部实现了——mount 能读出超级块和根 inode，read 能通过直接块和间接块读文件数据，readdir 能遍历目录项链表，lookup 能逐级解析路径。但这些能力目前只存在于 Ext2 类的内部方法中，Shell 的 ls/cat 命令走不到这里——它们通过系统调用进入内核，经过 VFS 层的路径解析和挂载点匹配，最终才到达后端文件系统的操作。换句话说，我们造了一台发动机，但还没把它接到轮子上。

这一章要做的就是"接轮子"——让 Ext2FileOps 和 Ext2DirOps 实现 InodeOps 接口（上一章其实已经写了，这里我们要确认它们被正确注册），把 ext2 文件系统挂载到 VFS 命名空间中，然后用 Shell 的 ls 和 cat 命令做端到端验证。完成本章后，你应该能在 Cinux 的 shell 里输入 `ls /` 看到 ext2 根目录下的文件列表，输入 `cat /hello.txt` 看到 "Hello from ext2!" 被打印出来。至此，tag 028 的完整目标——Cinux 能够从真实磁盘文件系统读写文件——就达成了。

知识前置：028-2/028-3（ext2 驱动核心实现）、027（VFS 框架、挂载点表、系统调用）、024（Shell 基础框架）。

## 概念精讲

### Ext2FileOps 和 Ext2DirOps —— 连接 ext2 和 VFS 的桥梁

在 tag 027 中，VFS 定义了 InodeOps 接口——包含 read、write、readdir、stat 等虚函数。ext2 驱动提供了两个 InodeOps 子类：Ext2FileOps 用于普通文件（实现 read、write、stat），Ext2DirOps 用于目录（实现 readdir、stat）。这两个类的构造函数都接收一个 Ext2 引用，所有的操作最终都委托给 Ext2 类的 read_block、read_disk_inode 等底层方法。

这里有一个关键的设计细节：每个 VFS Inode 的 ops 指针在 populate_vfs_inode 中被设置。当磁盘 inode 的类型是 Regular 时，ops 指向 Ext2FileOps 实例；当类型是 Directory 时，ops 指向 Ext2DirOps 实例。系统调用处理函数在拿到 Inode 之后，通过 `inode->ops->read(...)` 或 `inode->ops->readdir(...)` 调用对应的 ext2 实现——整个过程对系统调用层是透明的，它不知道也不需要知道底层是 ext2。

### 挂载到 VFS 命名空间

ext2 驱动的 mount() 方法负责从磁盘读出元数据并初始化内部状态，但"挂载到 VFS"是另一回事——这需要调用 vfs_mount_add 把 Ext2 对象注册到 VFS 的挂载点表中。挂载点的路径通常是 "/"，意味着根文件系统由 ext2 提供。挂载后，当 sys_open 调用 vfs_resolve 解析路径时，VFS 会匹配到 "/" 这个挂载点，把路径前缀剥离后调用 Ext2::lookup 来查找文件。

一个需要注意的顺序问题：Ramdisk（如果还在使用的话）和 ext2 不能同时挂载到同一个路径。我们的做法是在初始化时先尝试 ext2——如果 mount 成功就用 ext2 作为根文件系统，如果失败（比如没有 AHCI 磁盘或 Superblock 无效）则回退到 Ramdisk。这种 fallback 策略保证了内核在各种 QEMU 配置下都能启动。

### Shell 命令如何到达 ext2

以 `cat /hello.txt` 为例，完整的调用链是这样的：用户态 shell 解析命令后调用 sys_open("/hello.txt", O_RDONLY)，这触发 syscall 进入内核；内核的 sys_open handler 调用 vfs_resolve("/hello.txt")，VFS 匹配到 "/" 挂载点上的 Ext2 对象，调用 ext2.lookup("hello.txt")；ext2 的 lookup 在根目录（inode 2）的数据块中搜索 "hello.txt" 条目，找到后读取该条目对应的 inode，放入 cache 并返回 VFS Inode 指针；sys_open 把这个 Inode 包装成 File 对象放入 FDTable，返回 fd=3；用户态 shell 拿到 fd 后循环 sys_read(fd=3, buf, 256)，每次 read 最终走到 Ext2FileOps::read，通过块指针从磁盘读出文件数据；shell 把读到的内容通过 sys_write(fd=1) 打印到串口。

ls 命令的路径类似，区别在于它打开的是目录，调用的是 sys_getdents（内部走 Ext2DirOps::readdir），每次 readdir 调用返回一个目录项的名称。

### QEMU 磁盘配置

要让 QEMU 提供 ext2 磁盘给内核，需要在启动命令中配置 AHCI 设备。我们使用 `-drive` 和 `-device` 参数的组合：先定义一个 drive（指向 ext2.img），然后创建一个 AHCI 控制器，QEMU 会自动把 drive 连接到 AHCI 的第一个端口。内核的 AHCI 驱动（tag 025）会在 PCI 枚举阶段发现这个控制器并初始化端口，ext2 驱动随后通过 port_index=1 访问磁盘。

## 动手实现

### Step 1: 确认 InodeOps 实现完整

**目标**: 确保 Ext2FileOps 和 Ext2DirOps 的所有方法都已正确实现，并且能通过 InodeOps 虚函数表被调用。

**设计思路**: Ext2FileOps 提供 read（文件数据读取）、write（文件写入）、stat（获取文件元数据）三个方法。Ext2DirOps 提供 readdir（目录项遍历）、create（创建文件）、mkdir（创建目录）、unlink（删除条目）、stat 五个方法。每个方法首先检查参数合法性，然后通过 fs_private 指针获取 Ext2CachedInode，从中拿到磁盘 inode，最后调用 Ext2 的底层方法完成实际操作。

**实现约束**: create/mkdir/unlink 这三个写操作委托给 Ext2 类的 create/mkdir/unlink 方法，这些方法内部会处理 inode 分配、位图更新、目录项插入/删除、元数据写回等全套操作。stat 方法把磁盘 inode 的字段（mode、uid、gid、size、blocks、时间戳）填充到 struct stat 中返回。

**验证**: 编译通过，无未定义符号。

### Step 2: 在内核初始化中挂载 ext2

**目标**: 在内核的 main 函数或初始化序列中，在 AHCI 初始化之后，创建 Ext2 对象并调用 mount()，成功后调用 vfs_mount_add 挂载到 VFS。

**设计思路**: 初始化顺序是：PMM -> VMM -> 堆 -> AHCI -> ext2 mount -> vfs_mount_add。Ext2 对象可以是全局静态变量也可以在堆上 new 出来——我们的做法是使用静态全局变量，避免堆分配失败的风险。mount 成功后打印挂载信息，失败则打印警告并尝试 Ramdisk 回退。

**实现约束**: Ext2 构造函数需要传入 AHCI 引用和端口号（通常是 1）。mount() 内部会调用 ensure_dma_buffer，所以不需要单独的 buffer 初始化。挂载路径使用 "/"——这是唯一的根文件系统挂载点。

**踩坑预警**: 确保在调用 vfs_mount_add 之前 vfs_mount_init 已经被调用过（初始化挂载点表）。如果挂载点表没有初始化，vfs_mount_add 会崩溃。另外，如果 Ramdisk 和 ext2 都挂载到 "/"，后挂载的会覆盖先挂载的——所以要注意挂载顺序。

**验证**: 编译并在 QEMU 中运行：
```bash
cd build && cmake .. && make big_kernel_common
qemu-system-x86_64 -serial stdio -kernel big_kernel.bin \
  -drive file=../ext2.img,format=raw,if=none,id=ahci0 \
  -device ahci,id=ahci_ctlr 2>&1 | grep -E "\[EXT2\]|\[VFS\]"
```

预期看到 ext2 挂载信息以及 VFS 挂载点注册信息。

### Step 3: Shell ls 命令验证

**目标**: 在 QEMU shell 中执行 `ls /`，确认能看到 ext2 根目录下的文件和目录。

**设计思路**: ls 命令调用 sys_open("/") 打开根目录，然后循环 sys_getdents 读取目录项名称并打印。由于 ext2 的 readdir 在索引 0 和 1 分别返回 "." 和 ".."，你会在输出中先看到这两个特殊条目，然后是实际文件名（hello.txt、readme.txt、etc）。

**验证**: 在 QEMU shell 中输入：
```
ls /
```
预期输出类似：
```
.
..
hello.txt
readme.txt
etc
lost+found
```

### Step 4: Shell cat 命令验证

**目标**: 在 QEMU shell 中执行 `cat /hello.txt` 和 `cat /etc/passwd`，确认能正确读取 ext2 文件内容。

**设计思路**: cat 命令调用 sys_open 打开文件，循环 sys_read 读取内容并打印到串口，读完后 sys_close。对于嵌套路径（如 /etc/passwd），VFS 层的路径解析会逐级分解——先在根目录找 "etc"（得到目录 inode），再在 etc 目录里找 "passwd"（得到文件 inode），最终返回文件的 VFS Inode。

**验证**: 在 QEMU shell 中输入：
```
cat /hello.txt
cat /etc/passwd
```
预期输出：
```
Hello from ext2!
root:x:0:0
```

## 踩坑预警——常见错误清单

这一节我们把从 028-1 到 028-4 整个过程中最容易踩的坑汇总一下，方便遇到问题时快速排查。

**Magic 不匹配（0x0000 或乱码）**: 检查 QEMU 的 -drive 和 -device 参数是否正确配置了 AHCI 设备。如果 AHCI 控制器没有正确创建，或者 ext2.img 文件路径有误，内核读到的全是垃圾数据。用 `qemu-system-x86_64 ... 2>&1 | grep AHCI` 确认 AHCI 初始化信息。

**inode 编号从 1 开始的问题**: 所有计算 `group = (ino-1) / inodes_per_group` 和 `index = (ino-1) % inodes_per_group` 的地方，必须先减 1。如果你在某些地方忘了减 1，inode 2 会被错误地定位到 group 0 的第 2 个 inode（而不是第 1 个），导致所有路径解析都偏移一格。这个 bug 的特征是：ls 能看到文件名但 cat 读不到内容（或者读到错误的内容），因为文件名来自目录项（正确），但文件 inode 定位错误。

**rec_len vs name_len**: 遍历目录项时必须用 rec_len 步进。如果你用 name_len + 8 步进，在某些有条目填充的目录上会跳过有效条目或读到错误位置。这个 bug 的特征是 ls 输出缺少某些文件名，或者 ls 只显示 "." 和 ".." 就结束了。

**block_size 是左移值**: Superblock 的 s_log_block_size 不是块大小本身，而是一个指数——block_size = 1024 << s_log_block_size。如果你直接把 s_log_block_size 当块大小用，sectors_per_block 会算成 0（或 1），read_block 会只读一个扇区（512 字节）而不是一个完整的块。

**DMA buffer 单页限制**: 我们的 DMA buffer 只有一个物理页（4KB）。如果将来块大小设置为 4096，read_block 刚好用满整个 buffer——没问题。但如果需要同时缓存多个块的数据（比如读间接块时需要保留之前的块指针），就必须在两次 read_block 之间把需要的数据拷贝出来，否则第二次 read_block 会覆盖第一次的结果。

**Ext2 对象的生命周期**: Ext2 对象必须是全局静态的或者在内核堆上 new 出来的——绝对不能是栈变量。因为 VFS 挂载点表只存了一个指针，如果 Ext2 对象在函数返回后被销毁，所有后续的 VFS 操作都会访问已释放的内存。

## 构建与运行

完整的端到端验证流程：

```bash
# 1. 准备 ext2 镜像（如果还没有）
cd /path/to/cinux
dd if=/dev/zero of=ext2.img bs=1M count=8
mkfs.ext2 -b 1024 ext2.img
sudo mount -o loop ext2.img /mnt/ext2test
echo "Hello from ext2!" | sudo tee /mnt/ext2test/hello.txt
echo "Cinux OS kernel" | sudo tee /mnt/ext2test/readme.txt
sudo mkdir /mnt/ext2test/etc
echo "root:x:0:0" | sudo tee /mnt/ext2test/etc/passwd
sudo umount /mnt/ext2test

# 2. 构建内核
cd build && cmake .. && make big_kernel_common

# 3. 运行 QEMU
qemu-system-x86_64 -serial stdio -kernel big_kernel.bin \
  -drive file=../ext2.img,format=raw,if=none,id=ahci0 \
  -device ahci,id=ahci_ctlr

# 4. 在 shell 中测试
# ls /
# cat /hello.txt
# ls /etc
# cat /etc/passwd
```

## 调试技巧

**ls / 输出只有 "." 和 ".."**: 用 kprintf 在 Ext2DirOps::readdir 中打印遍历过程——看到多少个块、每个块遍历了多少字节、遇到了多少个有效条目。如果 total_blocks 是 0，说明目录 inode 的 i_size 不对。

**cat /hello.txt 返回 -1**: 在 sys_open 里打印 vfs_resolve 的返回值。如果返回 nullptr，检查 vfs_mount_add 是否被正确调用——也许挂载路径不是 "/" 而是别的，或者挂载还没发生就已经有进程在尝试打开文件了。

**cat /hello.txt 输出为空**: 在 Ext2FileOps::read 中打印 offset、to_read 和 total_read。如果 total_read 是 0 但 i_size > 0，说明块指针解析有问题——检查直接块指针是否为 0（如果是，返回全零）或者间接块读取是否成功。

**内核启动时卡死（无串口输出）**: 可能是 AHCI 初始化或 ext2 mount 在 DMA buffer 分配时卡住了。在 ensure_dma_buffer 中每一步都加打印，确认 PMM alloc_page 和 VMM map 是否正常返回。

**QEMU 退出时 ext2.img 被损坏**: 如果你在内核中实现了写操作（create/mkdir/unlink），异常退出可能导致元数据不一致。重新 mkfs.ext2 即可恢复。对于只读测试来说，镜像不会被修改。

## 本章小结

| 概念 | 要点 |
|------|------|
| Ext2FileOps | InodeOps 子类，提供 read/write/stat，ops 挂在 Regular 类型 inode 上 |
| Ext2DirOps | InodeOps 子类，提供 readdir/create/mkdir/unlink/stat |
| VFS 挂载 | vfs_mount_add("/", &ext2)，路径解析时自动匹配到 ext2 |
| 调用链 | sys_open → vfs_resolve → ext2.lookup → lookup_in_dir → 目录项匹配 |
| QEMU 配置 | -drive + -device ahci，ext2.img 作为 AHCI 磁盘 |
| 常见陷阱 | ino 从 1 开始、rec_len 步进、block_size 左移、DMA 单页覆盖 |
