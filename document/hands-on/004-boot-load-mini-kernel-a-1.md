## 导语

上一章我们费了九牛二虎之力从保护模式切到了 Long Mode，在 debugcon 里打出了一个漂亮的 `L` 字符。但说实话，光进 Long Mode 还不算完——CPU 是跑起来了，可我们连一行内核代码都没有加载。现在的状况就好比你把房子盖好了、电也通了，但家具还没搬进来，住不了人。这一章我们要做的是：在 Real Mode 还没离开之前，把两件依赖 BIOS 中断的关键事情办了——探测系统内存布局（E820）和从磁盘把 Mini Kernel 的 ELF header 读进内存（INT 13h）。完成之后，Bootloader 就可以在后续 tag 中解析 ELF header、做完整内核加载、最终跳转到 Mini Kernel 执行。

## 概念精讲

### E820 内存探测：给物理内存画一张地图

你可能会问：为什么我们需要知道内存布局？内存不就是"有多少用多少"吗？在应用程序的世界里确实是这样——操作系统把内存管理得妥妥帖帖，`malloc` 直接用就好。但在写操作系统内核的时候，没有人帮你管内存了，你必须自己知道哪些物理地址是可用的 RAM、哪些是硬件保留的、哪些是 BIOS 数据区。如果你不小心把内核的数据写到了硬件保留区，轻则数据丢失，重则硬件直接不响应，最惨的是那种"有时候能用有时候不行"的随机崩溃——因为保留区里有些地址是 memory-mapped I/O，写进去的行为取决于硬件当前状态，完全不可预测。

BIOS 提供了好几种内存探测方法，但 E820（INT 0x15, AX=0xE820）是其中最强大、信息最完整的，也是现代操作系统通用的选择。你可以把它理解为一次"迭代式地图测绘"——每次调用返回一条记录（描述一段连续内存区域的基地址、长度和类型），通过一个续接值（EBX）告诉你"下一次从哪里继续"，直到续接值变为 0 表示枚举结束。整个过程就像你在翻一本通讯录：第一次翻开第一页，每翻一页记下一个人的信息，直到翻到最后一页。

```
  调用流程（迭代式）：

  初始化:  EBX = 0（第一次调用）
           +------------------------------------------+
  循环 --> |  EAX = 0x0000E820                        |
           |  ECX = 24（buffer 大小，ACPI 3.0）        |
           |  EDX = 'SMAP'（签名）                     |
           |  ES:DI = 指向输出缓冲区                   |
           |  INT 0x15                                |
           +------------------------------------------+
                      |
           成功? -----+ CF=0 且 EAX='SMAP'
           失败? -----+ CF=1 或 EAX!='SMAP'
                      |
           +----------+----------+
           |  从 ES:DI 读取结果:  |
           |  base   (8 字节)    |
           |  length (8 字节)    |
           |  type   (4 字节)    |
           |  acpi   (4 字节)    |
           |                     |
           |  EBX = 0? --- 结束  |
           |  EBX != 0? -- 继续   |
           +---------------------+
```

每条记录的 type 字段告诉你这段内存的性质。对我们来说最关键的是 type=1（可用 RAM，就是内核可以放心使用的内存）和 type=2（保留区，绝对不能碰）。QEMU 默认会返回 6 到 7 条记录，其中只有 type=1 的条目是真正可用的。这里有一个非常隐蔽的坑：E820 返回了保留区并不意味着出错了——保留区的存在是正常的，它可能被 BIOS、ACPI 表、或者 memory-mapped I/O 占用。如果你不过滤 type 字段，把所有返回的内存区域都当成可用 RAM 来用，那么恭喜你，你会收获一个看起来能跑但随时可能随机崩溃的系统。这种 bug 的排查难度堪称地狱级——因为它不是每次都崩，只有在恰好用到保留区地址的时候才炸。

Intel SDM Vol.3A 15.3.1 对这个接口有完整定义。几个必须注意的细节：EAX 的完整值必须是 0x0000E820，高 16 位不能省略——如果你只写了 0xE820，高 16 位可能是上一次调用的残留值，BIOS 看到错误的 EAX 就会直接拒绝。ECX 应该设为 24 而不是 20——20 字节是旧版 BIOS 返回的结构大小，24 字节是 ACPI 3.0 扩展后的版本（多了一个 ACPI 属性字段）。虽然大多数 BIOS 即使你传 ECX=20 也能正常工作，但某些严格的 BIOS 实现会因为你传的 buffer 太小而返回不完整的数据，所以老老实实写 24 最安全。

### INT 13h 扩展读取：用 LBA 告别 CHS 的噩梦

磁盘读取是 Bootloader 的另一个核心任务。早期的磁盘读取使用 CHS（Cylinder-Head-Sector）寻址模式——你需要告诉 BIOS 磁盘的哪个柱面、哪个磁头、哪个扇区。这种方式在软盘时代还凑合，但到了硬盘时代就彻底力不从了：CHS 寻址上限只有大约 8GB，而且不同磁盘的几何参数不一样，你还得先探测参数。INT 13h AH=0x42（扩展读取）用 LBA（Logical Block Addressing）彻底解决了这个问题——磁盘被看作一个从 0 开始编号的扇区序列（每个扇区 512 字节），你想读第几个扇区就直说，不用关心物理几何结构。LBA 支持 48 位地址，理论上限约 144 PB，这辈子够用了。

扩展读取使用一个叫 DAP（Disk Address Packet）的数据结构来描述读取请求，总共 16 字节，布局如下。第一个字节是 DAP 自身的大小，固定为 16；第二个字节是保留字段，必须填 0；第三和第四字节组成一个 16 位值，表示要读取的扇区数量；第五和第六字节是目标缓冲区的偏移地址；第七和第八字节是目标缓冲区的段地址；最后八个字节是起始 LBA 号（64 位，低 16 位在前）。注意这些字段的排列顺序和字节序——x86 是小端序，LBA 的低字节存在低地址。

```
  DAP 结构（16 字节）：

  偏移  大小   含义
  ----------------------------------------------
  +0    1B     DAP 大小 = 16
  +1    1B     保留 = 0
  +2    2B     扇区数量（小端序）
  +4    2B     缓冲区偏移
  +6    2B     缓冲区段地址
  +8    8B     起始 LBA（64 位小端序）
```

调用时 DL 寄存器必须设为 0x80（第一块硬盘的 BIOS 设备号），AH 设为 0x42（扩展读取功能号），DS:SI 指向 DAP 结构——注意这里用的是 DS:SI 而不是 ES:DI，别和 E820 的 ES:DI 搞混了。成功返回时 CF=0 且 AH=0。有一个非常阴险的坑：某些 BIOS 实现在你忘记设 DL=0x80 的时候，会返回 AH=0x01（无效功能），但实际上数据可能已经读进缓冲区了——你以为是失败了，结果数据已经在了，只是设备号不对。所以如果你在调试时发现"明明返回错误但内存里的数据看起来是对的"，先检查 DL 寄存器。

本章我们只读取 ELF header 的前 4KB（8 个扇区），所以不需要分批读取。后续 tag 004B 在做完整内核加载时才会遇到 BIOS 单次读取上限（通常是 127 个扇区）的限制，届时需要分批循环读取。

### Real Mode 段地址计算：一个折磨了无数人的细节

如果你已经做了前面几章，对 Real Mode 的段地址机制应该不陌生，但这个知识点实在太容易出错了，值得再强调一遍。在 Real Mode 下，CPU 访问内存用的是"段地址:偏移"的表示法，物理地址的计算公式是 `segment << 4 + offset`。比如你想访问物理地址 0x5000，段地址应该是 0x0500（0x0500 << 4 = 0x5000），偏移是 0x0000。如果你手滑把段地址写成了 0x5000，那实际访问的物理地址就变成了 0x50000，差了十倍——而且这种错误编译器不会报、链接器不会报、运行时也不一定立刻崩溃（因为 0x50000 处的内存可能恰好可写），只有等到后续代码去 0x5000 读数据发现全是零的时候才会暴露，排查起来非常痛苦。

### 为什么必须在 Real Mode 做

这是一个很自然的问题：既然我们最终要在 Long Mode 里运行内核，为什么 E820 和磁盘读取不能留到 Long Mode 再做？答案很简单——因为 BIOS 中断只在 Real Mode 下可用。一旦我们切换到保护模式（更别说 Long Mode 了），INT 0x15 和 INT 0x13 就彻底失效了——保护模式下没有中断向量表来路由这些软中断。有些人可能会想"那我可以在保护模式下切回实模式再调用 BIOS"，理论上可以，但实际上这比直接在实模式做完要复杂得多：你需要保存保护模式的状态、切回实模式、调 BIOS、再切回保护模式、恢复状态——每一步都可能出错。所以在实模式阶段把所有依赖 BIOS 的事情一次性做完，是最简单也最可靠的方案。

## 动手实现

### Step 1: 创建公共引导模块

**目标**：创建 `boot/common/boot.S` 文件，作为 Real Mode 下内存探测和磁盘读取的公共模块。

**设计思路**：上一章我们把 Long Mode 相关代码独立成了 `boot/common/long_mode.S`，同样的思路在这里也适用。E820 和 INT 13h 的功能是通用的——不管以后 Bootloader 怎么演进，只要还需要通过 BIOS 获取信息，这两个函数就会一直存在。把它们从 stage2.S 中剥离出来，一方面避免 stage2.S 无限膨胀，另一方面也让代码职责更清晰：stage2.S 负责流程编排（先做什么后做什么），boot.S 负责具体的硬件操作。这个文件用 `.code16` 标记，因为所有操作都在 Real Mode 下进行。

**实现约束**：文件需要声明两个 `.global` 函数——`query_memory_map` 和 `load_kernel_from_disk`。错误处理标号 `disk_read_failed` 和 `disk_too_large` 作为 panic 跳转目标。错误消息字符串（E820 失败、磁盘读取失败、内核过大）放在 `.rodata` 段。文件头部需要用 `.extern` 声明外部依赖的函数和变量（panic 错误处理函数等）。

**验证**：文件创建后，确认路径正确即可，构建验证在后续步骤完成。

### Step 2: 定义 E820 相关常量

**目标**：在 `boot/common/boot.S` 开头用 `.set` 伪指令定义 E820 所有的魔数和地址常量。

**设计思路**：GNU AS 不能直接 `#include` C 头文件，所以常量必须用 `.set` 在汇编文件里重新定义。这些常量需要和 `boot/boot_info.h` 中的定义保持一致——如果两边对不上（比如 E820 条目大小一边是 24 一边是 20），内核读到的数据就会错位，后续内存管理全部乱套。常量分为四组：缓冲区地址（0x5000 区域的段地址和偏移分解）、MemoryMapEntry 结构的字段偏移（base=0、length=8、type=16、size=24）、BIOS 调用魔数（E820 签名 'SMAP' = 0x534D4150、功能号 0x0000E820），以及配置限制（最多 32 条记录）。

**实现约束**：GNU AS 的 `.set` 语法是 `.set SYMBOL, VALUE`。E820 的缓冲区我们固定放在物理地址 0x5000——这个地址是安全区域（BIOS 数据区之上、Bootloader 代码之下）。前 4 字节存放条目计数（count），接下来从偏移 4 字节开始存放条目数组，每条 24 字节，最多 32 条。段地址 0x0500（不是 0x5000！）对应物理地址 0x5000。

**踩坑预警**：这里有一个我们实打实踩过的坑——E820_CMD 的值必须写成 0x0000E820，不能偷懒只写 0xE820。在 AT&T 汇编里，`movl $0xE820, %eax` 会被汇编器截断成 16 位立即数然后零扩展到 32 位，所以 0xE820 实际上就是 0x0000E820——在 GAS 里这碰巧是对的。但如果你换了一个汇编器（比如 NASM），行为可能不同。为了保险和可读性，始终写完整的 32 位值 0x0000E820。另一个坑是 ECX 的值：必须传 24（ACPI 3.0 扩展大小），不是 20（旧版大小）。某些 BIOS 在你传 20 的时候会返回不完整的条目（缺 ACPI 字段），虽然大多数情况下这不会导致功能问题，但为了数据的完整性，24 是更安全的选择。

**验证**：常量定义不需要单独构建验证，但可以和后续步骤一起验证。

### Step 3: 实现 query_memory_map 函数

**目标**：实现 E820 内存枚举函数，将结果保存到 0x5000 缓冲区。

**设计思路**：函数的整体结构是一个循环——初始化续接值 EBX=0，然后把 ES:DI 指向缓冲区（跳过前 4 字节的 count 字段），每次循环调用 INT 0x15，成功后 DI 前进 24 字节指向下一条目的位置，count 加 1，检查 EBX 是否归零决定是否继续。循环退出后，count 的值拷贝到 CX 作为返回值，方便调用者知道有多少条目。函数开头用 `pushaw` 保存所有 16 位通用寄存器（包括 AX、BX、CX、DX、SI、DI、BP），以及 `pushw %es` 和 `pushw %ds`，因为我们会修改 ES（指向缓冲区）和 DS（指向 count 字段）。退出时对称恢复。

在循环体内，每次调用前需要重新加载 EDX='SMAP' 和 EAX=0x0000E820——因为 BIOS 可能会破坏这些寄存器的值（Intel SDM 明确说 BIOS 可能修改 EDX），不能假定它们在循环间保持不变。ECX=24 也需要每次重设。调用后检查三个条件：CF=0（没有进位，表示调用成功）、EAX='SMAP'（BIOS 签名确认）、ECX>=20（BIOS 至少返回了 20 字节的数据）。三个条件全部满足才算成功，任何一个不满足就跳转到错误处理。

**实现约束**：ES 需要设为 E820 条目数组的段地址（0x0500），DI 初始值为 0x0004（跳过前 4 字节的 count）。DS 也需要设为 0x0500，因为 count 字段在物理地址 0x5000，也就是段地址 0x0500 偏移 0x0000 处。条目数组从偏移 0x0004 开始。循环计数用物理地址 0x5000 处的 32 位 count 字段，而不是寄存器——这样设计是为了让内存探测结果即使在函数返回后也持久保存在固定地址，后续内核启动时可以直接从 0x5000 读取。最大条目数限制为 32，超过后强制退出，防止缓冲区溢出。

**踩坑预警**：最大的坑是段寄存器的设置。要访问物理地址 0x5000，段地址是 0x0500——但如果你手滑写成了 0x5000，CPU 会去访问物理地址 0x50000（320KB 位置），那里可能恰好也是可写内存，不会立刻崩溃，但写入的数据和后续读取的地址完全对不上。这个坑我们在开发的时候踩过，排查了半天才发现是一个 0 的问题。另一个容易犯的错误是 DI 的递增量——每条记录 24 字节，所以 DI 每次要加 24（E820_ENTRY_SIZE），不是 20。

**验证**：在 GDB 中可以在 `query_memory_map` 返回后检查 0x5000 处的数据。`x/4bx 0x5000` 查看 count（应该是 QEMU 返回的条目数，通常 6-7），`x/24bx 0x5004` 查看第一条记录的 base、length、type 字段。

### Step 4: 定义磁盘读取常量和 DAP 结构

**目标**：定义 INT 13h 扩展读取所需的所有常量，包括 DAP 结构的字段偏移、磁盘布局参数和加载地址。

**设计思路**：磁盘读取的常量包括 DAP 结构的 6 个字段偏移量（size=0、reserved=1、count=2、buffer_offset=4、buffer_segment=6、lba=8）、DAP 在低内存中的固定位置（0x7B00，MBR 用完后可安全重用），以及磁盘布局参数。本章只读 ELF header 的 8 个扇区（4KB）到物理地址 0x10000，对应段地址 0x1000、偏移 0x0000。0x10000 这个位置是经典的"临时加载区"——xv6 和 Linux 0.01 都把 ELF header 读到这个地址，它不与 MBR、Stage2、E820 缓冲区、VESA 缓冲区冲突。

**实现约束**：DAP 结构放在物理地址 0x7B00，段地址 0x07B0。Mini Kernel 的起始 LBA 是 16（前面 1 个扇区给 MBR、15 个扇区给 Stage2），读 8 个扇区。加载到物理地址 0x10000，对应段地址 0x1000、偏移 0。DAP 的 size 字段必须为 16，reserved 必须为 0。

**踩坑预警**：DAP 结构里的 LBA 字段是 64 位的，高 32 位必须显式清零——不能假定内存里是干净的。如果 DAP 所在的 0x7B00 区域有残留数据，高位的垃圾值会让 BIOS 尝试读取一个天文数字般的 LBA，直接返回错误。

**验证**：常量定义和后续步骤一起验证。

### Step 5: 实现 load_kernel_from_disk 函数

**目标**：实现磁盘读取函数，将 Mini Kernel 的 ELF header（前 4KB）从磁盘 LBA 16 读取到物理内存 0x10000。

**设计思路**：函数的结构很直接——构建 DAP 结构（设置 size=16、count=8、buffer 地址 0x1000:0x0000、LBA=16 且高 32 位清零），然后调用 INT 0x13 AH=0x42，检查 CF 和 AH 判断成功与否。函数用 `pushaw`/`popaw` 保存/恢复 16 位寄存器，以及单独保存 ES 和 DS。成功后 AX 返回读取的扇区数（8），失败则跳转到 `disk_read_failed` 打印错误信息并停机。

这种一次性读取 8 个扇区到临时区域的设计是刻意的。本章的目标不是加载完整的内核（那需要在保护模式下做，因为 Real Mode 有 1MB 内存限制），而是拿到 ELF header 的前 4KB。有了这 4KB，后续 tag（004B）就可以在保护模式下解析 Program Header Table、获取内核总大小、做有针对性的完整加载。

**实现约束**：DAP 构建时先设 ES 指向 0x07B0，DI 指向 0x0000，然后逐字段写入。INT 13h AH=0x42 要求 DS:SI 指向 DAP（不是 ES:DI），DL=0x80 表示第一块硬盘。BIOS 调用可能破坏寄存器，所以 DAP 构建和参数设置都在调用前完成。成功标志是 CF=0 且 AH=0。

**踩坑预警**：有三个我们实打实踩过的坑。第一个是 DS:SI vs ES:DI 的混淆——INT 13h AH=0x42 要求 DAP 通过 DS:SI 传递，不是 ES:DI。如果你按 E820 的习惯设了 ES:DI 然后调 INT 0x13，BIOS 会从错误的地址读取 DAP，返回一个莫名其妙的错误码。第二个是 DL 必须设为 0x80——如果 DL 里恰好还是之前 E820 调用留下的残留值，BIOS 返回 AH=0x01（Invalid Function），但神奇的是有些情况下数据其实已经读成功了。CF 才是最终的成功标志。第三个是 LBA 的高 32 位必须显式清零——在 QEMU 里可能碰巧是零，但在真实硬件上几乎必炸。

**验证**：在 GDB 中可以在函数返回后检查 0x10000 处的数据。Mini Kernel 的 ELF 文件头应该出现在这里——`x/4bx 0x10000` 应该看到 `0x7f 0x45 0x4c 0x46`（ELF magic number）。如果看到全零或者乱码，说明磁盘读取有问题。

### Step 6: 在 stage2.S 中集成调用

**目标**：修改 `boot/stage2.S`，在 VESA 初始化完成后、进保护模式之前调用 `query_memory_map` 和 `load_kernel_from_disk`。

**设计思路**：stage2.S 是整个 Bootloader 的流程编排文件——它按照固定顺序调用各个功能模块。在 tag 003 中，stage2 的调用链是：打印消息 -> 开启 A20 -> VESA 初始化 -> 进保护模式 -> 页表设置 -> 进 Long Mode。现在我们需要在"VESA 初始化完成"和"进保护模式"之间插入两个新的调用：先 `query_memory_map` 再 `load_kernel_from_disk`。这个顺序是有讲究的——E820 不依赖磁盘状态，可以先做；磁盘读取也不依赖 E820 结果，但按"先探测后加载"的逻辑顺序排列更清晰。两个函数都必须在 CLI（关中断）之前调用，因为 INT 0x15 和 INT 0x13 都是 BIOS 中断，需要中断处于开启状态。

**实现约束**：在 stage2.S 中用 `.extern` 声明 `query_memory_map` 和 `load_kernel_from_disk`。调用点在 `vesa_save_framebuffer_info` 之后、CLI 之前。两个函数之间不需要额外的寄存器保存——因为 `query_memory_map` 内部已经做了完整的寄存器保存/恢复。调用完成后 stage2 继续原有的保护模式切换流程。

**踩坑预警**：调用时机千万别搞错——必须在 VESA 之后（因为 VESA 还在用 BIOS 中断）且在 CLI 之前（因为后续的 BIOS 调用需要中断开启）。如果你把这两个调用放到了 CLI 之后，`int $0x15` 和 `int $0x13` 就不会执行了——中断被屏蔽了。函数不会报错，但也不会做任何事情，内存里全是零，后续内核启动直接爆炸。

**验证**：用 `objdump -d build/boot/stage2` 反汇编，确认在 VESA 调用和 CLI 之间能看到 `call query_memory_map` 和 `call load_kernel_from_disk`。

### Step 7: 创建 Mini Kernel 占位

**目标**：创建 `kernel/mini/` 目录下的小内核源文件，作为后续 Bootloader 跳转的目标。

**设计思路**：Mini Kernel 在当前 tag（004A）阶段只需要做一个最简单的事情——证明它能被编译和写入磁盘镜像。所以我们先写一个最精简的入口点：一个 `cli; hlt` 的死循环，用 `extern "C"` 包裹以避免 C++ 的 name mangling。现阶段它的唯一意义是让构建系统有一个可编译、可链接、可写入磁盘镜像的 ELF 文件——后续 tag 会让它真正跑起来，加入 SSE 启用、栈设置、BSS 清零等初始化流程。

链接脚本 `kernel/mini/linker.ld` 定义了内存布局：物理加载地址（LMA）是 0x200000（2MB，大页对齐），入口点 `_start`。这个地址在后续 tag 中会配合 higher-half 设计，通过页表映射到虚拟地址 0xFFFFFFFF80020000。

**实现约束**：Mini Kernel 的 C++ 代码使用 `-ffreestanding -fno-exceptions -fno-rtti -mcmodel=large -mno-red-zone` 编译选项。`-ffreestanding` 表示不依赖标准库，`-fno-exceptions` 和 `-fno-rtti` 禁用 C++ 异常和 RTTI（没有标准库支持这些特性），`-mcmodel=large` 允许编译器生成访问任意 64 位地址的代码（因为我们的内核最终会在 higher-half 运行），`-mno-red-zone` 禁用 System V AMD64 ABI 的 red zone 优化（内核代码不能依赖 red zone，因为中断可能在任何时候打断执行）。

**踩坑预警**：`-mcmodel=large` 是必须的，不能用默认的 `-mcmodel=small`。small 模型假设所有符号都在低 2GB 地址空间内，但我们的内核 VMA 是 0xFFFFFFFF80000000，远超 2GB 范围。如果用 small 模型编译，链接器生成的地址引用会截断，运行时访问到错误地址。

**验证**：确认 `kernel/mini/` 目录下有 `main.cpp`、`linker.ld` 和 `CMakeLists.txt`。

### Step 8: 更新构建系统

**目标**：更新 CMake 构建系统和镜像构建脚本，使 Mini Kernel 被编译并打包到磁盘镜像中。

**设计思路**：构建系统的改动分三层。第一层是 Mini Kernel 自身的构建：`kernel/mini/CMakeLists.txt` 定义了 `mini_kernel` 目标，编译 Mini Kernel 的所有源文件，链接生成 ELF 可执行文件，然后用 `objcopy -O binary` 把 ELF 转换成纯二进制文件（`mini_kernel.bin`）。ELF 格式包含了段头信息、符号表等元数据，这些在运行时是不需要的——Bootloader 直接跳到入口点执行，不需要 ELF 加载器。objcopy 的 `-O binary` 选项把所有可加载段按地址顺序提取出来，去掉所有 ELF 元数据，生成一个可以直接写入磁盘镜像的纯二进制文件。

第二层是磁盘镜像构建：`scripts/build_image.sh` 脚本需要知道 Mini Kernel 的二进制文件路径和磁盘布局。磁盘布局是——Sector 0 给 MBR（512 字节），Sector 1-15 给 Stage2（最多 15 个扇区 = 7680 字节），Sector 16 开始给 Mini Kernel。脚本先用 `dd` 创建一个全零的空白镜像，然后依次写入 MBR、Stage2 和 Mini Kernel。写入 Mini Kernel 时用 `bs=512 seek=16` 参数，意思是"跳到第 16 个扇区开始写入"。

第三层是 CMake 的 `cmake/qemu.cmake`：`run` 目标需要依赖 `mini_kernel`——确保每次 `make run` 之前 Mini Kernel 都会被重新编译。`build_image.sh` 的调用参数中需要加入 Mini Kernel ELF 文件的路径。

**实现约束**：Mini Kernel 的编译需要包含 `boot/` 目录到 include 路径中，因为 `main.cpp` 可能引用 `boot_info.h`。链接脚本 `linker.ld` 指定了 `-T` 选项。Post-build 命令使用 `objcopy -O binary` 做 ELF 到 flat binary 的转换。`build_image.sh` 的第三个参数是 Mini Kernel ELF 文件路径。

**验证**：执行完整构建后，确认 `build/kernel/mini/mini_kernel.bin` 文件存在且大小合理（应该至少有几 KB）。确认 `build/cinux.img` 文件存在且大于 4KB（因为包含了 Mini Kernel 的数据）。

### Step 9: 验证完整启动链

**目标**：在 Long Mode 入口中输出 `J` 字符，验证整个启动链条到 ELF header 读取这一步全部正确。

**设计思路**：这是整个 tag 004A 的最后一步。在 Long Mode 入口 `long_mode_entry` 中，我们已经在 003 里输出了 `L` 字符。现在我们追加一个 `J` 字符（0x4A -> debugcon 端口 0xE9）表示"磁盘读取完成，准备在后续 tag 中跳转到小内核"。当前 tag 暂时不会真正跳转——真正的跳转逻辑（解析 ELF、加载段、跳转 entry）留到 004B 和 004C。

**验证**：这是最终验证步骤。完整构建并运行后，debugcon 日志应该包含以下字符序列：`P`（保护模式切换成功）-> `L`（Long Mode 切换成功）-> `J`（磁盘读取完成，准备跳转）。如果缺少某个字符，说明对应的步骤出了问题——缺 `P` 或 `L` 说明前面 tag 的代码有问题，缺 `J` 说明磁盘读取有问题。

## 构建与运行

在项目根目录执行完整构建。确保 CMake 配置正确（Debug 模式），然后构建并运行：

```
cmake -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build
make run
```

QEMU 启动后，串口终端会输出 Bootloader 的启动信息（`Stage2 OK`、`Mode info OK, switching...`）。按 `Ctrl+C` 退出 QEMU（或者等内核进入停机循环），然后检查 debugcon 日志：

```
cat debug.log
```

预期看到完整的字符序列：`PLJ`。其中每个字符的含义是：`P` = 保护模式、`L` = Long Mode、`J` = 磁盘读取完成（准备跳转）。如果缺少某个字符，说明对应的步骤出了问题。

## 调试技巧

### E820 失败排查

如果 E820 调用失败（函数跳转到 `panic`），最可能的原因有三个。第一，段地址设错了——用 GDB 检查调用前 DS 和 ES 的值，应该是 0x0500 而不是 0x5000。第二，EAX 的值不是 0x0000E820——检查常量定义是否正确。第三，ECX 不是 24——某些 BIOS 对这个值很敏感。在 GDB 中可以在 INT 0x15 之前设断点，检查所有寄存器的值是否符合预期：

```
(gdb) break *query_memory_map
(gdb) info registers eax ebx ecx edx es ds
```

### 磁盘读取失败排查

如果磁盘读取失败（debugcon 出现错误信息），检查几个方面。首先确认 DL=0x80——这是最常见的遗漏。然后用 GDB 检查 DAP 的内容是否正确：`x/16bx 0x7B00`，第一个字节应该是 16（DAP 大小），第二字节是 0（保留），第三四字节是扇区数（8），第八字节开始是 LBA。如果 DAP 内容看起来正确但还是失败，试着确认 Mini Kernel 确实被写入了磁盘镜像——用 `xxd build/cinux.img | head -n 100` 查看镜像内容，在 LBA 16（偏移 0x2000）的位置应该能看到 ELF magic。

### 用 GDB 跟踪完整启动链

完整的调试流程是在一个终端启动 QEMU 调试模式，在另一个终端连接 GDB：

```
# 终端 1
make run-debug

# 终端 2
gdb -ex "target remote :1234" build/boot/stage2
(gdb) break query_memory_map
(gdb) break load_kernel_from_disk
(gdb) continue
```

在 `query_memory_map` 返回后检查 E820 数据：

```
(gdb) x/4bx 0x5000
(gdb) x/6gx 0x5004
```

在 `load_kernel_from_disk` 返回后检查加载的内核数据：

```
(gdb) x/4bx 0x10000
```

应该看到 `0x7f 0x45 0x4c 0x46`（ELF magic）。

## 本章小结

| 概念 | 要点 |
|------|------|
| E820 内存探测 | INT 0x15 AX=0x0000E820，迭代调用，EBX 续接，结果存 0x5000 |
| E820 条目格式 | base(8B) + length(8B) + type(4B) + acpi(4B) = 24 字节/条 |
| E820 type 字段 | 1=可用 RAM，2=保留区，必须过滤 type 后再使用 |
| INT 13h 扩展读取 | AH=0x42，DAP 通过 DS:SI 传递（不是 ES:DI） |
| DAP 结构 | 16 字节：size + reserved + count + buffer_seg:off + LBA(64bit) |
| 本章读取策略 | 只读 8 扇区（4KB ELF header）到 0x10000，完整加载留到 004B |
| 段地址计算 | 物理地址 = segment << 4 + offset，0x5000 -> seg=0x0500 |
| Mini Kernel | 当前阶段只有 cli; hlt 死循环，验证构建链路 |
| 验证字符序列 | P -> L -> J |
