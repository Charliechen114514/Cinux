## 导语

上一篇我们给内核装上了串口驱动，现在内核能"说话"了——但只能说固定的字符串。如果你调用 `puts("Hello")`，终端就老老实实显示 Hello，一个字都改不了。这对于调试来说远远不够：你想知道某个变量的值是多少，想看看地址 0x7000 处的 BootInfo 指针是否正确，想输出一个十进制整数或者十六进制地址——这些都需要格式化输出，也就是内核版的 `printf`。

这一章我们要实现一个叫 `kprintf` 的格式化输出函数，它支持 `%d`（有符号十进制）、`%u`（无符号十进制）、`%x`/`%X`（十六进制）、`%s`（字符串）、`%c`（字符）、`%p`（指针地址）、`%b`（二进制）和 `%%`（百分号本身）这些格式符。同时还要支持宽度修饰符（比如 `%04d` 表示四位零填充），这对输出对齐的调试日志很有用。

这个 `kprintf` 的设计有一个比较精巧的地方：格式化逻辑和输出设备是解耦的。格式化函数本身不知道字符最终去了串口还是 debugcon——它只管把格式化结果送到一个"字符输出函数"里。`kprintf` 用串口的 `putc` 作为输出函数，`kdebugf` 用 debugcon 的 `putc` 作为输出函数，同一个格式化引擎服务两个输出通道。这种设计在 C++ 里用模板函数来实现，既避免了虚函数的开销，又保持了代码的复用性。

## 概念精讲

### 可变参数机制：va_list 的黑魔法

C 语言的 `printf` 能够接受任意数量、任意类型的参数，靠的是可变参数机制。这个机制由四个宏组成：`va_list`、`va_start`、`va_arg`、`va_end`。

`va_list` 是一个类型，代表"可变参数列表"的抽象——在 x86-64 的 System V ABI 下，它的底层实现比较复杂（前几个参数通过寄存器传递，多余的通过栈传递），但使用时不需要关心这些细节。

`va_start` 初始化一个 `va_list`，你需要告诉它最后一个固定参数的名字（在我们的场景下就是 `format` 字符串），它据此计算出可变参数在栈/寄存器中的起始位置。

`va_arg` 每次从列表中取出一个参数。你需要指定参数的类型——`va_arg(args, int)` 取出一个 int，`va_arg(args, uint64_t)` 取出一个 64 位无符号整数。每次调用 `va_arg` 都会让列表的"游标"前进一步。

`va_end` 做清理工作。

在内核环境下使用可变参数有一点需要注意：我们用 `-ffreestanding` 编译，没有标准库，所以 `va_list` 等定义不是来自 `<stdarg.h>`，而是来自编译器内置的 `__builtin_va_*` 系列。不过 GCC 会自动处理这个问题——`#include <stdarg.h>` 在 freestanding 模式下也能工作，它内部会重定向到编译器的内置实现。

有一个非常重要的陷阱：`va_arg(args, char)` 是**未定义行为**。C 标准规定，比 int 窄的整型参数在传递时会被提升为 int，所以 `va_arg(args, char)` 实际上读的是不对的。正确的写法是 `va_arg(args, int)` 然后转型为 char。同理，`float` 会被提升为 `double`。

### 数字到字符串的转换算法

把一个整数变成人能读的字符串，说起来简单，但细节很值得琢磨。以十进制为例，最直觉的想法是"除以 10 取余数"——但这样得到的数字序列是反的：12345 除以 10 的余数序列是 5、4、3、2、1，你得把它翻转过来才是 "12345"。我们的实现思路是先把反向序列存到一个临时数组里，然后反向复制到输出缓冲区。

十六进制也是一样的套路——除以 16（或者说右移 4 位然后取低 4 位）得到每个"数字"，数字到字符的映射通过一个查找表完成（`"0123456789abcdef"` 或 `"0123456789ABCDEF"`）。

二进制格式化比较特殊。如果按除以 2 取余的方式做，64 位整数最多产生 64 个字符，临时数组和翻转操作都是浪费。我们的做法是先从高位开始扫描，找到最高的那个置 1 的 bit，然后从那个 bit 开始往低位输出——这样自然就压缩了前导零，0b0001010 输出为 "1010" 而不是 "0000000000001010"。

有一个非常关键的边界情况：`INT64_MIN`（-9223372036854775808）。这个值的绝对值超出了 int64_t 的表示范围（int64_t 最大正值是 9223372036854775807），所以你不能简单地 `value = -value`。我们的处理方式是对这个特殊值硬编码字符串——直接把 "-9223372036854775808" 这个字面量复制到输出缓冲区。虽然丑，但这是保证正确性的唯一方法。

### 模板化设计：OutputFn 策略模式

`kprintf` 和 `kdebugf` 的格式化逻辑完全相同，唯一的区别是输出字符去了哪里——前者去串口，后者去 debugcon。如果写两份格式化代码，不但冗余而且以后改格式符的时候很容易改了一边忘了另一边。

C++ 的模板函数完美地解决了这个问题。我们写一个模板函数 `vkprintf_impl`，它接受一个"输出函数"作为模板参数。格式化引擎每产生一个字符，就调用这个输出函数来发送它。`kprintf` 调用 `vkprintf_impl` 时传入一个 lambda，lambda 内部调用 `Serial::putc()`；`kdebugf` 传入另一个 lambda，调用 `debugcon_putc()`。编译器在编译时就能把模板实例化为两份具体的代码，零运行时开销。

在实际代码中，`vkprintf_impl` 被放在独立的头文件 `private/vkprintf_impl.h` 中，这样 Host 端测试也可以 include 它来测试格式解析逻辑，而不需要链接串口驱动或 I/O 端口代码。

这种模式在 C++ 里有个名字叫"策略模式"（Strategy Pattern），用模板实现在编译期绑定比用虚函数在运行时绑定高效得多。Linux 内核的 `printf` 实现也用了类似的方式——函数指针作为输出策略，只是它用的是 C 的函数指针而不是 C++ 的模板。

## 动手实现

### Step 1: 实现数字格式化函数

**目标**：在 `kernel/mini/lib/private/format.h` 和 `format.cpp` 中实现三个格式化函数——`format_decimal`、`format_hex`、`format_binary`，它们把整数转换成字符串存入缓冲区。

**设计思路**：这三个函数的签名统一：接受一个值、一个输出缓冲区指针、一个缓冲区大小参数，返回写入的字符数（不含末尾的 null 终止符）。`format_hex` 多一个布尔参数控制大小写。

`format_decimal` 处理有符号 int64_t。首先检查 INT64_MIN 特殊值——直接把硬编码的字面字符串复制到缓冲区返回。对于其他负数，取反后处理。取绝对值后进入 `do-while` 循环：每次 `% 10` 取最低位数字，`/ 10` 去掉最低位，直到商为 0。这个循环用 `do-while` 而不是 `while` 的原因是处理 value=0 的情况——`do-while` 至少执行一次，0 的余数是 0，写入 '0'，正确。如果用 `while` 循环，0 的情况直接跳过，缓冲区里什么都没写，输出为空。

`format_hex` 处理无符号 uint64_t。循环体是 `value & 0xF` 取最低 4 位，`value >>= 4` 右移去掉已处理的位。数字到字符的转换通过一个 16 字符的查找表完成。循环结束后反向复制临时数组。

`format_binary` 的做法是从 bit 63 开始往下扫描，找到第一个为 1 的位（或者扫到 bit 0 发现全是零），然后从那个高位开始往低位逐位输出 '0' 或 '1'。

所有三个函数都要检查 `buffer_size`，防止缓冲区溢出——如果缓冲区太小就只写入能放下的部分，然后在末尾加 null 终止符。

**实现约束**：这三个函数放在 `private/` 子目录下，因为它们是 `kprintf` 的内部实现细节，不应该被外部代码直接调用。头文件的 include guard 要用 `#pragma once`，保持项目风格一致。

**踩坑预警**：`format_decimal` 的 `do-while` vs `while` 选择是一个经典的 bug 来源。如果你不小心用了 `while (abs_val > 0)` 循环，传 0 进去就会输出空字符串。这个 bug 在 Host 端测试里很容易发现（测试用例直接检查 0 的输出），但在内核里如果你的 kprintf 输出了一个格式化为 %d 的 0 值却什么都不显示，你会以为内核挂了，实际上只是格式化函数没处理零值。

**验证**：这是最适合在 Host 端做单元测试的模块。下一篇的测试框架会覆盖这些函数，但现在你可以手动验证：在一个临时测试文件里调用 `format_decimal(42, buffer, 64)` 然后检查 buffer 内容是否为 "42"；调用 `format_hex(0xDEADBEEF, buffer, 64, true)` 检查是否为 "deadbeef"；调用 `format_binary(0b101010, buffer, 64)` 检查是否为 "101010"。

### Step 2: 实现 vkprintf_impl 格式解析引擎

**目标**：在 `kernel/mini/lib/kprintf.cpp` 中实现核心的格式字符串解析函数，它是整个 kprintf 的引擎。

**设计思路**：`vkprintf_impl` 是一个模板函数，模板参数是一个"输出函数"类型（`OutputFn`）。函数签名接受三样东西：输出函数（右值引用，完美转发）、格式字符串指针、`va_list` 参数列表。

函数的主循环遍历格式字符串的每个字符。如果当前字符不是 `%`，直接调用输出函数发送这个字符。如果遇到 `%`，就进入格式符解析流程：

第一步，解析宽度修饰符。先看 `%` 后面是不是 `0`——如果是，设置零填充标志，跳过这个 `0`。然后继续读连续的数字字符（`'0'` 到 `'9'`），把它们拼成一个十进制整数作为宽度值。比如 `%04d` 的解析结果是零填充标志=true，宽度=4。

第二步，读取格式类型字符（`d`、`u`、`x`、`X`、`s`、`c`、`p`、`b`、`%` 之一），根据类型进行分支处理。`%` 输出一个百分号字符；`%c` 通过 `va_arg(args, int)` 取一个 int 然后转成 char 输出；`%s` 取一个 `const char*` 指针，逐字符输出（如果指针是 NULL 则输出 "(null)"）；`%d` 和 `%u` 调用 `format_decimal`（有符号取 `int64_t`，无符号取 `uint64_t`）；`%x` 和 `%X` 调用 `format_hex`；`%b` 调用 `format_binary`；`%p` 输出 "0x" 前缀后调用 `format_hex` 输出 64 位地址值。

第三步，处理宽度填充。数值型格式符（`d`、`u`、`x`、`X`、`b`）产生的字符串长度如果小于指定的宽度，需要用空格或零在前面填充。多个 case 共用一个 `do_padding` 标签来完成这个后处理——格式化函数先把结果写入临时缓冲区并返回长度，然后统一跳到 `do_padding` 做填充和输出。

**实现约束**：`do_padding` 用的是 `goto`。是的，在 C++ 代码里用 `goto` 看起来不太"现代"，但这种场景下它是最干净的解决方案——多个 switch case 需要共享同一段后处理代码，提取成函数的话要传递一堆参数，而 `goto` 直接跳过去，零开销。Linux 内核的 `printf` 实现也用了类似的方式。

**踩坑预警**：宽度填充的实现里有一个容易搞错的地方——填充字符应该在格式化结果的前面，不是后面。也就是说 `%04d` 对应 7 应该输出 "0007"（前面补零），而不是 "7000"。这意味着填充循环在输出格式化字符串之前执行。另外，`%p` 格式符要记得先输出 "0x" 前缀——如果忘了，地址看起来就是一串裸的十六进制数字，很难辨认。

**验证**：这个函数需要配合 Step 3 的 `kprintf`/`kdebugf` 包装函数一起验证。但你可以先在 Host 端测试——下一篇的测试框架会详细测试各种格式符和边界条件。

### Step 3: 实现 kprintf 和 kdebugf 的包装函数

**目标**：在 `kernel/mini/lib/kprintf.h` 和 `kprintf.cpp` 中实现两个公开接口——`kprintf`（走串口）和 `kdebugf`（走 debugcon）。

**设计思路**：这两个函数是 `vkprintf_impl` 的简单包装。它们各自用 `va_start`/`va_end` 初始化和清理可变参数列表，然后用不同的输出函数调用 `vkprintf_impl`。

`kprintf` 的输出函数是一个 lambda，它捕获 Serial 单例的引用，内部调用 `Serial::putc()`。`kdebugf` 的输出函数是另一个 lambda，内部调用 `debugcon_putc()`（单条 `outb` 到端口 0xE9）。

函数声明放在头文件里，使用 `extern "C"` 风格的声明（虽然函数本身不是 extern "C"，但需要确保 C++ name mangling 正常工作，因为它们会被内核的 C++ 代码调用）。

**实现约束**：头文件需要声明 `kprintf` 和 `kdebugf`，但不能暴露 `vkprintf_impl` 这个模板函数的细节——它应该只在内部可见（实际代码中放在 `private/vkprintf_impl.h`，由 `kprintf.cpp` include）。`debugcon_putc` 可以声明为文件作用域的 static 函数或匿名命名空间里的函数，不需要在头文件里暴露。

**踩坑预警**：`va_start` 和 `va_end` 必须成对出现。如果你在某个错误路径上 `va_start` 了但没有 `va_end` 就 return 了，在某些平台上会导致栈损坏。虽然 GCC 在大多数情况下不会因此出问题，但这是未定义行为，值得注意。另外，`va_start` 的第二个参数必须是函数签名中 `...` 前面的那个参数的名字——在我们的场景下就是 `format`。如果写错了参数名，`va_start` 会从错误的位置开始读参数，后续所有的 `va_arg` 取值都是错的。

**验证**：现在可以写一个综合性的测试了。在 `main.cpp` 的 `mini_kernel_main()` 里，用 `kprintf` 输出一段包含各种格式符的字符串：

- 纯字符串：`kprintf("Hello, Cinux!\n")`
- 十进制整数：`kprintf("Decimal: %d\n", -12345)`
- 十六进制：`kprintf("Hex: %x\n", 0xDEADBEEF)`
- 指针：`kprintf("Address: %p\n", some_pointer)`
- 宽度填充：`kprintf("[%04d] [%4d]\n", 7, 7)`

构建并运行：

```
cmake --build build
cd build && make run
```

串口终端应该出现格式化后的输出：

```
Hello, Cinux!
Decimal: -12345
Hex: deadbeef
Address: 0x20000
[0007] [   7]
```

同时检查 `debug.log` 里是否出现了通过 `kdebugf` 输出的内容。

如果 `kprintf` 输出了乱码但 `kdebugf` 输出正确，问题不在格式化函数而在串口驱动——因为两者使用同一套格式化引擎，只是输出后端不同。这就是模板化设计的调试优势：你可以通过对比两个通道的输出来快速缩小问题范围。

## 构建与运行

本篇涉及的文件清单：

- `kernel/mini/lib/private/format.h` — 格式化函数声明
- `kernel/mini/lib/private/format.cpp` — 格式化函数实现
- `kernel/mini/lib/kprintf.h` — kprintf/kdebugf 声明
- `kernel/mini/lib/kprintf.cpp` — 核心引擎和包装函数
- `kernel/mini/main.cpp` — 内核入口，调用 kprintf 输出

确保 `kernel/mini/CMakeLists.txt` 里把 `driver/serial.cpp`、`lib/kprintf.cpp`、`lib/private/format.cpp` 加到了源文件列表中。构建并运行后，`main.cpp` 中的 kprintf 调用应该能在串口终端产生格式化输出。

如果编译时报 "undefined reference to format_decimal" 之类的链接错误，说明 `format.cpp` 没加到 CMakeLists.txt 的源文件列表里。如果编译通过但运行时输出乱码或格式不对，先检查格式字符串本身有没有写错（比如漏了 `\n`），然后下一篇的 Host 端测试可以精确定位是哪个格式化函数出了问题。

另外值得注意的是，实际代码中的 `vkprintf_impl` 被放在了 `kernel/mini/lib/private/vkprintf_impl.h` 这个独立的头文件中，而不是直接嵌入 `kprintf.cpp`。这种拆分是有意的：`vkprintf_impl` 是纯模板代码（不依赖任何硬件），把它单独放在头文件中可以让 Host 端测试直接 include 它来测试格式解析逻辑，而不需要链接串口驱动。`kprintf.cpp` 通过 `#include "private/vkprintf_impl.h"` 引入它。

实际的格式化引擎还支持左对齐（`%-Ns` 格式符），通过在宽度数字前加 `-` 标志来启用。这个特性在后续的内核输出中会用到，比如让表格对齐。

## 调试技巧

### 利用双通道对比排查格式化 bug

`kprintf` 和 `kdebugf` 使用同一套格式化引擎，所以如果 `kprintf` 输出的十进制数不对，但 `kdebugf` 输出的同样数是对的，那问题不在格式化函数而在串口驱动（字符在传输过程中丢了或错了）。反过来，如果两个通道的输出都不对，问题就锁定在格式化函数本身。

### 宽度填充的调试

宽度填充是最容易出错的格式化特性。如果 `%04d` 对应 7 的输出是 "7" 而不是 "0007"，可能是宽度解析部分没有正确读取 `0` 标志；如果输出是 "0007" 但紧接着下一行对不齐，可能是填充字符选错了（应该填 '0' 但填了空格）。逐一测试 `%d`、`%04d`、`%4d`、`%x`、`%08x` 这几种组合能快速定位问题。

### 在 Host 端快速迭代

下一篇会详细讲测试框架，但这里先提一下思路：`format.cpp` 是纯算法代码，不依赖任何硬件。你可以直接在 Linux 用户态写一个 `main()` 函数来测试它——用 `printf` 来对比输出结果。这样可以避免每次改动都要重新构建内核镜像、启动 QEMU 的漫长循环。

### 关于 %d 和 %u 的参数类型差异

`%d` 通过 `va_arg(args, int)` 取参数，`%u` 通过 `va_arg(args, unsigned int)` 取参数。两者都先转型为 `int64_t` 再传给 `format_decimal`。这种做法看起来有点奇怪——为什么 `%u` 用 `unsigned int` 而不是直接用 `uint64_t`？原因是保持和标准 `printf` 的兼容性：`%u` 对应 `unsigned int`，`%d` 对应 `int`。如果你传了一个 `uint64_t` 给 `%u`，在 x86-64 上 `va_arg(args, unsigned int)` 只会读出低 32 位——这是你的调用约定错误，不是格式化函数的 bug。

### debugcon_putc 的实现位置

`debugcon_putc` 函数在 `kprintf.cpp` 中定义，位于匿名命名空间里（文件作用域 static 的 C++ 写法）。这意味着它只在 `kprintf.cpp` 内部可见，其他文件无法直接调用它。这是有意为之的——外部代码应该通过 `kdebugf` 来使用 debugcon 输出，而不是绕过格式化引擎直接调用底层的字符输出函数。

| 概念 | 要点 |
|------|------|
| va_list 可变参数 | `va_start`/`va_arg`/`va_end` 三件套，`va_arg(args, char)` 是 UB 必须用 int |
| format_decimal | 反向取余 + 翻转，`do-while` 处理零值，INT64_MIN 硬编码 |
| format_hex | 逐 nibble 提取 + 查找表映射，大小写由表决定 |
| format_binary | 从高位扫描找到最高 set bit，自然压缩前导零 |
| 模板化 vkprintf_impl | OutputFn 策略模式，编译期绑定零开销 |
| 格式字符串解析 | `%` 触发解析 -> 宽度修饰符 -> 类型字符 -> 格式化 -> 填充输出 |
| kprintf vs kdebugf | 前者走串口 Serial::putc，后者走 debugcon outb(0xE9) |
| do_padding goto | 多 case 共享后处理代码，C 风格但实用 |
| 缓冲区保护 | 所有 format 函数检查 buffer_size，防溢出 |
