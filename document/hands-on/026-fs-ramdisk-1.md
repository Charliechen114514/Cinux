# 026-fs-ramdisk-1: USTAR 格式与八进制解析

## 导语

到这里，我们的内核已经有了串口输出、内存管理、进程调度、用户态切换、系统调用和 AHCI 磁盘驱动。但仔细想想——我们还从来没有从"文件"这个层面碰过磁盘上的数据。

AHCI 驱动能读写扇区，但扇区是一堆原始字节，不是文件。我们需要一个文件系统，而文件系统的起点，是先把一些文件弄到内核能碰到的地方。这就是 initrd（initial ramdisk）的用武之地。

我们现在的方案是：把一个 ustar 格式的 tar 归档直接编译进内核镜像里，内核启动时解析这个归档，列出里面的文件。这个 tag 做的就是这件事——而本篇我们聚焦在最底层：ustar 格式的定义和八进制 ASCII 解析。

完成本篇后，你会拥有一个完整定义的 ustar 头部结构和可靠的八进制转换函数，为下一篇的构建系统集成和最终的 ramdisk 驱动打下基础。前置知识：上一个 tag（025）的 AHCI 驱动不需要，本篇是纯软件层。

## 概念精讲

### 什么是 ustar 格式？

ustar 是 POSIX.1-1988 定义的"Unix Standard TAR"归档格式。它的设计哲学简单到令人感动：把所有文件一个接
接拼在一起，每个文件前面加一个 512 字节的头部描述这个文件的元信息（名字、大小、类型等），文件内容也按
512 字节对齐填充。整个归档用两个全零的 512 字节块标记结束。

你完全可以把 ustar 想象成一本书：每"页"是 512 字节，每个文件的"标题页"是头部，后面跟着若干页的"正文"。
翻到一页全零的标题页，就说明书读完了。这个类比其实非常精确——tar 的全称是 "Tape ARchive"，最初就是为
磁带设计的，磁带本身就是线性顺序读取的介质，所以 tar 的线性结构并非偶然，而是和底层存储介质的高度适配。

为什么选 ustar 而不是别的格式？因为它的解析逻辑极其简单——没有压缩、没有校验和验证（可以跳过）、没有
复杂的索引结构。对于教学 OS 来说，这就是最合适的 initrd 格式。OSDev Wiki 也推荐 tar 作为入门 initrd
格式，因为它只需要一个简单的线性扫描就能列出所有文件。与之相比，cpio 格式虽然也被 Linux 的 initramfs
使用，但它的头部格式更复杂（有多种变体），gzip/bzip2 压缩则需要先解压才能解析，增加了不必要的复杂度。

### ustar 格式的历史背景

tar 格式诞生于 1970 年代末的 Unix V7，最初的目的就是在磁带之间传输文件。磁带是顺序访问介质——你不能
随机跳转到某个位置，只能从头到尾线性读取。这解释了为什么 tar 的结构是线性的：每个文件按顺序排列，头部
和数据交替出现。你不可能像 FAT 或 ext2 那样维护一个文件分配表，因为在磁带上"跳转"到文件分配表的位置
是非常慢的操作。

随着硬盘的普及，tar 的用途从"磁带归档"扩展为通用的文件打包格式。虽然后来出现了更多高级格式（zip 的
随机访问、7z 的高压缩率），但 tar 的简单性使它在 OS 开发领域保持着特殊的地位——当你需要在内核里实现
一个最小化的文件读取时，tar 几乎是唯一一个能在几百行代码内完成解析的格式。

### 八进制 ASCII 编码——来自磁带时代的遗产

ustar 头部里有一个非常"复古"的设计决策：所有数字字段都用八进制的 ASCII 字符串表示，而不是二进制整数。
比如一个 100 字节的文件，它的 size 字段存的是字符串 "00000000144"（144 的八进制等于 100 的十进制），
每个字符都是 '0' 到 '7' 的 ASCII 码。

这个设计来自 1970 年代的磁带时代——用 ASCII 文本存储数字可以跨平台兼容，因为不同机器的二进制整数字节序
不同（大端序 vs 小端序），整数字长也可能不同（16 位 vs 32 位），但 ASCII 字符在所有系统上都是一样的。
一个 '5' 的 ASCII 码 0x35，无论在 PDP-11、VAX 还是现代 x86 上都是同一个值。

字符串以 null 字符或空格字符终止——有些 tar 实现用 null，有些用空格，有些两者混合使用，我们的解析器
必须同时处理这两种情况。这种编码方式虽然浪费空间（12 字节只能表示到约 8GB），但在那个年代 8GB 是不可
想象的数字，所以完全够用。

我们需要一个函数把这个八进制 ASCII 字符串转回 64 位无符号整数。思路很直接：从左到右逐字符读取，遇到
null 或空格就停，每个数字位的值等于字符值减去 '0' 的 ASCII 码，然后 result 等于 result 乘以 8 加上
当前数字值。用位运算的话就是每次左移 3 位再加上数字值——位移的语义更清晰，因为八进制的基数 8 恰好是
2 的 3 次方。

### ustar 头部的字段布局

一个 ustar 头部固定 512 字节，我们来详细了解每个字段的含义和位置。

前 100 字节（偏移 0-99）是文件路径，以 null 字符终止，包含完整的相对路径，比如 "etc/passwd" 或
"hello.txt"。偏移 100-107 是 mode 字段，8 字节的八进制 ASCII 表示文件权限（比如 "0000644 " 表示
用户可读写、其他人只读）。偏移 108-115 和 116-123 分别是 uid 和 gid，各 8 字节的八进制用户/组 ID。

偏移 124-135 是 size 字段，12 字节的八进制 ASCII 表示文件大小——这是我们最关心的字段之一。偏移
136-147 是修改时间（mtime），同样 12 字节八进制。偏移 148-155 是校验和，8 字节——虽然可以用来验证
头部完整性，但我们的实现选择跳过校验和验证以简化逻辑。

偏移 156 是 typeflag，只有 1 个字节，决定了这条记录的类型：'0' 是普通文件，'5' 是目录，'7' 是连续
文件（和普通文件等价），'1' 到 '6' 分别是硬链接、软链接、字符设备、块设备和 FIFO。偏移 157-256 是
链接目标名称，100 字节，仅当 typeflag 为链接类型时有效。

偏移 257 开始是 ustar 扩展字段——magic 字段（偏移 257-262，6 字节）必须是 "ustar" 加一个 null，
如果不是，说明这不是一个合法的 ustar 头部。这是我们在 mount 遍历中验证头部合法性的关键检查。后面还有
版本号（263-264，值为 "00"）、用户名和组名（各 32 字节）、设备主次号（各 8 字节）和路径前缀（155
字节，用于超过 99 字节的长文件名）。最后是 12 字节的填充把头部凑满 512 字节。

### 数据块的对齐规则

除了头部之外，ustar 还有一个重要的对齐规则需要理解。文件的原始数据紧跟在头部后面，但必须按 512 字节
对齐。也就是说，如果文件大小是 100 字节，它在归档中实际占用 512 字节（一个完整的数据块，剩余部分用零
填充）；如果文件大小是 600 字节，实际占用 1024 字节（两个数据块）；如果文件大小恰好是 512 字节，占用
512 字节（一个数据块，恰好填满）。

当文件大小为 0 时不占任何数据块——下一个头部直接紧跟在当前头部后面。目录条目的 size 通常也是 0，所以
目录头部后面也是直接接下一个条目。这个对齐规则直接影响 mount 时如何从一个条目跳到下一个条目：跳转量
等于 512（头部大小）加上向上取整（文件大小除以 512）再乘以 512（数据块）。

## 动手实现

### Step 1: 定义 ustar 常量

**目标**: 创建配置头文件 `ramdisk_config.hpp`，定义 ustar 相关的常量和类型标志。

**设计思路**: 我们把 ustar 格式的定义和实际的 ramdisk 驱动逻辑分成两个独立的头文件。配置层只负责描述
"ustar 是什么样的"，不涉及任何内核特定功能（不包含 kprintf 头文件、不依赖链接器符号）。这样 host 端
的单元测试也能包含这个头文件来测试格式定义，不需要链接整个内核。所有定义放在 cinux::fs 命名空间下，
避免全局命名空间污染。

**实现约束**: 定义一个 32 位无符号常量表示块大小（512 字节），这个常量会在数据块数计算和 mount 偏移
跳转中使用。定义一个命名空间包含 8 个类型标志字符常量：普通文件、目录、硬链接、软链接、字符设备、
块设备、FIFO 和连续文件。定义一个字符串常量 "ustar" 作为 magic 验证值。

类型标志用 constexpr char 而不是枚举，因为 typeflag 在头部中就是单个 char 字段，直接用 char 比较
更自然，避免了枚举和 char 之间的反复类型转换带来的代码冗余。

**踩坑预警**: 常量必须用 static constexpr 而不是宏定义，这样才能在编译期断言和模板中使用，而且有类型
安全。magic 字符串用 static constexpr char 数组定义，包含 null 终止符共 6 个字符。

**验证**: 编写测试确认块大小常量值为 512，类型标志字符与 POSIX 定义一致（'0' 到 '7'），magic 字符串
前 5 个字符确实是 "ustar" 且第 6 个字符是 null 终止符。

### Step 2: 定义 UstarHeader 结构体

**目标**: 定义与 ustar 头部完全对应的内存布局结构体。

**设计思路**: ustar 头部是 512 字节的固定格式，每个字段在固定偏移量上。我们用一个 packed struct 来
精确映射这个布局，这样可以直接把指向归档数据的指针重解释为这个结构体指针来访问字段。这在 OS 开发中是
非常常见的模式——当你需要直接把硬件寄存器、协议头部或磁盘数据映射为 C/C++ 结构体时，packed struct 是
标准做法。

**实现约束**: 结构体必须使用 GNU 特定的 packed 属性防止编译器插入填充字节，所有字段都是 char 数组
（对应 ASCII 字符串）。字段按 POSIX 规范的偏移量严格排列：name 数组 100 字节在偏移 0，mode 数组 8
字节在偏移 100，size 数组 12 字节在偏移 124，typeflag 单字节在偏移 156，magic 数组 6 字节在偏移
257，prefix 数组 155 字节在偏移 345，最后 padding 数组 12 字节凑满 512 字节。

必须在结构体定义后立即加编译期断言验证大小为 512——如果编译失败，说明字段定义有误。

**踩坑预警**: 如果不用 packed 属性，编译器可能会在 typeflag（只有 1 字节）后面插入填充字节让后续字段
对齐到合理边界，导致结构体大小超过 512 字节。这种情况下用指针重解释映射的数据会全部错位，而且这个 bug
非常隐蔽——你可能只发现文件名变成了乱码，却不知道根因在结构体布局上。编译期断言是你唯一的安全网。

另外，字段顺序必须和 POSIX 规范一致——如果你把两个字段写反了，编译不会报错（它们类型相同），但运行时
所有偏移量都会错位。

**验证**: 编写测试检查每个关键字段的偏移量：name 在偏移 0，mode 在偏移 100，size 在偏移 124，typeflag
在偏移 156，magic 在偏移 257，prefix 在偏移 345。具体做法是创建一个头部实例，获取其基地址的 uint8_t
指针，然后分别计算每个字段成员的地址与基地址的差值。

### Step 3: 实现 octal_to_uint 转换函数

**目标**: 实现将八进制 ASCII 字符串转换为 64 位无符号整数的函数。

**设计思路**: ustar 的 size、mode、uid 等数字字段都是八进制 ASCII。我们需要一个通用的转换函数，接受
字符指针和最大长度，返回解析后的整数。函数要能正确处理 null 终止和空格终止两种情况。

作为自由函数而非头部结构体的成员方法，因为它不操作任何实例，而且需要被 host 测试和内核代码分别独立
调用。声明在 config 头文件中，实现在 ramdisk.cpp 中。

**实现约束**: 函数签名接受一个 const char 指针和一个 size_t 长度参数，返回 uint64_t。循环遍历字符，
遇到 null 或空格就跳出。每个有效字符的数字值等于字符的 ASCII 码减去 '0' 的 ASCII 码，累积公式用位移
实现：每次左移 3 位再加上当前数字值。

**踩坑预警**: 有几个容易忽略的边界情况需要注意。空字符串（长度为 0）应该返回 0。全是空格的字段也应该
返回 0。前导空格同理返回 0。长度参数大于实际字符串长度时，null 终止符会正确截断——这一点很重要，因为
ustar 的 size 字段是 12 字节，但实际有效数字可能只有后面几位。

**验证**: 编写测试验证典型输入："00000000012" 解析为 10，"00000001000" 解析为 512。同时测试 null 终止
、空格终止、零长度、全零、大值（全 7 字符串等于约 8.6GB）等边界情况。

## 构建与运行

本篇只涉及头文件定义，不需要单独运行内核。但你可以运行 host 端单元测试来验证格式定义和转换函数的正确性：

```bash
cd build && cmake --build . --target test_ramdisk
ctest -R ramdisk --output-on-failure
```

预期输出：所有八进制转换测试通过（约 15 个测试用例），头部结构体大小和偏移量测试通过（约 8 个测试用例）
。你应该能看到类似 "ramdisk_ustar_header: sizeof(UstarHeader) is 512 ... OK" 和 "ramdisk_octal:
'12' returns 10 ... OK" 的输出。

如果某个偏移量测试失败，测试会打印期望值和实际值，方便你定位是哪个字段定义有误。

## 调试技巧

如果你发现结构体大小不等于 512，首先检查是否漏掉了 packed 属性。如果某个字段的偏移量不对，检查字段
排列顺序是否与 POSIX 规范一致——顺序错了的话中间的字段全都会偏移。在调试时，可以创建一个全零的头部
实例，获取其基地址指针，然后分别计算每个字段成员的地址与基地址的差值，与预期偏移量做比较。

如果八进制转换函数对某些输入返回了错误值，打印输入字符串的每个字符和它的 ASCII 码值，确认不是隐藏的
非 ASCII 字符在捣鬼。特别要注意制表符和其他空白字符——它们不是空格也不是 null，函数会尝试把它们当作
八进制数字来解析，结果会出错。

如果你怀疑 tar 工具生成的归档格式有问题，可以用 hexdump 检查前几个头部的内容，确认各字段确实是 ASCII
八进制字符串。具体命令是 hexdump -C initrd.tar | head -40，你应该能在偏移 257 处看到 "ustar" 字样。
