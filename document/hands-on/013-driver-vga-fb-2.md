# 013-2 字模引擎：PSF2 位图字体解析与渲染

## 导语

上一章我们把帧缓冲区搞定了——可以画像素、填矩形了。但光有像素还不够，我们要在屏幕上显示文字，就需要一套字体。这一章要做的是：把一个 PSF2 格式的位图字体嵌入内核二进制，然后解析它的头部结构，逐像素地把字形渲染到帧缓冲区上。完成这一章后，你会能够在屏幕上的任意位置渲染单个 ASCII 字符——虽然还不能自动换行和滚动，但"能写字"这个关键能力就算具备了。

知识前置：上一章的 Framebuffer 类必须已经可用（put_pixel 工作正常），你需要理解位图字体的基本概念（每个字符是一个像素矩阵，1 表示前景色，0 表示背景色）。此外，你还需要了解 GAS 汇编中 `.incbin` 伪指令的作用——它能在编译时将二进制文件的内容直接嵌入到目标文件中。

## 背景与动机

在图形模式下显示文字，本质上是一个"查表 + 画像素"的过程：每个字符对应一张位图（bitmap），位图中的每个 bit 决定对应的像素是前景色还是背景色。这种"位图字体"（bitmap font）是最简单的字体技术——相比 TrueType/OpenType 的矢量字体（需要贝塞尔曲线光栅化），位图字体不需要任何数学运算，直接查表即可。

位图字体的缺点是不支持缩放——8x16 的字体在 4K 显示器上会显得非常小，放大后又会出现锯齿。但对于内核控制台来说这不是问题：我们不需要多种字号，也不需要抗锯齿渲染。Linux 内核的 fbcon 在启动阶段也是用同样的位图字体。

PSF2 格式选择的原因：PSF2 是 Linux 内核实际使用的字体格式，有丰富的现成字体文件可用（在 `/usr/share/consolefonts/` 目录下），而且格式足够简单（32 字节头部 + 连续的字形数据）适合教学。OSDev Wiki 有完整的格式文档，遇到问题很容易查到参考资料。

## 与其他系统的对比

Linux 内核的字体嵌入方式与 Cinux 不同——Linux 把字形数据直接硬编码在 C 源文件中（比如 `lib/fonts/font_8x16.c` 是一个巨大的 unsigned char 数组），不需要 `.incbin` 或者 objcopy。Cinux 选择 `.incbin` 方案是因为它保持了字体文件的独立性——修改字体只需要替换 .psf 文件，不需要编辑 C 源码。

xv6 完全不需要软件字体渲染——它使用 VGA 文本模式的硬件字符发生器。每个字符只需要 2 字节（1 字节编码 + 1 字节颜色属性），写入 0xB8000 处的显存即可，硬件自动把字符编码转换为屏幕上的字形。这是最简单的方案，但只支持 80x25 的分辨率和 256 种预定义字符。

SerenityOS 把字体渲染放到了用户空间的 LibGfx 库中，内核只负责把渲染好的像素数据 blit 到帧缓冲区。这种设计的优势是内核代码更简洁，字体格式的支持可以在用户空间灵活扩展（包括 TrueType、BitmapFont 等）；劣势是内核启动早期需要一个独立的简单字体渲染器。

## 概念精讲

### PSF2 字体格式

PSF2（PC Screen Font version 2）是 Linux 内核控制台使用的位图字体格式。它的结构非常简单：一个 32 字节的头部，后面跟着所有字形（glyph）的位图数据。头部包含魔数（0x864AB572）、字形数量、每个字形占多少字节、字形的宽和高等信息。OSDev Wiki 的 PC Screen Font 页面有完整的格式说明。

头部由 8 个 uint32_t 字段组成（共 32 字节）：magic（魔数，必须是 0x864AB572）、version（版本号，通常为 0）、header_size（头部大小，标准为 32 字节）、flags（标志位，bit 0 表示是否有 Unicode 映射表）、length（字形总数）、charsize（每个字形占用的字节数）、height（字形高度，像素）、width（字形宽度，像素）。

对于 8x16 的字体来说，每个字形占 16 字节（16 行，每行 1 字节恰好编码 8 个像素）。第 0 个字形对应字符编码 0，第 1 个对应编码 1，依此类推——如果没有 Unicode 映射表的话。字形的第 N 个字节表示第 N 行的像素分布：最高位（bit 7）是最左边的像素，最低位（bit 0）是最右边的像素。

为什么是 bit 7 在左边而不是 bit 0？这是因为 PSF2 格式沿用了传统位图字体的编码习惯——字节中 bit 7 是"高位"，对应屏幕上的"左端"。如果你搞反了（把 bit 0 当最左边），渲染出来的文字会是水平镜像的。这个错误很容易犯，而且不会导致崩溃——只是画面上所有字母都是反的，盯着看半天才能发现问题。

### 内核嵌入二进制数据的方法

内核没有文件系统（至少现在还没有），所以不能在运行时加载字体文件。标准做法是用汇编器的 `.incbin` 伪指令把二进制文件直接嵌入到 ELF 的 `.rodata` 段中。GAS（GNU Assembler）会自动生成 `_start`、`_end`、`_size` 三个符号，C++ 代码通过 `extern "C"` 声明这些符号就能访问嵌入的数据了。

另一种方案是用 `objcopy` 把 .psf 文件转成 .o 目标文件再链接——Cinux 选择 `.incbin` 方案，因为它更直观，只需要一个 .S 文件。objcopy 方案自动生成的符号名基于文件路径（比如 `_binary_assets_font_psf_start`），既不直观又不稳定——换个路径所有引用都得改。而 `.incbin` 方案的符号名完全由我们控制。

还有一个值得注意的细节：`extern "C"` 声明是必须的。C++ 的 name mangling 会改变符号名（比如把 `font_psf_start` 编码成 `_Z14font_psf_startv` 之类的东西），而 GAS 生成的符号名是原样的字符串。加上 `extern "C"` 后，链接器就能正确地把 C++ 代码中的符号引用和汇编中定义的符号匹配起来。

`.incbin` 伪指令的文件路径是相对于汇编器的 include path 的。CMake 构建时需要确保项目根目录在 include path 中，这样 `.incbin "assets/font.psf"` 才能正确找到文件。如果路径找不到，汇编阶段就会报错——这是比较好定位的问题类型。

## 动手实现

### Step 1: 生成 PSF2 字体文件

**目标**: 用 Python 脚本生成一个 8x16、256 字符的 PSF2 字体文件。

**设计思路**: IBM PC 的经典 8x16 位图字体（CP437 编码）是公开可用的。我们用 Python 脚本将字形数据打包成 PSF2 格式：先写 32 字节的头部（magic=0x864AB572, version=0, headersize=32, flags=0, numglyph=256, bytesperglyph=16, height=16, width=8），然后写 256*16 = 4096 字节的字形数据。前 128 个字符（0x00-0x7F）是标准的 ASCII 字形，后 128 个留空。

字形数据来源于 IBM PC 的经典 CP437 编码 8x16 位图字体。这是一个公开可用的字体——任何人都可以使用，不受版权限制。Cinux 脚本中硬编码了 0x20（空格）到 0x7E（波浪号）之间可打印 ASCII 字符的字形字节值，以及部分控制字符和扩展字符。0x80-0xFF 的高半区字形全部填充为空白——Cinux 目前不需要 CP437 的制表符和方块字符。

**实现约束**: 脚本用 Python 3 编写，接受一个可选的输出路径参数（默认 assets/font.psf）。使用 `struct.pack` 以小端序写入头部（格式字符串用 `<` 前缀）。字形数据用一个列表的列表表示：外层 256 个元素，每个元素是 16 个字节的列表。输出文件总大小应该是 32 + 256*16 = 4128 字节。

**踩坑预警**: 字形数据中的每个字节代表一行像素，bit 7 是最左边的像素。如果搞反了（把 bit 0 当最左边），渲染出来的文字会是水平镜像的。

**验证**: 运行脚本后检查输出文件大小：应该是 32 + 256*16 = 4128 字节。用 `xxd assets/font.psf | head -3` 检查头部前 4 字节应该是 `72 B5 4A 86`（小端序的魔数）。

### Step 2: 用 .incbin 嵌入字体数据

**目标**: 编写一个汇编文件，用 GAS 的 `.incbin` 伪指令将 PSF2 字体文件嵌入内核。

**设计思路**: 在 `.rodata` 段中声明三个全局符号：`font_psf_start`（字体数据起始地址）、`font_psf_end`（字体数据结束地址）、`font_psf_size`（4 字节的大小值）。`.incbin` 指令直接将二进制文件的内容插入到当前位置。

`font_psf_start` 和 `font_psf_end` 之间的差值就是字体数据的总字节数（4128）。`font_psf_size` 用 `.long` 伪指令把这个差值存储为一个 4 字节的整数。C++ 代码通常只需要 `font_psf_start` 和头部中的 `header_size` 字段就能定位字形数据——`font_psf_end` 和 `font_psf_size` 是辅助性的，用于边界检查或调试。

**实现约束**: 文件放在 `kernel/drivers/video/font_data.S`。使用 `.section .rodata` 将数据放在只读段。汇编器的 include path 必须包含项目根目录，这样 `.incbin "assets/font.psf"` 才能正确找到文件。`.type` 伪指令标注符号类型为 @object。C++ 代码通过 `extern "C"` 声明这些符号为 `const uint8_t font_psf_start[]` 和 `const uint32_t font_psf_size[]`。

**踩坑预警**: 如果 CMake 没有正确设置汇编器的 include path，`.incbin` 会在编译时报 "file not found"。需要在 CMakeLists.txt 中为目标添加包含项目根目录的汇编 include 目录。

**验证**: 构建成功后，用 `nm` 或 `readelf` 检查内核 ELF 中是否存在 `font_psf_start` 和 `font_psf_size` 符号。

### Step 3: 实现 PSFFont 类

**目标**: 编写字体解析和字符渲染类。

**设计思路**: PSFFont 类在 `init()` 时解析嵌入的 PSF2 头部，提取字形数量、每个字形的字节数、宽度和高度，然后计算字形数据的起始指针（头部大小偏移之后）。`render_char()` 方法接收一个 Framebuffer 引用、字符编码、像素坐标和前景/背景色，逐行逐列地读取字形位图并调用 Framebuffer 的 put_pixel 写入像素。

这种渲染方式是"全覆盖"的——每个字符占据一个完整的 `8x16` 像素块，前景像素和背景像素都会被写入。这意味着连续渲染两个字符时不会有像素残留的问题，但也意味着字体渲染会覆盖已有的画面内容。在控制台场景下这完全没问题——每个字符位置都会被重新渲染。

每个字符的渲染需要调用 `width * height = 8 * 16 = 128` 次 `put_pixel`。对于控制台文本输出来说，这个开销完全可接受。如果将来需要高性能文本渲染（比如 cat 一个大文件），可以考虑用 `fill_rect` 先画背景再逐像素写前景来减少函数调用开销。

**实现约束**: 类放在 `cinux::drivers` 命名空间。内部成员包括字形数据指针（const uint8_t*）、每个字形字节数、字形总数、宽度和高度。头部结构体用 `__attribute__((packed))` 标注，因为它需要精确匹配文件布局。魔数验证使用常量 0x864AB572。

`render_char` 方法的核心逻辑：对于超出范围的字符编码，降级到字形 0（通常是一个空格或占位符）。字形指针 = 字形数据基址 + 字符编码 * 每字形字节数。双重循环：外层遍历行（0 到 height-1），内层遍历列（0 到 width-1）。对于每一行的每一列，测试字形字节中的对应位（`bits >> (7 - col) & 1`），根据结果写入前景色或背景色。

需要前向声明 Framebuffer 类（因为头文件中只有引用，不需要完整定义）。

**踩坑预警**: PSF2 头部结构体必须在 `.cpp` 文件中定义（不是头文件），因为它是内部实现细节。字节序问题：PSF2 是小端序格式，在 x86_64 上直接读取 uint32_t 就行，不需要做字节序转换。

**验证**: 在 kernel_main 中初始化 PSFFont 后，检查 `width()` 返回 8、`height()` 返回 16。然后渲染字符 'A' 到 (0,0) 位置（白色前景、黑色背景），读取 (0,0) 到 (7,15) 范围内的像素，验证既存在白色像素（字形的前景部分）也存在黑色像素（背景部分）。

### Step 4: 在 CMakeLists.txt 中注册新文件

**目标**: 将 font.cpp 和 font_data.S 添加到内核构建目标中。

**实现约束**: 在 `kernel/CMakeLists.txt` 的 `big_kernel_common` 库中添加这两个文件。font_data.S 作为汇编源文件添加。确保汇编器的 include path 包含项目根目录（CMake 的 `target_include_directories` 或 `set_source_files_properties`）。

**验证**: `cmake --build build` 成功，没有 "undefined reference to font_psf_start" 之类的链接错误。

### Step 4.5: 编写字体渲染测试

**目标**: 在 `kernel/test/test_video.cpp` 中添加字体渲染相关的测试用例。

**设计思路**: 字体测试需要验证三个层面：头部解析是否正确（width=8, height=16）、单个字符渲染是否正确（写入已知字形后检查像素）、以及越界字符的降级处理（字符编码超过字形总数时降级到字形 0）。

具体测试用例：初始化 PSFFont 后验证 width() 返回 8、height() 返回 16。然后渲染字符 'A' 到 (0,0) 位置（白色前景、黑色背景），读取 (0,0) 到 (7,15) 范围内的像素，验证既存在白色像素（字形的前景部分）也存在黑色像素（背景部分）。

对于连续字符渲染的验证：渲染 'A' 在 (0,0)，渲染 'B' 在 (8,0)，然后检查 'B' 的像素出现在 x=8 到 x=15 的范围，同时 'A' 的像素没有被覆盖。

**实现约束**: 测试函数在 test_video.cpp 中定义，需要在匿名命名空间中声明一个全局 PSFFont 对象。测试顺序是先初始化帧缓冲区（前面已经完成），然后初始化字体，最后运行字体测试。

**验证**: 所有字体测试通过，确认 PSF2 头部解析和字形渲染逻辑工作正常。

### Step 4.6: 理解字体数据在内存中的布局

**目标**: 理解字体数据从 .psf 文件到内核内存的完整路径。

**设计思路**: 字体数据在内存中的布局是这样的：首先是 32 字节的 PSF2 头部（包含魔数、字形数量、字形尺寸等元信息），紧接着是 256 个连续的字形数据块，每个块 16 字节。PSFFont::init() 把 `font_psf_start + header_size` 存储为 `glyphs_` 指针——这就是第一个字形的起始地址。

`render_char()` 中计算字形地址的方式是 `glyphs_ + c * bytes_per_glyph_`。对于字符 'A'（编码 0x41），字形地址就是 `glyphs_ + 0x41 * 16 = glyphs_ + 1024`。这个计算假设字符编码就是字形索引——只有当 PSF2 的 flags 字段中 bit 0 为 0（没有 Unicode 映射表）时才成立。

字节序方面，PSF2 格式是小端序的，而 x86_64 也是小端序，所以直接以 uint32_t 读取头部字段就能得到正确的值。不需要做字节序转换。但如果将来要在大端序架构上使用同样的代码（比如 ARM big-endian），就需要用 `le32toh()` 等函数进行转换。

### Step 4.7: 理解 __attribute__((packed)) 的作用

**目标**: 理解为什么 PSF2 头部结构体需要 packed 属性。

**设计思路**: C/C++ 编译器默认会在结构体的字段之间插入填充字节（padding），以确保每个字段都对齐到其自然对齐边界。例如，一个包含 uint8_t 和 uint32_t 的结构体，编译器可能在 uint8_t 后面插入 3 个填充字节，使得 uint32_t 对齐到 4 字节边界。

对于 PSF2 头部来说，所有字段都是 uint32_t（4 字节），在 x86_64 上本身就自然对齐，所以即使不加 packed 属性，结构体的布局也是正确的。但 packed 属性是一种防御性编程——万一将来有人在结构体中间添加了一个 uint8_t 字段（比如某个扩展版本号），没有 packed 的话编译器会自动插入填充字节，导致结构体大小与文件布局不匹配。

需要注意的是，packed 属性有一个潜在的性能问题：访问未对齐的字段在某些架构上会导致性能下降甚至总线错误。但在 x86_64 上，CPU 硬件处理未对齐访问是透明的（只是稍微慢一点），所以这不是问题。

**实现约束**: PSF2 头部结构体只在 font.cpp 中定义（不暴露在头文件中），因为它是一个内部实现细节。这样即使将来结构体的定义发生变化，也不会影响其他文件的编译。

### Step 4.8: render_char 的性能特征分析

**目标**: 理解 render_char 的性能特征和潜在的优化方向。

**设计思路**: 每次调用 render_char 需要执行 `width * height = 8 * 16 = 128` 次 `put_pixel` 调用。每次 put_pixel 包含一次边界检查和一次 volatile 内存写入。对于 128 次调用，总共有 256 次条件判断和 128 次内存写入。

在 1024x768 分辨率下，一屏可以显示 128 * 48 = 6144 个字符。如果要一次性刷新整个屏幕（比如清屏后重写），需要 6144 * 128 = 786,432 次 put_pixel 调用。这在现代 CPU 上仍然很快（几毫秒），但如果将来需要更高的刷新频率（比如动画），就可能成为瓶颈。

一个简单的优化是"只画前景像素"——对于背景色与现有画面相同的场景，可以跳过背景像素的写入。但这个优化在控制台场景下不太适用，因为每个字符位置都需要完整的重绘。

另一个优化方向是批量写入——对于 fill_rect 这样的操作，可以用 `memset` 或者 SSE/AVX 指令一次性写入多个像素。但 volatile 约束使得这些优化需要额外的类型转换。

**验证**: 不需要特别的验证步骤，这一步是纯理解性的。如果你对性能感兴趣，可以在 render_char 的前后读取 PIT 的 tick 计数来测量渲染时间。

## 构建与运行

构建后启动 QEMU。此时内核初始化 PSF2 字体后应该打印 `[BIG] PSF2 font loaded: 8x16`。如果这条日志没出现或者字体尺寸是 0x0，说明头部解析失败——最可能的原因是 `.incbin` 没有正确嵌入数据。

值得注意的是，字体初始化失败不会导致内核崩溃——PSFFont 的 init() 会检查魔数，如果匹配失败就直接返回，所有成员保持零值。后续调用 render_char() 时 glyphs_ 为 nullptr，会安全退出。这种"静默失败"的设计在内核初始化代码中是合理的——此时 kprintf 可能还没完全初始化，没有地方打印错误信息。

## 调试技巧

**字体魔数不匹配**: 在 init 中加一个 kprintf 打印读到的 magic 值。如果不是 0x864AB572，说明数据没有正确嵌入或者指针算错了。可以用 `xxd assets/font.psf | head -3` 检查字体文件的前几个字节——应该是 `72 B5 4A 86`（小端序的魔数）。

**字符渲染但画面全白或全黑**: 检查前景色和背景色的值是否搞反了——在位图测试中 `on ? fg : bg`，如果测试的是 `(bits >> col) & 1` 而不是 `(bits >> (7-col)) & 1`，字形会是镜像的。另一个可能原因是前景色和背景色用了相同的值——那样的话无论位图怎么取，渲染出来的都是同一种颜色。

**编译错误 "font_psf_start undeclared"**: 确认 font_data.S 被添加到了 CMakeLists.txt，并且 `extern "C"` 声明在 font.cpp 中。用 `nm` 检查 .o 文件中的符号名。

**渲染出来的字符位置偏移**: 检查像素坐标的计算是否正确——x 坐标应该是 `col * font_width`，y 坐标应该是 `row * font_height`。如果少了乘法或者乘反了，字符会挤在一起或者散得很开。

**字体文件路径找不到（汇编报错）**: CMake 构建时汇编器的 include path 必须包含项目根目录。检查 CMakeLists.txt 中是否为目标添加了正确的 include 目录，使得 `.incbin "assets/font.psf"` 能正确解析。

## 本章小结

| 概念 | 关键点 |
|------|--------|
| PSF2 格式 | 32 字节头部 + N * bytesperglyph 字形数据 |
| 魔数 | 0x864AB572（小端序存储） |
| .incbin | GAS 伪指令，直接嵌入二进制文件到 .rodata 段 |
| 字形渲染 | 逐行读取字节，bit 7 = 最左像素 |
| 8x16 字体 | 每字形 16 字节，256 字形共 4096 字节 |
| extern "C" | 必须，防止 C++ name mangling 导致符号找不到 |

## 扩展方向

- **支持 PSF1 格式**: PSF1 的头部只有 4 字节（magic + mode + charsize），是更简单的字体格式。添加 PSF1 兼容性解析，可以直接使用更多的现有字体文件（很多 Linux 发行版仍然提供 PSF1 格式的字体）。难度：⭐
- **运行时字体加载**: 将来有了文件系统后，从磁盘加载字体文件而不是编译时嵌入，支持用户自定义字体。这需要把 PSFFont 的 init() 改为接受一个文件路径参数，内部调用文件系统的 read() 来加载数据。难度：⭐⭐
- **抗锯齿渲染**: 对于高分辨率显示器，可以用子像素渲染或灰度抗锯齿来改善文字显示质量。这需要支持 alpha 混合的帧缓冲区操作，以及包含 alpha 信息的字体格式（PSF2 本身不支持 alpha，需要扩展格式）。难度：⭐⭐⭐
