# 001_boot_real_mode（调试篇） —— 构建系统、QEMU 调试与常见启动故障排查

## 导语

前两篇我们把 MBR 和 Stage2 写完了，CMake 能编译出磁盘镜像，QEMU 里能看到文字输出和图形模式切换。但如果事情不顺利呢——黑屏、反复重启、字符串不显示、跳转后崩溃——这些几乎是每个写 bootloader 的人都会遇到的情况。这一篇我们换个角度，不讲新功能，而是把构建系统的细节、QEMU 调试手段、以及最常见的启动故障排查方法整理成一篇完整的实战指南。说实话，这一篇里的内容有一半是笔者自己踩坑之后总结出来的，如果早点有人告诉我，至少能省两天调试时间。

前置要求：完成前两篇，MBR 和 Stage2 的代码已经写好，构建系统正常工作。

## 构建系统——从源码到磁盘镜像的全链路

### build_image.sh 做了什么

我们每次运行 `cmake --build build` 后，最终的磁盘镜像 `cinux.img` 是由 `scripts/build_image.sh` 这个脚本组装的。它的工作流程很直观：先用 dd 创建一个 1MB 的全零文件作为磁盘镜像骨架，然后把 mbr.bin 写到扇区 0（偏移 0 字节），再把 stage2.bin 写到扇区 1 开始的位置（偏移 512 字节）。

这里有几个细节值得注意。dd 命令的 `conv=notrunc` 参数不能少——没有它 dd 会截断输出文件，写入 Stage2 时会把后面还没用到的零填充空间丢掉。`seek` 参数让 dd 在输出文件中跳过指定数量的扇区，精确控制 Stage2 的写入位置。脚本还包含了验证逻辑：读出镜像偏移 510-511 的两个字节检查是否为 MBR 签名（0x55 0xAA），以及计算 Stage2 的实际扇区数是否超过 15（MBR 的 DAP 里硬编码了读取 15 个扇区，超过的部分不会被读进来）。

### 修改源码后构建不更新？检查 DEPENDS

这是一个非常常见的坑。CMake 的 `add_custom_command` 里 `DEPENDS` 列表决定了什么时候触发重新构建。如果只写了 `DEPENDS mbr` 而忘了 `stage2`，修改 stage2.S 后重新构建不会更新磁盘镜像——因为 CMake 认为镜像不依赖 stage2 目标。正确的写法是 `DEPENDS mbr stage2`，确保两个目标都构建完成后才执行镜像组装。

### objcopy 的作用

链接器输出的是 ELF 格式的可执行文件，包含段头、符号表等元数据。但 BIOS 不认识 ELF——它只认原始的二进制机器码。`objcopy -O binary` 把 ELF 转成纯二进制：剥离所有 ELF 头和段头，只保留代码和数据的原始内容。对 MBR 来说输出应该恰好 512 字节（因为 `.org 510` 加上 `.word 0xAA55` 精确填充到了 512 字节末尾），如果超过说明代码太大了。

### 验证构建产物

构建完成后，可以用几条命令验证产物是否正确。检查 MBR 大小是否恰好 512 字节，检查 MBR 签名是否正确（用 xxd 或 hexdump 查看最后两个字节），检查 Stage2 的大小是否在 15 个扇区以内（7680 字节）。

验证命令参考：

```
ls -la build/boot/mbr.bin build/boot/stage2.bin
xxd -s 510 -l 2 build/boot/mbr.bin
```

MBR 大小必须是恰好 512 字节——不多不少。如果 mbr.bin 大于 512 字节，说明代码段超过了 510 字节（加上 2 字节签名），BIOS 只会读取第一个 512 字节，签名位置对不上，结果就是无限重启。如果小于 512 字节（虽然几乎不可能出现），说明 objcopy 或者链接脚本有问题。Stage2 的大小通常在 1-2KB 左右，只要不超过 7680 字节（15 个扇区 × 512 字节）就没问题。

还有一个容易被忽略的验证点：检查 `build/cinux.img` 的大小。它应该是 1048576 字节（1MB = 2048 个扇区）。如果大小不对，说明 build_image.sh 的 dd 命令出了问题——最常见的原因就是前面提到的 `conv=notrunc` 缺失。

### 链接脚本的作用

MBR 和 Stage2 使用不同的链接脚本，理解它们的差异对排查地址相关的 bug 至关重要。MBR 的链接脚本起始地址是 0x7C00——这是 BIOS 加载 MBR 的实际物理地址，所以链接地址等于运行地址。Stage2 的链接脚本起始地址是 0——这是因为 MBR 用 `ljmp $0x0800, $0` 跳转到 Stage2，CS=0x0800 提供了段基地址（0x0800 × 16 = 0x8000），链接地址只需要从 0 开始的段内偏移就够了。

如果 Stage2 的链接脚本错误地写成了 `. = 0x8000`，会发生什么？代码能执行（因为 CS:IP 计算正确），但所有数据访问都会多算一个 0x8000——字符串的链接地址变成了 0x8000 + offset，而 DS:SI 实际访问的地址是 CS × 16 + 链接地址 = 0x8000 + 0x8000 + offset，完全不对。这个 bug 非常隐蔽，因为代码看起来能跑，只是打印出来的全是乱码或者干脆不显示。

## QEMU 调试实战

### 基本调试启动方式

QEMU 提供了丰富的调试支持。最常用的启动方式是 `-S -gdb tcp::1234`——`-S` 让 QEMU 启动后立即暂停（不执行任何指令），`-gdb tcp::1234` 开放一个 GDB 调试端口。然后在另一个终端启动 GDB 连接上去。实模式调试必须 `set architecture i8086`，否则 GDB 会按 64 位模式反汇编，看到的全是乱码。

调试启动命令参考：

```
qemu-system-x86_64 -S -gdb tcp::1234 -drive format=raw,file=build/cinux.img -serial stdio
```

### QEMU Monitor 命令

QEMU 运行时按 `Ctrl+A, C` 可以进入 monitor 模式。几个常用的 monitor 命令：`xp /512bx 0x7c00` 查看 MBR 是否被正确加载到内存（最后两个字节应该是 55 aa）；`xp /16bx 0x6400` 查看帧缓冲信息是否正确保存；`info registers` 查看当前寄存器状态。退出 monitor 再按一次 `Ctrl+A, C`。

### GDB 实模式调试要点

实模式下 GDB 的地址显示需要特别注意——GDB 显示的是线性地址（CS × 16 + IP），不是 CS:IP 对。所以在 0x7C00 设断点时要直接用 `break *0x7c00`，不需要手动计算段偏移。

Stage2 在 0x8000 开始执行，但因为 CS=0x0800，GDB 中设断点同样直接用物理地址 `break *0x8000` 就行。单步执行时注意 `si`（指令级单步）和 `ni`（跳过函数调用）的区别——BIOS INT 调用用 `ni` 跳过比较方便，否则会一头扎进 BIOS 内部看不到头。

GDB 里查看内存的几个常用命令也值得记一下。`x/16bx 0x7c00` 以十六进制字节方式查看 16 字节内存，适合查看机器码和签名。`x/4xw 0x6400` 以十六进制 word 方式查看 4 个 word，适合查看帧缓冲信息。`info registers` 查看所有通用寄存器和段寄存器的值——调试段寄存器问题时特别有用。`x/i $pc` 反汇编当前位置的指令，确认 CPU 到底在执行什么。

有一点很重要：GDB 连接后 CPU 会在第一条指令处暂停（因为用了 -S 参数）。如果你想从 BIOS 执行过程就开始观察，第一条指令的地址是 0xFFFFFFF0（现代 x86 CPU 的复位向量），但实模式下 GDB 只能看到低 20 位地址，所以断点要设在 0xFFFF0。不过通常我们不需要调试 BIOS 本身——在 0x7C00 设断点等待 MBR 加载就足够了。

### 调试 Stage2 的 VESA 流程

VESA 初始化是最难调试的部分，因为它跨越了"文本模式"和"图形模式"两个阶段。我们的策略是分阶段设断点：先在 `vesa_get_controller_info` 调用前设断点，确认 ES:DI 和签名正确；然后在 `vesa_get_mode_info` 调用后设断点，检查 0x6200 处的 ModeInfoBlock 内容；最后在 `vesa_set_mode` 调用前设断点，确认 BX 的值是 0x4118（模式号 + 线性帧缓冲标志）。

每个 VESA 函数调用后都要检查 AL 和 AH 的值：AL=0x4F 表示功能被支持，AH=0 表示调用成功。如果 AL 不是 0x4F，说明显卡不支持 VBE——这在 QEMU 里几乎不会发生，但在真机上可能遇到。如果 AL=0x4F 但 AH 不是 0，检查 ES:DI 是否正确指向了缓冲区。

### 串口调试

QEMU 加上 `-serial stdio` 参数后，后续内核的串口输出会直接显示在终端里。不过当前 tag 的实模式代码还没有初始化串口，这个参数要到后续 tag 才会发挥作用。提前加上也没坏处，至少不会报错。

### QEMU 日志输出

QEMU 还支持 `-d` 参数输出各种调试日志。`-d int` 打印所有中断和异常信息，排查三重故障时非常有用。`-d cpu_reset` 打印 CPU 重置原因。`-d in_asm` 打印执行的汇编指令（量大，慎用）。日志默认输出到 stderr，可以用 `-D qemu.log` 重定向到文件。

对于排查"QEMU 突然重启"的问题，推荐组合是 `-d int,cpu_reset -D qemu.log`，运行后检查日志文件中的最后几条记录，就能看到是哪次中断或异常导致了重置。在实模式下最常见的触发重置的原因是三重故障——CPU 在处理一个异常时又触发了另一个异常，连续三次就彻底重置。

### 用 objdump 反汇编 ELF

在不用 GDB 的情况下，可以用 objdump 反汇编编译产物来验证代码是否正确。命令是 `objdump -d -m i8086 build/boot/mbr`（注意是 ELF 文件，不是 bin 文件）。这会显示每条指令的地址和机器码，方便对照源码检查。特别有用的场景是确认 ljmp 是否正确生成、DAP 常量是否被正确编码。

如果 objdump 输出看起来像 32 位或 64 位指令，说明 `-m i8086` 参数没起作用或者代码段的模式标记不对。MBR 用 `.code16` 标注，objdump 加上 `-m i8086` 应该能正确反汇编。

## 常见故障排查

### QEMU 反复重启循环

这是最常见的故障——QEMU 启动后屏幕一闪然后又重启，无限循环。原因几乎总是 MBR 签名缺失或错误。BIOS 检查 0x7DFE-0x7DFF 处的两个字节，不是 0x55 0xAA 就拒绝执行，然后尝试下一个启动设备，找不到就重启。排查方法：用 GDB 在 0x7C00 设断点，检查 0x7DFE 处的值。常见原因包括 objcopy 输出超过 512 字节（签名被截断）、链接脚本 `.org 510` 写错了位置、或者构建时根本没有把签名编译进去。

### 字符串不显示

MBR 的打印函数能正常工作需要三个前提条件：CS 和 DS 一致（都是 0）、方向标志清除（DF=0）、SI 指向正确的字符串地址。如果字符串不显示，先在 GDB 里检查 DS 和 EFLAGS.DF 的值。另一个可能的原因是 `ljmp` 那条远跳转没写——某些 BIOS 上 CS 不为 0，字符串的链接地址和 DS:SI 实际访问的地址不匹配。

### Stage2 跳转后崩溃

MBR 跳转到 Stage2 后如果立刻崩溃或重启，排查思路分两路。第一条路：磁盘读取是否成功——用 GDB 在 `INT 0x13` 调用后检查 CF 标志，CF=0 说明成功。也可以直接检查 0x8000 处的内存内容，看有没有合理的机器码。第二条路：地址是否正确——Stage2 的链接脚本起始地址必须是 0（不是 0x8000），因为 CS=0x0800 已经提供了段偏移。

### INT 0x13 返回失败（CF=1）

磁盘读取失败通常有两个原因。第一是 DAP 字段填错了——偏移 1 的保留字节必须为 0，有些 BIOS 会检查这个字段。第二是启动盘号不对——DL 寄存器在 MBR 入口时由 BIOS 设置，必须在第一时间保存，不能被后续操作覆盖。用 GDB 在调用 INT 0x13 前检查 DL 的值和 DAP 的各个字段。

用 GDB 检查 DAP 的方法：DAP 在 0x7B00，共 16 字节。`xp /16bx 0x7B00` 可以看到完整的 DAP 内容。对照规范逐字段核对：偏移 0 应该是 0x10（结构大小），偏移 1 应该是 0x00（保留），偏移 2-3 是扇区数（0x0F 0x00 = 15），偏移 4-5 是缓冲区偏移（0x00 0x80 = 0x8000），偏移 6-7 是缓冲区段（0x00 0x00），偏移 8-15 是 64 位 LBA（0x01 0x00 0x00 0x00 0x00 0x00 0x00 0x00 = LBA 1）。注意所有数值都是小端序——如果你看到的是 0x00 0x80 而不是 0x80 0x00，那才是正确的小端表示。

### 构建后镜像没有更新

如果你修改了源码但 QEMU 运行结果没变化，首先确认构建确实执行了。检查 `ls -la build/cinux.img` 的时间戳是否更新了。如果没有，可能是 CMake 的依赖声明缺少某个目标——检查 cmake/qemu.cmake 中的 DEPENDS 列表是否同时包含 mbr 和 stage2。还有一种可能是 CMake 缓存了旧的配置，试试 `cmake --fresh -B build` 重新配置。

### VESA 初始化失败

VESA 的三个步骤（控制器信息、模式信息、设置模式）都有可能失败。最常见的是模式号不被支持——0x118 在 QEMU 上通常没问题，但在某些真机或旧版 QEMU 上可能不可用。排查方法：先确认控制器信息获取成功（AL=0x4F 且 AH=0），然后遍历 VbeInfoBlock 中的模式列表，用 INT 0x10 AX=0x4F01 逐个查询，找到一个支持 1024×768×32bpp 的模式号。

### 设置图形模式后无法验证

这是设计上的"正常困难"——设置模式后文本输出就废了，不能再用 INT 0x10 打印信息。唯一的验证手段是 GDB：在 Stage2 的 halt 循环处设断点，检查 0x6400 处的帧缓冲信息。PhysBasePtr 应该是 0xFD000000（QEMU 的典型值），pitch 应该是 0x1000（4096 字节），宽高分别是 1024 和 768。如果想验证帧缓冲本身能写，可以在 GDB 里直接向帧缓冲地址写颜色值——具体命令在前一篇的验证部分有写。

### "代码偶尔能跑偶尔不能跑"——最令人崩溃的 bug

这种不确定性 bug 在 bootloader 开发中特别常见，根本原因通常是以下几个之一。第一是 CS 不确定：没有写 ljmp 规范化 CS 的代码，在某些 BIOS 上 CS=0（正常），某些上 CS=0x07C0（地址偏移），结果就是同样的代码在不同环境下表现不同。第二是寄存器未初始化：BIOS 跳转后只有 DL 是确定的，其他所有寄存器的值都依赖 BIOS 实现的"善意"——大部分 BIOS 会把它们清零，但不保证。第三是内存布局冲突：栈、DAP、BIOS 数据区之间的边界没有留够安全距离，在某些 BIOS 上 BIOS 的栈使用量更大，踩到了我们的数据区。

排查这种 bug 的方法是构造一个"确定性环境"——用 GDB 从第一条指令开始单步执行，记录每个寄存器和关键内存位置的初始值和每次变化。然后在另一个 BIOS（比如换 SeaBIOS 为 Bochs BIOS，或者在真机上）上重复同样的过程，找出差异。QEMU 可以用 `-bios` 参数指定不同的 BIOS 固件，方便对比测试。

### 三重故障（Triple Fault）

三重故障是 x86 特有的灾难性错误——CPU 遇到无法处理的异常时彻底重置。在 bootloader 阶段最常见的原因是 IDT（中断描述符表）还没设置好就发生了异常（比如除零错误、一般保护错误）。实模式下 IDT 使用 BIOS 预设的 IVT，通常不会出问题；但如果你的代码不小心覆盖了 IVT 区域（0x0000-0x03FF），所有中断处理都会飞掉。QEMU 检测到三重故障会自动重启虚拟机，日志里会有 "Triple fault" 的记录。排查方法：在 QEMU 启动命令加上 `-d int` 参数，QEMU 会在控制台打印每次中断和异常的详细信息。

## 内存布局速查

调试时经常需要查某个地址放了什么数据，这里整理一份完整的内存布局供快速查阅。

实模式引导阶段的内存布局从低到高依次是：0x0000-0x03FF 是 IVT（中断向量表，256 个 4 字节入口），BIOS 依赖它处理中断，千万别覆盖；0x0400-0x4FFF 是 BIOS 数据区，同样不能碰；0x5000-0x5FFF 是空闲区域；0x6000 是 512 字节的 VbeInfoBlock（VESA 控制器信息）；0x6200 是 256 字节的 ModeInfoBlock（VESA 模式信息）；0x6400 是 16 字节的 FB Info（帧缓冲物理地址、pitch、宽高）；0x7000 是 MBR 栈底向下增长；0x7B00 是 16 字节的 DAP 结构；0x7C00-0x7DFF 是 MBR 代码（512 字节）；0x8000 开始是 Stage2 代码（最多 7.5KB）；Stage2 的栈在 SS=0x0900, SP=0xFFFE（物理地址 0x9FFFE 附近）。

GDB 里查这些地址用 `xp` 命令：`xp /16bx 0x7DFE` 查 MBR 签名，`xp /256wx 0x6000` 查 VbeInfoBlock，`xp /16bx 0x6400` 查帧缓冲信息。

## objcopy 常见陷阱

objcopy 看起来简单（ELF 转二进制），但有几个容易踩的坑。

第一个坑是 `.rodata` 段的位置。如果 `.rodata`（只读数据，字符串常量通常放在这里）在 ELF 中位于 `.text` 后面且有空隙，objcopy 输出的二进制会包含这个空隙（用零填充），导致二进制尺寸远大于预期。Cinux 的解决方案是在链接脚本里把 `.rodata` 合并到 `.text` 段中：`*(.text)` 后面紧跟 `*(.rodata)`，确保代码和数据紧密排列。

第二个坑是 `.bss` 段的处理。`.bss` 是未初始化的全局变量存放的地方，在 ELF 中它只记录大小不占文件空间。objcopy 默认会包含 `.bss`（用零填充），这可能导致二进制文件末尾有一大段多余的零。MBR 的 `.org 510` 指令会精确控制大小，所以这个问题对 MBR 影响不大，但对 Stage2 来说可能让 bin 文件变大。

第三个坑是符号表和调试信息。objcopy -O binary 会剥离所有元数据，包括符号名和调试信息。这意味着你不能用 GDB 的符号名来设断点——只能用地址。如果需要带调试信息的调试，可以直接用 ELF 文件而不是 bin 文件，或者在 objcopy 之前先用 objdump 反汇编查看符号地址。

## 调试工作流推荐

这里总结一个高效的 bootloader 调试工作流，适合从零开始排查未知问题。

第一步是"看能不能活"：构建后运行 QEMU，不加任何调试参数。如果看到 `Cinux Booting...` 说明 MBR 基本正常。如果无限重启循环，问题出在 MBR 签名或代码本身。如果什么都没显示就停了，可能是代码逻辑有误（比如提前 hlt）。

第二步是"分段排查"：用 GDB 在关键位置设断点——0x7C00（MBR 入口）、0x8000（Stage2 入口）、halt 循环地址。运行到每个断点后检查寄存器和关键内存位置，确认上一阶段的执行结果是否符合预期。哪个阶段过不去就集中排查哪个阶段。

第三步是"逐步缩小范围"：在出问题的阶段内部设更多断点，逐条指令单步执行，跟踪关键寄存器（CS、DS、SI、DL）和内存（DAP、ModeInfoBlock、FB Info）的变化。大多数 bug 在这个过程中就能定位到具体是哪条指令出了问题。

第四步是"修改-编译-验证"循环：定位到 bug 后修改代码，重新构建（确认镜像更新了），用 GDB 验证修复效果。这一步要特别注意确认镜像确实被更新了——有时候你改了代码但忘了保存，或者 CMake 没有检测到文件变化（特别是修改了 .h 文件时可能需要 touch 一下 .S 文件触发重新编译）。

## 常见构建错误

不是所有问题都是代码 bug，有时候构建配置本身就有问题。这里列举几个常见的构建错误和排查方法。

"undefined reference to _start"：链接器找不到入口符号。原因通常是 `.global _start` 没写，或者源文件路径不对导致目标文件是空的。检查 CMakeLists.txt 里的源文件列表是否正确引用了 .S 文件。

"relocation truncated to fit"：链接时地址超出了范围。MBR 的 `.org 510` 和 512 字节限制可能导致这个问题——如果代码+数据超过了 510 字节，`.word 0xAA55` 就写不进去了。检查 objcopy 输出的 mbr.bin 大小。

"cannot represent relocation type"：16 位实模式代码和 32 位目标文件格式之间的兼容性问题。确保所有汇编文件都用 `.code16` 或 `.code16gcc` 标注，编译选项用 `-Wa,--32`，链接选项用 `-Wl,-m,elf_i386`。

"file not recognized: File format not recognized"：objcopy 输入的不是 ELF 文件。可能是因为编译步骤失败但链接步骤仍然被执行了，导致目标文件是空的或格式不对。先 `cmake --build build` 看看有没有编译错误。

## 多种 BIOS 环境测试

在 QEMU 中可以切换不同的 BIOS 固件来测试兼容性。默认使用 SeaBIOS（传统 BIOS 实现），可以用 `-bios /path/to/bios.bin` 指定其他 BIOS。Bochs BIOS 在一些边界情况下的行为和 SeaBIOS 不同，是测试兼容性的好选择。

某些 bug 只在特定 BIOS 下复现——比如前面提到的 CS 初始值问题，SeaBIOS 和 Bochs BIOS 可能有不同的行为。如果你的代码在一种环境下能跑但另一种不行，几乎可以确定是依赖了未定义行为（比如假设了某个寄存器的初始值）。

## 调试心态

最后说一点调试心态方面的事情。bootloader 调试和普通应用调试最大的不同是"没有 printf"。在实模式下你能用的输出手段极其有限——INT 0x10 打字（图形模式切换后也废了）、GDB 查内存、QEMU Monitor 查内存。这意味着你必须在脑子里模拟 CPU 的行为，理解每条指令对寄存器和内存的影响。刚开始会觉得非常痛苦，但习惯之后你会发现对 x86 架构的理解深入了一个层次——这种"被迫理解硬件"的体验恰恰是自己写 bootloader 最大的价值。

笔者的建议是：遇到 bug 不要急着改代码，先用 GDB 定位问题的准确位置。知道"哪里错了"比"怎么修"更重要——修的方法通常很直观，但定位问题可能需要耐心和系统性的排查。

## 收尾

到这里，实模式引导的三个篇章全部完成了——第一篇写了 MBR 的核心逻辑，第二篇搞定了 Stage2 的 A20 和 VESA 初始化，这一篇把构建系统和调试方法整理到位。说实话调试 bootloader 的体验很独特：你面对的是一个没有任何输出手段的环境（图形模式切换后连 printf 都没有），唯一的眼睛是 GDB 和 QEMU Monitor。习惯这种调试方式之后，后续内核开发中遇到什么奇怪的问题都不至于手足无措。

下一篇我们要离开实模式了——设置 GDT，切换到保护模式，然后继续推进到 64 位长模式。真正的 x86 内核开发才刚刚开始。
