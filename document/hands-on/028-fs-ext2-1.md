# 028-1 Hands-on: ext2 磁盘布局与核心数据结构

## 导语

上一个 tag 我们搭建了 VFS 框架，Inode、InodeOps、FileSystem 抽象基类、FDTable 全部就位，Ramdisk 也成功接入了 VFS 命名空间——在 shell 里 `cat /hello.txt` 能看到文件内容。但说实话，Ramdisk 只是个玩具：它把 ustar 归档整个加载到内存里，文件系统后端的"磁盘"其实就是一段 RAM。真正的操作系统需要真正的磁盘文件系统，而 ext2 是 Linux 世界里最经典的入门选择——结构清晰、文档齐全、实现友好，而且不需要处理日志（不像 ext3/ext4）那些让人头疼的复杂特性。

这一章我们先不写任何驱动代码，只做一件事：搞清楚 ext2 的磁盘布局。读完这一章之后，你应该能拿着一个 ext2 镜像文件，用宿主机上的工具（dumpe2fs、hexdump）把 Superblock、Block Group Descriptor、Inode、Directory Entry 的每一个字段都指认出来。这些知识是下一章实现挂载流程的绝对前提——你不理解磁盘上数据的排列方式，就无从下手写驱动。

知识前置：你需要完成 tag 025（AHCI 驱动）和 tag 027（VFS 框架），理解 SATA 磁盘的基本读写机制以及 VFS 中 Inode/InodeOps 的设计意图。

## 概念精讲

### 为什么是 ext2

ext2（Second Extended Filesystem）诞生于 1993 年，由 Rémy Card 设计，作为 ext（First Extended Filesystem）的继任者。在 ext3 于 2001 年引入日志功能之前，ext2 曾是 Linux 发行版的默认文件系统长达近十年。时至今日，虽然桌面和服务器早已迁移到 ext4 甚至 Btrfs，但 ext2 在教学 OS 领域的地位依然无可替代——原因很简单，它在"功能性"和"可实现性"之间取得了教科书级别的平衡。

我们具体来对比一下可选方案。FAT32 虽然结构极简（只有 FAT 表和目录项），但它缺乏 Unix 语义——没有 inode 的概念，没有权限位，没有符号链接支持，文件名大小写不敏感。如果你把 FAT32 作为内核的"主文件系统"，后续实现 execve 加载 ELF 时会非常别扭，因为 Unix 文件权限（可执行位）无处存放。ext4 的另一个极端是过于复杂——它引入了 extent 树（替代传统的间接块指针）、延迟分配、多块分配、日志（jbd2），代码量是 ext2 的好几倍。而 ext2 恰好处于两者之间：有完整的 Unix 语义（inode、权限、硬链接），但数据结构扁平直观，一个本科生在一个学期的课程中就能实现基本可用版本。

更重要的是生态支持。OSDev Wiki 上有 ext2 的完整字段级文档（https://wiki.osdev.org/Ext2），每个结构的每一个字节偏移都列得清清楚楚。Linux 宿主机自带 mkfs.ext2 和 dumpe2fs 工具，你可以随时创建镜像、查看内部结构、验证你的驱动读取是否正确。这种调试体验是其他文件系统很难提供的——你甚至不需要写一行代码就能在宿主机上把 ext2 的每一个字节都摸清楚。

### 块（Block）—— ext2 的基本 I/O 单位

ext2 把磁盘空间划分为固定大小的"块"（block），块大小可以是 1024、2048 或 4096 字节——必须是 1024 的整数倍，且不超过一个页面（4096）。这个概念和磁盘扇区不同：扇区是硬件层面的最小读写单位（通常 512 字节），而块是 ext2 文件系统层面的逻辑单位。一个块可能跨越多个扇区——比如 1024 字节的块 = 2 个扇区，4096 字节的块 = 8 个扇区。Cinux 的 ext2 驱动使用 1024 字节块大小（mkfs.ext2 的默认值），这意味着每次 read_block 需要读 2 个扇区。

块大小不是直接存储在 Superblock 里的——Superblock 存的是一个指数值 `s_log_block_size`，实际块大小等于 1024 左移这个值。也就是说，`s_log_block_size` 为 0 时块大小 1024，为 1 时 2048，为 2 时 4096。这种设计看起来多此一举，但这是 1993 年的存储节省习惯——用 4 字节存储一个可能的大数，而不是直接存一个实际值。

### 块组（Block Group）—— 局部性原理的体现

ext2 最核心的设计思想是块组：整个磁盘被划分为若干个块组，每个块组包含自己的数据块位图、inode 位图、inode 表和数据块区域。你可以把它想象成一栋大楼被分成若干楼层，每层楼都有自己的前台（inode 表）和储物间（数据块），不需要每次都跑到一楼去取东西。

这种设计的直接好处是局部性——一个文件的 inode 和它的数据块尽量放在同一个块组里，减少磁盘寻道时间。你可以想象一下，如果不分组，所有 inode 集中在磁盘开头、所有数据集中在磁盘后面，那么读取一个小文件就需要磁头在磁盘的前半部分和后半部分之间来回移动，这在机械硬盘上会导致显著的延迟。块组把相关的元数据和数据在物理上聚集在一起，一次寻道就能同时拿到 inode 和数据块。

对于我们的内核来说，这个设计还带来了一个意外的简化：只要理解了一个块组的结构，你就理解了所有块组，它们是完全同构的。每个块组描述符（Block Group Descriptor）只有 32 字节，记录了该组的位图和 inode 表的起始块号以及空闲计数。

我们还需要知道一个块组内各部分的排列顺序。从低地址到高地址依次是：Superblock 备份（部分块组才有）、Block Group Descriptor Table 备份（部分块组才有）、Data Block Bitmap（占一个块）、Inode Bitmap（占一个块）、Inode Table（占若干个块，大小取决于每组的 inode 数量除以每块的 inode 数量）、最后是 Data Blocks。不过我们驱动不需要关心这个排列顺序——我们只需要通过 BGDT 中记录的块号来定位每个部分即可。

### Superblock —— 文件系统的"身份证"

Superblock 位于磁盘的字节偏移 1024 处（不是 0！不是 0！不是 0！），固定 1024 字节大小。这个位置是历史遗留——ext2 的前 1024 字节留给 boot sector（引导扇区），Superblock 从第 1024 字节开始。对于 1024 字节块大小来说，Superblock 恰好占据 block 1（block 0 是引导块）。

Superblock 里最重要的字段有这么几个：`s_magic`（偏移 56 字节，值必须是 0xEF53，这是 ext2 的签名）、`s_log_block_size`（偏移 24 字节，用来计算块大小）、`s_inodes_count` 和 `s_blocks_count`（文件系统的 inode 和块总数）、`s_blocks_per_group` 和 `s_inodes_per_group`（每组的块数和 inode 数）、`s_rev_level`（修订版本号，0 表示老格式，1 表示有扩展字段）。如果 `s_rev_level >= 1`，还需要关注 `s_inode_size`（偏移 88 字节，inode 结构的实际大小，老版本固定 128 字节）和 `s_first_ino`（第一个非保留 inode 的编号，老版本是 11）。

我们驱动挂载时的第一步就是读 Superblock、验 magic、算 block_size——如果 magic 不对，后面全是白搭。

### Block Group Descriptor —— 每组的导航表

Block Group Descriptor Table（简称 BGDT）紧跟在 Superblock 所在的块之后。对于 1024 字节块，Superblock 在 block 1，BGDT 从 block 2 开始；对于更大的块，Superblock 在 block 0 的偏移 1024 处，BGDT 从 block 1 开始。每个描述符固定 32 字节，里面最重要的三个字段是：`bg_block_bitmap`（该组数据块位图所在的块号）、`bg_inode_bitmap`（该组 inode 位图所在的块号）、`bg_inode_table`（该组 inode 表的起始块号）。

这三个地址是我们后续"根据 inode 编号找到 inode 数据"和"分配新块/inode"的关键。拿到 BGDT 之后，我们就知道了每一组的 inode 表在哪里、位图在哪里——所有后续操作都依赖这些地址。

### Inode —— 文件的元数据容器

ext2 的 inode 是一个 128 字节的结构（老版本）或更大的结构（新版本，由 `s_inode_size` 决定，但前 128 字节的布局是固定的）。每个 inode 代表一个文件系统对象（文件、目录、符号链接等）。关键字段包括 `i_mode`（类型加权限，用掩码 0xF000 提取类型部分，0x8000 是普通文件，0x4000 是目录）、`i_size`（文件字节大小）、`i_links_count`（硬链接计数）。

最核心的部分是 `i_block` 数组——15 个 32 位块指针。前 12 个（`i_block[0]` 到 `i_block[11]`）是直接块指针，每个直接指向一个数据块。第 13 个（`i_block[12]`）是单间接指针——它指向的块不是数据，而是一个"指针数组"，数组里每个元素才是真正的数据块号。第 14 个是双间接指针，第 15 个是三间接指针。对于 1024 字节块大小，单间接块能存 256 个块指针（1024/4），也就是说通过单间接块可以额外寻址 256KB 数据。对于我们的教学 OS 来说，实现直接块加单间接块已经完全够用了——双间接和三间接留给读者作为练习。

有一个非常容易踩坑的地方：**inode 编号从 1 开始**，不是从 0 开始。inode 0 不存在（它被用作"无效"标记），inode 1 是 bad blocks inode，inode 2 是根目录。所有涉及 inode 编号的计算都必须先减 1。这个设计看起来很反直觉——程序员习惯了从 0 开始索引——但 Unix 的 inode 编号就是这么规定的，从 V7 Unix 开始就是如此。原因已经淹没在历史中，可能是为了让 0 作为一个"无效"sentinel 值使用。

另外需要说明的是，前 10 个 inode（1-10）是保留的，分别有特殊用途——比如 inode 2 是根目录、inode 8 是 journal（如果有的话）。普通文件的 inode 从 `s_first_ino`（revision 0 时为 11）开始分配。不过在我们的驱动中，只需要关注 inode 2（根目录），其余保留 inode 的内容我们不需要关心。

### Directory Entry —— 变长链表

目录在 ext2 里本质上就是一个"文件"，它的 inode 指向的数据块里存储的不是普通数据，而是一组目录项（directory entry）。每个目录项是一个变长结构：前 4 字节是 inode 编号，接着 2 字节是本条目的总长度 `rec_len`，然后 1 字节是文件名长度 `name_len`，1 字节是文件类型 `file_type`，最后是变长的文件名。

遍历目录的方式是：从第一个条目开始，每次按 `rec_len` 步进到下一个条目。`rec_len` 包含了文件名加填充的总长度——文件名要求 4 字节对齐，所以实际 `rec_len` 可能大于 8 + `name_len`。如果某个条目的 `inode` 字段为 0，说明这是一个空位（被删除后留下的空洞），跳过即可。条目不能跨越块边界——如果一个块的最后剩余空间不够放一个新条目，上一个条目的 `rec_len` 会被扩展到块末尾，把剩余空间"吃掉"。

这种变长链表设计的好处是删除操作非常简单——只需要把前一个条目的 `rec_len` 加上被删除条目的 `rec_len`，或者把被删除条目的 `inode` 置 0。缺点是长时间使用后目录会产生碎片（空洞），但我们暂时不需要操心这个。

## 动手实现

这一章不需要编写内核代码，我们的任务是创建一个 ext2 镜像，用宿主机工具验证我们学到的磁盘布局。

### Step 1: 创建 ext2 测试镜像

**目标**: 在项目根目录下创建一个 8MB 的 ext2 镜像文件，挂载后放入几个测试文件。

**设计思路**: 使用 `dd` 创建空文件，`mkfs.ext2` 格式化为 ext2，然后通过 loop 设备挂载并写入测试数据。这些文件将在后续章节中作为 Cinux 内核读取的目标。

**验证**:
```bash
dd if=/dev/zero of=ext2.img bs=1M count=8
mkfs.ext2 -b 1024 ext2.img
sudo mkdir -p /mnt/ext2test
sudo mount -o loop ext2.img /mnt/ext2test
echo "Hello from ext2!" | sudo tee /mnt/ext2test/hello.txt
echo "Cinux OS kernel" | sudo tee /mnt/ext2test/readme.txt
sudo mkdir /mnt/ext2test/etc
echo "root:x:0:0" | sudo tee /mnt/ext2test/etc/passwd
sudo umount /mnt/ext2test
```

### Step 2: 用 hexdump 看看前 1024 字节——引导块

**目标**: 确认 block 0（字节 0-1023）确实是全零的引导块，建立"Superblock 不在磁盘开头"的直觉。

**验证**:
```bash
hexdump -C ext2.img -s 0 -n 64
```

预期输出是全零。这 1024 字节原本是给 boot loader 留的，在 QEMU 中我们不使用它，所以全是零。接下来你就会理解为什么 Superblock 要从 1024 开始——它给引导扇区留了空间。

### Step 3: 用 dumpe2fs 查看 Superblock 信息

**目标**: 理解 Superblock 中的关键字段及其含义。

**验证**:
```bash
dumpe2fs ext2.img 2>/dev/null | head -30
```

预期输出中你应该能看到：Inode count、Block count、Block size（1024）、Blocks per group、Inodes per group、Magic number 等信息。对照我们上面讲的字段含义逐条核实。

### Step 4: 用 hexdump 直接查看 Superblock 的 magic

**目标**: 验证 Superblock 位于字节偏移 1024，magic 为 0xEF53。

**验证**:
```bash
# Superblock 从字节 1024 开始，magic 在 Superblock 内偏移 56 字节
# 所以 magic 在字节 1024+56 = 1080 处
hexdump -C ext2.img -s 1080 -n 2
```

预期输出应该包含 `53 ef`（小端序，即 0xEF53）。如果你看到别的值，要么是偏移算错了，要么是镜像没有正确格式化。

### Step 5: 用 hexdump 查看 BGDT

**目标**: 验证 BGDT 紧跟 Superblock 之后（block 2，即字节偏移 2048）。

**验证**:
```bash
# 对于 1024 字节块，BGDT 从 block 2 = 字节 2048 开始
# 每个描述符 32 字节
hexdump -C ext2.img -s 2048 -n 32
```

输出中你应该能看到三组 4 字节值，分别是 block bitmap、inode bitmap、inode table 的块号。

### Step 6: 查看 dumpe2fs 的块组信息

**目标**: 验证块组结构。

**验证**:
```bash
dumpe2fs ext2.img 2>/dev/null | grep -A5 "Group 0"
```

这会显示第 0 组的详细信息：block bitmap、inode bitmap、inode table 的位置，以及空闲计数。

### Step 7: 用 hexdump 查看根目录的 inode 数据

**目标**: 手动定位 inode 2（根目录）的磁盘位置，验证它确实是目录类型（mode 高 4 位为 0x4000）。

**设计思路**: inode 2 在 group 0 中，组内索引为 (2-1)%inodes_per_group = 1。inode table 起始块号可以从 BGDT 中读出（上一个 step 看到的 inode table 值），假设是 block N，那么 inode 2 的字节偏移 = N * 1024 + 1 * 128。先用 dumpe2fs 确认 inode table 的块号，再手动计算偏移。

**验证**:
```bash
# 先获取 inode table 的块号（从 dumpe2fs 输出）
dumpe2fs ext2.img 2>/dev/null | grep "Inode table"
# 假设输出是 "Inode table at 5-8"，则 inode table 从 block 5 开始
# inode 2 = block 5, offset (2-1)*128 = 128 字节处
# 字节偏移 = 5*1024 + 128 = 5248
# 查看 128 字节的 inode 数据
hexdump -C ext2.img -s 5248 -n 128
```

输出的前 2 字节是 i_mode（小端序），如果根目录是目录类型，高字节应该是 0x41（0x4000 | 0x01ED = 0x41ED，即目录 + rwxr-xr-x 权限）。紧接着的 2 字节是 i_uid，然后 4 字节是 i_size（目录的大小，通常是一个块的大小即 1024）。再往下跳到偏移 40 处（i_block 数组的起始），你应该能看到根目录数据块的块号——这个块里存放的就是 "."、".."、"hello.txt" 等目录项。

## 调试技巧

**hexdump 看不到 0xEF53**: 最常见的原因是偏移算错了。记住 Superblock 在字节 1024，不是块 0。在 1024 字节块大小下，block 0 是引导块（全零），block 1 才是 Superblock。如果你从字节 0 开始看，看到的全是零——那是正常的，但那不是 Superblock。

**mkfs.ext2 报错 "not a block device"**: 普通文件也能用 mkfs.ext2 格式化，但可能需要加 `-F` 参数强制执行：`mkfs.ext2 -F -b 1024 ext2.img`。另外，8MB 的镜像用默认参数格式化后，block 数大约 8192、inode 数大约 2048——这些数值在 dumpe2fs 的输出中都可以确认。

**dumpe2fs 输出太多**: 管道到 `head` 或 `grep` 来过滤。我们主要关注 Superblock 信息和 Group 0 的描述。一个有用的命令是 `dumpe2fs ext2.img 2>/dev/null | grep -E "Inode count|Block count|Block size|blocks per group|inodes per group"`，这会只显示我们驱动最关心的几个参数。

**hexdump 看不到 i_block 中的块号**: 如果你定位 inode 正确但 i_block 数组全为 0，说明你可能在看错误的 inode——检查你的偏移计算是否正确。在 1024 字节块大小下，inode table 起始块号通常是 5 或更大的值（前面有 boot block、superblock、BGDT、block bitmap、inode bitmap），所以 inode 2 的偏移不会特别小。

## 本章小结

| 概念 | 要点 |
|------|------|
| Block | ext2 的基本 I/O 单位，大小 1024/2048/4096 字节 |
| Block Group | 磁盘被划分为多个块组，每组包含位图+inode表+数据块 |
| Superblock | 偏移 1024，1024 字节，magic=0xEF53，存全局元数据 |
| BGDT | 紧跟 Superblock 之后，每组 32 字节，存位图和 inode 表地址 |
| Inode | 128 字节，12 直接块 + 1 单间接 + 1 双间接 + 1 三间接 |
| DirEntry | 变长链表，rec_len 步进，inode=0 表示空洞 |
| 关键陷阱 | Superblock 在偏移 1024；inode 编号从 1 开始；block_size 是左移值 |
