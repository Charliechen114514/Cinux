# 006-1 内存字面量与基础定义

## 导语

上一章我们让内核成功跑了起来——它能通过串口告诉我们 BootInfo 里有什么、E820 内存映射长什么样。但说实话，现在的内核就像一个只会看不会动手的旁观者，它能看到物理内存的布局，却连一个字节都分配不了。这一章我们要开始改变这个局面，第一步是搭建内存管理的基础设施：让代码里不用再写满屏幕的 `4096` 和 `0x100000`，而是用人类可读的方式表达内存大小。

完成本文后，你将拥有一套零运行时开销的内存字面量工具和对齐辅助函数，它们会在后续所有内存管理代码中被反复使用。

前置知识：上一章的 BootInfo 结构和 E820 内存映射，以及 C++ 基础的 `constexpr` 概念。

---

## 概念精讲

### 为什么需要内存字面量

写过底层代码的人一定对这种东西不陌生：`4096`、`0x100000`、`0x40000000`。单看数字你完全不知道它代表多大——4KB？1MB？还是 1GB？你只能靠数零来猜，而且很容易数错。C++11 引入的用户定义字面量（User-Defined Literals）可以很好地解决这个问题：我们定义 `4_KB`、`1_MB`、`4_GB` 这样的后缀，让编译器在编译期完成换算，运行时零开销。

这个机制的底层原理非常简单。我们定义一些名字形如 `operator""_KB` 的 `constexpr` 函数，编译器看到 `4_KB` 这个 token 时，会自动调用 `operator""_KB(4)` 并把返回值内联到使用处。因为整个计算都是编译期常量表达式，生成的机器码和你手写 `4096` 完全一样。

### 对齐操作的位运算原理

内存管理里到处需要对齐操作——页对齐（4KB）、大页对齐（2MB）等等。对齐的本质就是把地址的低位清零。对于 2 的幂次对齐值（4KB = 2^12），有一个经典的位运算技巧：对齐值减一之后，低位全部是 1（比如 `4096 - 1 = 0xFFF`，12 个 1），取反就是低位全 0 的掩码。拿这个掩码和地址做 AND 操作，就完成了向下对齐；先加上 `align - 1` 再做 AND，就是向上对齐。

举个例子：`align_up(0x1234, 0x1000)` 的过程是 `0x1234 + 0xFFF = 0x2233`，然后 `0x2233 & ~0xFFF = 0x2000`。结果 0x1234 被向上对齐到了 0x2000，即第二个 4KB 页的起始地址。

## 动手实现

### Step 1: 创建内存字面量头文件

**目标**：在 `kernel/mini/mm/` 目录下创建 `memory_literals.h`，提供 `_KB`、`_MB`、`_GB`、`_TB` 四个字面量后缀。

**设计思路**：这些字面量本质上就是乘法运算的语法糖。`_KB` 乘 1024，`_MB` 乘 1024 的平方，以此类推。我们把它们放在 `cinux::mini::mm::literals` 命名空间里，然后在上一级命名空间 `cinux::mini::mm` 中用 `using namespace literals` 导入，这样在该命名空间内可以直接写 `4_KB` 而不用加限定符。

**实现约束**：

- 每个字面量运算符接收 `unsigned long long` 参数，返回 `uint64_t`
- 必须全部标记为 `constexpr`，确保编译期计算
- 后缀名必须以下划线开头（`_KB`），因为非下划线后缀是标准保留的
- 不依赖任何标准库头文件，只包含 `<stdint.h>`

**踩坑预警**：后缀名如果不用下划线开头（比如直接用 `KB`），标准规定只有标准库才能用不带下划线的字面量后缀，你的代码虽然可能编译通过，但属于未定义行为。另外别忘了乘法要用 `1024ULL` 而不是裸 `1024`，否则在 32 位平台上中间结果会溢出。

**验证**：这一步只是头文件，编译通过即可。在后续文件中 include 它，如果能正常使用 `4_KB`、`1_MB` 等表达式且值正确，就说明没问题。

---

### Step 2: 创建内存管理通用定义

**目标**：在同一个目录下创建 `mm_defines.h`，提供页大小常量和对齐辅助函数。

**设计思路**：这个头文件是整个内存管理子系统的"公共入口"——它 include 了上面的 `memory_literals.h`，然后定义了三级页大小常量（4KB、2MB、1GB）和三个对齐工具函数。其他内存管理代码只需要 include 这一个头文件，就能获得所有基础定义。

**实现约束**：

- 三个页大小常量：4KB 对应标准页，2MB 对应大页，1GB 对应巨型页（都是 x86-64 硬件支持的页大小）
- `align_up(addr, align)` 将地址向上对齐到最近的 align 边界
- `align_down(addr, align)` 将地址向下对齐到最近的 align 边界
- `is_aligned(addr, align)` 检查地址是否已经是对齐的
- 对齐参数必须是 2 的幂，函数不做运行时检查（教学内核，保持简单）
- 所有函数都是 `constexpr`，可以在编译期使用

**踩坑预警**：`align_up` 里有一个经典的整数溢出陷阱——如果 `addr + align - 1` 溢出了 `uint64_t` 的范围，结果就会出错。不过在 64 位地址空间下，除非你在处理接近 `UINT64_MAX` 的地址，否则一般不会遇到。另一个容易忽略的点是：`align_down` 不需要加法，所以不存在溢出问题，但如果传入的地址本身比 align 小，结果会是 0，这在某些场景下可能是意外的。

**验证**：同样只验证编译。你可以在测试文件中加几行静态断言来确认常量值是否正确，比如确认 `4_KB` 等于 4096，`align_up(0x1234, 0x1000)` 等于 0x2000 等。因为都是 `constexpr`，这些断言在编译期就能生效。

---

### Step 3: 验证编译集成

**目标**：确认这两个头文件能被项目中的其他文件正确 include。

**设计思路**：`mm_defines.h` 已经 include 了 `memory_literals.h`，所以我们只需要确认 `mm_defines.h` 能正确编译即可。这两个文件都是纯头文件，不需要修改 `CMakeLists.txt`——CMake 的 `target_include_directories` 已经把 `kernel/mini/` 目录加入了搜索路径，其他文件用 `#include "mm/mm_defines.h"` 就能找到它们。

**实现约束**：

- `mm_defines.h` include `memory_literals.h` 时使用相对路径（同一个 `mm/` 目录下）
- 命名空间约定：字面量定义在 `cinux::mini::mm::literals`，然后通过 `using namespace` 提升到 `cinux::mini::mm`
- 两个文件都使用 `#pragma once` 作为 include guard

**验证**：

```bash
# 在项目根目录
cmake --build build 2>&1 | head -20

# 如果编译报错找不到头文件，检查 CMakeLists.txt 中
# target_include_directories 是否包含了 kernel/mini/ 目录
```

编译成功就说明头文件的路径和命名空间配置都是正确的。

## 构建与运行

本文只是基础设施，还没有修改内核的运行行为。编译通过即可：

```bash
cd /path/to/cinux
mkdir -p build && cd build
cmake -DCMAKE_BUILD_TYPE=Debug -B . -S ..

```


## 调试技巧

### 常见问题排查

**问题 1：编译报错 "no matching literal operator"**

这说明编译器没找到字面量运算符的定义。最常见的原因是忘记在 `using namespace cinux::mini::mm;` 的作用域内使用字面量，或者根本没有 include `mm_defines.h`。检查 include 路径和命名空间是否正确。

**问题 2：`constexpr` 函数在运行时计算**

这不会导致错误，但意味着你丢掉了编译期优化的好处。确保传给 `align_up` 等函数的参数都是编译期常量——如果你用运行时变量调用它们，`constexpr` 就只是个普通函数修饰符，不会报错但也不会在编译期求值。

## 本章小结

本文创建了两个纯头文件，为后续的物理内存管理打下了基础。`memory_literals.h` 用 C++ 用户定义字面量提供了 `4_KB`、`1_MB` 等人类可读的内存大小表达方式，`mm_defines.h` 则在此基础上定义了页大小常量和对齐辅助函数。两个文件都是零运行时开销的编译期工具。

| 文件 | 提供的能力 | 存放位置 |
|------|-----------|---------|
| `memory_literals.h` | `_KB`/`_MB`/`_GB`/`_TB` 字面量 | `kernel/mini/mm/` |
| `mm_defines.h` | 页大小常量 + `align_up`/`align_down`/`is_aligned` | `kernel/mini/mm/` |
