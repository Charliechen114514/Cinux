# 002-1 · 保护模式概念与汇编修正

## 导语

如果你跟着上一章把实模式的 MBR 和 Stage2 跑通了，现在手里应该有一个能通过 BIOS 中断在屏幕上打印字符串、切换 VESA 图形模式、并且正确保存 framebuffer 信息的 bootloader 雏形。但实模式的 1MB 地址上限和没有任何内存保护的裸奔状态，终究撑不起一个现代操作系统。

接下来我们要做的，就是把 CPU 从实模式（Real Mode）推进到保护模式（Protected Mode）—— 这是 x86 处理器迈向 32 位寻址和特权级保护的第一道大门。完成 tag 002 的全部修改后，你会看到 QEMU 的 debugcon 终端输出一个字母 `P`，这个看似微不足道的字符背后是整个段描述符机制的建立和 CPU 模式的彻底切换。

如果你还没有完成 tag 001 的实模式环境搭建，请先回到上一章把基础链路跑通，否则接下来的改动在错误的代码基础上大概率会直接 triple fault。

tag 002 涉及五个文件的变更：`boot/common/serial.S` 的汇编模式修正、`boot/CMakeLists.txt` 的 linker script 修正、`cmake/qemu.cmake` 新增 debugcon 支持、`boot/stage2.S` 新增 GDT 定义和保护模式切换代码。所有代码都是纯汇编，没有引入 C 代码。

本篇是 tag 002 的第一篇，我们先搞清楚保护模式的核心概念——GDT、段描述符、段选择子、平坦内存模型——然后把 `serial.S` 的汇编模式隐患修干净。这些概念理解不到位，后面几篇的实操基本就是盲人摸象。

## 概念精讲

### 保护模式（Protected Mode）

你可以把实模式理解为一个没有法律的蛮荒小镇：任何程序都能直接读写任意物理内存地址，没有权限检查，没有内存隔离，一个野指针就能把整个系统搞崩。在单任务的 DOS 时代这还能凑合，但如果你要跑多任务操作系统，这就完全不可接受了。

保护模式就是 CPU 进入"法治社会"的第一步。它引入了三个核心机制。

第一，段描述符（Segment Descriptor）替代了原来粗暴的段地址方式，用一张描述符表来精确控制每个内存段的基地址、大小和访问权限——CPU 每次访问内存时都要先查表，确认你有权限访问这片区域，然后才能进行地址翻译。

第二，特权级（Ring 0 到 Ring 3）把内核和用户程序隔离开来，应用程序再也没法随便碰硬件——内核跑在 Ring 0（最高特权），用户程序跑在 Ring 3（最低特权），任何越权访问都会被 CPU 拦截并触发异常。

第三，分页机制的前置条件——虽然我们在本章不启用分页，但保护模式是后续开启分页、进入长模式的必经之路。从启动流程上看，BIOS 把控制权交给 bootloader 时 CPU 还在实模式，我们需要手动完成切换，这条路没有任何捷径可以绕过。

保护模式是 x86 处理器在 80286 时代引入的一种运行模式（后来被 80386 扩展到 32 位），Intel SDM Vol.3A Section 10.9.1 给出了完整的切换流程，我们后续会逐步展开。说到底，实模式和保护模式的切换不是"可选的优化"，而是 x86 启动流程中必须经过的一道门——不管你写的是 Linux 还是 xv6 还是一个教学 OS，只要目标是 32 位或 64 位，你就必须走完这条路。

关于保护模式下的地址翻译，有一个非常关键的机制值得提前理解。在实模式下，线性地址（此时也等于物理地址）的生成方式是 `段寄存器 * 16 + 偏移`，简单粗暴，但上限只有 1MB（20 位地址总线）。

到了保护模式，段寄存器里存的不再是一个段地址，而是一个叫段选择子的 16 位索引值。CPU 拿着这个索引去 GDT 里查对应的段描述符，从中取出一个 32 位的基地址（Base），然后把这个 Base 加上偏移量，得到线性地址。

如果开启了分页，线性地址还要再经过页表翻译才能得到最终的物理地址；但在我们的 bootloader 阶段，分页还没开启，线性地址就等于物理地址。这个"段选择子 -> GDT 查询 -> 得到 Base -> 加偏移 -> 线性地址"的流程，就是保护模式下内存寻址的核心机制。

### GDT（全局描述符表）

GDT 是保护模式下的"段定义手册"。在实模式下，段寄存器（CS、DS、ES 这些）直接存的就是段地址，CPU 用 `段地址 * 16 + 偏移` 的方式算出物理地址。但到了保护模式，段寄存器存的不再是段地址了，而是一个叫"段选择子"（Segment Selector）的索引值，这个索引指向 GDT 中的某个条目，每个条目叫一个"段描述符"，占 8 个字节，里面打包了这段内存的基地址、大小限制和访问权限。

x86 的段寄存器不是简单的 16 位寄存器——每个段寄存器内部都有一个"隐藏部分"（Hidden Part，也叫 Segment Cache），存储着当前段的 Base Address、Limit 和 Access Rights。这些隐藏信息是在段寄存器被加载时由 CPU 自动从 GDT 中读取并缓存的，后续所有的内存访问都直接使用缓存中的值，不会每次都去 GDT 查。

理解了这个缓存机制，后面讲"为什么远跳转不可省略"就水到渠成了——因为远跳转会强制刷新 CS 的隐藏缓存，让 CPU 从 GDT 中重新加载段描述符的属性。

每个段描述符的 8 字节布局比较紧凑，字段被拆散在不同的位置上（这是 x86 历史包袱的经典体现）。低两个字节是 Limit 的低 16 位，接下来两个字节是 Base 的低 16 位，第五个字节是 Base 的中间 8 位（Base 23:16），第六个字节是 Access Byte（访问权限），第七个字节是 Flags 加上 Limit 的高 4 位（Limit 19:16），最高字节是 Base 的高 8 位（Base 31:24）。

这种奇怪的排列方式是为了向后兼容 80286——286 的段描述符只有 6 字节（Byte 0-5），386 扩展到了 8 字节，新增的 Base 高 8 位和 Flags+Limit 高 4 位被巧妙地塞进了空隙里。Intel SDM Vol.3A Section 3.4.5（Figure 3-8）给出了完整的格式，初看确实不直观，但这就是 x86 的历史包袱，习惯了就好。

### Access Byte 详解

Access Byte 里的各个位各有讲究。P 位（Present）表示这个段是否在内存中——必须是 1 才能用，如果 P 为 0 则任何试图使用这个描述符的内存访问都会触发段不存在异常（#NP）。

DPL（Descriptor Privilege Level）两位表示特权级，我们用 Ring 0（即 DPL=00），这也是内核代码运行在的级别。后续如果要支持用户态程序，就需要额外定义 DPL=3 的描述符。

S 位区分这是代码/数据段描述符还是系统段描述符（TSS、LDT 等），我们定义的是代码段和数据段，所以 S 位为 1。Type 四位定义具体的段类型——对于代码段，Type 表示可执行、可读等属性；对于数据段，Type 表示可写、扩展方向等属性。

对于我们的代码段描述符，Access Byte 的值是 0x9A，拆开来看：P=1（段存在）、DPL=00（Ring 0）、S=1（代码/数据段）、Type=1010（可执行的代码段、可读）。对于数据段描述符，Access Byte 的值是 0x92：P=1、DPL=00、S=1、Type=0010（可读写的数据段）。

### Flags 字段详解

Flags 字节同样包含多个重要信息。G 位（Granularity）控制粒度——G=0 时 Limit 以字节为单位，G=1 时 Limit 以 4KB 页为单位。D/B 位表示默认操作数宽度——在保护模式下 D/B=1 表示 32 位，这意味着 push/pop/call 的默认宽度都是 32 位。

L 位是后来为 64 位长模式添加的——当 L=1 且 D/B=0 时，表示这是一个 64 位代码段。Flags 的低 4 位则是 Limit 的高 4 位，和低 16 位一起构成完整的 20 位 Limit 值。

我们的 Flags 值是 0xCF，拆开来看：G=1（4KB 粒度）、D/B=1（32 位默认操作数）、L=0（非 64 位）、高 4 位 Limit=0xF。配合 Limit 低 16 位的 0xFFFF，完整的 20 位 Limit 就是 0xFFFFF。再配合 G=1（4KB 粒度），实际覆盖范围是 0xFFFFF * 4096 = 4GB - 1。

### Flat Memory Model 与段选择子

我们的 GDT 采用"平坦内存模型"（Flat Memory Model），意思是代码段和数据段的基地址都是 0，大小限制都拉满到 4GB。这样一来，整个 4GB 地址空间对所有段都是平铺的、统一的，不再有传统意义上的"分段" —— 实际的内存保护留给后续的分页机制完成。

这不是我们拍脑袋想出来的设计，xv6、Linux、SerenityOS，几乎所有现代 x86 操作系统在 bootloader 阶段都这么做。具体来说，我们让 Base 全部为 0x00000000，Limit 全部为 20 位最大值（即 0xFFFFF），Granularity 位（G）设为 1 表示以 4KB 为单位。

代码段选择子的值是 `0x08`（索引 1，指向第二个描述符），数据段选择子的值是 `0x10`（索引 2，指向第三个描述符）。

在 x86 的历史中，分段保护（Segmentation）是最早的内存保护机制——不同段有不同的 Base 和 Limit，程序不能越界访问。但在现代操作系统里，分段保护基本被废弃了，取而代之的是分页机制（Paging），它提供了更灵活的 4KB 粒度的内存保护。

Linux、Windows、macOS 以及所有教学 OS 都使用 Flat Memory Model：GDT 里代码和数据都覆盖整个地址空间，Base 为 0，不做任何分段隔离，真正的内存隔离交给后续的页表实现。我们选择 Flat Model 的原因有三个。

第一，这是行业标准的做法，参考 xv6 的 `bootasm.S` 就会发现它的 GDT 和我们几乎一模一样。第二，在 Bootloader 阶段做分段保护没有意义——此时系统里只有一个"程序"在跑，就是 Bootloader 本身，没有隔离的需求。第三，后续实现分页时会重新定义内存布局，GDT 只是保护模式的一个"入场券"，不需要承担太多职责。

段选择子是 16 位值，高 13 位是 GDT 中的索引号（每个描述符 8 字节，所以实际地址是 `GDT_base + Index * 8`），中间的 TI 位（Table Indicator）表示用 GDT（TI=0）还是 LDT（TI=1），低两位是请求特权级（RPL）。

bootloader 阶段 TI 和 RPL 始终为 0。所以代码段选择子 `0x08` 就是 Index=1、TI=0、RPL=0；数据段选择子 `0x10` 就是 Index=2、TI=0、RPL=0。你在做计算的时候可以这样验证：`0x08 >> 3 = 1`，`0x10 >> 3 = 2`，正好是 GDT 中的索引。

### Null 描述符的必要性

GDT 的第一个条目（索引 0）必须是 null descriptor——8 字节全零。Intel 硬件规定这个条目必须存在且必须全零，段选择子值为 `0x00` 时指向它，CPU 试图用 null selector 访问内存会触发 General Protection Fault（#GP）。

这不是可选的装饰，是强制的。这个设计的目的是让"忘记初始化段寄存器"这种编程错误能被立即捕获，而不是静默地读到随机数据——在 C 语言里，未初始化的变量默认值是 0，如果段选择子也是 0，那么 null descriptor 就能确保这种"未初始化"的情况被立即检测到。

这是一个非常典型的硬件辅助调试设计——x86 架构里有很多类似的"帮你踩刹车"的机制，比如对齐检查、栈溢出检测等。

### 实模式到保护模式切换流程概览

Intel SDM Volume 3A Section 10.9.1 给出了严格的切换步骤，顺序不可调换，任何一步缺失都可能导致不可预测的 CPU 行为。整个流程是六步。

第一步，用 `cli` 关闭中断。进入保护模式后 IDT（中断描述符表）还没建立，如果这时候发生中断，CPU 会尝试从默认的 IDT 地址读取，大概率触发 triple fault——CPU 先是 #GP，然后 #DF，最后 triple fault 导致系统重启。

第二步，确保 DS 寄存器为 0。`lgdt` 在实模式下使用的是 `DS * 16 + offset` 的地址计算方式，如果 DS 不为 0，GDT 的物理地址就会算错——这一点特别容易被忽略，但查起来让人血压拉满，因为 `lgdt` 本身不会报错，错误要等到后续第一次段访问时才会暴露。

第三步，用 `lgdt` 指令把 GDT 的位置和大小加载到 GDTR 寄存器。注意 `lgdt` 本身不校验 GDT 内容的合法性，它只是做了一次内存拷贝。如果 GDT 里的描述符是乱写的，`lgdt` 不会报错，错误会在后续 `ljmp` 尝试加载 CS 时才暴露。

第四步，设置 CR0 寄存器的 PE 位（Protection Enable，bit 0）为 1，这一步"通知"CPU 进入保护模式。但此时 CPU 的流水线里还缓存着 16 位的指令，CS 寄存器的隐藏部分还是实模式的旧值——CPU 仍然在用旧的段信息翻译地址、用 16 位模式解码指令。

第五步，执行一个远跳转（far jump），目标操作数包含新的代码段选择子和 32 位入口点的偏移地址，这个跳转会强制刷新 CS 寄存器和 CPU 流水线，让 CPU 真正以 32 位保护模式开始取指令。

第六步，进入 32 位代码后，立即把所有数据段寄存器（DS、ES、FS、GS、SS）加载为数据段选择子，并设置新的栈指针。

这六步里最容易被忽略、也最容易导致炸掉的是第五步的远跳转。如果不做这个跳转，CPU 虽然已经设置了 CR0.PE，但 CS 还指向实模式的段，流水线里的指令还在按 16 位解码，后续的 32 位指令会被错误解析，几乎必定 triple fault。具体的代码实操我们留到第三篇展开，这里先建立整体认知。

### GDTR 寄存器与 gdt_ptr 结构

`lgdt` 指令需要读取一个 6 字节的数据结构——前 2 字节是 GDT 的大小减 1（limit 字段），后 4 字节是 GDT 的线性基地址。Intel SDM Vol.3A Section 2.4.1 对此有详细说明。

在我们的设计中，GDT 有三个描述符（null + code32 + data32），每个 8 字节，总共 24 字节，所以 limit 的值是 23（即 0x17）。limit 之所以要减 1，是因为 limit 字段表示的是"GDT 的最后一个有效字节地址"，也就是总大小减 1。

基地址部分则直接使用 GDT 标号的链接地址——因为我们已经把 linker script 的 base 设成了 `0x8000`，链接器会自动算出正确的绝对地址。这里有一个非常关键的点需要注意：`lgdt` 指令在实模式下执行时，CPU 计算物理地址的方式是 `DS * 16 + offset`，所以我们在调用 `lgdt` 之前必须先确保 DS 为 0。

## 动手实现

### Step 1: 修正 16 位汇编调用约定

**目标**: 把 `serial.S` 中的汇编模式指示符从 `.code16gcc` 改为 `.code16`，同时把所有的 push/pop 指令改为显式的 16 位宽度后缀（`pushw`/`popw`）。

**设计思路**: 这里有一个经典的坑需要先说清楚。GNU Assembler 提供了两个 16 位模式指示符：`.code16gcc` 和 `.code16`。

`.code16gcc` 是专门为 GCC 生成的 16 位代码设计的，它允许汇编器生成 32 位指令编码（因为 GCC 即使在 16 位目标下也可能输出 32 位操作）。但 `.code16` 则严格要求所有指令按 16 位编码生成。两者的差异看似细微，但在栈操作指令上的影响是致命的。

问题出在 push 和 pop 这样的指令上。在 `.code16gcc` 模式下，没有显式后缀的 push 可能被汇编为 32 位操作，这就导致每次压栈消耗 4 个字节而不是 2 个字节。

如果你在某个函数里用 push 保存了三个寄存器，函数末尾用 pop 恢复，但一个是 32 位而另一个是 16 位，栈指针就会错位。

SP 从 0xFFFE 变成 0x000A 这种离谱的值就是这么来的，然后函数的返回地址就完全乱了，CPU 直接跳到随机地址执行。

所以我们的修正是：统一使用 `.code16` 确保编码一致性，同时把所有裸写的 push/pop 加上显式的 16 位宽度后缀，明确告诉汇编器我们就是要操作 16 位宽度。需要修改的文件是 `boot/common/serial.S`，改动涉及文件头部的 directive 和文件中所有函数里的栈操作指令，总共二十多处。

具体来说，受影响的函数包括 `print_string`、`enable_a20`、`vesa_get_controller_info`、`vesa_get_mode_info`、`vesa_save_framebuffer_info`——每个函数里都有若干 push/pop 需要加上宽度后缀。

不过需要注意一个细节：在 `serial.S` 的 `vesa_get_controller_info` 函数中，正常返回路径上有一个 pop 漏掉了宽度后缀——错误返回路径上是有后缀的但正常路径上没有。这种不一致性就是最容易踩的坑。汇编器在 `.code16` 下默认推断为 16 位，但显式写出宽度后缀是最安全的做法，可以让代码意图完全明确，不留下任何歧义。

从设计决策的角度来说，我们选择 `.code16` 而非 `.code16gcc` 的核心理由是确定性：`.code16` 要求所有指令宽度都是显式的，汇编器不会自作主张地加 32 位前缀。

我们的 Bootloader 是纯手写汇编，每条指令的宽度应该由程序员明确控制，而不是让汇编器猜。如果你翻看 xv6 的 `bootasm.S`（https://github.com/mit-pdos/xv6-public/blob/master/bootasm.S ），你会发现它从第一行起就是 `.code16`——这个选择在所有手写 16 位引导代码中是共识。

`.code16gcc` 的存在是为了给 GCC 的 C 编译器输出用的，GCC 在 `-m16` 模式下生成的汇编可能混合 16 位和 32 位操作，`.code16gcc` 告诉汇编器"我知道这里可能有 32 位指令，别报错"。但我们是手写汇编，完全没必要接受这种不确定性。

**踩坑预警**: 如果你发现 Stage2 在某些函数调用后跑飞了，SP 的值变得莫名其妙，十有八九是 push/pop 宽度不一致导致的栈错位。

排查方法是连上 GDB，在被调函数的入口和 ret 指令处设断点，分别打印 SP 的值，如果两次的差值不等于你 push 的总字节数，那就是宽度搞错了。

这个 bug 的可怕之处在于隐蔽性——在简单场景下（比如函数里只 push/pop 一次），32 位和 16 位 push 的结果可能恰好"看起来对"，因为实模式下的 32 位 push 只是把 SP 多减了 2 而已，数据本身还在栈上。

但当调用链变深、多个函数嵌套 push/pop 时，栈偏移就累积起来了，最终在某次 ret 时跳到错误地址——而且这个错误发生的位置跟真正的 bug 位置可能隔了十万八千里。

**验证**: 修改完成后，重新构建项目，确认 tag 001 的所有功能（Stage2 OK 打印、A20 使能、VESA 模式切换）不受影响：

```bash
mkdir -p build && cd build
cmake -DCMAKE_BUILD_TYPE=Debug -B . -S ..
make
make run
```

串口输出应该仍然能看到 `Stage2 OK` 和 `Mode info OK, switching...` 等消息。如果函数调用后 QEMU 直接重启，大概率就是 push/pop 宽度修正没到位导致的栈错位。

另外你也可以用 GDB 连上 QEMU，在 `_start` 和各个被调函数的入口处设断点，观察 SP 的变化是否符合预期——每次 call 之前 SP 应该是同一个值（因为 push 和 pop 严格配对），如果 SP 在函数返回后跟进入时不一致，就说明某个函数内部的栈操作宽度出了问题。

更进一步地，你可以在 GDB 中使用 `info registers` 查看 SP 的值，然后用 `x/4hx $sp` 查看栈顶的内容——如果 push 是 32 位的（多压了 2 个字节），你会看到栈上多了两个零字节。这种逐步验证虽然繁琐，但对于排查栈错位类的问题是最直接的方法。

另外值得注意的是，如果你在修改过程中发现 `make` 阶段就报错了（比如 "no such instruction" 或者 "suffix or operands invalid"），大概率是你在某个 push/pop 后面加了错误的后缀——比如写成了 `pushb` 而不是 `pushw`，或者在 `.code16` 模式下使用了 `pushl`。检查汇编错误信息中给出的行号，回到 `serial.S` 对应位置修正即可。

## 本篇小结

本篇我们理清了保护模式的核心概念体系：GDT 是保护模式下段定义的基础设施，段描述符用 8 字节描述一个内存段的基地址、限长和权限，段选择子是索引这些描述符的 16 位值，Flat Memory Model 是现代 OS 的共识选择。

Null 描述符虽然内容全是零，但它的存在是 Intel 硬件的强制要求，用于捕获未初始化段寄存器的编程错误。GDTR 寄存器通过 `lgdt` 指令加载，它读取一个 6 字节的伪描述符结构（2 字节 limit + 4 字节 base）。

Intel SDM Vol.3A Section 10.9.1 定义了严格的六步切换流程：CLI、DS 清零、LGDT、CR0.PE、远跳转、段寄存器重载。同时我们把 `serial.S` 的 `.code16gcc` 改成了 `.code16`，消除了 push/pop 宽度不一致的隐患——这个 bug 的隐蔽性极高，在简单的调用链下可能不暴露，但在嵌套调用时栈偏移会累积最终导致跑飞。

从工程实践的角度来看，这次修正体现了一个通用的原则：在手写汇编中，所有指令的语义应该尽可能显式——不要依赖汇编器的隐式推断，不要依赖默认行为。每个 push 和 pop 都应该明确标注宽度，每条分支指令都应该标明是长跳转还是短跳转，每个内存操作数都应该标明访问大小。这种"显式优于隐式"的编码风格，虽然写起来啰嗦一点，但能帮你避免大量的隐晦 bug。

下一篇我们将继续打地基——修正 linker script 的基地址、配置 QEMU debugcon、在 `stage2.S` 中定义完整的 GDT。这几个改动是保护模式切换的前置条件，每一个不动都会在后续让你收获一个漂亮的三重故障。

回顾一下本篇涉及的核心数值，你需要确保自己能脱口而出：GDT 中的 null 描述符占索引 0（选择子 0x00），代码段占索引 1（选择子 0x08），数据段占索引 2（选择子 0x10）。代码段的 Access Byte 是 0x9A，数据段是 0x92，Flags 都是 0xCF。Base 全部为 0x00000000，Limit 全部为 4GB（Flat Model）。切换流程六步严格有序：CLI -> DS 清零 -> LGDT -> CR0.PE -> 远跳转 -> 段寄存器重载。

如果你对任何一步的原理还不够清楚，建议回头重新阅读对应的概念精讲部分——后面的实操篇不会再重复这些概念，而是直接上手修改代码。概念理解不到位就去写代码，跟蒙着眼睛走雷区没什么区别。
