# 002 · GDT 与进入保护模式

## 导语

如果你跟着上一章把实模式的 MBR 和 Stage2 跑通了，现在手里应该有一个能通过 BIOS 中断在屏幕上打印字符串、切换 VESA 图形模式、并且正确保存 framebuffer 信息的 bootloader 雏形。但实模式的 1MB 地址上限和没有任何内存保护的裸奔状态，终究撑不起一个现代操作系统。接下来我们要做的，就是把 CPU 从实模式（Real Mode）推进到保护模式（Protected Mode）—— 这是 x86 处理器迈向 32 位寻址和特权级保护的第一道大门。

完成本章后，你会看到 QEMU 的 debugcon 终端输出一个字母 `P`，这个看似微不足道的字符背后是整个段描述符机制的建立和 CPU 模式的彻底切换。如果你还没有完成 tag 001 的实模式环境搭建，请先回到上一章把基础链路跑通，否则接下来的改动在错误的代码基础上大概率会直接 triple fault。

## 概念精讲

### 保护模式（Protected Mode）

你可以把实模式理解为一个没有法律的蛮荒小镇：任何程序都能直接读写任意物理内存地址，没有权限检查，没有内存隔离，一个野指针就能把整个系统搞崩。在单任务的 DOS 时代这还能凑合，但如果你要跑多任务操作系统，这就完全不可接受了。

保护模式就是 CPU 进入"法治社会"的第一步。它引入了三个核心机制：段描述符（Segment Descriptor）替代了原来粗暴的段地址方式，用一张描述符表来精确控制每个内存段的基地址、大小和访问权限；特权级（Ring 0 到 Ring 3）把内核和用户程序隔离开来，应用程序再也没法随便碰硬件；以及分页机制的前置条件 —— 虽然我们在本章不启用分页，但保护模式是后续开启分页、进入长模式的必经之路。从启动流程上看，BIOS 把控制权交给 bootloader 时 CPU 还在实模式，我们需要手动完成切换，这条路没有任何捷径可以绕过。

### GDT（全局描述符表）

GDT 是保护模式下的"段定义手册"。在实模式下，段寄存器（CS、DS、ES 这些）直接存的就是段地址，CPU 用 `段地址 * 16 + 偏移` 的方式算出物理地址。但到了保护模式，段寄存器存的不再是段地址了，而是一个叫"段选择子"（Segment Selector）的索引值，这个索引指向 GDT 中的某个条目，每个条目叫一个"段描述符"，占 8 个字节，里面打包了这段内存的基地址、大小限制和访问权限。

每个段描述符的 8 字节布局比较紧凑，字段被拆散在不同的位置上（这是 x86 历史包袱的经典体现），我们用 ASCII art 画一下它的结构：

```
  Byte 0   Byte 1   Byte 2   Byte 3   Byte 4   Byte 5   Byte 6   Byte 7
+--------+--------+--------+--------+--------+--------+--------+--------+
| Limit  | Limit  | Base   | Base   | Base   | Access | Flags  | Base   |
| 15:0   | 15:0   | 15:0   | 15:0   | 23:16  | Byte   |+Limit  | 31:24  |
|        |        |        |        |        |        | 19:16  |        |
+--------+--------+--------+--------+--------+--------+--------+--------+
                                          |       |          |
                                          |       |          +-- Granularity (G)
                                          |       |              Default size (D)
                                          |       |              Limit 高 4 位
                                          |       |
                                          |       +-- Present (P)
                                          |           DPL (2 bits)
                                          |           S (Descriptor type)
                                          |           Type (4 bits)
                                          |
                                          +-- Base 和 Limit 的低 16 位
```

这里最关键的字段是 Access Byte（第 5 字节）和 Flags + Limit 高 4 位（第 6 字节）。Access Byte 里的 P 位表示这个段是否在内存中（必须是 1 才能用），DPL 两位表示特权级（我们用 Ring 0），S 位区分这是代码/数据段还是系统段，Type 四位定义具体的段类型（可执行、可读、可写等）。第 6 字节的高 4 位里，G 位控制粒度（0 表示以字节为单位，1 表示以 4KB 为单位），D 位表示默认操作数宽度（保护模式下 D=1 表示 32 位）。

### Flat Memory Model

我们的 GDT 采用"平坦内存模型"（Flat Memory Model），意思是代码段和数据段的基地址都是 0，大小限制都拉满到 4GB。这样一来，整个 4GB 地址空间对所有段都是平铺的、统一的，不再有传统意义上的"分段" —— 实际的内存保护留给后续的分页机制完成。这不是我们拍脑袋想出来的设计，xv6、Linux、SerenityOS，几乎所有现代 x86 操作系统在 bootloader 阶段都这么做。

具体来说，我们让 Base 全部为 0，Limit 全部为 `0xFFFFF`（20 位最大值），Granularity 位（G）设为 1 表示以 4KB 为单位。这样 `0xFFFFF * 4096 = 4GB`，整个线性地址空间对代码段和数据段都是完全可见的。代码段选择子的值是 `0x08`（索引 1，指向第二个描述符），数据段选择子的值是 `0x10`（索引 2，指向第三个描述符）。

### 段选择子（Segment Selector）

段选择子是 16 位值，结构如下：

```
  15                    3  2  1  0
  +--------------------+--+--+--+
  |     Index          |TI| RPL |
  +--------------------+--+--+--+
```

Index 占 13 位，是 GDT 中的索引号（每个描述符 8 字节，所以实际地址是 `GDT_base + Index * 8`）。TI 位表示用 GDT（0）还是 LDT（1），我们只用 GDT。RPL 两位是请求特权级，bootloader 阶段始终为 0。所以代码段选择子 `0x08` 就是 Index=1、TI=0、RPL=0；数据段选择子 `0x10` 就是 Index=2、TI=0、RPL=0。

### 实模式到保护模式切换流程

Intel SDM Volume 3A Section 9.9 给出了严格的切换步骤，顺序不可调换，任何一步缺失都可能导致不可预测的 CPU 行为。整个流程是六步：

第一步，用 `cli` 关闭中断，因为进入保护模式后 IDT（中断描述符表）还没建立，如果这时候发生中断，CPU 会尝试从默认的 IDT 地址读取，大概率触发 triple fault。第二步，确保 DS 寄存器为 0，因为 `lgdt` 在实模式下使用的是 `DS * 16 + offset` 的地址计算方式，如果 DS 不为 0，GDT 的物理地址就会算错。第三步，用 `lgdt` 指令把 GDT 的位置和大小加载到 GDTR 寄存器。第四步，设置 CR0 寄存器的 PE 位（Protection Enable，bit 0）为 1，这一步"通知"CPU 进入保护模式，但此时 CPU 的流水线里还缓存着 16 位的指令。第五步，执行一个远跳转（far jump），目标操作数包含新的代码段选择子和 32 位入口点的偏移地址，这个跳转会强制刷新 CS 寄存器和 CPU 流水线，让 CPU 真正以 32 位保护模式开始取指令。第六步，进入 32 位代码后，立即把所有数据段寄存器（DS、ES、FS、GS、SS）加载为数据段选择子，并设置新的栈指针。

这六步里最容易被忽略、也最容易导致炸掉的是第五步的远跳转。如果不做这个跳转，CPU 虽然已经设置了 CR0.PE，但 CS 还指向实模式的段，流水线里的指令还在按 16 位解码，后续的 32 位指令会被错误解析，几乎必定 triple fault。

## 动手实现

### Step 1: 修正 16 位汇编调用约定

**目标**: 把 `serial.S` 中的汇编模式指示符从 `.code16gcc` 改为 `.code16`，同时把所有的 `push`/`pop` 指令改为显式的 `pushw`/`popw`（16 位宽度）。

**设计思路**: 这里有一个经典的坑需要先说清楚。GNU Assembler 提供了两个 16 位模式指示符：`.code16gcc` 和 `.code16`。`.code16gcc` 是专门为 GCC 生成的 16 位代码设计的，它允许汇编器生成 32 位指令编码（因为 GCC 即使在 16 位目标下也可能输出 32 位操作）。但 `.code16` 则严格要求所有指令按 16 位编码生成。

问题出在 `push` 和 `pop` 这样的指令上。在 `.code16gcc` 模式下，没有显式后缀的 `push` 可能被汇编为 32 位的 `pushl`，这就导致每次压栈消耗 4 个字节而不是 2 个字节。如果你在某个函数里用 `push` 保存了三个寄存器，函数末尾用 `pop` 恢复，但 push 是 32 位而 pop 是 16 位（或者反过来），栈指针就会错位。SP 从 0xFFFE 变成 0x000A 这种离谱的值就是这么来的，然后函数的返回地址就完全乱了，CPU 直接跳到随机地址执行。

所以我们的修正是：统一使用 `.code16` 确保编码一致性，同时把所有 `push`/`pop` 改为显式的 `pushw`/`popw`，明确告诉汇编器我们就是要操作 16 位宽度。不过需要注意一个细节：在 `serial.S` 的 `vesa_get_controller_info` 函数中，有一个 `pop %ax` 漏掉了 `w` 后缀，这种地方就是最容易踩的坑 —— 汇编器在 `.code16` 下默认推断为 16 位，但显式写出 `popw` 是最安全的做法。

**踩坑预警**: 如果你发现 Stage2 在某些函数调用后跑飞了，SP 的值变得莫名其妙，十有八九是 push/pop 宽度不一致导致的栈错位。排查方法是连上 GDB，在被调函数的入口和 `ret` 指令处设断点，分别打印 SP 的值，如果两次的差值不等于你 push 的总字节数，那就是宽度搞错了。

**验证**: 修改完成后，重新构建项目，确认 tag 001 的所有功能（Stage2 OK 打印、A20 使能、VESA 模式切换）不受影响。构建命令在后面的"构建与运行"部分给出。

### Step 2: 修正 Linker Script 基地址

**目标**: 把 Stage2 的 linker script 中 `.text` section 的起始地址从 `0x0` 改为 `0x8000`，同时新增一个 `.gdt` section 放在 `.text` 之后、`.rodata` 之前，并设置 `ALIGN(8)` 确保 8 字节对齐。

**设计思路**: 为什么 linker base 地址必须匹配加载地址？因为 MBR 把 Stage2 从磁盘读出来后，是直接加载到物理内存 `0x8000` 的。如果 linker script 里写的 base 地址是 `0x0`，那链接器在解析所有绝对地址引用时（比如 GDT 指针里的 base 地址），都会基于 `0x0` 来计算。但运行时这段代码实际在 `0x8000`，于是 `lgdt` 读到的 GDT 地址会指向错误的位置，轻则加载了垃圾数据，重则直接 triple fault。

把 base 改成 `0x8000` 之后，链接器算出来的所有绝对地址都会加上 `0x8000` 的偏移，和运行时的物理地址完全匹配。新增的 `.gdt` section 是为了让 GDT 数据结构有自己独立的、8 字节对齐的存储区域，而不是混在 `.text` 或 `.rodata` 里被指令对齐要求影响。linker script 还需要加一个断言：Stage2 的总大小不能超过 `0x10000`（即 32KB，从 0x8000 到 0x10000），这是 MBR 加载时的空间限制。

**验证**: 构建完成后，用 `objdump` 检查 Stage2 ELF 文件中的符号地址，确认所有地址都在 `0x8000` 以上。具体来说，`_start` 应该在 `0x8000`，GDT 相关的符号（比如 `gdt`、`gdt_ptr`）应该在 `0x8xxx` 范围内。

```bash
objdump -t build/boot/stage2 | grep -E '(gdt|_start|pm_entry)'
```

### Step 3: 定义 GDT

**目标**: 在 `stage2.S` 中新增一个 `.gdt` section，在里面定义三个段描述符（null、code32、data32）和 GDT 指针结构 `gdt_ptr`。

**设计思路**: 我们先理清楚需要定义什么，再逐个解释每个字段的含义。

首先是一个 Null 描述符，占 8 字节全零，放在 GDT 的第 0 个位置（索引 0）。Intel 硬件规定这个条目必须存在且必须全零，段选择子值为 `0x00` 时指向它，CPU 试图用 null selector 访问内存会触发异常。这不是可选的装饰，是强制的。

第二个是 32 位代码段描述符，放在索引 1，选择子值为 `0x08`。这个描述符定义了保护模式下的代码段：基地址（Base）全部为 0，大小限制（Limit）的低 16 位填 `0xFFFF`、高 4 位填 `0xF`，合起来 20 位就是 `0xFFFFF`；粒度位 G 设为 1（表示以 4KB 为单位），所以 `0xFFFFF * 4096 = 4GB`，覆盖整个地址空间。Access Byte 的值是 `0x9A`，我们拆开来看：最高位 P=1 表示段在内存中存在，接下来两位 DPL=00 表示 Ring 0 内核级权限，S=1 表示这是代码/数据段（不是系统段），Type 字段的 4 位是 `1010`，在代码段类型中表示可执行且可读。Flags 字节是 `0xCF`，其中高 4 位是 `1100`：G=1（4KB 粒度）、D=1（32 位默认操作数大小），剩下两位是 Limit 的高 4 位 `0xF`。

第三个是 32 位数据段描述符，放在索引 2，选择子值为 `0x10`。布局和代码段几乎一模一样，唯一的区别在 Access Byte：值是 `0x92`，P=1、DPL=00、S=1，Type 是 `0010` 表示可读写的数据段。Limit 和 Base 的值与代码段完全相同，实现平坦内存模型。

然后是 GDT 指针结构 `gdt_ptr`，这是 `lgdt` 指令需要读取的 6 字节数据：前 2 字节是 GDT 的大小减 1（limit 字段），后 4 字节是 GDT 的线性基地址。大小可以用汇编表达式 `gdt_end - gdt - 1` 自动计算，基地址直接写 `gdt` 标号即可 —— 因为 linker script 已经把 base 设成了 `0x8000`，链接器会自动算出正确的绝对地址。

**实现约束**: `.gdt` section 必须用 `.align 8` 确保 8 字节对齐；null 描述符必须全零；`gdt_ptr` 的格式是严格的 2 字节 limit + 4 字节 base，顺序不能反。

**验证**: 编译通过后，检查 `.gdt` section 在 ELF 文件中的大小和内容：

```bash
objdump -s -j .gdt build/boot/stage2
```

你应该能看到三个 8 字节的描述符条目和最后的 6 字节 gdt_ptr。null 描述符全零，code 描述符第 5 字节是 `0x9A`、第 6 字节是 `0xCF`，data 描述符第 5 字节是 `0x92`。

### Step 4: 切换到保护模式

**目标**: 在 `stage2.S` 的 `_start` 函数中，VESA 初始化完成后，依次执行：关中断 -> DS 清零 -> lgdt -> CR0.PE 置 1 -> 远跳转到 `.code32` 入口点。

**设计思路**: 让我们一步步解释为什么每一步都不可少，以及它们之间的依赖关系。

首先是 `cli` 关中断。进入保护模式后，BIOS 的中断向量表（IVT）就失效了，而我们的 IDT 还没建立。如果这时候发生任何中断（包括定时器中断），CPU 会尝试查一个不存在的 IDT，直接触发 double fault，然后 triple fault，QEMU 重启。所以在切换开始前必须把中断关掉。

然后是 `movw $0, %ax; movw %ax, %ds` 把 DS 清零。这一点特别容易被忽略。在实模式下，所有内存访问的物理地址都是 `DS * 16 + offset`。当我们执行 `lgdt gdt_ptr` 时，CPU 读 `gdt_ptr` 的物理地址就是 `DS * 16 + gdt_ptr`。如果 DS 不为 0（比如还是之前 VESA 调用后残留的某个值），CPU 就会从错误的位置读取 GDT 指针，加载一堆垃圾到 GDTR 里。这个问题查起来特别难定位，因为 `lgdt` 本身不会报错 —— 它只是默默地把数据搬进 GDTR，错误要等到后续第一次段访问时才会暴露。

接下来是 `lgdt gdt_ptr`，这条指令从内存中读取 6 字节的 GDT 指针结构（2 字节 limit + 4 字节 base），加载到 GDTR 寄存器。Intel SDM Volume 3A Section 2.4.1 对此有详细说明。注意 `lgdt` 本身不校验 GDT 内容的合法性，它只是做了一次内存拷贝。

然后是设置 CR0 的 PE 位：把 CR0 读到 EAX，把最低位（bit 0）或上 1，再写回 CR0。这一步是"通知"CPU 进入保护模式，但此时 CPU 流水线里还缓存着按 16 位实模式取的指令，CS 寄存器还指向实模式的段。所以 PE 位设完之后，必须马上做一个远跳转。

远跳转是整个流程中最关键的一步。远跳转的操作数同时指定了新的代码段选择子（`0x08`）和目标偏移地址（`pm_entry`），CPU 在执行这个跳转时会把 CS 刷新为新选择子指向的段描述符，同时清空流水线中的预取指令队列，从新的 CS:EIP 开始以 32 位模式重新取指令。如果不做这个跳转，后续的 32 位指令会被当作 16 位来解码，几乎必定执行非法指令然后崩溃。

关于远跳转的编码，有一个好消息：在 `.code16` 模式下，GAS 会自动把 `ljmp $0x08, $pm_entry` 编码为正确的 16 位远跳转格式。你不需要手动拼机器码，也不需要操心编码宽度的问题，汇编器会帮你处理好。

**踩坑预警**: 如果你在 lgdt 之前忘记把 DS 清零，表面上不会报任何错误，但后续远跳转时 CPU 会从错误的 GDT 条目里取段描述符，大概率触发 General Protection Fault 或者 triple fault。另一个常见问题是有人在远跳转的目标地址上手动减去了加载偏移（比如写 `pm_entry - 0x8000`），这是不必要的 —— 因为 linker script 已经把 base 设成了 `0x8000`，链接器算出来的 `pm_entry` 地址本身就是正确的物理地址，再减一次就跳到错误的内存去了。

**验证**: 在远跳转指令处设 GDB 断点，确认 CR0 的 PE 位已经被置 1：

```bash
gdb build/boot/stage2
(gdb) target remote :1234
(gdb) break *pm_entry
(gdb) continue
(gdb) info registers cr0
```

CR0 的值应该是 `0x...1`（最低位为 1）。

### Step 5: 保护模式段寄存器初始化与验证输出

**目标**: 在 `.code32` 的 `pm_entry` 入口点中，把 DS、ES、FS、GS、SS 全部加载为数据段选择子 `0x10`，设置栈指针到 `0x90000`，然后向端口 `0xE9` 输出字母 `P`。

**设计思路**: 远跳转完成后，CS 已经被自动刷新为代码段选择子 `0x08`，但其他段寄存器（DS、ES、FS、GS、SS）还残留着实模式的值。在保护模式下，这些段寄存器也要通过段选择子来寻址，如果不更新，任何内存访问都会因为使用无效的实模式段值而触发异常。所以进入保护模式的第一件事就是统一加载所有数据段寄存器为 `0x10`（数据段选择子）。

栈段的设置也不能遗漏。SS 必须指向有效的数据段描述符，栈指针我们设在 `0x90000` —— 这个地址低于 Stage2 加载地址 `0x8000` 的上方空间，不会被代码覆盖，并且和实模式阶段我们用的 `0x9000:0xFFFE`（线性地址 `0x90000 + 0xFFFE`）在同一个区域，保持了一致性。

最后是验证输出。进入保护模式后，BIOS 中断全部失效，我们不能再用 INT 10h 打印字符了。QEMU 提供了一个非常方便的调试接口叫 debugcon：向 I/O 端口 `0xE9` 写入一个字节，QEMU 就会把这个字节输出到 debugcon 设备。我们向它写入 `0x50`（字母 `P` 的 ASCII 码），如果一切顺利，终端上就会看到这个 `P`。这是保护模式下最简单的输出手段，不需要初始化 UART、不需要配置串口，一个 `outb` 搞定。QEMU 的 debugcon 端口号固定为 `0xE9`，需要在 QEMU 启动参数中用 `-debugcon` 选项启用。

**验证**: 构建 Stage2 并打包到磁盘镜像，然后运行 QEMU。如果你在终端的 debugcon 输出中看到了字母 `P`，恭喜你 —— 实模式到保护模式的切换已经成功了。完整的构建和运行命令见下一节。

## 构建与运行

完整的构建流程如下。确保你在项目根目录下操作：

```bash
mkdir -p build && cd build
cmake -DCMAKE_BUILD_TYPE=Debug -B . -S ..
make
```

构建完成后，`build/boot/stage2.bin` 就是 Stage2 的裸二进制文件，`build/cinux.img` 是完整的磁盘镜像。

运行 QEMU 的命令：

```bash
cd build
make run
```

这个 `make run` 目标背后实际执行的 QEMU 命令包含了以下关键参数：

`-serial stdio` 把串口输出定向到终端，你在实模式阶段用 INT 10h 打印的字符串会从这里出来。`-debugcon file:debug.log` 把 debugcon 端口（`0xE9`）的输出写入 `build/debug.log` 文件，我们保护模式下输出的字母 `P` 就在这里面。`-global isa-debugcon.iobase=0xe9` 指定 debugcon 设备的 I/O 端口号。如果你想直接在终端看到 debugcon 输出而不是写到文件，可以手动运行 QEMU 并把 `-debugcon file:debug.log` 改为 `-debugcon stdio`。

如果你只想快速验证保护模式是否成功进入，最简单的方式是：

```bash
cd build
make run
# 等待几秒后 Ctrl+C 退出 QEMU
cat debug.log
# 应该能看到字母 P
```

## 调试技巧

### GDB 断点调试

在保护模式切换这种底层代码里，GDB 是你最可靠的朋友。启动 QEMU 的调试模式：

```bash
cd build
make run-debug
```

然后在另一个终端连上 GDB：

```bash
gdb build/boot/stage2
(gdb) target remote :1234
```

注意这里要用 ELF 文件（`build/boot/stage2`）而不是 bin 文件（`build/boot/stage2.bin`）。ELF 文件包含符号表和调试信息，GDB 才能正确识别函数名和行号；bin 文件只是裸二进制，没有任何元数据，GDB 拿到它就像看一堆字节，断点和符号全都对不上。

在 GDB 中可以设置几个关键断点：在 `_start` 处确认 Stage2 入口正常，在远跳转指令处确认 lgdt 已经执行完毕，在 `pm_entry` 处确认保护模式入口被正确跳转到。到达 `pm_entry` 后，用 `info registers` 查看 CR0 的 PE 位是否为 1，段寄存器是否指向正确的选择子。

### QEMU 中断日志

如果 CPU 在某个时刻 triple fault 了（QEMU 窗口直接关闭或者重启），用 `-d int` 参数启动 QEMU 可以看到所有中断和异常的记录：

```bash
qemu-system-x86_64 -drive format=raw,file=build/cinux.img -d int -D qemu_int.log -serial stdio -debugcon stdio -no-reboot
```

然后在 `qemu_int.log` 里搜索 `exception` 或 `fault` 关键字，看看是哪种异常、发生在什么地址。最常见的三种：General Protection Fault 通常是段选择子或描述符有问题，Page Fault 在这个阶段不太可能（还没开分页），Double/Triple Fault 说明前一个异常的处理又触发了异常。

### debugcon 作为最简验证手段

在整个 bootloader 开发过程中，debugcon（端口 `0xE9`）是最轻量的调试输出方式。它不需要初始化任何硬件，不需要配置波特率，只需要一个 `outb` 指令。你可以在代码的任何可疑位置插入一个 `outb`，输出不同的字符来标记执行到了哪里。比如在 lgdt 前后各输出一个字符，在 CR0 写入后输出一个字符，在远跳转后输出一个字符 —— 通过观察哪些字符出现了、哪些没出现，就能快速定位崩溃发生在哪一步。这比 GDB 单步调试快得多，尤其是你只是想知道"有没有执行到这里"的时候。

## 本章小结

到这里，我们已经完成了 bootloader 开发中最关键的里程碑之一：把 CPU 从 16 位实模式成功推进到 32 位保护模式。下面是本章涉及的关键概念和数值总结：

| 概念 | 值/位置 | 说明 |
|------|---------|------|
| `.code16` | serial.S, stage2.S | 严格 16 位指令编码，替代 `.code16gcc` |
| `pushw`/`popw` | serial.S | 显式 16 位栈操作，防止宽度不一致 |
| Linker base | `0x8000` | Stage2 的加载地址，linker script 必须匹配 |
| `.gdt` section | stage2.S | 独立、8 字节对齐的 GDT 存储区 |
| Null 描述符 | 索引 0, 8 字节全零 | Intel 硬件强制要求 |
| Code32 描述符 | 索引 1, 选择子 `0x08` | Base=0, Limit=4GB, 可执行可读, Ring 0 |
| Data32 描述符 | 索引 2, 选择子 `0x10` | Base=0, Limit=4GB, 可读写, Ring 0 |
| Access Byte (code) | `0x9A` | P=1, DPL=00, S=1, Type=1010(代码/可执行/可读) |
| Access Byte (data) | `0x92` | P=1, DPL=00, S=1, Type=0010(数据/可读写) |
| Flags (code/data) | `0xCF` | G=1(4KB), D=1(32位), Limit高4位=0xF |
| `gdt_ptr` | 2 字节 limit + 4 字节 base | `lgdt` 指令读取的 6 字节伪描述符 |
| CR0.PE | bit 0 = 1 | Protection Enable，开启保护模式 |
| 远跳转 | `ljmp $0x08, $pm_entry` | 刷新 CS 和流水线，真正进入 32 位 |
| debugcon 端口 | `0xE9` | QEMU 调试输出，`outb` 写入即可 |
| 栈指针 | `0x90000` | 保护模式下的 ESP 初始值 |
