# 014-1 控制器握手：8042 PS/2 初始化

## 导语

上一章我们把屏幕点亮了——FrameBuffer、PSF2 字体、Console 三件套齐全，kprintf 可以同时往串口和屏幕输出。但这个内核到现在还是个"单行道"：只出不进。你按键盘，内核完全不知道。这一章我们要把输入通道打通，让内核能听到键盘说了什么。而这一切的起点，就是跟那颗古老但依然在位的 Intel 8042 PS/2 控制器完成握手。

这一章我们要解决的问题是：PS/2 键盘控制器的初始化。具体来说，我们需要通过 I/O 端口向 8042 控制器发送一系列命令，配置它的工作模式（启用键盘 IRQ、开启扫描码翻译），然后通过自检确认控制器活着，最后重新启用键盘端口。完成之后，8042 就准备好了——它会在每次按键时向 CPU 发送 IRQ1 中断，但这一章我们只做控制器握手，中断处理是后面两章的事。

知识前置：你需要理解 x86 I/O 端口编程（`inb`/`outb` 指令），以及我们在 tag 010-011 中搭建的 IDT + PIC 中断框架（IRQ1 对应 IDT 向量 0x21）。

## 概念精讲

### PS/2 控制器与 Intel 8042

PS/2 键盘控制器听起来像个接口标准，但它背后实际上是一颗独立的微控制器——Intel 8042。这颗芯片的历史可以追溯到 IBM PC/AT 时代（1984 年），现在虽然物理上早就被集成到了主板的 Super I/O 芯片或者 LPC 总线桥里，但从软件角度看，它的行为和 40 年前一模一样。这就是 PC 兼容性的"威力"——或者"诅咒"，取决于你怎么看。

8042 控制器支持两个设备通道：第一端口（Port 1）接键盘，第二端口（Port 2）接鼠标。CPU 通过两个 I/O 端口和它通信：端口 0x60 是数据寄存器（读写字节数据），端口 0x64 是状态/命令寄存器（读取状态或者写入命令）。当你往 0x64 写一个字节时，控制器把它解释为一条命令；当你从 0x64 读一个字节时，你拿到的是状态寄存器——其中 bit 0 表示输出缓冲区是否有数据可读（1=有），bit 1 表示输入缓冲区是否已满（1=满，此时不能写入）。

8042 控制器内部有两个缓冲区：一个输出缓冲区（存放键盘发来的扫描码或者命令响应，等待 CPU 读取），一个输入缓冲区（存放 CPU 写入的命令或数据，等待控制器处理）。这两个缓冲区各只有一个字节深度——这意味着如果你读慢了，新的数据就会覆盖旧的；如果你写快了，上一次的命令还没处理完就会被新命令冲掉。这就是为什么我们在每次 I/O 操作前都要检查状态寄存器的原因。

### 控制器命令协议

和 8042 通信有一套严格的握手机制。每次发送命令之前，你必须先检查状态寄存器的 bit 1——如果输入缓冲区是满的，说明控制器还没处理完上一次的写入，此时你继续写就会丢数据。所以我们的模式是：轮询等待 bit 1 变成 0，然后往 0x64 写入命令字节；如果命令需要附带数据（比如写配置字节命令 0x60 之后要跟一个数据字节），再等 bit 1 变 0，然后往 0x60 写入数据。

对于命令响应的读取，流程是类似的：先发命令，等控制器处理完毕，然后检查状态寄存器 bit 0（输出缓冲区满），如果为 1 就从 0x60 读回响应数据。每条命令的响应格式不同——有的命令不返回数据（比如禁用/启用端口），有的返回一个字节（比如读配置字节返回配置值，自检返回 0x55 或错误码），有的返回多个字节（比如设备识别返回 0xAB, 0x83 两字节）。

一些关键命令你需要知道：0xAD 禁用第一端口（键盘），0xA7 禁用第二端口（鼠标），0xAE 启用第一端口，0x20 读取控制器配置字节，0x60 写入配置字节，0xAA 控制器自检（期望返回 0x55）。配置字节各位的含义：bit 0 控制是否在第一端口有数据时产生 IRQ1，bit 1 控制第二端口的 IRQ12，bit 6 控制是否将键盘原生的扫描码集 2 翻译成集 1。

### 扫描码翻译模式

这里有一个非常关键的设计决策。键盘本身发送的是扫描码集 2（Scan Code Set 2），这是 PS/2 键盘的默认协议。但 8042 控制器可以在硬件层面把这个翻译成扫描码集 1——也就是古老的 XT 键盘使用的格式。集 1 的规则极其简单：按下（Make）时发送一个字节（bit 7 为 0），松开（Break）时发送同一个字节但 bit 7 置 1。比如字母 A 按下是 0x1E，松开就是 0x9E。我们选择开启翻译模式（配置字节 bit 6 = 1），这样软件处理的始终是集 1 格式，逻辑干净得多。

翻译模式的另一个好处是兼容性——几乎所有 x86 教学材料和 OSDev Wiki 的示例代码都基于扫描码集 1 编写。如果你不开启翻译模式，收到的就是集 2 的编码，它的 break code 规则完全不同（不是 bit 7 置 1，而是在 make code 前面加一个 0xF0 前缀），你需要写更复杂的状态机来处理。对于一个教学 OS 来说，能省一点复杂度就省一点。

### OSDev Wiki 的十步初始化

OSDev Wiki 上有一个著名的 8042 完整初始化序列，一共 10 步——从 USB 遗留支持检测到双通道探测再到设备复位。Cinux 做了合理的简化：我们跳过 USB 遗留支持检测（QEMU 不需要）、跳过双通道检测（只关心键盘）、跳过设备复位和识别（不需要），把初始化压缩成 6 步。这在 QEMU 环境下完全够用，在真实硬件上也大概率能工作。

> 参考：[OSDev Wiki - I8042 PS/2 Controller](https://wiki.osdev.org/I8042_PS/2_Controller)

## 动手实现

### Step 1: 定义 PS/2 常量和辅助函数

**目标**: 建立和 8042 通信所需的所有常量定义和底层辅助函数。

**设计思路**: 我们需要三组常量。第一组是端口地址——数据端口 0x60、状态端口 0x64、命令端口 0x64（状态和命令共用端口号，区别在于读还是写）。第二组是命令字节——禁用/启用端口、读写配置、自检等。第三组是状态寄存器的位掩码——输出缓冲区满（bit 0）和输入缓冲区满（bit 1）。这些常量放在命名空间中分组（Ps2Port、Ps2Cmd、Ps2Status），每个常量用 `constexpr uint16_t` 或 `constexpr uint8_t`。

然后我们需要两个辅助函数。第一个是等待输入缓冲区变空：轮询状态端口的 bit 1，带一个超时计数器（比如 100000 次循环），超时了就直接返回而不是死等——万一控制器挂了，至少不会把内核卡死。轮询循环体内插入 `pause` 指令（`__asm__ volatile("pause")`），这告诉 CPU "我在自旋等待"，让超线程体系中的另一个逻辑核能更高效地执行。第二个辅助函数是发送命令：先等输入缓冲区空，然后往命令端口 0x64 写入命令字节。

**实现约束**: 辅助函数放在匿名命名空间中，使用已有的 `io_inb` 和 `io_outb` 函数（定义在 `kernel/arch/x86_64/io.hpp`）。文件放在 `kernel/drivers/keyboard/keyboard.cpp`，对应头文件 `kernel/drivers/keyboard/keyboard.hpp`。

**踩坑预警**: 状态寄存器的 bit 0 和 bit 1 含义一定要搞清楚。bit 0 = 1 表示输出缓冲区有数据（你可以读 0x60），bit 1 = 1 表示输入缓冲区满（你不能写）。很多教程把这两个搞混，导致初始化序列死锁。另外，`io_wait()` 函数（往端口 0x80 写一个无用字节）在某些步骤之间可能需要调用，给控制器一点反应时间——不过在 QEMU 中通常不需要。

**验证**: 这一步没有独立验证手段，因为辅助函数在后续步骤中才会被调用。但你可以暂时在 `kernel_main` 中构造一个测试：调用 `send_command(0xAA)`（自检），然后从 0x60 读取返回值，检查是否为 0x55。如果读到 0x55，说明你的 I/O 端口通信是正确的。

### Step 2: 实现 Keyboard::init() 初始化序列

**目标**: 按照 6 步序列完成 PS/2 控制器初始化。

**设计思路**: 初始化序列分为以下 6 步：

第一步，禁用两个 PS/2 设备端口——发送命令 0xAD 禁用键盘端口，发送命令 0xA7 禁用鼠标端口。这样做的原因是在初始化过程中，我们不希望有任何设备数据干扰配置流程。如果键盘缓冲区里有残留的扫描码，在配置过程中被读出来就乱套了。

第二步，刷新输出缓冲区——轮询状态端口 bit 0，如果为 1 就读 0x60 丢弃数据，直到 bit 0 变成 0。这一步清除所有残留的陈旧数据。

第三步，读取当前配置字节——发送命令 0x20，然后从数据端口 0x60 读回一个字节。这个字节包含了控制器的当前工作模式。

第四步，修改配置字节——我们需要设置三个位：bit 0 置 1（启用 IRQ1，让键盘数据到来时触发中断），bit 1 清 0（禁用 IRQ12，我们没有鼠标驱动），bit 6 置 1（启用扫描码集 2 到集 1 的翻译模式）。然后用命令 0x60 写回修改后的配置字节。bit 6 这一步极其重要——如果不开启翻译模式，键盘发来的就是集 2 扫描码，我们后面写的整个解码逻辑都会错。

第五步，控制器自检——发送命令 0xAA，从数据端口读回结果，期望值是 0x55。如果拿到 0x55，说明控制器内部逻辑正常。如果不等于 0x55……在 QEMU 中这种情况基本不会发生，但在真实硬件上如果自检失败，通常意味着控制器有硬件故障，整个键盘子系统就没法用了。

第六步，重新启用第一端口——发送命令 0xAE，键盘端口恢复工作。从这一刻开始，如果有按键，8042 就会把翻译后的集 1 扫描码放到输出缓冲区，并且（因为 bit 0 = 1）触发 IRQ1 中断。不过现在我们的 IRQ1 handler 还是个空壳，所以中断来了也不会有什么效果——那是后面两章的事。

最后，重置驱动内部状态——环形缓冲区的 head/tail 指针归零，三个 modifier 标志位（shift_held_、ctrl_held_、alt_held_）全部清除。

**实现约束**: 整个序列封装在 `Keyboard::init()` 静态方法中。每一步之后用 kprintf 打印日志，方便调试。自检结果用条件判断：成功打印 `[KBD] PS/2 controller self-test passed.`，失败打印包含实际返回值的错误信息。所有 I/O 操作通过 Step 1 定义的辅助函数完成。

**踩坑预警**: 有两个经典坑点。第一个：OSDev Wiki 指出，命令 0xAA（自检）在某些硬件上会重置控制器配置。也就是说，你在自检之前写的配置可能在自检之后就丢了。标准做法是自检之后再写一次配置。Cinux 目前的实现是先写配置再自检，在 QEMU 中没问题（QEMU 不会重置配置），但如果你将来要在真实硬件上跑，建议把写配置移到自检之后。第二个：如果你忘记刷新输出缓冲区（第二步），控制器可能在你读配置字节时返回一个残留的扫描码而不是真正的配置值——这个 bug 非常隐蔽，因为配置字节和扫描码的值域是重叠的。

**验证**: 构建并启动内核，串口输出应该包含以下日志（顺序可能不完全一致，取决于其他初始化步骤的位置）：

```
[KBD] PS/2 controller self-test passed.
[KBD] Keyboard driver initialised.
```

如果自检失败，你会看到 `FAILED (got 0xXX, expected 0x55)` 的错误信息。如果内核在 init 期间 triple fault，大概率是 I/O 端口访问出了问题——检查 IDT 是否正确加载了（tag 010 的内容）。

### Step 3: 修改 interrupts.S 绑定 IRQ1 handler

**目标**: 在汇编中断处理文件中，将 IRQ1 的处理函数从默认的空操作切换到键盘专用的 handler。

**设计思路**: 回顾 tag 010-011 中我们搭建的中断框架——`interrupts.S` 里用 `ISR_NOERRCODE` 宏为每个 IRQ 生成了一个 stub，stub 在保存上下文后调用对应的 C 处理函数。之前 IRQ1（向量 0x21）绑定的是 `irq_default_handler`——一个什么都不做的空函数。现在我们要把它改成 `keyboard_irq1_handler`。

这里有一个关键的链接问题：`interrupts.S` 中的 stub 是纯汇编代码，而 `keyboard_irq1_handler` 是一个 C 函数。但我们的 `Keyboard::irq1_handler` 是 C++ 命名空间 `cinux::drivers::Keyboard` 中的静态方法，经过 C++ name mangling 后符号名会变得面目全非。所以在 `keyboard.cpp` 的末尾，我们需要一个 `extern "C"` 桥接函数——它的名字就是 `keyboard_irq1_handler`，内部调用 `Keyboard::irq1_handler(frame)`。汇编侧只需要声明这个 C 链接的函数名就行了。

**实现约束**: 修改位置在 `kernel/arch/x86_64/interrupts.S` 的 IRQ 注释块下方。只需要把 `irq1_stub` 绑定的处理函数名从 `irq_default_handler` 改成 `keyboard_irq1_handler`，一个词的改动。桥接函数放在 `keyboard.cpp` 的命名空间外部，用 `extern "C"` 修饰。注意桥接函数的参数类型是 `cinux::arch::InterruptFrame*`，必须和汇编 stub 传递的参数类型匹配。同时记得在 `kernel/CMakeLists.txt` 中把 `drivers/keyboard/keyboard.cpp` 加入 `big_kernel_common` 库的源文件列表。

**踩坑预警**: 如果你改了汇编文件中的函数名但忘了在 C++ 侧提供对应的 `extern "C"` 定义，链接阶段会报 "undefined reference to keyboard_irq1_handler"。反过来，如果你在 C++ 侧定义了但忘了改汇编文件，那 IRQ1 中断来了之后还是会走默认的空 handler，键盘完全没反应——而且没有任何错误提示，这种"静默失败"特别折磨人。

**验证**: 构建成功（没有链接错误）就说明汇编侧和 C++ 侧的符号对接上了。更完整的验证需要等后面实现 IRQ1 handler 之后——在 QEMU 中按键看是否有反应。

### Step 4: 在 kernel_main 中集成并移除 PIT 噪音

**目标**: 将键盘初始化插入到内核启动流程的正确位置，同时清理 PIT 的调试输出。

**设计思路**: kernel_main 中已有的初始化顺序是：Serial -> GDT -> IDT -> PIC -> PIT -> Framebuffer -> Font -> Console。键盘初始化应该放在 Console 之后（因为 init 里的 kprintf 需要控制台已经就绪），在 `sti`（开中断）之前。具体来说，顺序变成：Console 初始化完毕 -> `Keyboard::init()` -> `PIC::unmask(0)` + `PIC::unmask(1)` -> `sti` -> 进入空闲循环。

为什么要同时 unmask IRQ0（PIT）和 IRQ1（键盘）？因为 IRQ1 不 unmask 的话，即使 8042 发了中断信号，PIC 也会把它屏蔽掉，CPU 永远收不到。我们之前只 unmask 了 IRQ0（定时器），现在加上 IRQ1。

另外还有一个细节：PIT 驱动里之前有一个每秒打印 uptime 的日志（`[TICK] uptime: Xs`）。这个日志在串口调试时很有用，但现在我们要让键盘回显字符了，每秒一条 TICK 日志会把屏幕弄得乱七八糟——你敲一个 `a`，屏幕上可能出现 `[TICK] uptime: 3s`a 这种交错输出。所以我们需要把 PIT 中的那段 per-second 打印代码删掉——只保留 tick 计数器的递增和 EOI 发送。

**实现约束**: 在 `kernel/main.cpp` 中，在 Console 初始化之后、PIC unmask 之前，调用 `Keyboard::init()`。把 `PIC::unmask(0)` 后面加上 `PIC::unmask(1)`。更新日志信息从 `IRQ0 unmasked` 改为 `IRQ0+IRQ1 unmasked`。idle loop 暂时还是 `while(1) { hlt; }` 的空壳——后面两章才会加入键盘轮询。在 `kernel/drivers/pit/pit.cpp` 中删除 `if ((tick_count_ % freq_hz_) == 0)` 那段打印代码。

**踩坑预警**: `Keyboard::init()` 必须在 `PIC::unmask(1)` 之前调用。如果你先 unmask 了 IRQ1 再 init，那么 init 过程中如果有残留中断信号，就会在 handler 还没准备好的时候触发——虽然我们的 handler 不会 crash（它只是读端口 0x60 然后 enqueue），但逻辑上不应该让这种情况发生。另外，`sti` 必须是最后一个操作——在所有硬件初始化和 IRQ unmask 都完成之后才能开中断。

**验证**: 构建并启动 QEMU，串口输出的初始化日志应该包含：

```
[BIG] Console initialised -- dual output active.
[KBD] PS/2 controller self-test passed.
[KBD] Keyboard driver initialised.
[BIG] IRQ0+IRQ1 unmasked, enabling interrupts...
[BIG] Interrupts enabled. Keyboard echo active.
```

注意顺序：Console 在前，KBD 在后，unmask 在更后面，sti 在最后。如果你看到 `[TICK] uptime` 日志还在出现，说明 PIT 的修改没有生效——检查 `pit.cpp` 是否正确删除了那段代码。

## 构建与运行

构建命令不变。启动 QEMU 后你应该能看到内核正常启动，串口和屏幕都输出初始化日志。此时 IRQ1 已经 unmask 了，键盘控制器也初始化了，但我们还没有实际处理按键事件，所以按键不会有任何反应——这是预期的。如果你在 QEMU 窗口中按键，8042 会把扫描码放入输出缓冲区并触发 IRQ1，但由于我们的 handler 暂时还是空的（或者后面才实现完整版），中断来了就立刻返回，扫描码被丢弃。

另外注意一点：此时 PIT 的每秒日志已经消失了，屏幕应该很安静。如果还看到 TICK 日志，说明 Step 4 的 PIT 清理没有生效。

## 调试技巧

**自检失败（got 0xXX, expected 0x55）**: 在 QEMU 中这几乎不可能发生。如果出现了，首先检查你的 I/O 端口操作是否正确——`send_command` 函数有没有正确等待输入缓冲区变空。用 GDB 在 `Keyboard::init()` 入口设断点，单步跟踪每一步的端口读写值。

**内核在 init 期间 triple fault**: 大概率不是键盘驱动本身的问题，而是 IDT 没有正确加载。检查 `idt_init()` 是否在 `Keyboard::init()` 之前被调用了。如果 IDT 没加载，那么任何中断（包括 init 过程中可能产生的 PIT 中断）都会触发 double fault -> triple fault。

**构建时 undefined reference to keyboard_irq1_handler**: 检查三个地方：`interrupts.S` 中的函数名拼写、`keyboard.cpp` 中 `extern "C"` 桥接函数的存在、以及 `CMakeLists.txt` 中是否添加了 `keyboard.cpp`。三个环节缺一不可。

**构建时 undefined reference to Keyboard::init() 等**: 说明 `keyboard.cpp` 没有被加入构建。在 `kernel/CMakeLists.txt` 的 `big_kernel_common` 库中添加 `drivers/keyboard/keyboard.cpp`。

| 8042 PS/2 控制器 | I/O 端口 0x60（数据）和 0x64（命令/状态），40 年不变的接口 |
| 缓冲区深度 | 输出和输入缓冲区各 1 字节，读写前必须检查状态位 |
| 输入缓冲区检查 | 发送命令前必须轮询状态寄存器 bit 1 = 0 |
| 配置字节 | bit 0 = IRQ1 使能，bit 1 = IRQ12 禁止，bit 6 = 扫描码翻译 |
| 扫描码翻译 | 集合 2 -> 集合 1，使 XT 兼容的 make/break 规则生效 |
| 自检命令 | 0xAA 发送，期望返回 0x55 |
| 初始化顺序 | 禁用端口 -> 刷新缓冲 -> 配置 -> 自检 -> 启用端口 |
| IRQ1 绑定 | interrupts.S 中将 irq1_stub 绑定到 keyboard_irq1_handler |
| extern "C" 桥接 | 解决 C++ name mangling 和汇编符号名不匹配的问题 |
| PIT 噪音清除 | 删除 per-second uptime 打印，避免与键盘回显交错 |
