# 030-3 鼠标交互与 GUI 集成

## 导语

前两章我们分别搞定了输入（鼠标驱动+事件队列）和输出（Window + WindowManager + compositing），但它们还是割裂的。这一章要把两者串起来——让窗口管理器真正响应鼠标事件，实现窗口拖拽、关闭、焦点切换，并且把整个 GUI 的启动流程集成到内核的初始化序列里。另外我们还要处理键盘事件到焦点窗口的路由，以及一个有趣的问题：QEMU 里为什么会出现两个光标。

完成本章后，你会看到一个完整可交互的 GUI 桌面：鼠标移动时光标跟随，点击窗口可以 raise 和拖拽，点击关闭按钮可以销毁窗口，键盘输入会送到焦点窗口。

知识前置：前两章的全部内容。

## 概念精讲

### 鼠标事件的状态机

窗口管理器处理鼠标事件本质上是一个简单的状态机。Idle 状态下收到 MouseDown 时，先做 hit test（从顶层到底层遍历窗口数组，找到第一个包含点击坐标的可见窗口），然后根据点击位置决定行为：如果命中关闭按钮就销毁窗口，如果命中标题栏就进入 Dragging 状态并记录拖拽偏移量，如果命中内容区就只 raise。Dragging 状态下收到 MouseMove 时，用当前鼠标坐标减去偏移量算出新位置，更新焦点窗口的坐标，触发重绘。收到 MouseUp 时退出 Dragging 状态回到 Idle。

拖拽偏移量（drag offset）的设计很关键。它记录的是"点击时鼠标相对于窗口左上角的偏移"，这样在 MouseMove 时用鼠标坐标减去偏移量就能得到窗口应该移到的新位置，而不是让窗口左上角直接跳到鼠标位置。没有这个偏移量的话，窗口会在你点击标题栏的那一刻突然跳一下，体验非常差。

### hit test 的遍历方向

hit test 必须从最顶层（数组末尾）向下遍历，因为上面的窗口遮挡下面的。如果我们从 index 0 开始遍历，点击一个被其他窗口遮挡的窗口时，底层窗口会"抢到"这个点击事件——显然不对。倒序遍历保证了最上面的窗口总是先被检测到。

### PIT tick 回调驱动的 GUI 事件循环

我们的 GUI 没有独立的事件循环线程（至少在这个 tag 里没有），而是借用 PIT 定时器的 tick 回调来驱动。每次 PIT 中断（100Hz，每 10ms 一次），回调函数就会做三件事：第一，清空事件队列，把每个事件分发到窗口管理器；第二，轮询所有终端窗口的 shell 输出；第三，调用 composite 合成一帧。这意味着 GUI 的刷新率被 PIT 频率绑定了——100Hz 的 PIT 给我们最多 100fps 的 GUI 刷新。

这种设计的好处是实现简单，不需要额外的线程调度。缺点是 compositing 占用的时间会吃掉一部分 PIT 中断的处理时间。不过对于我们的教学 OS 来说完全够用。

### 延迟工作队列

在 PIT tick 回调里有些操作是不能安全执行的，比如 fork + execve 创建新的 shell 终端窗口。fork 需要操作进程表、复制地址空间，这些在中断上下文里做太危险了。所以我们用一个简单的 Atomic 变量作为标志——tick 回调检测到桌面图标被点击时设置标志，一个专门的 gui_worker 内核线程在主循环里轮询这个标志，发现有待处理的工作就在正常上下文里执行。这是一个经典的"中断上下文 defer to 进程上下文"模式。

## 动手实现

### Step 1: 窗口管理器的鼠标事件处理

**目标**: 实现 handle_mouse 方法，处理 MouseDown、MouseMove、MouseUp 三种事件。

**设计思路**: handle_mouse 首先更新内部跟踪的鼠标坐标。然后根据事件类型分发：

MouseDown 时做 hit test，从顶层到底层找第一个被点击到的可见窗口。没找到就检查是否点到了桌面图标或者空白桌面——点空白区域清除焦点。找到了就检查关闭按钮命中（调用 Window::is_close_button_hit，把鼠标坐标从屏幕空间传入），命中则销毁窗口。没命中关闭按钮就 raise 这个窗口，然后检查点击是否在标题栏（鼠标 Y 减去窗口 Y < TITLE_BAR_HEIGHT），是的话进入拖拽状态并记录偏移量。

MouseMove 时如果正在拖拽且焦点窗口非空，用当前鼠标坐标减去偏移量得到新位置，调用 set_position 更新窗口坐标，重绘标题栏和内容，调用 composite。

MouseUp 时清除拖拽状态。

**踩坑预警**: 拖拽过程中的 composite 调用可能很频繁（每次 MouseMove 都触发）。如果发现拖拽时系统卡顿，可以考虑只在 MouseUp 时做最终 composite，MouseMove 时只更新坐标不做渲染——不过目前我们的窗口数量很少，每帧都 composite 也不会有明显问题。

**验证**: 创建 2-3 个窗口，用鼠标点击不同位置确认：点击标题栏能拖拽，点击关闭按钮能销毁，点击内容区只 raise 不拖拽，点击空白区清除焦点。

### Step 2: 键盘事件路由

**目标**: 把键盘事件转发给焦点窗口。

**设计思路**: handle_key 方法极其简单——如果焦点窗口非空，就调用它的 on_key 虚函数，把 KeyEvent 传进去。具体窗口怎么处理键盘事件取决于子类实现，WindowManager 不需要知道。这是标准的观察者/策略模式。

**验证**: 如果有 Terminal 子类，可以用键盘输入字符，确认字符出现在焦点终端里。非焦点终端不应该接收到键盘事件。

### Step 3: 光标渲染

**目标**: 在 composite 的最后阶段渲染鼠标光标。

**设计思路**: 使用一个 16x16 的黑白箭头位图。位图用 uint16_t 数组表示（16 行，每行 16 位），bit 为 1 的像素画灰色，周围四个方向各画一个黑色描边像素增加对比度。光标位置从 Mouse::x() 和 Mouse::y() 获取，直接在屏幕画布上画。因为 draw_pixel 已经有边界检查，光标移到屏幕边缘时多余的像素会自动被裁剪。

**踩坑预警**: 你可能会在 QEMU 里看到两个光标——一个是我们画的箭头，一个是 QEMU VNC 自带的圆点。两者之间有一个固定偏移，方向取决于我们的初始光标位置。这是因为 PS/2 鼠标只报告相对位移，QEMU 的 VNC 光标用的是宿主机绝对坐标，两者的"原点"不同。这不是 bug，是 PS/2 协议在虚拟化环境下的固有缺陷。临时方案是把初始光标位置设为 (0, 0) 减小偏移，长期方案是实现 USB HID 驱动获取绝对坐标，或者在 QEMU 启动参数里加 `-usb -device usb-tablet`。

**验证**: 移动鼠标，确认灰色箭头跟随移动。移到屏幕右下角，确认不会超出屏幕。

### Step 4: GUI 初始化流水线

**目标**: 把鼠标驱动、窗口管理器、PIT tick 回调串进内核启动序列。

**设计思路**: 初始化分两个阶段。第一阶段在 kernel_main 里完成（中断开启之前）：创建屏幕 Canvas，调用 gui_init 初始化窗口管理器并渲染 demo 画面，然后 unmask IRQ12。第二阶段在 kernel_init_thread 里完成（调度器启动之后）：调用 gui_start，这里初始化鼠标驱动、设置屏幕边界、注册桌面图标、安装 PIT tick 回调。最后创建 gui_worker 内核线程来处理延迟工作。

之所以分两阶段，是因为鼠标驱动的 init 需要中断已经开启（要等 ACK），而窗口管理器的 init 需要一个可以工作的 Canvas。PIT tick 回调需要在调度器运行之后注册，因为回调里会调用 composite 等重操作。

**实现约束**: gui_init 和 gui_start 是 gui 命名空间下的两个独立函数，main.cpp 和 init.cpp 通过 gui_init.hpp 头文件引入。当 CINUX_GUI 宏未定义时，这些调用通过条件编译跳过。

**验证**: `make run` 后观察完整启动日志：[GUI] 初始化消息 → [MOUSE] 初始化消息 → 桌面画面渲染 → 鼠标移动响应。

## 构建与运行

```
make run
```

启动后你应该能看到深色桌面背景，上面有几个彩色矩形（demo 画面），移动鼠标能看到灰色箭头光标跟随。如果一切正常，点击左键不会有特别反应（因为还没有窗口），但光标应该能正常移动。

如果遇到 QEMU 双光标问题影响体验，可以在启动脚本里加 QEMU 参数：
```
-usb -device usb-tablet
```

## 调试技巧

如果鼠标完全没反应，第一步检查 IRQ12 是否被 unmask（在 main.cpp 的 CINUX_GUI 块里）。第二步检查 IDT 里向量 0x2C 是否注册了 irq12_stub。第三步在 mouse_irq12_handler 里加串口打印确认中断是否触发。

如果鼠标能动但窗口拖拽不工作，检查 handle_mouse 是否被调用（在 PIT tick 回调里加打印）。如果被调用了但窗口不动，检查 dragging_ 标志是否正确设置，drag_offset_x_/y_ 是否有合理值。

如果 composite 后画面闪烁，检查是否在每次 composite 前都做了 clear——不清屏的话旧位置的窗口画面会残留在屏幕上。

## 本章小结

| 概念 | 关键点 |
|------|--------|
| 拖拽状态机 | Idle → MouseDown(hit title) → Dragging → MouseUp → Idle |
| 拖拽偏移量 | 记录点击时鼠标相对窗口左上角的偏移，避免窗口跳动 |
| hit test 方向 | 从顶层到底层（数组末尾到开头）遍历 |
| 键盘路由 | 焦点窗口的 on_key 虚函数 |
| PIT tick 驱动 | 100Hz 回调清空事件队列 + composite |
| 延迟工作队列 | Atomic 标志，中断设置，工作线程消费 |
| QEMU 双光标 | PS/2 相对定位 vs VNC 绝对定位的固有偏移 |
| GUI 启动两阶段 | kernel_main: Canvas+WM+IRQ12; init_thread: Mouse+PIT callback |
