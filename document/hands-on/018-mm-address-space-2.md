# 018-2 构造、析构与 Move 语义

## 导语

上一篇我们把 AddressSpace 的类接口搭好了，init_kernel() 也在启动时成功保存了内核 PML4 模板，编译运行一切正常。
现在该实现最核心的部分了——构造函数负责分配和初始化一张全新的 PML4 表，析构函数负责在进程销毁时递归释放所有用户空间的页表页。
此外，因为拷贝被禁用了，我们还必须正确实现移动构造和移动赋值，让 AddressSpace 对象可以被安全地转移所有权（比如从临时对象移入进程表）。
说实话，这一篇是整个 AddressSpace 模块里最"硬核"的部分——递归释放子树的逻辑稍有不慎就会导致物理内存泄漏或者 use-after-free，而 move 语义的自赋值检查写漏了就是 double-free 的温床。
慢慢来，我们一步一步走。

知识前置：理解 PML4 的用户/内核分割策略（条目 0-255 属于用户空间，256-511 属于内核空间），以及 x86-64 四级页表的树形结构。你需要对 RAII 模式有基本的认知——构造获取资源，析构释放资源，移动转移所有权。如果你对 tag 016 中 VMM 的 map/unmap/translate 接口不熟悉，建议先回顾一下——因为这一篇的构造函数会调用 VMM 的 map 来创建用户空间映射。

这一篇和上一篇的关系是：上一篇画了设计蓝图（类接口、init_kernel 的调用时机、PML4 分割策略），这一篇把蓝图变成可运行的代码。你可以把上一篇看作"规划"，这一篇看作"施工"。

## 概念精讲

### 构造函数：三步创建一个干净的地址空间

在动手之前，先搞清楚构造函数的职责边界。构造函数只做三件事：分配、清零、复制。它不做映射——映射是用户代码通过 map() 方法在构造之后按需建立的。它也不做激活——激活是 activate() 的工作。构造函数的产出物是一张"干净但有内核映射"的 PML4 表——用户空间部分全零，内核空间部分和模板一致。

创建一个新的地址空间本质上就是创建一张"空白但有内核映射"的 PML4 表。操作分三步走，每一步都不可或缺。第一步，从 PMM 申请一页物理内存作为新的 PML4 表。
一张 PML4 表恰好是 4 KB（512 个条目，每个 8 字节，512 x 8 = 4096），和一个物理页的大小完全吻合——这不是巧合，而是 x86 分页设计时就精心安排的。
PMM 返回的是这页的物理基地址（4 KB 对齐的），我们把它保存在 pml4_phys_ 成员变量中。

第二步，把整张 PML4 表清零。这一步看起来多余，但 PMM 分配的物理页不保证内容是干净的——它可能之前被别的页表用过，里面还有残留的旧数据。
如果不清零就复制内核条目，用户空间部分（条目 0-255）的 PML4 条目可能指向已经释放的物理页——后续创建映射时，VMM 发现某个 PML4 条目已经标记为 present，就会顺着这个垃圾指针往下遍历，后果完全不可预测。
清零确保了用户空间部分是一张真正的白纸。但这里有一个技术细节：我们怎么访问这个刚分配的物理页来清零？内核代码运行在虚拟地址空间里，不能直接用物理地址。
Cinux 使用 higher-half 直接映射的方式来解决——物理地址 phys 对应的虚拟地址是 `phys + KERNEL_VMA`（其中 KERNEL_VMA = 0xFFFFFFFF80000000，
和我们在 tag 016 中使用的值相同）。把物理地址加上这个偏移量，就能通过虚拟地址来读写物理页的内容了。这个映射是 tag 016 中 VMM 初始化时建立的，覆盖了整个物理内存范围——只要 PMM 分配的物理地址在有效范围内，phys_to_virt 就能正确工作。

第三步，从我们之前保存的内核 PML4 模板中，把条目 256 到 511 逐个复制过来。注意这里复制的是 PML4 条目的 raw 值——一个 64 位整数。这个整数里面包含了下一级页表（PDPT）的物理地址和标志位。
所以所有 AddressSpace 的 PML4[256..511] 条目和内核 PML4 的对应条目指向同一组 PDPT 表——这就是"共享内核映射"的本质。内核空间的页表结构只有一份，但所有进程的 PML4 都指向它。
如果以后内核需要修改内核空间的映射（比如映射新的 MMIO 区域），只需要修改内核 PML4 模板中的条目，然后同步更新所有已创建的 AddressSpace 的对应条目——当然，目前的实现中我们不做这个同步，
因为内核映射在启动后就是固定的。

有一点值得特别强调：这个"三步走"的每一步都不可或缺。如果跳过第一步（分配），pml4_phys_ 就是 0，后续所有操作都无效。如果跳过第二步（清零），新 PML4 的用户空间部分可能包含垃圾数据——PMM 分配的物理页不保证内容是干净的，它可能之前被别的页表用过，里面还有残留的旧条目。如果跳过第三步（复制内核条目），新地址空间的内核映射就是空的——一旦 activate() 切换到这个地址空间，CPU 连下一条指令都取不到，直接 triple fault。

### 析构函数：递归释放子树

析构要做的事情是构造的逆操作，但比构造复杂得多。
构造时我们只分配了一页 PML4，但在这张 PML4 的生命周期内，VMM 的 map 操作会在用户空间部分创建 PDPT、PD、PT 等中间级页表页——每映射一个新的 512 GB 区域（新的 PML4 条目被填充），
就至少多分配了三个中间页表页（PDPT、PD、PT 各一页）。析构时必须把这些中间页表页全部归还给 PMM，否则进程创建和销毁一次就会泄漏掉好几页物理内存。

递归释放的遍历逻辑需要从 PML4 开始，只处理用户空间部分（条目 0-255）。对于每个 present 的条目，它指向一个 PDPT 表。我们递归进入这个 PDPT，遍历它的 512 个条目。
对于每个 present 的 PDPT 条目，递归进入 PD；再从 PD 递归进入 PT。
到了 PT 这一层，停止递归——因为 PT 条目指向的是数据页（进程的实际数据），这些数据页不由 AddressSpace 管理，应该由进程自己的内存管理逻辑来释放。

AddressSpace 释放"页表结构"本身的物理页（PDPT、PD、PT 表本身占用的物理页），同时在 PT 层也会释放 PT 条目指向的数据页。
这意味着 AddressSpace 析构时，该地址空间中所有用户空间映射关联的物理页都会被归还给 PMM。
在当前阶段这是合理的——每个 AddressSpace 独占其用户空间的物理页，没有共享机制，析构时全量释放是最简单的策略。
未来如果实现共享内存或 COW fork，需要修改 free_subtree 的行为——在 PT 层不释放数据页，改为由引用计数机制管理数据页的生命周期。

这个设计决定也影响了 free_subtree 的递归终止条件。我们用常量 LEVEL_PDPT = 3、LEVEL_PD = 2、LEVEL_PT = 1 来表示页表的层级。当 level == LEVEL_PT（值为 1）时不再递归——因为 PT 条目指向的是数据页，不属于页表基础设施。
注意终止条件是 level > LEVEL_PT 才继续递归，而不是 level >= LEVEL_PT。如果你把终止条件写成了大于等于，那么在 PT 层仍然会执行 free_page，把 PT 条目指向的数据页释放掉——这种 bug 不会立刻暴露，但会在未来导致数据损坏。

递归在每一层做的事情是相同的：遍历 512 个条目，对每个 present 条目先递归处理下一层（如果还没到 PT 层的话），然后释放当前条目指向的页表页。注意释放的顺序——先递归处理子树（释放更深层级的页），再释放当前层级。
这是因为一旦当前层级的页被释放并归还给 PMM，它的内容就可能被覆盖（PMM 会把这一页分配给其他用途），递归时再读取这个页的内容就是 undefined behavior。

### Move 语义：安全转移所有权

拷贝被删除的原因很好理解——两个 AddressSpace 共享同一个 pml4_phys_ 意味着两个析构函数都会尝试释放同一组物理页，第二次释放就是 double-free。
PMM 的 free_page 如果不做去重检查（我们的 PMM 确实做了 bitmap 检查，同一页 free 两次只是 no-op 而不是真正释放），那还好；
但如果 PMM 的实现改了（比如用 free list），double-free 可能导致同一个物理页出现在 free list 上两次——后续两次 alloc_page 返回同一个地址，两个不同的调用者以为各自独占了一页，结果互相踩数据。

但移动是安全的：移动构造函数把源对象的 pml4_phys_ 拿过来赋给目标对象，然后把源对象的值置 0。这样源对象析构时检查到 pml4_phys_ == 0 就直接 return（no-op），只有目标对象会执行真正的释放逻辑。
整个过程只有一份数据被转移，没有复制。移动赋值稍微复杂一点，因为目标对象可能已经持有资源——它之前可能创建了自己的 PML4 表并在里面 map 了页面。
所以在接管源对象的 pml4_phys_ 之前，必须先释放目标对象当前持有的用户空间子树和 PML4 页。这个释放逻辑和析构函数完全一样：遍历用户空间条目、递归释放子树、归还 PML4 页。释放完旧资源后再接管新资源，最后把源对象置 0。

自赋值安全（`this != &other`）是移动赋值必须处理的边界情况。
虽然写 `as = std::move(as)` 看起来像是脑子进水了，但在泛型代码或者容器操作中（比如 std::swap、std::sort），这种情况有可能间接发生。
如果不做自赋值检查，移动赋值会先释放自己的资源，然后试图接管"源对象"（其实就是自己）的资源——但资源已经被释放了，等于在释放后使用悬空指针。
加上 `if (this != &other)` 的保护就简单粗暴地避开了这个问题——检测到自赋值直接返回，什么都不做。

### 资源释放的顺序问题

递归释放中的"先深入、后释放"顺序不是随意选择的。free_subtree 在每个层级先递归处理子树（释放更深层级的页），然后才释放当前层级的页表页。
这是因为一旦当前层级的页被释放并归还给 PMM，它的内容就可能被覆盖（PMM 会把这一页分配给其他用途），递归时再读取这个页的内容就是 undefined behavior。
先处理子树，保证释放父级页表页之前子级已经处理完毕——这是树形数据结构遍历的经典模式。

但这里有一个微妙的安全点：在递归释放子级之后，我们仍然通过 table[i].phys_addr() 读取当前条目。这是安全的，因为 free_subtree 释放的是子级页表页，但释放只是把页帧标记为空闲，内存内容不会立即被清零或覆盖。当前层级的 table 指针指向的页在所有子条目处理完毕之后才由上一级的 free_page 释放。不过严格来说，如果 PMM 的实现变了（比如释放时清零页内容），这段代码就可能出问题——这是一个隐式假设。

## 动手实现

### Step 3: 实现构造函数

**目标**: 在 address_space.cpp 中实现 AddressSpace 的默认构造函数，完成 PML4 分配、清零和内核条目复制。

**设计思路**: 构造函数按照前面分析的三步来实现。第一步，调用 `g_pmm.alloc_page()` 分配一页物理内存。如果分配失败（返回 0），打印一条 FATAL 日志然后 return——此时 pml4_phys_ 保持为 0（成员使用了默认值初始化），后续析构时会检查到 0 直接跳过，不会出问题。第二步，通过 higher-half 直接映射（`phys + KERNEL_VMA`）把物理地址转换成虚拟地址，然后循环 512 次把每个 PML4 条目的 raw 字段设为 0。第三步，同样通过 higher-half 映射访问内核 PML4 模板，循环复制条目 256-511 的 raw 值。

需要在匿名命名空间中定义几个常量来提高可读性：USER_PML4_START = 0、USER_PML4_END = 256（这两个界定了用户空间 PML4 条目的范围）、PT_ENTRIES = 512（每张页表的条目数）、
KERNEL_VMA = 0xFFFFFFFF80000000ULL（higher-half 偏移量）。
以及一个辅助函数 phys_to_virt(phys)，接受物理地址，返回加上 KERNEL_VMA 后的 PageEntry* 指针——注意返回类型是我们 tag 016 中定义的页表项联合体指针。

**实现约束**: 构造函数在 `cinux::mm` 命名空间内。需要 include kernel/arch/x86_64/paging.hpp（获取 PageEntry 类型和常量）和 kernel/arch/x86_64/paging_config.hpp（获取 PT_ENTRIES 等常量），以及 kernel/mm/pmm.hpp（获取 g_pmm）。phys_to_virt 函数定义在匿名命名空间中，不暴露给外部。匿名命名空间是 C++ 的"内部链接"惯用法——相当于 static 函数，但不污染全局命名空间，也不会和其他编译单元的同名函数冲突。

另外需要在 .cpp 文件的匿名命名空间之外定义几个常量来提高可读性：USER_PML4_START = 0、USER_PML4_END = 256（这两个界定了用户空间 PML4 条目的范围）、PT_ENTRIES = 512（每张页表的条目数）、KERNEL_VMA = 0xFFFFFFFF80000000ULL（higher-half 偏移量）。以及 LEVEL_PDPT = 3、LEVEL_PD = 2、LEVEL_PT = 1 这三个常量，用于 free_subtree 的递归深度控制。

**踩坑预警**: KERNEL_VMA 的值是 0xFFFFFFFF80000000ULL——注意是 ULL 后缀。如果写成 0xFFFFFFFF80000000，在某些编译器配置下会被当成有符号 32 位整数，高位截断后结果完全错误，phys_to_virt 会返回一个错误的虚拟地址，写入时大概率触发 page fault 导致三重故障。另外，分配失败后不能抛异常（我们编译内核时禁用了异常），也不能调用 panic（因为此时 AddressSpace 对象还没构造完），所以用日志 + 早返回是最安全的处理方式。构造函数失败后 pml4_phys_ 为 0，对象处于一种"僵尸"状态——存在但不可用。后续的 map/translate 操作会委托给 VMM，传入 &pml4_phys_ 即传入一个指向 0 的指针，VMM 会检测到并返回 false 或 0。

这种"构造失败但不崩溃"的模式在内核 C++ 中很常见——我们编译内核时禁用了异常处理（-fno-exceptions），也没有标准库支持，所以不能用 throw 或 std::optional 来报告错误。用成员变量作为状态标志（pml4_phys_ 是否为 0）是一种简单有效的替代方案。虽然它意味着调用者需要检查对象状态，但在内核环境中这是可以接受的——内核代码通常会假设资源分配不会失败，如果真的失败了，说明系统已经处于不可恢复的状态。

**验证**: 这个步骤暂时没法单独验证——要等到析构函数也实现之后才能创建和销毁 AddressSpace 对象，否则析构时调用的 free_subtree 还是空的，编译会报未定义符号。现在只要求编译通过。

你可以先写一个最小化的 free_subtree 空实现来让编译通过——只是占位，不要做任何释放操作。等到下一篇实现了 free_subtree 的完整逻辑后再替换。或者你也可以选择一口气实现完所有方法，然后一起测试——这取决于你的开发风格。

### Step 4: 实现 free_subtree 递归释放和析构函数

**目标**: 实现 free_subtree(table_phys, level) 递归辅助函数和析构函数，正确释放所有用户空间页表页。

**设计思路**: free_subtree 是一个私有成员函数，接受两个参数：table_phys 是当前层页表的物理地址，level 是当前层级。我们用常量 LEVEL_PDPT = 3、LEVEL_PD = 2、LEVEL_PT = 1 来代替裸数字，提高可读性。函数的逻辑是这样的：首先通过 phys_to_virt 把物理地址转成虚拟地址，拿到当前级页表的指针。然后遍历 512 个条目。对于每个 present 的条目，如果 level > 1（即还没到 PT 层），先递归释放下一层——递归调用 free_subtree(table[i].phys_addr(), level - 1)。递归返回后，释放当前条目指向的页表页——调用 g_pmm.free_page(table[i].phys_addr())。

这里有一个容易搞混的地方需要特别说清楚。对于 PT 层（level == 1），条目指向的是数据页。当 level == 1 时，不递归。但 free_page 仍然会被调用——在我们的实现中，PT 层的 free_page 会释放数据页。
PT 页本身（作为一个物理页）是由上一层（PD 层，level == 2）的 free_subtree 释放的——PD 层遍历到某个 present 条目时，先递归到 PT（level - 1 = 1），在 PT 层处理数据页；
然后回到 PD 层，释放 PD 条目指向的 PT 页。所以 PT 页是被正确释放的，释放动作发生在 PD 层。

析构函数的逻辑更简单：先检查 pml4_phys_ 是否为 0（如果是，说明对象被移动过了或者是分配失败的对象，直接 return），然后遍历 PML4 的用户空间部分（条目 0-255），
对每个 present 的条目调用 free_subtree(pml4_phys, LEVEL_PDPT)——注意起始层级是 3（PDPT），不是 4（PML4），因为 PML4 条目指向的就是 PDPT。
遍历完所有用户空间条目后，调用 g_pmm.free_page(pml4_phys_) 释放 PML4 页本身，把 pml4_phys_ 置 0。

**踩坑预警**: 这里最容易犯的错误是在 PT 层释放数据页。如果你把终止条件写成 `level >= LEVEL_PT`（大于等于）而不是 `level > LEVEL_PT`（大于），那么当 level == 1 时仍然会执行递归调用 free_subtree，传入的是 PT 条目指向的数据页的物理地址——这不是一张合法的页表，遍历它的 512 个"条目"实际上是在读取用户数据，后果完全不可预测。不过在我们的实现中，free_page 在 level=1 时仍然会被调用——它会释放 PT 条目指向的数据页。这在当前阶段是可接受的（每个 AddressSpace 独占其数据页，没有共享机制），但如果你计划实现共享内存或 COW fork，需要修改 free_subtree 在 level=1 时不调用 free_page。

另一个容易出错的地方是忘记在析构函数最后释放 PML4 页本身——如果你只递归释放了子树但忘了释放根，每创建销毁一个 AddressSpace 就泄漏一页（4KB）。在频繁创建和销毁进程的场景下，这种泄漏会快速耗尽物理内存。

**验证**: 可以写一个简单的测试来验证——创建一个 AddressSpace，在其中 map 几个用户空间页面（这会创建中间级页表），然后销毁它。在销毁前后检查 PMM 的已分配页数，如果差值为 0（或者等于构造前的基线），说明所有页都被正确释放了。这个验证需要到下一篇写测试代码时才能执行。

在深入实现之前，用一个具体的例子来理解 free_subtree 的执行流程会很有帮助。假设一个 AddressSpace 在虚拟地址 0x20000000（对应 PML4 索引 1）映射了一个页面。那么它的页表结构是：PML4[1] -> PDPT 表 -> PDPT[0] -> PD 表 -> PD[0] -> PT 表 -> PT[0] -> 数据页。析构时，free_subtree 从 PML4[1] 开始，level = LEVEL_PDPT（3）：
1. 进入 PDPT 表（level=3），遍历 512 个条目，找到 PDPT[0] 是 present 的
2. 因为 level (3) > LEVEL_PT (1)，递归调用 free_subtree(PDPT[0].phys_addr(), 2)，进入 PD 表
3. 在 PD 表（level=2）中找到 PD[0] 是 present 的，因为 level (2) > LEVEL_PT (1)，递归调用 free_subtree(PD[0].phys_addr(), 1)，进入 PT 表
4. 在 PT 表（level=1）中，level 不再大于 LEVEL_PT，不递归。但对每个 present 条目调用 free_page 释放该 PT 条目指向的物理页
5. 回到 PD 层（level=2），free_page 释放 PD[0] 指向的 PT 表页本身
6. 回到 PDPT 层（level=3），free_page 释放 PDPT[0] 指向的 PD 表页本身

最终效果：PDPT 表页、PD 表页、PT 表页和 PT 条目指向的数据页全部被归还给 PMM。PML4 页本身由析构函数在遍历完所有用户空间条目后释放。

### Step 5: 实现 Move 构造和 Move 赋值

**目标**: 实现移动构造函数和移动赋值运算符，支持 AddressSpace 对象的所有权安全转移。

**设计思路**: 移动构造函数很简单：用源对象的 pml4_phys_ 初始化目标对象的 pml4_phys_（在初始化列表中完成），然后把源对象置 0。三行代码搞定，不需要额外的错误处理——移动操作不会失败（不涉及 PMM 分配），所以标记为 noexcept。移动赋值运算符稍微复杂：首先检查自赋值（`if (this != &other)`），这是安全的前提。然后判断目标对象当前是否持有资源（pml4_phys_ != 0），如果是，执行和析构函数完全相同的释放逻辑——遍历 PML4 用户空间条目、递归 free_subtree、释放 PML4 页。释放完毕后，接管源对象的 pml4_phys_，把源对象置 0。

**实现约束**: 两个函数都标记为 noexcept。移动赋值中的释放逻辑和析构函数完全重复——这在一个小型 RAII 类中是可以接受的。如果你对代码重复有洁癖，可以把释放逻辑提取到一个名为 release() 或 destroy() 的私有方法中，让析构函数和移动赋值都调用它。但在目前这个规模下（释放逻辑只有几行），直接复制更清晰。

**踩坑预警**: 移动赋值中释放旧资源的逻辑和析构函数一模一样——这意味着如果你改了析构函数的逻辑（比如加了一个新的清理步骤），必须同步修改移动赋值中的释放代码。漏改的结果就是：通过移动赋值替换的 AddressSpace 会泄漏资源，而正常析构的不会。建议在代码中加一个注释提醒自己这两个地方需要保持同步。

**验证**: 编写测试用例来验证 move 语义。创建 AddressSpace 对象 A，记录它的 pml4_phys() 值。然后执行移动构造：`auto B = std::move(A)`。检查 A 的 pml4_phys() 为 0，B 的 pml4_phys() 等于 A 之前的值。然后分别销毁 A 和 B——A 的析构应该是 no-op，B 的析构应该正确释放资源。这些测试在下一篇的集成测试中实现。

移动赋值的测试稍微复杂一些——你需要创建两个 AddressSpace 对象（各有各的 PML4），然后执行 `as2 = std::move(as1)`。检查 as2 的 pml4_phys() 变成了 as1 之前的值，as1 的 pml4_phys() 变成了 0。而且 as2 之前持有的旧 PML4 应该被释放了——你可以在移动赋值前后检查 PMM 的分配计数来验证。

你也可以先做一个手动的冒烟测试：在构造函数实现后，在 kernel_main() 中创建一个 AddressSpace 对象，观察串口输出是否有异常。但注意不要在测试之外调用 activate()——因为我们还没有写测试代码来恢复 CR3，手动 activate 后如果忘了恢复会导致三重故障。

## 调试技巧

**析构时三重故障**: 如果你在销毁一个 AddressSpace 时立刻三重故障，最可能的原因是——你正在释放的 PML4 页恰好就是 CR3 当前指向的那张。CPU 正在使用这张页表翻译地址，你却把它的 PML4 物理页还给了 PMM，CPU 下一次翻译时发现 PML4 条目全是垃圾数据，直接炸了。解决方法是：在销毁 AddressSpace 之前，先把 CR3 切回内核 PML4（调用 `write_cr3(AddressSpace::kernel_pml4())`）。在测试代码中尤其要注意这一点——下一篇写测试时会反复强调。

**递归释放导致 PMM double-free**: 如果 PMM 报了 double-free 错误（或者你在 PMM 里加了断言触发了 panic），说明 free_subtree 把同一个物理页释放了两次。检查你的递归逻辑——是不是在 PT 层既释放了 PT 条目指向的数据页（不应该），又在上一层的递归中释放了 PT 页本身（应该）？或者是不是 move 赋值的自赋值检查没生效，导致同一个 pml4_phys 被释放了两次？

**phys_to_virt 返回错误地址**: 如果 higher-half 直接映射的偏移量 KERNEL_VMA 写错了（比如少了一个 F），phys_to_virt 返回的虚拟地址就是错的，写入时会触发 page fault。在 QEMU 中可以用 `-d int` 参数追踪异常，看 CR2 的值（出错的虚拟地址）是否落在 higher-half 映射范围内（0xFFFFFFFF80000000 附近）。这个 bug 在 tag 016 中也容易出现——higher-half 映射偏移量在多个文件中使用，必须保持一致。如果 address_space.cpp 中的 KERNEL_VMA 和 paging 代码中的值不同，构造函数复制内核条目时就会读写错误的虚拟地址。

**析构后内核立刻崩溃**: 如果在销毁一个 AddressSpace 之后内核立刻崩溃（串口输出突然中断），说明析构函数可能释放了它不该释放的东西。检查 free_subtree 的遍历范围是否正确——它应该只处理 PML4[0..255]（用户空间），不应该触及 PML4[256..511]（内核空间）。如果 USER_PML4_END 的值被错误地设为 512 而不是 256，析构函数就会遍历并释放内核空间的页表页，CPU 的地址翻译立即失效。

**构造后 pml4_phys() 返回 0**: 这说明 PMM 分配失败了。在正常运行中这不应该发生——PMM 在启动时管理了几十 MB 的物理内存，分配一页 4KB 完全不成问题。但如果你的代码中有内存泄漏（比如之前的 AddressSpace 没有正确析构，PMM 的空闲页被逐渐耗尽），到了某个时刻 PMM 就会返回 0。检查是否有 AddressSpace 对象在离开作用域时没有被正确析构——特别要注意被移走的对象是否也参与了析构（它们应该跳过释放）。

**递归深度导致栈溢出**: free_subtree 的最大递归深度是 3（PDPT -> PD -> PT），每次递归的栈帧很小（几个局部变量），所以栈溢出的风险几乎不存在。但如果你未来修改了递归逻辑（比如加入了 5 级分页的支持，或者修改了终止条件导致更深层的递归），就需要注意栈空间的使用。内核栈通常只有 8KB（2 页），非常有限。

## 设计决策

**为什么不提取公共的 destroy() 方法**: 移动赋值和析构函数中都有相同的释放逻辑（遍历用户空间条目、递归 free_subtree、释放 PML4 页）。提取一个私有的 destroy() 方法来消除重复是合理的重构。但在当前阶段（释放逻辑只有几行），直接复制更清晰——读析构函数的人不需要跳转到另一个方法就能理解完整逻辑。如果将来释放逻辑变复杂（比如需要释放 VMA 链表、通知其他模块等），再提取也不迟。

**为什么不使用 std::unique_ptr 管理物理页**: std::unique_ptr 的删除器可以自定义来调用 g_pmm.free_page()，看似很优雅。但 AddressSpace 管理的不是单一资源——除了 PML4 页本身，还有用户空间的整棵页表子树。析构时需要递归释放子树，这不是一个简单的 free_page 调用能搞定的。手动管理生命周期虽然代码多一点，但逻辑更透明。

**Move 后原对象仍然"可用"**: 如果你发现 move 之后的源对象还能正常调用 map/translate，说明你忘了在 move 操作中把源对象的 pml4_phys_ 置 0。此时两个对象持有相同的物理地址，析构时就会 double-free。

**移动赋值和析构函数的同步维护**: 移动赋值中释放旧资源的逻辑和析构函数一模一样。这意味着如果你改了析构函数的逻辑（比如加了一个新的清理步骤），必须同步修改移动赋值中的释放代码。漏改的结果就是：通过移动赋值替换的 AddressSpace 会泄漏资源，而正常析构的不会。建议在代码中加注释提醒这两个地方需要保持同步，或者把释放逻辑提取到一个私有的 destroy() 方法中。

## 本章小结

这一篇实现了 AddressSpace 生命周期管理的核心逻辑。构造函数分三步创建一张"空白但带内核映射"的 PML4 表——分配、清零、复制内核条目。
析构函数递归释放用户空间的所有页表页和数据页（从 PDPT 到 PT，包括 PT 条目指向的数据页），最后归还 PML4 页本身。Move 语义安全地转移 PML4 的所有权——源对象置 0 变成 no-op 壳，目标对象接管所有资源。

到这里 AddressSpace 的核心实现已经完成了大约 80%——类的骨架、构造、析构、move 语义全部就位。我们还没有实现的是四个公开方法：map()、unmap()、translate() 和 activate()。前三个是单行委托，实现非常简单；activate() 也是一行 write_cr3，但它背后的硬件行为（TLB 刷新、地址空间切换）需要仔细理解。
