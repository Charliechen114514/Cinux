# 032-1 Canvas 位图渲染与透明机制

## 导语

到现在为止，我们的 Canvas 类已经能画矩形、画线、画文字，甚至能把一个 Canvas 的内容 blit 到另一个 Canvas 上——这些基本够用来搭建窗口管理器了。但如果你仔细想一想，桌面上最显眼的那些元素——回收站、终端图标、文件管理器——全都是小图片，不是用 draw_rect 画出来的色块。我们今天要做的，就是给 Canvas 加上"画一张小图片"的能力。

这个功能的核心需求其实很简单：给定一个像素数组，把它原封不动地画到 Canvas 的指定位置。但这里有一个容易忽略的关键点——小图片的形状通常不是完美的矩形，图标有圆角、有不规则的边缘，这意味着我们需要某种透明机制来告诉 Canvas "这个像素不要画"。这就是本篇要解决的两个核心问题：位图渲染和透明度处理。

完成本篇后，我们将在 Canvas 上拥有一个能正确处理透明像素、自动裁剪越界的位图绘制能力，为后续的桌面图标系统打下基础。前置要求是上一篇 tag（031_gui_native_app）的窗口管理器和 Canvas 双缓冲系统已经就绪。

## 概念精讲

### 像素数组与位图——从数据到画面

所谓"位图"在字面意义上就是"用位组成的图"——一个二维的像素网格，每个像素用固定长度的数值来表示颜色。在我们的系统里，每个像素是一个 32 位无符号整数，格式是 `0x00RRGGBB`，也就是高 8 位留空、接下来 8 位红色、8 位绿色、8 位蓝色。一个 32x32 的图标需要 1024 个这样的数值，按行优先顺序排列：第一行的 32 个像素排在最前面，第二行紧随其后，以此类推。这种存储方式叫做 row-major order（行优先序），是图形编程中最常见的数据布局方式，几乎所有图形 API 和文件格式都采用这种排列。

你可以把 draw_bitmap 想象成一块印章——我们把印章（像素数组）按到纸（Canvas 后端缓冲区）的指定位置上，印章上每个像素对应纸上的一个点。和真实印章不同的是，我们的"印章"有些位置是空的（透明的），这些位置不应该在纸上留下任何痕迹。

渲染的数学过程很直观但值得仔细理解。假设我们要在 Canvas 坐标 (dst_x, dst_y) 处画一个宽 w 高 h 的位图。对于位图中的每一行 row（从 0 到 h-1）和每一列 col（从 0 到 w-1），目标像素在 Canvas 后端缓冲区中的索引是 `(dst_y + row) * stride + (dst_x + col)`，其中 stride 等于 pitch 除以 4，代表每行的 uint32_t 元素数。而源像素在位图数组中的索引是 `row * w + col`——注意这里用的是位图宽度 w 而不是 Canvas 的 stride，因为位图数组是紧凑存储的，行与行之间没有任何填充字节。

这个区别听起来很小，但搞混了后果很严重。如果你用 stride 去索引源像素数组，每行就会多跳过几个元素（stride 通常大于等于 w），导致图标被水平拉伸或者出现条纹状的错位。这是一个在图形编程中反复出现的坑——源和目标的行步长（stride）不一定相同，必须分别使用正确的值。OSDev Wiki 的 "Drawing In a Linear Framebuffer" 页面特别强调了这一点：像素位置的正确计算公式是 `pixel = vram + y * pitch + x * pixelwidth`，pitch 可能和 width * bytes_per_pixel 不同。

### 透明色的选择——为什么用 0x00000000

透明有几种做法，每种做法在正确性、性能和实现复杂度之间做了不同的权衡。最"正统"的是 alpha blending：每个像素有一个额外的 alpha 通道（0-255），表示这个像素的不透明度，渲染时需要和背景做加权混合。具体来说，Porter-Duff "source over" 公式是这样的——对于红色通道：`result_R = src_R * src_A / 255 + dst_R * (255 - src_A) / 255`，绿色和蓝色通道同理。这意味着每个像素需要 6 次乘法和 3 次加法。SerenityOS 的 Painter 类就是用这种方式实现的，他们通过模板参数在编译时选择不同的混合路径来优化性能——有 SrcAlpha、DstAlpha、BothAlpha、NoAlpha 四种模式，还有一个专门的 issue（#69）追踪 SIMD 优化 alpha blending 的进展。

另一种更简单的方案叫 color-key transparency（色键透明）：约定一个特定的颜色值代表"透明"，遇到这个值就跳过不画。Windows 95/98 时代的位图就是用这种方式处理透明的——经典的选择是把粉红色（0x00FF00FF）或者纯黑色作为透明色。这种方案对 CPU 来说几乎零开销——只需要一个整数比较和一次条件跳转，比 alpha blending 的 6 次乘法加 3 次加法廉价得多。

我们选择 `0x00000000` 作为透明色的理由是多方面的。首先是语义上的自然性——这个值在各通道上都是零，红色是零、绿色是零、蓝色也是零，一个"什么颜色都没有"的像素，本身就是"不可见"的最佳语义表达。其次是格式上的安全性——我们用的是 0x00RRGGBB 格式（高 8 位始终为零），所以 0x00000000 永远不会和一个"真正的"可见颜色混淆，因为任何可见的颜色在 RGB 通道上至少有一个不为零。最后是实现上的简洁性——一行 `if (color == 0x00000000) continue` 就完成了全部透明处理，不需要任何额外的计算。

当然，color-key 也有它的局限——它只支持"完全透明"和"完全不透明"两种状态，不支持半透明效果。但想想我们的使用场景：桌面图标需要的是清晰的像素边界，不是半透明的羽化边缘。color-key 方案在这个场景下完全够用。

### 裁剪（Clipping）——越界怎么办

当位图被画到 Canvas 的边缘或者完全超出 Canvas 范围时，我们不能让像素写到缓冲区以外的地方——那会踩到别的内存，后果不堪设想。在内核态写越界内存不会有段错误提示你出了问题，而是直接把别的数据结构搞乱，然后你在完全不相关的地方遇到诡异的 crash，调试起来血压拉满。

裁剪的本质就是在写每个像素之前检查"这个位置还在 Canvas 范围内吗"。我们采用的策略非常直接——在内层循环中检查列方向越界（`x + col >= width_` 时 break 当前行），在外层循环中检查行方向越界（`y + row >= height_` 时 break 整个循环）。这种 break-based clipping 虽然不是最高效的做法——更优化的方式是先计算可见区域的矩形交集（像 SerenityOS 的 `rect.intersected(clip_rect())` 那样），然后只遍历交集部分——但它简单正确，而且对于 32x32 这种小位图来说，多几次比较的开销完全可以忽略。

这种策略自然覆盖了三种情况。第一种情况是位图完全在 Canvas 范围内：所有 break 条件都不触发，正常渲染全部像素。第二种情况是位图部分超出 Canvas 范围：超出部分的行或列被 break 跳过，只渲染落在 Canvas 内的部分——比如一个 32x32 的图标放在 x=1010, y=750 的位置，在 1024x768 的 Canvas 上只有右下角的 14x18 像素可见。第三种情况是位图完全在 Canvas 范围外：外层循环的第一个 row 就触发 break，循环体一次也不执行，什么都不画。

### draw_bitmap 和 blit 的区别

Canvas 已经有一个 `blit()` 方法用于 Canvas 之间的区域复制。如果你熟悉 blit 的实现，可能会问——draw_bitmap 和 blit 有什么不同，能不能复用 blit 的逻辑？答案是两者在源数据的处理方式上有根本差异，不太适合复用。

blit 的源是另一个 Canvas 对象——它有自己的后端缓冲区、自己的 pitch、自己的宽高。blit 在读取源像素时需要用源 Canvas 的 stride（`src.pitch_ / 4`），写入目标时用目标 Canvas 的 stride。blit 还支持负的 dst_x / dst_y——窗口被拖到屏幕边缘外时，需要计算列偏移 col_skip 来跳过源数据中不可见的部分。

draw_bitmap 的源是一个裸的像素数组——没有 Canvas 对象包装它、没有 pitch 信息、行紧密排列（stride 就是 w）。draw_bitmap 不需要负坐标支持（图标不会被拖动到屏幕外），所以裁剪逻辑更简单。draw_bitmap 还有一个 blit 没有的功能——透明像素跳过。blit 做的是无条件拷贝（直接覆盖目标像素），draw_bitmap 做的是条件拷贝（透明像素保留目标的原有内容）。这三个差异加在一起，让两个方法的内部逻辑有了足够的区别，各自独立实现更清晰。

## 动手实现

### Step 1: 在 Canvas 中添加 draw_bitmap 方法声明

**目标**: 在 canvas.hpp 中声明新的 draw_bitmap 方法，并编写完整的文档注释。

**设计思路**: 方法的参数设计遵循 Canvas 已有方法的风格——目标坐标在前（x, y），尺寸在中（w, h），数据源在后（pixels）。这和 draw_rect(x, y, w, h, color) 的参数顺序保持一致。所有坐标和尺寸参数使用 uint32_t（无符号），因为图标不会被拖到负坐标位置。pixels 参数使用 `const uint32_t*`，指向不可修改的像素数据（因为图标数据是编译时常量）。

**实现约束**: 文档注释必须说明三件事——像素格式（0x00RRGGBB 的 32 位颜色值）、透明色约定（值为 0x00000000 的像素被跳过）、裁剪行为（超出 Canvas 边界的像素不写入）。方法的访问级别是 public，和 draw_pixel、draw_rect 等方法一样。

**踩坑预警**: 别忘了在方法声明的末尾加分号——这看起来很蠢但确实犯过。另外，如果你之前用 forward declaration 声明过某些类型（比如 Framebuffer 或 PSFFont），确保新方法的参数类型不需要额外的 forward declaration。

**验证**: 添加声明后保存头文件，先不实现方法体。此时编译会报 undefined reference 错误——这是预期的，说明声明被正确识别了但缺少实现。

### Step 2: 实现 draw_bitmap 的主循环

**目标**: 在 canvas.cpp 中实现 draw_bitmap 方法，包含防御性检查、行扫描循环、透明判断和裁剪逻辑。

**设计思路**: 方法入口先做两个防御性检查——back_buf_ 为空说明 Canvas 没有初始化，pixels 为空说明调用者传了空指针。两种情况都直接返回，不做任何操作。然后进入双层循环：外层遍历行，内层遍历列。在内层循环中，依次执行列方向裁剪检查（break）、读取源像素、透明检查（continue）、写入目标缓冲区。外层循环在每行开始时做行方向裁剪检查（break）。

**实现约束**: stride 的计算必须在循环之前完成（`pixels_per_row = pitch_ / 4`），避免在每次迭代中重复计算。源像素的索引使用 `row * w + col`，目标像素的索引使用 `(y + row) * pixels_per_row + (x + col)`。透明判断使用严格等于比较（`color == 0x00000000`），不做范围检查。

**踩坑预警**: 这里最容易犯的错误有两个。第一个是把源像素索引写成 `row * pixels_per_row + col`——这会导致当 pixels_per_row 不等于 w 时读取错误的数据。第二个是把写入和裁剪的顺序搞反——如果你先读取像素再做裁剪检查，那么即使当前像素在 Canvas 外也会被读取（虽然不会写入），这不仅浪费了一次内存访问，在某些极端情况下（位图数据恰好紧邻一个不可读的内存页）甚至可能触发 page fault。

**验证**: 实现完成后项目应该能编译链接成功。此时 draw_bitmap 的逻辑已经完整，但还没有视觉输出可以验证——我们会在后续的测试章节中用像素级验证来确认正确性。

### Step 3: 验证透明和裁剪行为

**目标**: 通过测试用例验证 draw_bitmap 的透明跳过和边界裁剪行为符合预期。

**设计思路**: 验证需要覆盖五个场景——不透明像素的精确位置和颜色值、透明像素不覆盖已有内容、右边缘和底边缘的裁剪、完全超出 Canvas 时不写入任何像素、空指针防护不崩溃。每个场景的验证方式是：先准备一个已知状态的 Canvas（用 clear 填充特定颜色），执行 draw_bitmap 操作，然后逐像素检查关键位置的值。

**踩坑预警**: 在验证透明行为时，一定要用一个非零的背景色（比如 0x000000FF 蓝色）——如果你用 clear(0) 填充黑色背景，然后画透明像素，检查结果是否为零，你无法区分"透明跳过保留了零"和"写入了零"两种情况。非零背景色可以明确区分这两种行为。

**验证**: 所有测试用例通过后，draw_bitmap 的行为就是可信的。在实际渲染中，你可以看到带透明区域的位图被正确地叠加在桌面背景上——透明的部分透出背景色，不透明的部分覆盖背景色。

## 构建与运行

整个 draw_bitmap 的实现不需要特殊的 QEMU 参数，因为我们使用的仍然是之前 tag 已经设置好的 VBE 图形模式。编译时确保 `CINUX_GUI` 宏被定义——这个宏控制了 Canvas 类和相关 GUI 代码的编译条件。

构建和运行命令与之前 tag 完全一致。如果你在 QEMU 中看到之前 tag 的窗口系统仍然正常工作，说明 draw_bitmap 的添加没有破坏任何既有功能——因为我们只是在 Canvas 类中增加了一个新方法，没有修改任何现有方法的逻辑。这一点很重要——在添加新功能时，不破坏现有功能是最基本的要求，也是重构安全的标志。

## 调试技巧

**像素位置偏移**: 如果渲染出来的图案歪了或者缺了一行，最先怀疑的是行偏移计算。用串口打印 pitch_ 和 width_ 的值，确认 `pitch_ / 4` 是否等于 `width_`。如果不等，说明 VBE 模式有行对齐填充字节，此时必须使用 pitch 而非 width 来计算行偏移。

**透明不生效**: 如果所有像素都被画出来了（包括应该透明的部分），检查透明判断是不是写错了——比如写成 `if (color != 0x00000000) continue`（逻辑反了）或者 `if (color == 0xFFFFFFFF) continue`（值写错了）。

**完全看不到位图**: 如果位图画上去后什么都看不到，排查方向有三个——坐标是否在合理范围内（用串口打印 x、y、w、h 的值）、像素指针是否为空、位图数据本身是否全是透明的。最后一个方向容易被忽视——如果你刚定义了一个新的图标但忘了填充图案数据，所有像素默认是零（透明），画上去当然什么都看不到。

**部分像素颜色不对**: 如果某些像素的颜色不是你期望的值，检查像素格式的字节序。我们的系统使用 0x00RRGGBB 格式（红在高字节、蓝在低字节），但某些图形系统和文件格式使用的是 0x00BBGGRR 格式（蓝在高字节、红在低字节）。如果格式搞混了，红色和蓝色会被交换——图标看起来像是用蓝色画了本该是红色的部分。用串口打印几个像素的原始值（十六进制），和你的预期做对比就能定位问题。

## 本章小结

| 概念 | 要点 |
|------|------|
| 位图渲染 | 逐行逐像素从源数组复制到 Canvas 后端缓冲区，源索引用 w，目标索引用 pitch / 4 |
| 透明色 | 0x00000000 表示完全透明，遇到时 continue 跳过写入，保留背景内容 |
| 裁剪 | 利用 height_ 和 width_ 做边界检查，越界时 break，自然覆盖三种越界情况 |
| pitch 意识 | 行偏移用 pitch / 4 而非 width，适配 VBE 行对齐填充字节 |
| 防御性编程 | 入口检查缓冲区和像素指针是否为空，所有错误情况静默处理 |
| 与 blit 的区别 | 源不同（裸数组 vs Canvas）、裁剪方式不同、有无透明处理 |
