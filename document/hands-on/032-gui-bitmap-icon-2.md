# 032-2 编译期图标数据——constexpr 打造嵌入式像素

## 导语

上一篇我们已经让 Canvas 具备了 draw_bitmap 的能力，但还留下一个悬而未决的问题：像素数据从哪里来？在成熟的操作系统里，图标通常是 PNG 或者 SVG 文件，运行时从磁盘加载、解码、渲染。但我们的内核还没有文件系统（或者说，即使有 ramdisk 也不一定有现成的 PNG 解码器），更别提什么 GUI 资源管理器了。所以我们需要一种更原始的方式来存储图标——把它直接编译进内核二进制。

这一篇要讲的就是 Cinux 的编译期图标数据系统。我们会看到如何用 C++ 的 constexpr 模板在编译时把人类可读的 hex 字符串转换成 32x32 的像素数组，以及如何用调色板系统让图标的定义变得紧凑又直观。完成本篇后，我们将拥有两个可以在内核代码中直接使用的预定义图标（Shell 终端和 Calculator 计算器），它们的数据完全在编译时生成，零运行时开销。

## 概念精讲

### 为什么不从文件加载

在正式的桌面系统里，图标资源的加载流程通常是：文件系统读取 → 图像解码（PNG/BMP/SVG）→ 内存中的像素缓冲 → 渲染到屏幕。这条链路需要文件系统驱动、堆内存分配器、图像格式解析器——全都是我们已经实现或正在实现的子系统。但对于 32x32 的小图标来说，这套流程实在太重了。一个 32x32 的像素数组只有 1024 个 uint32_t，也就是 4KB——直接塞进内核二进制里完全没问题，还能省掉运行时加载和解码的开销。

这种做法并不罕见。Linux 内核的启动 logo 就是这么干的：在 `include/linux/linux_logo.h` 里用预处理的像素数组把企鹅图片编译进内核。Cinux 把这个思路往前推了一步——不是用预处理宏，而是用 C++17 的 constexpr 函数和模板在编译时完成"字符串到像素"的转换，类型安全且可验证。

### 调色板系统——16 色编码

如果你直接写 1024 个 `0x00RRGGBB` 数值来定义一个 32x32 的图标，代码会变成一坨完全不可读的魔法数字。调色板系统的思路是：定义一个最多 16 种颜色的查找表（palette），然后用一个 hex 字符（0-9, a-f）来代表一种颜色。这样每个像素只需要一个字符来描述，整行 32 个像素就是 32 个字符——非常接近 ASCII art，人类可以一眼看出图标的形状。

调色板的好处不仅在于可读性。不同的图标可以共享某些颜色定义但使用不同的调色板，这意味着同一个"深灰色"在不同图标的调色板里可以被映射到不同的实际颜色值。比如在 Shell 图标里，调色板索引 2 代表标题栏的灰色；在 Calculator 图标里，索引 2 代表边框的深灰色——两者共享同一个 hex 字符但实际颜色不同。

### constexpr build_icon 模板——编译时的"图片解码器"

`build_icon` 是整个系统的核心。它是一个 constexpr 模板函数，接受两个参数：一个 16 元素的颜色查找表和一个包含 32 个 C 字符串的数组。函数在编译时遍历每个字符串的每个字符，把 hex 字符转换为 0-15 的数字（nibble），然后从调色板中查找对应的颜色值，写入结果数组。最终产出的是一个 `std::array<uint32_t, 1024>`，完全在编译时计算完毕，运行时就是一个普通的常量数组。

这里有一个设计上的细节值得注意：为什么用字符串而不是用数字数组来定义图案？答案是可读性。当你看到一行 `"01111111111111111111111111111110"` 时，你能直观地看出这是一个左右各有一个透明像素、中间全是深色的矩形条。如果把同样的数据写成数字数组 `{0, 1, 1, 1, ...}`，可读性会大打折扣。

## 动手实现

### Step 1: 定义图标公共接口

**目标**: 创建图标的公共头文件，定义图标尺寸常量和命名空间。

**设计思路**: 图标尺寸被定义为编译时常量（32x32），放在专门的 `cinux::gui::icons` 命名空间中。这个头文件作为图标的唯一对外入口，其他模块只需要 include 这一个文件就能获得所有图标相关的定义和数据。

**实现约束**: 头文件使用 pragma once 防止重复包含。常量使用 constexpr 确保是编译时求值。数据文件（icon_data.hpp）通过 include 指令从这个接口头文件引入，形成一个清晰的依赖层次。

**验证**: 头文件添加后项目应正常编译。可以通过在某个测试文件中引用这些常量来确认命名空间和值是否正确。

### Step 2: 实现调色板和 build_icon 模板

**目标**: 在 icon_data.hpp 中定义颜色调色板和编译期图标构建器。

**设计思路**: 首先定义一组全局颜色常量（黑色、白色、灰色等），这些颜色可以被多个调色板复用。然后定义一个 namespace detail 中的辅助函数链：hex_nibble 把一个 hex 字符转成 0-15 的数字，palette_lookup 根据索引从调色板查颜色，build_icon 把整个 32x32 的字符串矩阵展开成像素数组。

**实现约束**: 所有函数都标记为 constexpr 或 inline constexpr。build_icon 使用模板参数约束行数必须为 32（通过 static_assert）。结果类型是 `std::array<uint32_t, 1024>`。

**踩坑预警**: hex_nibble 必须同时处理大写和小写的 hex 字符（'A'-'F' 和 'a'-'f'）。如果只处理小写，那么用大写定义的图案数据会在编译时产生错误的结果（所有大写字母被映射为 0）。另一个容易忽视的点是 build_icon 的字符串数组参数类型——必须是指向 const char 的指针的 const 引用数组（`const char* const (&rows)[Rows]`），否则编译器无法正确推导模板参数。

**验证**: 可以通过编写一个简单的编译期断言来验证——比如检查某个已知位置的像素值是否等于预期的颜色。

### Step 3: 定义 Shell 和 Calculator 图标

**目标**: 使用 build_icon 模板和各自的调色板定义两个 32x32 的图标。

**设计思路**: 每个图标有自己的调色板（最多 16 种颜色，实际只用到 7-8 种）和 32 行 hex 字符串。Shell 图标的视觉设计是：圆角深色终端窗口、标题栏上有红黄绿三个小圆点（模拟 macOS 风格的窗口按钮）、中间有一个白色的 ">_" 命令提示符和绿色光标。Calculator 图标的视觉设计是：灰色圆角机身、顶部是绿色 LCD 显示屏显示 "123"、下面是灰色的按钮网格、最右边一列有橙色的等号按钮。

**实现约束**: 每个图标声明为 `inline constexpr std::array<uint32_t, 1024>`，inline 确保头文件包含不违反 ODR（One Definition Rule），constexpr 确保编译时求值。

**踩坑预警**: 定义图案字符串时，每一行必须恰好是 32 个字符——不多不少。如果多了一个或少了一个空格，build_icon 的 static_assert 不会报错（它检查的是行数而不是每行的长度），但后续像素的读取会错位，导致整个图标变形。建议在定义完每一行后仔细对齐字符数。

**验证**: 编译成功后，图标数据已经嵌入二进制。后续的测试章节会通过 draw_bitmap 把它们渲染到屏幕上做视觉验证。在开发阶段，你也可以用串口打印几个关键位置的像素值来确认调色板映射是否正确。

## 构建与运行

本篇新增的文件是纯头文件（icon.hpp 和 icon_data.hpp），不需要修改 CMakeLists.txt 中的源文件列表。只要确保 icon.hpp 被 WindowManager 或测试文件正确 include 即可。

编译命令不变，但要注意一点：icon_data.hpp 中的 build_icon 使用了 constexpr 模板，这在 C++17 下完全支持。但如果你在代码中不小心用到了 C++20 的 consteval（编译器会在编译时报错），需要注意内核构建可能使用的是较老的 C++ 标准。这也是为什么 const 相关的测试放在宿主端而非内核端的原因之一。

## 调试技巧

**图标显示乱码**: 如果渲染出来的图标看起来像噪声，最可能的原因是调色板映射错误或者图案字符串长度不对。用串口打印图标数组的前几个像素值，对照调色板定义检查是否匹配。

**编译时间增长**: constexpr 的 build_icon 在编译时展开 32x32 = 1024 个像素的计算，对于两个图标来说是 2048 次 constexpr 求值。这在现代编译器上完全不是问题，但如果你扩展到更多更大的图标（比如 64x64），可能会感受到编译时间的增加。

**链接错误——重复定义**: 如果忘记在 `std::array` 变量前加 `inline`，在多个编译单元 include 同一个头文件时会出现链接错误（multiple definition）。`inline constexpr` 是 C++17 头文件定义全局常量的标准写法。
