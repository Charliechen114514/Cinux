# ext2 文件读写与间接块

> 标签：ext2、文件读写、间接块、Inode Cache、块分配、get_or_alloc_block
> 前置：[028_fs_ext2 第 2 篇](028-fs-ext2-2.md)

## 前言

前两篇我们把 ext2 驱动的骨架搭好了——Superblock 解析、BGDT 加载、inode 寻址、DMA buffer 就位。骨架有了，该让它"动"起来了。

这一篇我们要实现的是 ext2 驱动中最有实质性意义的功能：文件的读和写。从 inode 的块指针数组出发，经过直接块和间接块的寻址逻辑，最终在用户空间的 buffer 和磁盘扇区之间搬运数据。写入路径尤其值得一提——它涉及块按需分配（get_or_alloc_block）、读-改-写（read-modify-write）的部分块写入策略、inode 元数据写回等一整套机制。相比只读驱动，这是一个质的飞跃。

我们还会讨论 inode 缓存的设计权衡，以及 Cinux 的方案和 Linux、SerenityOS 的差异。说实话，写路径的代码量是读取路径的两倍还多，但只要搞清楚每一个步骤"为什么这样做"，整条路径其实非常清晰。

## Inode 的块指针数组：文件数据的索引表

每个 ext2 inode 有 15 个块指针（`i_block[0..14]`），它们的作用是把"文件内第 N 个逻辑块"映射到"磁盘上第 M 个物理块"。前 12 个（`i_block[0..11]`）是直接指针——直接指向数据块。第 13 个（`i_block[12]`）是单间接块指针，它指向的块不是数据，而是一个"指针数组"，数组中每个元素指向一个数据块。第 14 个（`i_block[13]`）和第 15 个（`i_block[14]`）分别是双间接和三间接块指针，理论上可以支撑到 16 GB 的文件。

这种设计的核心思想是"小文件快、大文件能"——大多数文件（小于 12 个块，对于 1K 块大小就是 12KB）完全通过直接指针访问，不需要任何额外 I/O。超过 12 块后才开始使用间接块，每多一层间接就多一次磁盘读取。以 1024 字节块大小为例，各级间接块的寻址能力如下：

```
级别            指针数        数据块数        容量
直接 (0-11)     12            12              12 KB
单间接          256           256             256 KB
双间接          256*256       65,536          64 MB
三间接          256^3         16,777,216      16 GB
```

Cinux 当前实现了直接块和单间接块——对我们 4MB 的测试镜像来说完全足够。双间接和三间接在数学上只是单间接的多层嵌套，但每一层都需要额外的 `read_block` 调用来解析指针数组。如果你好奇 Linux 怎么处理这个问题——`fs/ext2/inode.c` 中的 `ext2_get_branch()` 函数用递归式的链式查找处理了全部三级间接，原理和 Cinux 的单间接完全一样，只是多了几层 `read_block`。

## Ext2FileOps 与 Ext2DirOps：虚基类继承，不是函数指针表

在讨论读写路径之前，我们需要先了解 Cinux 的 InodeOps 架构。在 tag 027 中，VFS 定义了 `InodeOps` 作为文件系统操作的抽象基类，包含 `read`、`write`、`readdir`、`create`、`mkdir`、`unlink` 等虚方法。ext2 驱动通过两个子类来实现多态分发：

```cpp
class Ext2FileOps : public InodeOps {
public:
    explicit Ext2FileOps(Ext2& ext2);
    int64_t read(const Inode* inode, uint64_t offset,
                 void* buf, uint64_t count) override;
    int64_t write(Inode* inode, uint64_t offset,
                  const void* buf, uint64_t count) override;
    int64_t stat(const Inode* inode, struct stat* st) override;
private:
    Ext2& ext2_;
};

class Ext2DirOps : public InodeOps {
public:
    explicit Ext2DirOps(Ext2& ext2);
    int64_t readdir(const Inode* inode, uint64_t index,
                    char* name, uint64_t name_max) override;
    Inode*  create(Inode* dir, const char* name,
                   uint32_t namelen) override;
    Inode*  mkdir(Inode* dir, const char* name,
                  uint32_t namelen) override;
    int64_t unlink(Inode* dir, const char* name,
                   uint32_t namelen) override;
private:
    Ext2& ext2_;
};
```

你会发现，`Ext2FileOps` 专注于文件读写——它重写了 `read`、`write` 和 `stat`；`Ext2DirOps` 专注于目录操作——它重写了 `readdir`、`create`、`mkdir`、`unlink` 和 `stat`。两者都持有对 `Ext2` 实例的引用（`ext2_`），通过这个引用访问底层的 `read_block`、`write_block`、`alloc_block` 等方法。不支持的操作自然回退到 `InodeOps` 基类的默认实现（返回 -1 或 nullptr）。

这种通过引用而不是全局指针来访问 Ext2 对象的设计有一个明显的好处：理论上支持同一系统上挂载多个 ext2 实例。虽然 Cinux 当前只有一个 AHCI 端口挂 ext2，但代码架构上没有硬编码的 singleton 限制。这与 Linux 的 `ext2_sb_info`（嵌入在 super_block 中）思路类似——每个挂载实例都有独立的私有数据，不依赖全局状态。SerenityOS 的做法更彻底——`Ext2FS::Inode` 类本身继承自 VFS 的 `Inode` 基类，直接把文件系统操作作为 Inode 的虚方法。Cinux 的方案介于两者之间：使用 C++ 虚基类（比 Linux 的纯 C 函数指针表多了一层类型安全），但 ops 对象和 inode 是分离的（不像 SerenityOS 那样融合在一起）。

## 文件读取的完整路径：从 inode 到数据

`Ext2FileOps::read()` 是文件读取的入口。它的输入是一个 VFS inode（通过 `fs_private` 拿到缓存中的磁盘 inode 副本）、读取偏移量和长度。首先进行边界检查——如果 offset 超出文件大小直接返回 0，然后计算本次实际可读的字节数（`to_read = min(count, i_size - offset)`）。

整个流程的核心是一个 while 循环，每次迭代处理一个块范围内的数据。对于每次迭代，首先计算当前逻辑块号和块内偏移：

```cpp
uint64_t file_block   = (offset + total_read) / bs;
uint64_t block_offset = (offset + total_read) % bs;
uint64_t chunk        = bs - block_offset;
if (chunk > to_read - total_read)
    chunk = to_read - total_read;
```

这三个值决定了"从哪里读、读多少"。`file_block` 是文件内的逻辑块号，`block_offset` 是块内偏移（因为读取不一定从块开头开始），`chunk` 是本次迭代最多能读的字节数（取"块内剩余"和"还未读的总数"的较小值）。

然后根据 `file_block` 的大小选择寻址路径。如果小于 12，直接从 `disk.i_block[file_block]` 拿到物理块号。如果在 12 到 12+block_ptrs_per_block 之间（单间接范围），先读取间接块（`disk.i_block[12]`），再从间接块数组中索引得到物理块号：

```cpp
if (file_block < EXT2_DIRECT_BLOCKS) {
    disk_block = disk.i_block[file_block];
} else if (file_block < EXT2_DIRECT_BLOCKS + block_ptrs_per_block) {
    uint32_t indirect_block = disk.i_block[EXT2_INDIRECT_BLOCK];
    if (indirect_block == 0) break;
    if (!ext2_.read_block(indirect_block)) break;
    uint32_t idx = file_block - EXT2_DIRECT_BLOCKS;
    auto* indirect = reinterpret_cast<uint32_t*>(ext2_.dma_buf_virt());
    disk_block = indirect[idx];
}
```

拿到物理块号后，`read_block()` 把数据读入 DMA buffer，然后从 `dma_buf_virt() + block_offset` 处拷贝 `chunk` 字节到用户 buffer。这里有一个细节值得注意：如果 `disk_block == 0`（文件中的"空洞"，即从未写入过的块），读取不报错而是用循环逐字节填零返回。这是因为 ext2 允许文件有稀疏区域——逻辑上存在但物理上没有分配块的区域。POSIX 规定读取未分配的区域应返回零字节，Cinux 的实现正确地遵循了这个语义。

## 文件写入：从"能读"到"能写"的质变

如果说读取路径是"按图索骥"——按照已有的指针找到数据，那写入路径就是"逢山开路"——没有指针就分配，没有块就申请新的。`Ext2FileOps::write()` 的整体结构和 read 类似，也是 while 循环逐块写入，但每个步骤都多了一层"按需分配"的逻辑。

写入的核心在于 `get_or_alloc_block()` 这个方法。给定一个磁盘 inode 和一个逻辑块号，它的职责是：如果这个块已经有对应的物理块了，直接返回物理块号；如果没有，就分配一个新的物理块，把指针写回 inode，然后返回新块号。

对于直接块（file_block < 12），逻辑很直白——检查 `disk.i_block[file_block]` 是否为零，为零就调 `alloc_block()` 分配一块，清零后写入磁盘，然后把新的物理块号记录在 inode 里：

```cpp
if (disk.i_block[file_block] == 0) {
    uint32_t blk = alloc_block();
    if (blk == 0) return 0;
    // 清零新块
    auto* dma = reinterpret_cast<uint8_t*>(dma_buf_virt_);
    for (uint32_t i = 0; i < block_size_; ++i) dma[i] = 0;
    if (!write_block(blk)) { free_block(blk); return 0; }
    disk.i_block[file_block] = blk;
}
return disk.i_block[file_block];
```

对于单间接块范围，事情稍微复杂一些。首先要确保间接块本身存在——如果 `disk.i_block[12]` 为零，需要先分配一个块作为间接块（里面全是零，代表"所有数据块都还没分配"）。然后读取间接块，检查对应索引处的数据块指针是否为零，为零就分配新的数据块。这里有一个细节容易出错：分配完新数据块后，必须重新读取间接块（因为 `alloc_block` 内部会使用同一个 DMA buffer 读写位图块，间接块的数据已经被覆盖了），然后把新数据块的指针写入间接块，写回磁盘。这个"重新读取"的步骤是使用单 DMA buffer 架构必须付出的代价——每次涉及多层块操作时，中间结果会被覆盖，必须手动恢复。

回到 write 路径本身。拿到目标物理块号后，如果写入不是对齐到整块的（`block_offset != 0 || chunk != bs`），必须先把原始块从磁盘读出来（读-改-写策略），然后在 DMA buffer 中修改对应区域，最后整块写回。如果写入恰好覆盖整个块，就直接在 DMA buffer 中清零后填入新数据，免去了一次读盘：

```cpp
if (block_offset != 0 || chunk != bs) {
    if (!ext2_.read_block(disk_block)) break;  // 读-改-写
} else {
    auto* dma = reinterpret_cast<uint8_t*>(ext2_.dma_buf_virt());
    for (uint32_t i = 0; i < bs; ++i) dma[i] = 0;  // 全块覆盖，无需先读
}
auto* dst = reinterpret_cast<uint8_t*>(ext2_.dma_buf_virt()) + block_offset;
for (uint64_t i = 0; i < chunk; ++i)
    dst[i] = src[total_written + i];
if (!ext2_.write_block(disk_block)) break;
```

你会发现这个分支判断背后的逻辑是：只修改块的一部分时，必须保留未修改部分的数据。这是块设备 I/O 的基本约束——你不能只写扇区中间的几个字节，必须整块写入。Linux 的 buffer_head 子系统把块细分为更小的 buffer（最小 512 字节），每个 buffer 独立跟踪 dirty 状态，减少了不必要的读-改-写操作。Cinux 的单 DMA buffer 方案虽然每次都要整块操作，但在教学内核的场景下足够正确。

写入循环结束后，如果确实写入了数据（`total_written > 0`），还需要更新 inode 的大小字段和块计数，然后调用 `write_disk_inode` 把修改后的 inode 写回磁盘：

```cpp
if (total_written > 0) {
    uint64_t new_end = offset + total_written;
    if (new_end > disk.i_size) {
        disk.i_size = static_cast<uint32_t>(new_end);
        uint32_t sectors_used = ((disk.i_size + bs - 1) / bs) * (bs / 512);
        disk.i_blocks = sectors_used;
    }
    ext2_.write_disk_inode(static_cast<uint32_t>(inode->ino), disk);
    inode->size = disk.i_size;
}
```

`write_disk_inode` 本身也是一个读-改-写过程——先读取包含目标 inode 的块，在 DMA buffer 中覆盖 inode 数据，再把整块写回。这是 Cinux 中最常见的磁盘修改模式：read 原始内容，patch 修改的部分，write 整块回去。

## Inode Cache：避免重复读盘的简单策略

接下来问题来了——文件读写都依赖 inode 数据，而每次从磁盘读取 inode 都需要一次完整的块 I/O（计算块组、定位块、DMA 读取、从 buffer 中提取 128 字节）。如果一个文件被连续读写多次（比如 Shell 反复 `cat` 同一个配置文件），每次都从磁盘读取 inode 显然太浪费了。这就引出了 inode 缓存。

ext2 驱动中有一个固定 64 条目的 inode 缓存——`Ext2CachedInode inode_cache_[64]`。每个缓存条目组合了三样东西：磁盘 inode 的副本（`disk_inode`）、对应的 VFS inode（`vfs_inode`）和 inode 编号（`ino`），外加一个 `in_use` 标志位。

`get_cached_inode(ino)` 的查找策略是先线性扫描缓存查找已存在的条目（命中就返回），找不到就找一个空闲槽位从磁盘读取并填充。如果 64 个槽位全满，就驱逐一个——策略是简单的取模 `evict = 1 + (ino % 63)`，始终保护 slot 0（根 inode，mount 时放入）。这个驱逐策略非常朴素——不考虑 LRU 信息，不考虑引用计数，甚至不考虑这个 slot 是否还在被使用。但在 n=64 的规模下，线性扫描的 O(n) 开销完全可以忽略。

这个缓存的设计和 Linux、SerenityOS 有本质区别。Linux 的 inode 缓存是一个全局的哈希表加上 LRU 链表，缓存的 inode 用 `iget_locked()` 获取、`iput()` 释放，引用计数为零的 inode 在内存压力下可以被回收。Linux 还有独立的 dentry cache（目录项缓存），把"路径名到 inode"的映射关系缓存下来，避免每次路径解析都要读磁盘。SerenityOS 的 Inode 类本身就是一个带引用计数的缓存对象，通过 `BlockBasedFileSystem::get_inode()` 获取时会自动检查缓存，Inode 的生命周期由引用计数管理。

Cinux 的固定 64 条目缓存看起来很简陋，但对教学内核来说是合理的权衡。64 个条目已经能覆盖"打开一些文件、切换几个目录"的日常使用场景。固定数组避免了动态内存分配的复杂性（堆上的链表或哈希表引入了另一个出错的可能）。更重要的是，这个缓存在写入路径中扮演了关键角色——`Ext2FileOps::write()` 通过 `fs_private` 拿到的 `Ext2CachedInode` 中保存着磁盘 inode 的最新副本，写入时直接修改这个副本中的 `i_size` 和 `i_blocks`，然后调用 `write_disk_inode` 一次性写回。如果没有缓存，每次写入都要先从磁盘重新读取 inode，性能开销会非常可观。

## 单 DMA buffer 的代价与 SerenityOS 的 BlockView

我们在前面多次提到"单 DMA buffer"的限制，这里值得展开讲讲它对整体设计的影响。Cinux 的 ext2 驱动只有一个 4KB 的 DMA buffer（`dma_buf_virt_`），所有块 I/O——不管是读数据块、读位图、读间接块、写 inode——都要经过它。这意味着任何一次 `read_block` 或 `write_block` 调用都会覆盖上一次的内容。如果你需要在两个块之间来回操作（比如先读间接块拿到指针，再分配新块时要写位图，然后又要回间接块写入新指针），就必须在每次切换时重新读取。

`get_or_alloc_block()` 中间接块路径的代码完美展示了这个问题。分配间接块本身（`alloc_block`）会使用 DMA buffer 读写位图块，间接块的数据已经不在 buffer 里了，所以分配完后必须重新读取间接块才能写入新的数据块指针。这种"read-alloc-patch-write"的四步操作在单 buffer 架构下是不可避免的。

SerenityOS 用 `BlockView` 类优雅地解决了这个问题。`BlockView` 提供了一个块级别的读写抽象——你可以同时持有多个 `BlockView` 实例，每个对应不同的块，它们内部管理着各自的缓存。底层通过 `BlockBasedFileSystem` 的缓存层自动处理块的加载和卸载，调用者不需要关心 DMA buffer 的争用。Linux 的方案更复杂但也更高效——`buffer_head` 子系统为每个块维护一个独立的缓存对象（包含数据指针、dirty 标志、引用计数），多个 `buffer_head` 可以同时存在于内存中，通过块号作为键进行查找。`get_block()` 在 Linux 中返回的是一个已经缓存好的 buffer，不需要额外的磁盘 I/O。

Cinux 的单 buffer 方案在正确性上没有任何问题——只要每次操作后及时写回，就不会丢数据。但在性能上，频繁的重复读盘是无法避免的代价。对教学内核来说，这完全是一个合理的取舍。如果你以后想改进，最直接的方法是把 DMA buffer 扩展为一个简单的块缓存（用固定大小的数组 + LRU 驱逐），就能在不改变整体架构的前提下大幅减少磁盘 I/O 次数。

## write_disk_inode：元数据写回的读-改-写模式

在写入路径中，`write_disk_inode` 是被调用最频繁的方法之一——每次 `write`、`create`、`mkdir`、`unlink` 最后都会调用它。它和 `read_disk_inode` 的寻址逻辑完全一致（根据 inode 号计算块组、组内索引、目标块号），只是最后一步从"从 DMA buffer 拷贝到输出结构体"变成了"把输入结构体拷贝到 DMA buffer 再写回磁盘"。

具体来说，`write_disk_inode` 先读取包含目标 inode 的磁盘块到 DMA buffer（read），然后在 buffer 中定位到 inode 的偏移处，用 `memcpy` 覆盖 128 字节的 inode 数据（modify），最后把整个块写回磁盘（write）。这种"读-改-写"三步曲是 Cinux 磁盘操作的标准模式——不管是修改 inode、修改位图还是修改块组描述符，都是先读原始块、patch 修改部分、写回整块。原因很简单：磁盘 I/O 的最小单位是块（至少 512 字节扇区，ext2 中是 1024 字节块），你不可能只写 128 字节的 inode 而不碰同一块中的其他数据。这也是为什么一个 1024 字节的块可以容纳 8 个 inode（每个 128 字节），但修改其中任何一个都需要把整个块读出来再写回去。

## 收尾

这一篇我们完成了 ext2 驱动的完整读写路径。读取通过直接块和间接块定位数据，正确处理了稀疏文件的零填充语义。写入通过 `get_or_alloc_block` 按需分配新块，采用读-改-写策略处理部分块写入，更新 inode 元数据后写回磁盘。Ext2FileOps 和 Ext2DirOps 通过 C++ 虚基类继承 InodeOps，通过引用持有 Ext2 对象，避免了全局 singleton。inode 缓存虽然简单（固定 64 条目、取模驱逐），但在读写路径中都发挥了减少磁盘 I/O 的关键作用。我们也坦诚地讨论了单 DMA buffer 架构的代价——以及 Linux 的 buffer_head 和 SerenityOS 的 BlockView 如何用更复杂的缓存层解决这个问题。
