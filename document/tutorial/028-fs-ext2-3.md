# ext2 文件读取与间接块

> 标签：ext2、文件读取、间接块、Inode Cache、InodeOps
> 前置：[028_fs_ext2 第 2 篇](028-fs-ext2-2.md)

## 前言

前两篇我们把 ext2 驱动的骨架搭好了——Superblock 解析、BGDT 加载、inode 寻址、DMA buffer 就位。骨架有了，该让它"动"起来了。

这一篇我们要实现的是 ext2 驱动中最有实质性意义的功能：文件的读取和目录的遍历。从 inode 的块指针数组出发，经过直接块和间接块的寻址逻辑，最终在用户空间的 buffer 和磁盘扇区之间搬运数据。我们还会讨论 inode 缓存的设计权衡，以及 Cinux 的方案和 Linux、SerenityOS 的差异。

说实话，tag 028 是一个只读驱动——写操作被设计为返回 -1 的桩函数，真正的写入要等到 tag 028b。但只要搞清楚读取路径中每一个步骤"为什么这样做"，整条路径其实非常清晰。

## Inode 的块指针数组：文件数据的索引表

每个 ext2 inode 有 15 个块指针（`i_block[0..14]`），它们的作用是把"文件内第 N 个逻辑块"映射到"磁盘上第 M 个物理块"。前 12 个（`i_block[0..11]`）是直接指针——直接指向数据块。第 13 个（`i_block[12]`）是单间接块指针，它指向的块不是数据，而是一个"指针数组"，数组中每个元素指向一个数据块。第 14 个（`i_block[13]`）和第 15 个（`i_block[14]`）分别是双间接和三间接块指针，理论上可以支撑到 16 GB 的文件。

这种设计的核心思想是"小文件快、大文件能"——大多数文件（小于 12 个块，对于 1K 块大小就是 12KB）完全通过直接指针访问，不需要任何额外 I/O。超过 12 块后才开始使用间接块，每多一层间接就多一次磁盘读取。以 1024 字节块大小为例，各级间接块的寻址能力如下：

```
级别            指针数        数据块数        容量
直接 (0-11)     12            12              12 KB
单间接          256           256             256 KB
双间接          256*256       65,536          64 MB
三间接          256^3         16,777,216      16 GB
```

Cinux 当前实现了直接块和单间接块——对我们 4MB 的测试镜像来说完全足够。双间接和三间接在数学上只是单间接的多层嵌套，但每一层都需要额外的 `read_block` 调用来解析指针数组。如果你好奇 Linux 怎么处理这个问题——`fs/ext2/inode.c` 中的 `ext2_get_branch()` 函数用递归式的链式查找处理了全部三级间接，原理和 Cinux 的单间接完全一样，只是多了几层 `read_block`。

## InodeOps：静态函数指针表 + 全局实例指针

在讨论读取路径之前，我们需要先了解 tag 028 的 InodeOps 架构。Cinux 的 VFS 定义了 `InodeOps` 为一个包含三个函数指针（read、write、readdir）的 POD 结构体，不是虚基类。ext2 驱动用两个静态实例分别处理文件和目录：

```cpp
namespace {

int64_t ext2_file_read(const Inode* inode, uint64_t offset,
                       void* buf, uint64_t count);
int64_t ext2_file_write(Inode*, uint64_t, const void*, uint64_t) {
    return -1;  // tag 028: read-only driver
}
int64_t ext2_dir_readdir(const Inode* inode, uint64_t index,
                         char* name, uint64_t name_max);

InodeOps ext2_file_ops = {
    ext2_file_read,
    ext2_file_write,
    nullptr,  // no readdir for regular files
};

InodeOps ext2_dir_ops = {
    nullptr,           // no read on directories
    ext2_file_write,   // still returns -1
    ext2_dir_readdir,
};

static Ext2* g_ext2_instance = nullptr;

}  // anonymous namespace
```

你会发现，这种设计不使用继承，不使用虚函数——就是纯粹的 C 风格函数指针表。回调函数通过全局指针 `g_ext2_instance` 访问 Ext2 对象的 `read_block()`、`block_size()`、`dma_buf_virt()` 等公共方法。在 Ext2 构造函数中，`g_ext2_instance = this` 把当前实例注册为活跃的 ext2 对象。

`ext2_file_write` 在 tag 028 中只返回 -1。这是一个只读驱动——不支持创建、写入、删除。这些操作将在 tag 028b 中实现，届时 `InodeOps` 会被扩展为虚基类，消除全局指针的限制。

## 文件读取的完整路径：从 inode 到数据

`ext2_file_read` 是文件读取的入口。它是一个静态函数（不是类方法），通过 `g_ext2_instance` 访问 Ext2 对象。输入是一个 VFS inode（通过 `fs_private` 拿到缓存中的磁盘 inode 副本）、读取偏移量和长度。

整个流程的核心是一个 while 循环，每次迭代处理一个块范围内的数据。对于每次迭代，首先计算当前逻辑块号和块内偏移：

```cpp
uint64_t file_block   = (offset + total_read) / bs;
uint64_t block_offset = (offset + total_read) % bs;
uint64_t chunk        = bs - block_offset;
if (chunk > to_read - total_read)
    chunk = to_read - total_read;
```

这三个值决定了"从哪里读、读多少"。`file_block` 是文件内的逻辑块号，`block_offset` 是块内偏移，`chunk` 是本次迭代最多能读的字节数。

然后根据 `file_block` 的大小选择寻址路径。小于 12 走直接块，12 到 12+256 走间接块：

```cpp
if (file_block < EXT2_DIRECT_BLOCKS) {
    disk_block = disk.i_block[file_block];
} else if (file_block < EXT2_DIRECT_BLOCKS + block_ptrs_per_block) {
    uint32_t indirect_block = disk.i_block[EXT2_INDIRECT_BLOCK];
    if (indirect_block == 0) break;
    if (!ext2->read_block(indirect_block)) break;
    uint32_t idx = file_block - EXT2_DIRECT_BLOCKS;
    auto* indirect = reinterpret_cast<uint32_t*>(ext2->dma_buf_virt());
    disk_block = indirect[idx];
}
```

拿到物理块号后，`read_block()` 把数据读入 DMA buffer，然后从 `dma_buf_virt() + block_offset` 处拷贝 `chunk` 字节到用户 buffer。这里有一个细节值得注意：如果 `disk_block == 0`（文件中的"空洞"，即从未写入过的块），读取不报错而是用循环逐字节填零返回。这是因为 ext2 允许文件有稀疏区域——逻辑上存在但物理上没有分配块的区域。POSIX 规定读取未分配的区域应返回零字节，Cinux 的实现正确地遵循了这个语义。

间接块读取有一个微妙之处：第一次 `read_block` 把间接块读进 DMA buffer，我们从中取出目标块号存到 `disk_block` 局部变量；第二次 `read_block` 读数据块时会覆盖 DMA buffer 中的间接块数据——但没关系，因为 `disk_block` 已经提取出来了。这种"提取到局部变量"的模式在单 DMA buffer 架构下是标准做法。

## Inode Cache：避免重复读盘的简单策略

文件读取依赖 inode 数据，而每次从磁盘读取 inode 都需要一次完整的块 I/O。如果一个文件被连续读取多次（比如 Shell 反复 `cat` 同一个配置文件），每次都从磁盘读取 inode 显然太浪费了。这就引出了 inode 缓存。

ext2 驱动中有一个固定 64 条目的 inode 缓存——`Ext2CachedInode inode_cache_[64]`。每个缓存条目组合了三样东西：磁盘 inode 的副本（`disk_inode`）、对应的 VFS inode（`vfs_inode`）和 inode 编号（`ino`），外加一个 `in_use` 标志位。

`get_cached_inode(ino)` 的查找策略是先线性扫描缓存查找已存在的条目（命中就返回），找不到就找一个空闲槽位从磁盘读取并填充。如果 64 个槽位全满，就驱逐一个——策略是简单的取模 `evict = 1 + (ino % 63)`，始终保护 slot 0（根 inode，mount 时放入）。

这个缓存的设计和 Linux、SerenityOS 有本质区别。Linux 的 inode 缓存是一个全局的哈希表加上 LRU 链表，缓存的 inode 用 `iget_locked()` 获取、`iput()` 释放，引用计数为零的 inode 在内存压力下可以被回收。Linux 还有独立的 dentry cache（目录项缓存），把"路径名到 inode"的映射关系缓存下来，避免每次路径解析都要读磁盘。SerenityOS 的 Inode 类本身就是一个带引用计数的缓存对象，通过 `BlockBasedFileSystem::get_inode()` 获取时会自动检查缓存。

Cinux 的固定 64 条目缓存看起来很简陋，但对教学内核来说是合理的权衡。64 个条目已经能覆盖"打开一些文件、切换几个目录"的日常使用场景。固定数组避免了动态内存分配的复杂性。

## 单 DMA buffer 的代价与 SerenityOS 的 BlockView

我们在前面多次提到"单 DMA buffer"的限制，这里值得展开讲讲它对整体设计的影响。Cinux 的 ext2 驱动只有一个 4KB 的 DMA buffer（`dma_buf_virt_`），所有块 I/O 都要经过它。这意味着任何一次 `read_block` 都会覆盖上一次的内容。

间接块读取是这个问题最典型的例子。我们需要先读间接块拿指针，再读数据块拿内容——两次 `read_block` 之间，间接块的数据已经不在 buffer 里了。好在目标块号已经提取到局部变量，后续操作不再依赖间接块数据。但如果操作更复杂（比如 tag 028b 的写入路径需要同时操作位图块、间接块、数据块），单 buffer 的限制就会变得更加明显——每次切换都必须重新读取。

SerenityOS 用 `BlockView` 类优雅地解决了这个问题。`BlockView` 提供了一个块级别的读写抽象——你可以同时持有多个 `BlockView` 实例，每个对应不同的块，它们内部管理着各自的缓存。Linux 的 `buffer_head` 子系统更复杂但也更高效——为每个块维护独立的缓存对象。

Cinux 的单 buffer 方案在正确性上没有任何问题——只要每次操作后及时处理数据，就不会丢数据。但在性能上，频繁的重复读盘是无法避免的代价。对教学内核来说，这完全是一个合理的取舍。

## 收尾

这一篇我们完成了 ext2 驱动的完整读取路径。文件读取通过直接块和间接块定位数据，正确处理了稀疏文件的零填充语义。InodeOps 通过静态函数指针表实现回调分发，通过全局指针 `g_ext2_instance` 访问 Ext2 对象。inode 缓存虽然简单（固定 64 条目、取模驱逐），但在读取路径中发挥了减少磁盘 I/O 的关键作用。我们也坦诚地讨论了单 DMA buffer 架构的代价——以及 Linux 的 buffer_head 和 SerenityOS 的 BlockView 如何用更复杂的缓存层解决这个问题。
