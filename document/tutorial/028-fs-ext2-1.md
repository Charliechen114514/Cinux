# 028-1 Tutorial: 从 VFS 到 ext2——挂载真实磁盘文件系统

> 标签：ext2、文件系统、AHCI、磁盘布局、Superblock
> 前置：[027_fs_vfs](027-fs-vfs-1.md)

## 前言

写完 027 tag 的 VFS 之后，我们手上有了一套看起来还算像样的文件系统框架——

FileSystem 抽象基类、Inode/InodeOps 接口、挂载表、路径解析。

但说实话，这套东西跑的其实只有一个后端：Ramdisk。

而 Ramdisk 是什么？说白了就是把一块内存区域假装成文件系统。

断电即消失，写入无处安放，容量受限于内核启动时加载的那点数据。

这在教学阶段够用，但如果我们想让操作系统真正"可用"——

能持久存储文件、能从磁盘启动用户程序——

就必须对接一个真正的磁盘文件系统。

这不是什么可有可无的锦上添花，而是从"能跑的 demo"到"可用的 OS"的分水岭。

想想看，一个操作系统的文件系统如果断电就没了，那它和一段普通的演示程序有什么本质区别？

接下来问题来了：选哪个文件系统？

FAT32 简单但结构丑陋（FAT 表本质上就是一个巨大的数组），

NTFS 闭源且复杂到让人绝望，

ZFS/Btrfs 对一个教学内核来说完全是 overkill。

经过反复权衡，我们选择了 ext2——The Second Extended Filesystem。

这不是一个随意的选择，ext2 在操作系统开发社区中被公认为

"第一个真正的文件系统"的最佳学习对象。

它的数据结构清晰、文档完善

（OSDev Wiki 上的 ext2 条目是少数能直接当 spec 用的页面），

而且 Linux、SerenityOS 等知名项目都有完整的 ext2 实现可供参考对比。

ext2 由 Rémy Card 等人在 1993 年设计，

作为 ext（Extended Filesystem）的继任者。

虽然如今 Linux 桌面和服务器已经全面转向 ext4，

但 ext2 的核心数据结构与 ext3/ext4 几乎完全相同——

区别主要在于 ext3 加了日志（journal），ext4 加了 extent 和更大的块支持。

所以学会了 ext2，理解 ext3/ext4 也就是多几层概念的事情。

## 环境说明

这次的起点是 027_fs_vfs tag，在它的基础上我们新增了三个核心文件：

- `kernel/fs/ext2_types.hpp`（磁盘数据结构定义，约 310 行）
- `kernel/fs/ext2.hpp`（驱动类声明，约 476 行）
- `kernel/fs/ext2.cpp`（驱动实现，约 1810 行）

工具链不变——GCC 14 + CMake + QEMU，

但这次我们需要第二块磁盘：

一块 4MB 的 ext2 镜像挂载在 AHCI port 1 上。

创建镜像的脚本在 `scripts/create_ext2_disk.sh`，

使用了 mkfs.ext2 和 debugfs 两个来自 e2fsprogs 的工具。

如果你用的是 Debian/Ubuntu，`sudo apt install e2fsprogs` 就能装上。

如果你用的是 Arch Linux，这些工具默认就在 `e2fsprogs` 包里。

## 第一步——理解 ext2 的磁盘布局：块、块组与层级结构

在写一行代码之前，我们必须先搞清楚 ext2 是怎么把数据组织在磁盘上的。

ext2 的基本 I/O 单位是"块"（block），

大小可以是 1024、2048 或 4096 字节——

具体值由 Superblock 中的 `s_log_block_size` 字段决定，

计算公式是 `block_size = 1024 << s_log_block_size`。

这个设计比直接存一个整数稍微绕了一点，

但好处是 0 代表 1024、1 代表 2048、2 代表 4096，

用移位就能算出来，CPU 友好。

你可能会问为什么不直接存块大小的值？

原因在于 ext2 设计时考虑的是"最少存储空间表达最大信息量"——

一个 32 位整数如果直接存 1024/2048/4096 太浪费了，

而用指数表示只需要 2 个 bit 就能区分所有合法的块大小。

ext2 把整个磁盘空间划分成若干"块组"（Block Group），

每个块组包含自己的数据块位图（block bitmap）、

inode 位图（inode bitmap）、

inode 表（inode table）

和数据块区域。

这种设计不是随便画的——

它的核心目的是把"属于同一个文件的元数据和数据"在物理上放在一起，

减少磁盘寻道时间。

在机械硬盘时代这个优化非常重要，即使在 SSD 时代也依然有益于局部性。

每个块组内部的结构顺序是固定的：

首先是 block bitmap（占一个块），

然后是 inode bitmap（占一个块），

然后是 inode table（占若干块，具体取决于每组有多少 inode），

最后是数据块区域。

磁盘的第一个 1024 字节是引导块（boot block），

从字节 1024 开始是 Superblock。

对于 1024 字节块大小的情况，Superblock 位于 block 1

（block 0 是引导块）；

对于更大的块大小，Superblock 起始位置仍然是字节 1024

但它位于 block 0 的后半部分。

紧跟在 Superblock 之后的是块组描述符表

（Block Group Descriptor Table, BGDT），

每个条目 32 字节，描述一个块组的元数据位置。

这个布局可以用一张简单的图来表示：

```
Disk layout (1024-byte blocks):
  Block 0:  Boot block (1024 bytes, unused by ext2)
  Block 1:  Superblock (1024 bytes)
  Block 2+: Block Group Descriptor Table
  Block N:  Block Group 0: [Block Bitmap][Inode Bitmap][Inode Table][Data Blocks]
  Block M:  Block Group 1: [Block Bitmap][Inode Bitmap][Inode Table][Data Blocks]
  ...
```

你会发现，每个块组都有自己独立的 bitmap 和 inode table。

这意味着当你在某个目录下创建文件时，

ext2 可以把这个文件的 inode 和数据块都分配在同一个块组内，

大大减少了磁头的移动距离。

## 第二步——设计对比：Linux、SerenityOS 与 Cinux 的不同路线

理解了 ext2 的磁盘布局之后，

我们来看看其他操作系统是怎么实现 ext2 驱动的，

这对理解我们自己的设计选择非常有帮助。

### Linux 内核的 ext2 实现

Linux 内核的 ext2 实现（`fs/ext2/`）是生产级的，

代码分散在十几个文件里——

super.c 处理挂载和超级块，inode.c 处理 inode 读写，

balloc.c 和 ialloc.c 分别负责块分配和 inode 分配，

dir.c 处理目录操作。

Linux 使用 `buffer_head` 子系统来管理磁盘块的缓存和 I/O 调度——

每次读取一个块时，buffer_head 层会先检查页缓存（page cache），

命中就直接返回，未命中才发起真正的磁盘 I/O。

这意味着在 Linux 中连续两次读取同一个块，第二次几乎是免费的。

Linux 的 ext2 还实现了 Orlov 块分配器

（在 `fs/ext2/ialloc.c` 中），

它的目标是在创建新文件时智能地选择块组，

使得相关文件（比如同一个目录下的文件）尽量放在同一个块组里，

减少碎片化。

此外 Linux 完整支持三级间接块

（最大文件可达 2TB+），

使用 per-CPU 计数器来跟踪空闲块/inode 数量以避免锁竞争，

还实现了文件预读（readahead）来优化顺序读性能。

### SerenityOS 的 ext2 实现

SerenityOS 的 ext2 实现

（`Kernel/FileSystem/Ext2FS/`）

走的是 C++ 面向对象路线，和 Cinux 在架构上最为相似。

它定义了一个 `BlockBasedFileSystem` 基类

来抽象"以块为单位进行 I/O"的文件系统，

然后 `Ext2FS::FileSystem` 继承自它。

SerenityOS 最有意思的设计是 `BlockView`——

一个 RAII 风格的块访问封装，

使用时就像打开一个智能指针，用完自动释放回缓存。

这种设计把"获取块→使用块→释放块"的生命周期管理

变成了类型系统的一部分，非常优雅。

SerenityOS 还直接复用了 Linux 的 `ext2_fs.h` 磁盘定义

（通过 copy），

并且支持多种文件系统后端——

Ext2、FAT、ISO9660、RAMFS、ProcFS、SysFS 等——

都通过统一的 VFS 层暴露给用户态。

不过 SerenityOS 的 ext2 实现比 Cinux 复杂得多，

光是 Inode 类就有几百行代码，处理了符号链接、扩展属性等特性。

### Cinux 的定位

相比之下，Cinux 的 ext2 驱动介于 xv6 和 SerenityOS 之间——

架构上采用了类似 SerenityOS 的 C++ 接口继承

（Ext2 继承 FileSystem，Ext2FileOps/Ext2DirOps 继承 InodeOps），

但 I/O 层面更接近 xv6 的极简风格：

一个物理页的 DMA buffer，每次只读一个块，没有块缓存。

这意味着频繁访问同一个块会导致反复的磁盘 I/O，

性能代价很大，但代码复杂度大大降低——

对教学内核来说这是一个合理的取舍。

## 第三步——创建 ext2 磁盘镜像

理解了设计之后，我们先把测试用的磁盘镜像准备好。

`scripts/create_ext2_disk.sh` 做的事情很简单但很精巧——

用 dd 创建一个 4MB 的零填充镜像，

用 mkfs.ext2 格式化为 ext2

（指定 1024 字节块大小、不启用任何扩展特性、128 个 inode），

然后用 debugfs 往里写入测试文件。

```bash
IMAGE_SIZE=4
BLOCK_SIZE=1024
dd if=/dev/zero of="$OUTPUT" bs=1M count="$IMAGE_SIZE" 2>/dev/null
mkfs.ext2 -b "$BLOCK_SIZE" -O none -N 128 "$OUTPUT" >/dev/null 2>&1
```

debugfs 是 e2fsprogs 自带的调试工具，

它可以在不挂载文件系统的情况下直接操作 ext2 镜像的内部结构——

创建目录、写入文件、修改 inode，非常适合在构建脚本中使用。

脚本创建的目录结构包括 `/etc/motd`（欢迎信息）和 `/hello.txt`（测试文件），

如果提供了 shell ELF 路径还会创建 `/bin/sh`。

这里有两个细节值得注意。

第一，`-O none` 参数禁用了所有 ext2 扩展特性

（稀疏超级块、文件类型等），

让镜像结构尽可能简单。

第二，`-N 128` 指定只创建 128 个 inode，

对于 4MB 镜像来说绰绰有余而且节省空间。

创建完镜像之后，你可以用 `dumpe2fs ext2.img` 来查看文件系统的详细信息——

块大小、inode 数量、块组布局一目了然。

这个工具在调试 ext2 驱动的时候也非常有用，

当你怀疑某个数据结构读错了，可以先用 dumpe2fs 确认磁盘上的正确值。

## 第四步——QEMU 配置：第二块 AHCI 磁盘

镜像创建好了，接下来要把它挂到 QEMU 的 AHCI 控制器的第二个端口上。

在 QEMU 启动参数中，我们添加一块新磁盘并连接到 AHCI 的 port 1

（port 0 留给了 ramdisk 或者启动盘）。

具体配置取决于你的 QEMU 启动脚本，

核心是在 `-drive` 参数中指定 `file=ext2.img`

并且确保它连接到 AHCI 控制器的第二个端口。

这样在内核中，AHCI 驱动枚举到两个端口后，

我们用 `Ext2 ext2(ahci, 1)` 把 ext2 驱动绑定到 port 1，

然后调用 `ext2.mount()` 就能开始读取文件系统元数据了。

这块 ext2 磁盘在 Cinux 中是一个独立于 ramdisk 的存储设备。

挂载成功后，我们通过 `vfs_mount_add("/", &ext2)`

把 ext2 根目录挂载到 VFS 的某个挂载点上，

之后所有的文件操作就会透明地走到 ext2 驱动里。

如果你回头看 027 tag 的 VFS 设计，

会发现这套机制正好被用上了——

FileSystem 抽象基类的 `mount()` 和 `lookup()`

就是为这种场景设计的。

VFS 层存在的意义，现在终于体现出来了：

上层代码（syscall、shell）不需要知道底层是 Ramdisk 还是 ext2，

它只要调用 `vfs_lookup(path)` 就行。

## 收尾

这一篇我们完成了从"为什么要做 ext2"到"怎么准备测试环境"的全部铺垫。

磁盘镜像有了，QEMU 配置好了，ext2 的磁盘布局也理清楚了。

我们也对比了 Linux、SerenityOS 和 Cinux 三种不同的 ext2 实现路线，

理解了为什么 Cinux 选择"简单但正确"的设计策略。

下一篇我们就要真正深入代码——

从 AHCI 读取 Superblock、验证魔数、计算块大小、

加载 BGDT、读取根 inode。

这些都是 ext2 驱动的核心初始化流程，

每一步都有数学推导和陷阱等着我们。
