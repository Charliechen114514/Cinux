# 从零理解 USTAR——给内核一个文件系统

> 标签：ramdisk, initrd, ustar, tar, 文件系统, 八进制编码
> 前置：[025_driver_ahci](025-driver-ahci-1.md)

## 前言

说到这里，我们的内核已经干了不少事情了——串口输出、GDT/IDT、物理和虚拟内存管理
、进程调度、用户态切换、系统调用、一个能用的 shell，还有上一章刚搞定的 AHCI 
磁盘驱动。但有一个非常基本的能力我们还一直缺失：内核从来不认识什么叫"文件"。

AHCI 让我们能读写磁盘扇区，但扇区只是 512 
字节的原始数据块，上面没有文件名、没有目录结构、没有任何"文件"的概念。要想让内
核理解文件，我们需要一个文件系统。而文件系统的第一步，是先把一些文件弄到内核能
碰到的地方——这就是 initrd（initial ramdisk）的意义。

这一章我们要做的事情说起来很简单：定义一个 C++ 结构体精确映射 POSIX ustar 
归档的头部格式，然后写一个函数把八进制 ASCII 
字符串转回整数。但在这个"简单"的背后，涉及到格式规范的理解、内存布局的精确控制
、以及编译器行为的深入认知——这些都是 OS 开发中反复出现的基本功。

## 环境说明

我们现在的开发环境：x86_64 架构，GCC 14 工具链，CMake 构建系统，内核运行在 
QEMU 中。内核编译时禁用了标准库和异常（`-ffreestanding 
-fno-exceptions`），所有基础设施自给自足。C++ 内核代码使用现代 C++ 
特性（`constexpr`、类内初始化、命名空间常量），但受限于 no-STL 环境，不能使用 
`std::string`、`std::vector` 等容器。

## ustar 格式——一个来自 1970 年代的优雅设计

先来回答一个你可能想问的问题：为什么选 tar 格式？答案很简单——tar（确切地说是 
POSIX ustar 
变体）是所有归档格式中解析逻辑最简单的。没有压缩算法、没有索引表、没有复杂的状
态机。整个归档就是一个线性的"头部-数据-头部-数据-..."序列，从头到尾扫描一遍就
完事了。对于教学 OS 来说，这种"不用动脑就能解析"的格式就是最优解。

ustar 格式的核心单位是 512 
字节的"块"。每个文件在归档中占据至少一个块（头部），如果文件有内容，紧接着还有
若干个块存储文件数据（不足 512 
字节的部分用零填充）。归档末尾用两个全零块标记结束。

头部有 512 
字节，信息非常丰富，但对当前阶段来说我们真正关心的只有三个字段：name（文件路径
，偏移 0，100 字节）、size（文件大小，偏移 124，12 字节）和 
typeflag（类型标志，偏移 156，1 字节）。另外一个关键的验证字段是 magic（偏移 
257，6 字节），合法的 ustar 头部这里一定是 `"ustar\0"`。

现在我们来看 Cinux 中 `UstarHeader` 结构体的定义。整个结构体用 
`[[gnu::packed]]` 
属性标记，这个属性告诉编译器"不要在字段之间插入任何填充字节"。

```cpp
struct [[gnu::packed]] UstarHeader {
    char    name[100];     // 偏移 0:   文件路径
    char    mode[8];       // 偏移 100: 文件权限（八进制）
    char    uid[8];        // 偏移 108: 用户 ID（八进制）
    char    gid[8];        // 偏移 116: 组 ID（八进制）
    char    size[12];      // 偏移 124: 文件大小（八进制）
    char    mtime[12];     // 偏移 136: 修改时间（八进制）
    char    checksum[8];   // 偏移 148: 头部校验和
    char    typeflag;      // 偏移 156: 类型标志
    char    linkname[100]; // 偏移 157: 链接目标
    char    magic[6];      // 偏移 257: "ustar\0"
    char    version[2];    // 偏移 263: "00"
    char    uname[32];     // 偏移 265: 用户名
    char    gname[32];     // 偏移 297: 组名
    char    devmajor[8];   // 偏移 329: 设备主号
    char    devminor[8];   // 偏移 337: 设备次号
    char    prefix[155];   // 偏移 345: 路径前缀
    char    padding[12];   // 偏移 500: 保留/填充
};
```

你可能会想：为什么不用更简洁的 `memcmp` 或者某种自动偏移计算？原因在于，OS 
开发中"直接把内存指针重解释为结构体"是一种非常常见的模式——我们把归档数据的起始
地址加上偏移量，直接 cast 成 `UstarHeader*`，然后就能用 
`hdr->name`、`hdr->size` 
这样的语法访问字段。这要求结构体的内存布局和磁盘上的字节排列完全一致，一个字节
的偏差都会导致数据错乱。

紧跟在结构体后面的这行是整个定义中最重要的安全检查：

```cpp
static_assert(sizeof(UstarHeader) == 512, "UstarHeader must be exactly 512 
bytes");
```

`static_assert` 是编译期断言。如果结构体大小不是 512 
字节，编译直接失败，你立刻知道哪里出了问题。没有这个检查的话，如果你不小心在某
个字段后面加了一个额外字节，编译能通过，运行时数据全部错位，你会花半天时间排查
一个莫名其妙的问题。

## 八进制 ASCII——来自磁带时代的遗产

ustar 头部有一个在今天看来很"复古"的特性：所有数字字段都用八进制 ASCII 
表示。比如一个 100 字节的文件，size 字段存的是字符串 `"00000000144\0"`——因为 
144 的八进制等于 100 的十进制。每个字符都是 '0' 到 '7' 的可打印 ASCII 码。

这种设计的根源要追溯到 1970 年代的磁带时代。不同的 Unix 
系统使用不同的整数表示法（大端序/小端序、不同的字长），但 ASCII 
字符在所有系统上都是一样的。用 ASCII 
存储数字就自动解决了跨平台兼容性问题——虽然牺牲了一些空间和解析效率，但在那个年
代这是正确的权衡。

我们需要一个函数把这些八进制字符串转回正常的整数，实现如下：

```cpp
uint64_t octal_to_uint(const char* s, size_t len) {
    uint64_t result = 0;
    for (size_t i = 0; i < len; ++i) {
        char c = s[i];
        if (c == '\0' || c == ' ') {
            break;
        }
        result = (result << 3) + static_cast<uint64_t>(c - '0');
    }
    return result;
}
```

这里的逻辑很直观：从左到右逐字符读取，遇到 null 
或空格就停。每个字符的数字值是 `c - '0'`（ASCII 
码的差值恰好等于数字值），然后 `result = (result << 3) + digit`，左移 3 
位等于乘以 8。比如解析 `"144"`：先是 result=1，然后 result=1*8+4=12，最后 
result=12*8+4=100。

这里有一个值得注意的边界情况处理：ustar 规范说数字字段可以以 null 
或空格终止。有些 tar 工具用 
null，有些用空格，有些两者混合。我们的代码同时处理了这两种情况，对于 "全零" 
字段（`"00000000000"`），解析结果就是 0，这也是正确的。

## 其他 OS 怎么做？

我们来看看其他操作系统是怎么处理 initrd 的，理解不同设计决策背后的原因。

**xv6**（MIT 的教学 OS）采用了一种完全不同的方式。xv6 的 ramdisk 
不是一个文件解析器，而是一个**块设备模拟器**——它把 QEMU 
加载的整个文件系统镜像（fs.img）当作一块内存，通过 `ramdiskrw()` 
函数按块（1024 字节）读写。xv6 
有自己的文件系统实现（`fs.c`），在块设备之上提供文件级别的操作。

这种分层设计的好处是灵活性——ramdisk 
驱动不需要知道上面跑的是什么文件系统，它只管搬运数据块。坏处是你需要先实现一整
套文件系统才能读取里面的文件。对于 xv6 来说这不是问题，因为文件系统是 xv6 
教学的核心内容之一。

Cinux 的做法刚好相反——我们直接在 ramdisk 驱动里解析 ustar 格式，在 `mount()` 
时就把文件名列出来了。这种"扁平化"的设计牺牲了块设备抽象，换来的是极低的上手门
槛：只要格式定义和解析逻辑对了，马上就能看到结果。

**Linux** 的做法更为复杂。Linux 有两代 initrd 方案：早期的 initrd 
是一个完整的文件系统镜像（ext2 格式），被 bootloader 
加载到内存后作为块设备挂载；现代的 initramfs 是一个压缩的 cpio 
归档，内核直接解压到 ramfs 中，不需要块设备。Linux 用 initrd 
解决的核心问题是"鸡生蛋蛋生鸡"——内核需要磁盘驱动才能挂载根文件系统，但磁盘驱动
本身就在根文件系统的模块目录里。initrd 
提供了一个临时的根环境，在里面加载必要的驱动后再切换到真正的根。

Cinux 现在的阶段还不需要考虑这些问题——我们的 initrd 
是直接编译进内核的，不存在"先有驱动还是先有文件"的困境。但随着项目推进到真正的
文件系统（tag 027 VFS）和磁盘分区读取，这些设计决策会变得越来越重要。

## 收尾

到这里，ustar 格式的"图纸"就画完了。我们定义了精确映射磁盘布局的 packed 
结构体，声明了类型标志常量和 magic 
字符串，实现了八进制到整数的转换函数，还加了编译期断言确保一切正确。这些定义看
起来只是几个 struct 和常量，但它们是后续所有 ramdisk 
逻辑的基础——如果格式定义有错，后面的 mount、遍历、跳转全都会出问题。

下一篇我们要做的是把这个 ustar 
归档"塞进"内核二进制镜像里。这涉及到一个有趣的工具链技巧：用 objcopy 把 tar 
文件转成 ELF 目标文件，然后链接进内核。

（atime、ctime、xattr 等）。GNU tar 则有自己的长文件名和卷标扩展。不过对于教学
OS 来说，基础 ustar 格式已经完全够用——我们在 Cinux 中不打算支持这些扩展。

最后提一句：如果你想在自己的 OS 项目中复用这套格式定义，只需要把 ramdisk_config.hpp
拷贝过去即可——它是完全独立的，不依赖任何内核特定代码。结合 OSDev Wiki 上的 tar
解析教程，你应该能在几十行代码内实现一个最小化的 initrd 解析器。

值得注意的是，Cinux 选择把所有数字字段都声明为 char 数组而不是直接用整数类型，这
不是因为偷懒，而是刻意为之。ustar 规范中这些字段是八进制 ASCII 字符串，用 char
数组能最直观地反映实际的数据格式。如果用 uint32_t 或 uint64_t 来"直接映射"这些
字段，不仅会遇到字节序问题（ustar 规范不定义二进制整数的字节序），还会遇到对齐
问题（packed struct 中非 char 类型可能不在自然对齐的地址上）。char 数组加辅助转换
函数是最安全的做法。
