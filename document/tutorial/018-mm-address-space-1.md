# 从一张页表到无数张：为什么进程需要独立的地址空间

> 标签：地址空间, PML4, 进程隔离, Higher Half, CR3
> 前置：[017-1 从页到字节：为什么内核需要堆分配器](017-mm-heap-1.md)

## 前言

到目前为止，Cinux 的内存管理已经搭好了三层栈——PMM 管物理页，VMM 管虚拟映射，
Heap 管任意大小的动态分配。听起来像是一切就绪了对吧？但你仔细想想会发现一个很
严重的问题：整个内核从头到尾只用了一张页表。CR3 寄存器在启动时被写入一次之后
就没再变过，所有代码、所有数据、所有动态分配出来的内存，全部共享同一套
PML4 -> PDPT -> PD -> PT 的四级页表结构。

这种"全世界只有一张页表"的设计在单体内核阶段是完全合理的——反正就一个执行流
在跑，没有谁跟谁隔离的问题。可一旦我们想要"进程"这个概念，事情就完全不一样了。
进程的核心属性是什么？是隔离。进程 A 的内存不能被进程 B 看到，进程 B 的栈不能
被进程 A 踩到。如果两个进程共享同一张页表，进程 A 在虚拟地址 `0x400000` 写了个
数据，进程 B 在同一个地址也写了数据，它们就直接撞车了——因为在同一套页表里，
一个虚拟地址只能映射到一个物理页。

说实话，这个问题在操作系统课程里被讲过无数次了，每个教科书都会画那张经典的
"每个进程有自己的页表"的示意图。但当你真正坐下来写代码的时候，你会发现从"知道
概念"到"把它变成一组可编译、可测试的数据结构"之间有一大段距离。PML4 的 512 个
条目里哪些该复制、哪些该清零？析构的时候页表子树怎么递归释放才能不漏不重？
两个 `AddressSpace` 能不能共享同一个物理 PML4（答案是不能，但为什么）？
这些才是真正需要动手解决的问题。

tag 018 的目标就是在 VMM 之上再抽象一层——引入 `AddressSpace` 类，让每个进程
拥有自己独立的页表根（PML4），用户空间的映射互不可见，而内核空间的映射又保持
一致。整个 tag 涉及 10 个文件的变更，新增约 1470 行代码，核心是
`address_space.hpp`（144 行）和 `address_space.cpp`（198 行），另外还有
308 行 QEMU 集成测试和 771 行 Host 单元测试。

这一章我们先讨论"为什么需要独立地址空间"，把问题本身看清楚，然后研究 PML4 的
分割策略如何天然地适配了内核/用户的隔离需求，再对比 xv6 和 Linux 的经典设计，
最后看 Cinux 的 `AddressSpace` 类是如何被设计出来的。

## 环境说明

本章涉及的核心文件是 `kernel/mm/address_space.hpp`（144 行）。运行环境和前面
几个 tag 一致——GCC cross-compiler（`x86_64-elf`）、QEMU、higher-half 内核
（`KERNEL_VMA = 0xFFFFFFFF80000000`）。前置条件是 PMM 和 VMM 已经初始化完毕，
因为 `AddressSpace` 的构造函数需要调用 `g_pmm.alloc_page()` 来分配新的 PML4 页。
在启动流程中，`init_kernel()` 被安排在 `kernel_main()` 的 Step 9，紧跟 VMM
初始化之后、Heap 初始化之前——时间窗口是确定的，VMM 就绪后内核页表已经完全建立
好了，此时读到的 CR3 就是"标准内核页表"的物理地址。

## 问题：一个虚拟地址，多个进程，怎么办

我们先把这个问题的本质看清楚。x86-64 的四级分页机制下，CPU 做地址翻译时的起点
是 CR3 寄存器——它指向当前正在使用的 PML4 表的物理地址。根据 Intel SDM Vol.3A
Section 4.5.2 的描述，CR3 的 bit 51:12 存放着 PML4 表的物理页帧号，4KB 对齐。
PML4 有 512 个条目，每个条目控制 512 GB 的虚拟地址空间，总共覆盖 256 TB 的
48 位规范地址空间。

当 CPU 需要访问虚拟地址 `0x20000000` 时，它先从 CR3 找到 PML4，取出条目索引
`(virt >> 39) & 0x1FF`——对于 `0x20000000` 来说这个索引是 0，也就是第一个条目。
然后从这个条目找到 PDPT，再一路走到 PT，最终拿到物理地址。整个翻译过程是纯
硬件的，CPU 的 MMU 自动完成，每次内存访问都走这套流程。

现在假设内核创建了进程 A 和进程 B。进程 A 要在 `0x400000` 放自己的代码段，
进程 B 也要在 `0x400000` 放自己的代码段。如果两者共享同一张 PML4，那 `0x400000`
对应的 PTE 只能指向一个物理页——你不可能让同一个虚拟地址同时映射到两个不同的
物理页，同一张页表里一个条目只能存一个值。你可能会说"那让进程 B 用另一个虚拟
地址不就行了"——理论上确实可以，但这要求每个进程知道自己被分配了哪段虚拟地址
空间，编译器、链接器、动态加载器全部要配合，复杂度直接爆炸。更根本的问题是，
这违背了"每个进程拥有独立地址空间"的抽象——进程应该觉得自己独占整个地址空间，
从 `0x0` 到 `0xFFFFFFFFFFFFFFFF` 全是它自己的。

所以解法很自然：给每个进程一张独立的 PML4。进程 A 的 PML4 里，`0x400000` 映射到
物理页 X；进程 B 的 PML4 里，`0x400000` 映射到物理页 Y。切换进程时，把 CR3 从
A 的 PML4 换成 B 的 PML4，整个地址空间就切换了——后续所有的内存访问都走 B 的
页表，A 的映射对 B 完全不可见。这就是 Intel 硬件提供 CR3 寄存器的根本理由：
通过切换页表根来实现地址空间的隔离。

## PML4 的天然分割线：上半内核、下半用户

搞清楚了"需要多张 PML4"之后，下一个问题是：每张 PML4 的 512 个条目是不是全部
独立？显然不是——内核代码和数据必须始终可见，否则中断处理、系统调用、驱动程序
全都不能工作。想象一下进程 A 正在执行，来了一个时钟中断，CPU trap 进内核态——
此时如果内核的代码段和堆栈在 A 的 PML4 里没有映射，CPU 连中断处理函数的第一条
指令都取不到，直接 triple fault。

所以我们想要的效果是：PML4 的一部分（内核空间）在所有地址空间中完全一致，另一
部分（用户空间）各走各的。好在这个分割在 x86-64 的地址空间布局里是天然存在的。
根据 Intel SDM Vol.3A Section 4.5.4 的描述，48 位规范地址空间被分成两半：
下半部分从 `0x0000000000000000` 到 `0x00007FFFFFFFFFFF`（bit 47 为 0），
上半部分从 `0xFFFF800000000000` 到 `0xFFFFFFFFFFFFFFFF`（bit 47 为 1，高 16 位
做符号扩展）。PML4 的条目索引是虚拟地址的 bit 47:39，所以条目 0 到 255 对应
下半部分（用户空间），条目 256 到 511 对应上半部分（内核空间）。每个 PML4 条目
控制一个 512 GB 的区域，所以用户空间最多 128 TB，内核空间也是最多 128 TB，
对任何操作系统来说都绰绰有余。

这意味着一个极其优雅的实现策略：构造新的 `AddressSpace` 时，分配一个新的 PML4 页，
把条目 0 到 255 全部清零（空白用户空间），然后把条目 256 到 511 从内核的 PML4
逐条复制过来（共享内核映射）。如此一来，每个地址空间的用户空间从零开始互相隔离，
内核空间则保持一致。而且这个复制操作非常轻量——只需要拷贝 256 个 8 字节的条目，
总共 2 KB 的数据，不需要递归拷贝 PDPT/PD/PT 这些下层页表。内核条目指向的 PDPT
页是共享的，所有地址空间通过相同的 PML4 条目值访问同一套内核页表结构。

这种设计模式在 OS 开发圈子里叫做 Higher Half Kernel——内核被映射到地址空间的
高半部分，所有进程共享；用户空间占据低半部分，每个进程独立。OSDev Wiki 的
[Higher Half Kernel](https://wiki.osdev.org/Higher_Half_Kernel) 页面对这种布局
有详细的描述，几乎所有现代 OS（Linux、Windows、macOS）都采用了这种模式。
Cinux 的 `KERNEL_VMA = 0xFFFFFFFF80000000` 也是基于这种布局——内核的虚拟地址
全在 `0xFFFF800000000000` 以上的区域。

## 经典设计对比：xv6 与 Linux

在深入 Cinux 的实现之前，我们先看看两个经典 OS 是怎么处理"每进程页表"这个问题
的。对比是理解设计取舍的最好方式。

先看 xv6，MIT 的教学 OS（RISC-V 版本）。xv6 在 `kernel/proc.c` 的
`proc_pagetable()` 函数中创建进程页表。它调用 `uvmcreate()` 分配一个空白的
页表根——注意 xv6 跑在 RISC-V 上，用的是 Sv39 三级页表，所以它的页表根只有三层，
不像 x86-64 的四级。然后 xv6 通过 `mappages()` 把两个特殊页面映射到每个用户
页表中——trampoline 页存放 trap 入口代码，放在地址空间的最顶端（`MAXVA` 附近），
trapframe 页存放用户寄存器的保存区。这和 Cinux 复制 PML4[256..511] 的目的是
一样的——确保内核基础设施在每个地址空间中可见。

但 xv6 的做法有一个根本性的不同：它维护两套独立的页表——用户页表和内核全局
页表（`kernel_pagetable`）。从用户态 trap 进内核时，xv6 先把页表从用户页表切换
到 `kernel_pagetable`，在内核态运行期间一直使用这张全局页表，返回用户态之前再
切回用户页表。这种双页表设计的好处是内核页表不需要复制到每个用户页表里，节省了
内存——如果有 100 个进程，xv6 不需要复制 100 份内核页表条目。但代价是每次 trap
进出都要做一次页表切换，而且这种切换会导致 TLB 失效。另外，xv6 的双页表设计
还带来了一个著名的限制——内核无法直接通过用户虚拟地址访问用户内存（因为内核运行
在自己的页表上），必须通过 `copyin`/`copyout` 函数手动翻译。

再看 Linux。Linux 的做法和 Cinux 更像——内核映射存在于每个进程的页表中。
Linux 用 `mm_struct` 结构管理进程的地址空间，`task_struct->mm` 指向它。
`mm_struct->pgd` 保存着进程的 PGD（Page Global Directory，在 x86-64 上 PGD
就是 PML4）的物理地址。创建新进程时，`pgd_alloc()` 分配新的 PGD，然后通过
`pgd_ctor()` 把内核条目复制到 PGD 的对应位置——这和 Cinux 的构造函数逐条复制
PML4[256..511] 在逻辑上完全一致。进程切换时，`switch_mm()` 把新进程的 PGD 写入
CR3，和 Cinux 的 `activate()` 干的事情一模一样。

不过 Linux 的 `mm_struct` 比我们现在的 `AddressSpace` 复杂得多——它还包含 VMA
链表（`vm_area_struct`，描述每段连续虚拟内存区域的属性）、RSS 计数器（统计物理
内存使用量）、mmap 信号量（保护地址空间修改的并发安全）、缺页计数器等大量成员。
Cinux 的 `AddressSpace` 目前就是 Linux `mm_struct` 的最小子集——一个 PML4 物理
地址加上几个基本操作。VMA 管理、引用计数、COW fork 这些高级特性留给了后面的 tag。
Mel Gorman 在 *Understanding the Linux Virtual Memory Manager* 第 4 章中对
`mm_struct` 的结构和进程地址空间管理有详细的描述，如果你对 Linux 的完整实现
感兴趣，这是必读材料。

从 xv6 和 Linux 的对比中我们可以提炼出一个有意思的维度：xv6 选择"内核页表独立
于用户页表"来节省条目复制的开销，但付出了 trap 时页表切换的代价；Linux 选择
"内核映射存在于每个用户页表"来避免 trap 时的页表切换，但付出了一点复制开销和
内存占用。Cinux 选了 Linux 的路线——这在 x86-64 上是更自然的选择，因为 x86-64
的 syscall/sysret 指令对页表切换的支持不如 RISC-V 的 S 模式灵活，而且在同一张
页表里运行内核和用户代码，对 TLB 的利用率也更高。

## AddressSpace 类的设计概览

看完了对比，我们来看 Cinux 的类设计。`AddressSpace` 的公开接口非常精炼，但每一
个方法都有明确的职责和设计考量。

构造函数的核心逻辑是三步走：从 PMM 分配一个新的 4KB 物理页作为 PML4，把 512
个条目全部清零（这样用户空间从"完全空白"开始，所有虚拟地址访问都会触发 page
fault），然后把内核 PML4 的条目 256 到 511 逐条复制过来。析构函数做的事情恰好
反过来——遍历用户空间条目 0 到 255，对每个 present 的条目递归释放整个页表子树
（PDPT -> PD -> PT），最后释放 PML4 页本身。这里有一个很重要的设计决策：析构只
释放"页表结构"（PML4/PDPT/PD/PT 这四层页表页本身），不释放"数据页"（PT 条目
指向的实际物理页）。数据页的生命周期由上层的进程内存分配器管理，`AddressSpace`
不越权。

拷贝构造和拷贝赋值被 `= delete` 禁止。这不是偷懒——而是安全性的硬要求。试想如果
允许拷贝构造，两个 `AddressSpace` 对象的 `pml4_phys_` 就指向同一个物理 PML4 页。
当第一个对象析构时，它把 PML4 归还给 PMM，第二个对象手里就剩下一个悬空指针，
后续任何操作都会往已经不属于自己的物理内存上写东西——这是经典的 use-after-free。
禁止拷贝从编译期就杜绝了这种可能性。移动语义是允许的——被移走的对象 `pml4_phys_`
置 0，析构时检查到 0 就跳过释放。这样 `AddressSpace` 就可以被放进容器里了。

`init_kernel()` 是一个静态方法，整个类只有这一个"启动钩子"。它做的事情极其
简单——读 CR3，把值存到静态成员 `kernel_pml4_` 里。但它的调用时机非常关键：必须
在 bootloader 设置好内核页表之后、在创建第一个 `AddressSpace` 之前调用。为什么
不干脆在构造函数里直接读 CR3？因为构造函数读到的 CR3 是"当前正在使用的"页表根，
不一定是"标准内核页表"。如果进程 A 的 `AddressSpace` 被激活后 CR3 指向进程 A 的
PML4，此时再创建一个新的 `AddressSpace`，构造函数读到的就是 A 的 PML4 而不是
内核 PML4——复制出来的内核条目可能不是期望的。显式保存确保了模板来源是确定的、
一致的。

`map()`、`unmap()`、`translate()` 三个方法是对 VMM 的直接委托。它们把
`&pml4_phys_` 作为可选参数传给 `g_vmm` 的对应方法，让 VMM 知道"用哪个 PML4 做
页表遍历的起点"。这种设计让 `AddressSpace` 成了一个非常轻量的"所有权包装器"——
页表遍历和修改的复杂逻辑全部复用 VMM 已有的实现。`activate()` 是唯一一个不委托
给 VMM 的操作——它直接调用 `write_cr3(pml4_phys_)`，用内联汇编把 PML4 物理地址
写入 CR3 寄存器。根据 Intel SDM Vol.3A Section 4.10.2 的规定，写入 CR3 会隐式
刷新所有非全局的 TLB 条目，所以切换地址空间后 CPU 的地址翻译缓存自动失效。

整个类的数据成员只有两个：实例成员 `pml4_phys_`（当前空间的 PML4 物理地址）和
静态成员 `kernel_pml4_`（启动时保存的内核 PML4 物理地址）。加起来 16 字节。
你可以把它类比为一张"身份证"——上面写着一个物理地址，凭借这个地址就能找到
一整套四级页表结构。`pml4_phys_` 是否为 0 是整个对象状态的"哨兵"：构造成功后
非 0，移动后变为 0，析构时也变为 0，所有生命周期操作都先检查它。

## 收尾

到这里我们已经把"为什么需要独立地址空间"这个问题说清楚了——从单一张页表的
局限性，到 PML4 的自然分割如何优雅地适配了隔离需求，到 xv6 的双页表方案和
Linux 的共享映射方案的设计取舍，再到 Cinux `AddressSpace` 类的接口设计。
核心思想其实很简单：每个进程一张 PML4，上半复制内核映射，下半各管各的。

但设计概览只是图纸，真正的乐趣在实现细节里。下一章我们深入构造函数和析构函数
的代码，看看"分配 PML4 -> 清零 -> 复制内核条目"这三步在代码里是怎么走的，
析构时递归释放页表子树的 `free_subtree` 是如何保证不漏不重的，以及 move 语义
在"独占所有权"的约束下是如何安全转移的。
