# 028-2 Tutorial: ext2 驱动核心——超级块、块组与 inode

> 标签：ext2、Superblock、Block Group Descriptor、Inode、磁盘寻址
> 前置：[028_fs_ext2 第 1 篇](028-fs-ext2-1.md)

## 前言

上一篇我们搞清楚了 ext2 的设计动机和磁盘布局，

也准备好了 4MB 的测试镜像。

现在该动真格的了——

写代码把 Superblock 从磁盘读出来、

解析 Block Group Descriptor Table、

定位根 inode。

这一篇的内容是整个 ext2 驱动的骨架，

后续的文件读写、目录遍历、路径解析

全部建立在这些基础之上。

如果你把 ext2 驱动比作一栋楼，

那 Superblock 就是地基，BGDT 是楼层平面图，

inode 表是每个房间的编号表。

不把这些搞对，后面的一切都是空中楼阁。

## 第一步——DMA Buffer：磁盘数据进入内核的通道

在读取 Superblock 之前，我们得先解决一个底层问题：

从磁盘读出来的数据放哪？

AHCI 驱动通过 DMA（Direct Memory Access）进行块传输——

CPU 告诉 AHCI 控制器"把第 N 个扇区的数据写到物理地址 P"，

控制器完成后中断通知 CPU。

这意味着我们必须先分配一块物理内存，

映射到内核虚拟地址空间，

然后把物理地址交给 AHCI。

Cinux 的方案是分配一个物理页（4KB）作为 DMA buffer。

`ensure_dma_buffer()` 的实现很直接：

调用 PMM 的 `alloc_page()` 获取一个物理页，

然后通过 VMM 的 `map()` 把它映射到内核空间的预留虚拟地址

`KMEM_EXT2_DMA_BASE`。

映射完成后清零整个页面，标记 `dma_ready_ = true`。

```cpp
bool Ext2::ensure_dma_buffer() {
    if (dma_ready_) return true;
    dma_buf_phys_ = cinux::mm::g_pmm.alloc_page();
    if (dma_buf_phys_ == 0) return false;

    constexpr uint64_t flags =
        cinux::arch::FLAG_PRESENT | cinux::arch::FLAG_WRITABLE;
    dma_buf_virt_ = EXT2_DMA_VIRT_BASE;

    if (!cinux::mm::g_vmm.map(dma_buf_virt_, dma_buf_phys_, flags)) {
        cinux::mm::g_pmm.free_page(dma_buf_phys_);
        return false;
    }
    dma_ready_ = true;
    return true;
}
```

这里有一个非常重要的设计决策——

我们只用了一个 DMA buffer。

这意味着每次 `read_block()` 调用都会覆盖上一次读取的数据。

如果你读了 block 5，还没处理完就去读 block 8，

block 5 的数据就丢了。

这是一个有意为之的简化：

在 Linux 中，块 I/O 层会通过 buffer_head 和 page cache

自动缓存最近访问的块；

SerenityOS 的 BlockBasedFileSystem 也有内部的块缓存机制。

但 Cinux 目前不加缓存，每次读取都直接走磁盘。

原因很简单——

缓存策略是性能优化的范畴，而我们的首要目标是正确性。

一个块缓存意味着额外的内存管理、缓存一致性、淘汰策略等一大堆问题。

先跑通再说，以后有需要再加。

## 第二步——从 AHCI 读取 Superblock 并验证

万事俱备，我们来读 Superblock。

ext2 规范规定 Superblock 位于磁盘字节偏移 1024 处，固定 1024 字节。

对于 512 字节扇区来说，这就是 LBA 2 开始的连续 2 个扇区。

Cinux 的 `mount()` 函数用硬编码的常量来计算这个位置：

```cpp
constexpr uint64_t SB_LBA     =
    EXT2_SUPERBLOCK_OFFSET / EXT2_SECTOR_SIZE;   // 1024 / 512 = 2
constexpr uint16_t SB_SECTORS =
    EXT2_SUPERBLOCK_SIZE / EXT2_SECTOR_SIZE;      // 1024 / 512 = 2
```

`ahci_.read()` 把这两个扇区读入 DMA buffer，

然后我们用 `memcpy` 把 Superblock 结构体从 DMA buffer 拷贝到内核的

`sb_` 成员变量中。

这里有一个初学者特别容易踩的坑：

Superblock 不是从磁盘第 0 字节开始的，前面有 1024 字节的引导块。

如果你直接从 LBA 0 开始读，得到的是引导代码而不是 Superblock。

Cinux 通过 `EXT2_SUPERBLOCK_OFFSET = 1024` 这个常量来避免这个错误。

读取完成后，第一件事是验证魔数。

ext2 的魔数是 `0xEF53`，

存储在 Superblock 的 `s_magic` 字段（偏移 56 字节处）。

如果魔数不对，说明磁盘上不是有效的 ext2 文件系统——

可能是个空磁盘，也可能是格式化参数搞错了。

```cpp
if (sb_.s_magic != EXT2_SUPER_MAGIC) {
    kprintf("[EXT2] Invalid magic: 0x%x (expected 0x%x)\n",
            sb_.s_magic, EXT2_SUPER_MAGIC);
    return false;
}
```

验证通过后，我们计算文件系统的核心参数。

`block_size_` 通过移位得到：`1024U << sb_.s_log_block_size`。

每个块包含的扇区数是 `block_size_ / 512`。

inode 大小取决于修订版本——

revision 0 固定 128 字节，revision 1+ 从 `s_inode_size` 字段读取。

块组数通过向上取整计算：

`group_count_ = (s_blocks_count + blocks_per_group - 1) / blocks_per_group`。

这一步的输出是一组调试日志，让你确认参数是否正确：

```
[EXT2] Superblock valid: magic=0xef53
[EXT2]   block_size=1024  inode_size=128
[EXT2]   blocks=4096  inodes=128  groups=1
```

## 第三步——加载 Block Group Descriptor Table

Superblock 解析完了，下一步是加载 BGDT。

BGDT 的起始块号取决于块大小——

对于 1024 字节块，Superblock 在 block 1，BGDT 从 block 2 开始；

对于更大的块大小，Superblock 和 BGDT 都在 block 0 内

（Superblock 占用字节 1024-2047，BGDT 从字节 2048 开始）。

Cinux 的处理是一个简单的条件判断：

```cpp
uint32_t bgdt_block = (block_size_ == 1024) ? 2 : 1;
```

每个块组描述符 32 字节，包含四个关键位置信息

（block bitmap 块号、inode bitmap 块号、inode table 起始块号）

和三个统计信息（空闲块数、空闲 inode 数、目录数）。

我们来看看 `Ext2BlockGroupDescriptor` 的定义：

```cpp
struct [[gnu::packed]] Ext2BlockGroupDescriptor {
    uint32_t bg_block_bitmap;       // 块位图所在的块号
    uint32_t bg_inode_bitmap;       // inode 位图所在的块号
    uint32_t bg_inode_table;        // inode 表起始块号
    uint16_t bg_free_blocks_count;  // 本组空闲块数
    uint16_t bg_free_inodes_count;  // 本组空闲 inode 数
    uint16_t bg_used_dirs_count;    // 本组目录数
    uint16_t bg_pad;
    uint32_t bg_reserved[3];
};
```

加载 BGDT 时，我们按块逐个读取，

每块能容纳 `block_size / 32` 个描述符，

然后 `memcpy` 到内核的 `bgdt_[]` 数组中。

这个数组最大支持 128 个块组

（`EXT2_MAX_GROUPS = 128`），

对于 4MB 的测试镜像来说只会有一个块组，绰绰有余。

这里值得对比一下 Linux 的做法。

Linux 在挂载 ext2 时不会一次性把整个 BGDT 读入内存——

它通过 buffer_head 按需读取需要的块组描述符，

而且会检查 BGDT 的备份副本

（存放在块组 0、1 和最后一个块组中）。

Cinux 直接全量加载，简单粗暴但对小镜像完全够用。

## 第四步——Inode 查找的数学推导

有了 Superblock 和 BGDT，

我们现在能把任意 inode 号转换为磁盘上的物理位置。

这是 ext2 驱动最核心的寻址计算之一，推导过程如下。

给定 inode 号 `ino`（从 1 开始编号），

首先计算它所在的块组：

```
group = (ino - 1) / inodes_per_group
```

为什么减 1？

因为 inode 编号从 1 开始，

而除法和取模需要 0-based 的索引。

然后从 BGDT 中获取该组的 inode table 起始块号

`bg_inode_table`。

接下来计算 inode 在组内的索引：

```
index = (ino - 1) % inodes_per_group
```

这个索引乘以 inode 大小得到字节偏移：

```
byte_offset = index * inode_size
```

最后把这个字节偏移拆分成"块偏移 + 块内偏移"两部分：

```
block_offset = byte_offset / block_size
within_block_offset = byte_offset % block_size
target_block = inode_table_start + block_offset
```

以我们的 4MB 镜像为例：

inode 大小 128 字节，块大小 1024 字节，每个块能放 8 个 inode。

根 inode（ino=2）在组 0，组内索引 1，

字节偏移 128，块内偏移 128——

所以它和 inode 1 在同一个块里，但偏移了 128 字节。

Cinux 的 `read_disk_inode()` 函数忠实地实现了这套计算：

```cpp
bool Ext2::read_disk_inode(uint32_t ino, Ext2Inode& out_inode) {
    uint32_t group = (ino - 1) / inodes_per_group_;
    uint32_t inode_table_block = bgdt_[group].bg_inode_table;
    uint32_t index_in_group = (ino - 1) % inodes_per_group_;
    uint64_t byte_offset =
        static_cast<uint64_t>(index_in_group) * inode_size_;
    uint32_t block_offset =
        static_cast<uint32_t>(byte_offset / block_size_);
    uint32_t within_block_offset =
        static_cast<uint32_t>(byte_offset % block_size_);
    uint32_t target_block = inode_table_block + block_offset;

    if (!read_block(target_block)) return false;
    auto* block_data =
        reinterpret_cast<const uint8_t*>(dma_buf_virt_);
    memcpy(&out_inode, block_data + within_block_offset,
           sizeof(Ext2Inode));
    return true;
}
```

它还做了一个安全检查——

确保 inode 不会跨越块边界：

`within_block_offset + sizeof(Ext2Inode) <= block_size`。

在 128 字节 inode + 1024 字节块大小的情况下

这永远不会触发（最大偏移 896 + 128 = 1024），

但对于更大的 inode 尺寸

（比如 revision 1+ 的 256 字节 inode + 1024 字节块大小）

就可能出问题，这时候需要跨块读取。

Cinux 目前不支持这种情况，直接返回 false。

## 第五步——根 inode：为什么是 inode 2？

你可能注意到了，ext2 的根目录对应 inode 2

而不是 inode 1 或 inode 0。

这不是随意的选择——

inode 0 在 ext2 中被保留为"无效 inode"

（类似于 C 的 NULL），

inode 1 在 revision 0 中被保留为"坏块列表"（Bad blocks inode），

revision 1+ 中还可能用于其他特殊用途

（比如 ACL 索引或引导加载程序的预留区域）。

所以根目录从 inode 2 开始，

这是一个从 ext2 诞生就确定下来的约定。

`mount()` 的最后一步是读取根 inode，

把它放入 inode cache 的 slot 0，

然后通过 `populate_vfs_inode()`

把磁盘 inode 的字段转换成 VFS inode 格式。

这个转换函数检查 `i_mode` 的文件类型位：

如果是 `EXT2_S_IFDIR`（0x4000）

就设为 Directory 类型并绑定 `ext2_dir_ops`，

如果是 `EXT2_S_IFREG`（0x8000）

就设为 Regular 类型并绑定 `ext2_file_ops`。

`fs_private` 指针指向包含这个 VFS inode 的

`Ext2CachedInode` 结构——

这样在 InodeOps 的回调中，

我们可以通过 `inode->fs_private`

反向拿到完整的缓存条目和磁盘 inode。

## 设计对比再探：buffer_head vs 单 DMA buffer

这一节值得再深入对比一下 Linux 和 Cinux 在块 I/O 上的根本区别。

Linux 的 `buffer_head` 是一个相当复杂的子系统——

每个 buffer_head 描述一个磁盘块到内存页的映射，

包含块号、设备号、脏标记、锁、引用计数等。

读取一个块时，Linux 会先在 page cache 中查找

（通过块号和设备号做哈希），

命中就直接返回内存中的数据副本，

未命中才分配一个 buffer_head、发起磁盘 I/O、

然后把数据插入 page cache。

写入时，修改内存中的副本并标记为脏，

由后台的 pdflush/kworker 线程异步写回磁盘。

这套机制使得 Linux 中的文件系统驱动

几乎不需要关心"重复读取"的性能问题——

缓存层自动处理了。

Cinux 的单 DMA buffer 完全没有这些机制。

每次 `read_block()` 都是一个同步的 AHCI DMA 传输，

数据存放在一个固定的 4KB 虚拟地址上。

调用者必须在下一次 `read_block()` 之前

把数据处理完或者拷走。

这意味着像"遍历目录项时同时读 inode"这种操作

需要小心安排——

你不能在读 inode 的时候还依赖之前目录块的数据。

SerenityOS 的 `BlockView` 在这方面做得很好——

它本质上是一个引用计数的智能指针，

持有时块数据不会被回收，释放后自动归还缓存池。

这避免了 Cinux 中"必须立即拷走数据"的限制。

## 收尾

到这里，ext2 驱动的骨架已经立起来了。

Superblock 解析、BGDT 加载、inode 寻址、根 inode 读取——

这些是 ext2 驱动的初始化流程。

所有后续操作都依赖这些基础设施：

文件读取需要通过 inode 定位数据块，

目录遍历需要读取目录 inode 的数据块，

路径解析需要逐级查找 inode。

下一篇我们将深入文件读写的完整路径：

从 inode 的块指针数组出发，

通过直接块和间接块定位文件数据，

实现真正的文件读取。
