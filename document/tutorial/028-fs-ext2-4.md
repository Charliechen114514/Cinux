# ext2 目录操作与实战验证

> 标签：ext2、目录操作、路径解析、VFS 集成、块分配、inode 分配、QEMU 验证
> 前置：[028_fs_ext2 第 3 篇](028-fs-ext2-3.md)

## 前言

前三篇我们搭好了 ext2 驱动的数据结构、读取路径和写入路径。现在还差最后几块拼图——目录操作。文件系统的"目录"不是什么神秘的东西，本质上就是一个"名字到 inode 号"的映射表，存在磁盘上的数据块里。但在 ext2 中，这个映射表的格式有一些有趣的设计选择——变长记录、链表式连接、4 字节对齐——这些细节在遍历、插入和删除目录项时都需要正确处理。

这一篇我们要讲的不只是目录遍历，还有完整的文件系统突变能力：创建文件（create）、创建目录（mkdir）、删除文件（unlink），以及支撑这些操作的块/inode 分配器。最后，我们把 ext2 挂载到 VFS 上，用 QEMU 启动完整系统做端到端验证——这是整个 tag 028 的高潮。

## Ext2DirEntry：变长的链表式目录项

ext2 的目录数据块由一系列 `Ext2DirEntry` 结构体组成。每个结构体有一个 8 字节的固定头部（inode 号 4 字节 + rec_len 2 字节 + name_len 1 字节 + file_type 1 字节），后面紧跟变长文件名。整个条目的长度由 `rec_len` 决定，必须 4 字节对齐。所以即使文件名只有 3 字节，`rec_len` 也是 12（8 头部 + 3 名字 + 1 填充，对齐到 4）。

先别急，在写遍历代码之前，我们先搞清楚这里有几个非常容易踩的坑。第一，文件名在磁盘上不以 null 结尾——`name_len` 才是确定文件名长度的唯一依据。如果你用 `strcmp` 来比较文件名，你会读到垃圾数据——必须逐字节比较 `name_len` 个字符。第二，`rec_len` 可能远大于 `8 + name_len`——这是被删除条目留下的空洞被合并到前一个条目的结果，所以步进时必须用 `rec_len` 而不是 `name_len + 8`。第三，`inode == 0` 的条目是空闲的，遍历时必须跳过它。

这种变长设计比固定长度的目录项（比如 FAT 的 32 字节条目）灵活得多——文件名最长可达 255 字节，而且通过 rec_len 合并可以高效地回收已删除条目的空间。Linux 的 ext2 驱动在 `fs/ext2/dir.c` 中处理同样的逻辑，但它还支持 htree 索引（一种基于 DX 算法的目录项哈希树加速结构），用来加速大目录的查找。Cinux 当前不需要这种优化——我们的测试镜像里目录项数量很少，线性扫描就够了。

## 目录遍历：readdir 的实现

Cinux 的 `Ext2DirOps::readdir` 接受一个 VFS inode 和一个索引号，返回对应索引处的文件名。索引 0 和 1 被硬编码返回 "." 和 ".."——这是所有 Unix 目录都有的两个特殊条目，不需要从磁盘读取。索引 2+ 的条目需要扫描目录数据块。

扫描过程是：先计算目录占用的总块数（`dir_size / block_size`），然后对每个块调用 `read_block()` 读取数据，在块内按 `rec_len` 步进遍历每一个条目。实现中有一个值得注意的细节——扫描时显式跳过了磁盘上 "." 和 ".." 条目，避免和硬编码的索引 0/1 重复：

```cpp
if (entry->name_len == 1 && entry->name[0] == '.') {
    pos += entry->rec_len;
    continue;
}
if (entry->name_len == 2 && entry->name[0] == '.'
                          && entry->name[1] == '.') {
    pos += entry->rec_len;
    continue;
}
```

很好，现在当 `found` 计数器到达 `target`（即 `index - 2`）时就找到了目标条目，拷贝文件名到输出 buffer 并返回 1。这种"索引号"驱动的 readdir 接口和 Linux 的 `getdents` 系统调用有微妙的区别。Linux 的 `getdents` 返回的是从指定偏移开始的一批目录项（批量返回，减少系统调用次数），而 Cinux 的接口是"给我第 N 个条目"——更简单，每次只返回一个。在 Shell 的 `ls` 命令实现中用起来很方便：从 index=0 开始递增调用 readdir，直到返回 0（没有更多条目）。

## 路径解析：从 "/" 到目标 inode

有了目录遍历的能力，路径解析就是顺理成章的事情了。`Ext2::lookup(const char* path)` 实现的是经典的 Unix 路径解析算法：从根 inode（inode 2）开始，逐级分解路径组件，每级在当前目录中查找匹配的目录项。

解析过程从跳过前导 `/` 开始，然后进入 while 循环。每次迭代提取一个路径组件（到下一个 `/` 或字符串末尾的连续非斜杠字符），调用 `lookup_in_dir()` 在当前目录中查找。`lookup_in_dir()` 本身就是一次完整的目录扫描——读取目录 inode 的所有数据块，在每个块中按 `rec_len` 步进，逐字节比对每个条目的 `name_len` 和文件名内容。找到匹配就返回 inode 号，找不到返回 0。

如果中间某一级不是目录（通过读取 inode 的 `i_mode` 检查 `EXT2_S_IFDIR` 位），解析直接失败——你不能在普通文件后面继续跟路径组件。所有组件解析完毕后，调用 `get_cached_inode(current_ino)` 返回最终的 VFS inode。

这个实现虽然简单，但有一个性能特性值得了解：每次路径解析的每一级都需要从磁盘读取目录块。对于路径 `/etc/motd`，需要读根目录块（找到 `etc`），再读 `etc` 目录块（找到 `motd`）——至少两次磁盘读取。Linux 通过 dentry cache 把"目录项到 inode 号"的映射缓存下来，第二次访问同一路径时几乎不需要磁盘 I/O。Cinux 没有这一层缓存，但教学内核的场景下路径解析的频率很低（主要发生在 Shell 执行 `ls`、`cat` 时），影响可以接受。

## VFS 集成：Ext2FileOps 与 Ext2DirOps 的分发机制

ext2 驱动和 VFS 的对接点在 tag 027 定义好的 `InodeOps` 接口上。`populate_vfs_inode()` 在把磁盘 inode 转换为 VFS inode 时，根据文件类型自动绑定对应的 ops 对象——目录绑 `&dir_ops_`，普通文件绑 `&file_ops_`：

```cpp
uint16_t mode_type = disk.i_mode & EXT2_S_IFMT;
if (mode_type == EXT2_S_IFDIR) {
    cached.vfs_inode.type = InodeType::Directory;
    cached.vfs_inode.ops  = &dir_ops_;
} else if (mode_type == EXT2_S_IFREG) {
    cached.vfs_inode.type = InodeType::Regular;
    cached.vfs_inode.ops  = &file_ops_;
}
```

这种"ops 对象绑定到 inode"的模式和 Linux 的 `inode->i_fop` / `inode->i_op` 非常相似。Linux 中每个 inode 有两个操作表——`i_fop`（file operations，处理 read/write/lseek）和 `i_op`（inode operations，处理 lookup/create/unlink）。Cinux 合并为一个 `InodeOps`，简化了设计。SerenityOS 的做法更彻底——Inode 类本身就有虚方法如 `read_bytes()`、`write_bytes()`、`traverse_as_directory()`，不需要额外的 ops 指针，但代价是 Inode 类的体积更大、继承层次更深。

`Ext2DirOps` 中的 `create`、`mkdir`、`unlink` 方法都很薄——它们只是把 VFS Inode 中的 inode 号提取出来，转发给 `Ext2` 类的对应方法。真正的逻辑在 `Ext2::create()`、`Ext2::mkdir()`、`Ext2::unlink()` 中。

## 文件创建：create 的完整流程

`Ext2::create()` 的流程可以拆解为以下几个步骤。首先检查目标名称是否已经存在（`lookup_in_dir`），避免重复创建。然后通过 `alloc_inode()` 分配一个新的 inode 号，初始化一个全零的 `Ext2Inode`，设置 `i_mode = EXT2_S_IFREG | 0644`（普通文件，rw-r--r--），`i_links_count = 1`。接着把新 inode 写入磁盘（`write_disk_inode`），在父目录中添加一个目录项（`add_dir_entry`），最后更新父目录的 inode 并把新 inode 放入缓存返回。

`add_dir_entry()` 本身有一个很精巧的实现。它不是简单地追加一个新块，而是先扫描现有目录块，寻找已分配空间中的"缝隙"——如果某个目录项的 `rec_len` 远大于它实际需要的最小长度（`8 + name_len` 向上对齐到 4），就把多余的空间分给新条目。这种"条目分裂"的策略是：先缩小当前条目的 `rec_len` 到最小值，然后在紧随其后创建一个新条目，新条目的 `rec_len` 等于原来多出来的空间。只有在所有现有块都塞不下的情况下，才会分配新的数据块——新块中只有一个条目，`rec_len` 覆盖整个块。

你会发现，这种"利用缝隙"的策略最大化了磁盘空间利用率，避免了每创建一个文件就消耗一个整块（1024 字节）。Linux 的 `ext2_add_link()` 用了完全相同的策略——`fs/ext2/dir.c` 中的实现也是先尝试在已有条目间找空间，不够再分配新块。

## 目录创建：mkdir 与 "."/".." 的初始化

`Ext2::mkdir()` 比 `create()` 复杂一些，因为新目录需要初始化两个特殊条目 "." 和 ".."。流程是：分配 inode 和数据块各一个，初始化 inode 为 `EXT2_S_IFDIR | 0755`（目录，rwxr-xr-x），`i_links_count = 2`（一个来自自身的 "."，一个来自父目录的条目），`i_size = block_size_`（一个块）。然后在数据块中写入 "." 条目和 ".." 条目：

```cpp
auto* dot = reinterpret_cast<Ext2DirEntry*>(dma);
dot->inode     = new_ino;
dot->name_len  = 1;
dot->file_type = static_cast<uint8_t>(Ext2FileType::Directory);
dot->name[0]   = '.';
uint32_t dot_rec_len = (EXT2_DIR_ENTRY_HDR_SIZE + 1 + 3) & ~3u;  // 12
dot->rec_len   = static_cast<uint16_t>(dot_rec_len);

auto* dotdot = reinterpret_cast<Ext2DirEntry*>(dma + dot_rec_len);
dotdot->inode     = parent_ino;
dotdot->name_len  = 2;
dotdot->file_type = static_cast<uint8_t>(Ext2FileType::Directory);
dotdot->name[0]   = '.';
dotdot->name[1]   = '.';
dotdot->rec_len   = static_cast<uint16_t>(block_size_ - dot_rec_len);
```

这里有一个容易忽略的细节：创建子目录后，父目录的 `i_links_count` 也需要加 1——因为子目录的 ".." 条目是对父目录的一个硬链接。Cinux 的实现正确地处理了这一点，同时还更新了块组描述符中的 `bg_used_dirs_count`。这和 Linux 的 `ext2_mkdir()` 实现逻辑一致——Linux 在 `fs/ext2/namei.c` 中也做了同样的三件事：初始化 "." 和 ".."、增加父目录链接计数、更新组内目录计数。

## 删除：unlink 与资源回收

`Ext2::unlink()` 是三个突变操作中最复杂的一个。它首先调用 `remove_dir_entry()` 从父目录中移除目标条目，然后将目标 inode 的 `i_links_count` 减 1。如果链接数归零，就释放所有数据块和 inode 本身。

`remove_dir_entry()` 的实现也有讲究。如果被删除的条目不是块中的第一个条目，它的空间会被合并到前一个条目的 `rec_len` 中——这是 ext2 目录项链表的经典"合并"操作。如果被删除的是第一个条目，就不能简单地合并到"前面"（因为前面没有条目），只能把 `inode` 字段清零标记为空闲。这种不对称的处理是 ext2 目录项格式的固有特性，Linux 的 `ext2_delete_entry()` 也采用了完全相同的策略。

释放数据块的逻辑考虑了直接块和间接块两种情况。对于直接块（`i_block[0..11]`），逐个调用 `free_block()`。对于单间接块，先读取间接块本身，遍历其中所有非零指针逐个释放，最后释放间接块本身：

```cpp
for (uint32_t i = 0; i < EXT2_DIRECT_BLOCKS; ++i) {
    if (target_disk.i_block[i] != 0)
        free_block(target_disk.i_block[i]);
}
if (target_disk.i_block[EXT2_INDIRECT_BLOCK] != 0) {
    uint32_t indirect_blk = target_disk.i_block[EXT2_INDIRECT_BLOCK];
    if (read_block(indirect_blk)) {
        auto* indirect = reinterpret_cast<uint32_t*>(dma_buf_virt_);
        uint32_t ptrs_per_block = bs / sizeof(uint32_t);
        for (uint32_t i = 0; i < ptrs_per_block; ++i)
            if (indirect[i] != 0) free_block(indirect[i]);
    }
    free_block(indirect_blk);
}
```

释放完成后，inode 被清零（`i_size = 0`、`i_blocks = 0`）写回磁盘，然后通过 `free_inode()` 在 inode 位图中清除对应的位。如果是目录，还会递减父目录的 `i_links_count`（因为子目录的 ".." 链接被移除了）并更新 `bg_used_dirs_count`。最后，从 inode 缓存中移除被删除的条目，防止后续访问到过时的数据。

## 块与 inode 的位图分配器

`alloc_block()` 和 `alloc_inode()` 是支撑所有写入操作的基础设施。它们的逻辑结构相同：遍历所有块组，找到有空闲资源的组（检查 BGDT 中的 `bg_free_blocks_count` 或 `bg_free_inodes_count`），读取对应的位图块，在位图中逐位扫描寻找第一个 0 位（空闲）。

以 `alloc_block()` 为例，核心循环如下：

```cpp
for (uint32_t byte_idx = 0; byte_idx < bytes_needed; ++byte_idx) {
    if (bitmap[byte_idx] == 0xFF) continue;  // 整字节满，跳过
    for (uint32_t bit = 0; bit < 8; ++bit) {
        uint32_t local_block = byte_idx * 8 + bit;
        if (local_block >= blocks_in_group) break;
        if ((bitmap[byte_idx] & (1U << bit)) == 0) {
            bitmap[byte_idx] |= static_cast<uint8_t>(1U << bit);  // 置位
            write_block(bitmap_block);       // 写回位图
            // 更新超级块和块组描述符的空闲计数
            --sb_.s_free_blocks_count;
            --bgdt_[group].bg_free_blocks_count;
            write_superblock();
            write_bgdt(group);
            return first_block + local_block;  // 返回全局块号
        }
    }
}
```

你会发现，每次分配都伴随着三到四次磁盘写入（位图、超级块、块组描述符）。这是没有缓存层的直接后果——每次修改元数据都要立即刷盘。Linux 的 ext2 使用 Orlov 分配器来优化新 inode 和数据块在同一块组中的概率（减少磁盘寻道），并且使用 per-CPU 计数器来避免频繁更新全局空闲计数。SerenityOS 的分配器相对简单但也用了位图缓存，避免每次分配都读写磁盘。Cinux 的"扫描第一个空闲位"策略虽然简陋，但对教学内核来说正确性第一，性能以后再说。

`free_block()` 和 `free_inode()` 的逻辑是分配的逆操作：定位块组、读取位图、清除对应位、写回位图、增加空闲计数、写回超级块和块组描述符。对称且简洁。

## Linux 与 Cinux 目录操作的横向对比

在深入验证之前，值得把 Cinux 的目录操作实现和 Linux 的 ext2 做一个更系统的对比，帮助大家理解"教学简化"和"生产级实现"之间的差距到底在哪。

首先是目录项的查找。Cinux 的 `lookup_in_dir` 对目录块做线性扫描，逐条目比较文件名。Linux 的 `ext2_find_entry`（位于 `fs/ext2/dir.c`）在大多数情况下也是线性扫描，但如果目录启用了 htree 索引（通过 `EXT4_INDEX_FL` 标志），查找会先经过哈希树定位到包含目标条目的目录块，然后只扫描那一个块。htree 本质上是一个 B 树变体，根节点存储在目录的第一个块中，通过文件名的哈希值快速缩小搜索范围。Cinux 不需要这个优化——我们的目录通常只有十几个条目。

其次是目录项的插入。Cinux 的 `add_dir_entry` 在已有条目间找缝隙（条目分裂），不够就分配新块。Linux 的 `ext2_add_link` 做的事情完全一样，但有一个额外的优化：如果启用了 htree，在分裂条目后需要更新索引树的结构（调整 DX 节点中的边界哈希值）。Cinux 没有索引树，所以插入逻辑更简单。

最后是目录项的删除。Linux 的 `ext2_delete_entry` 和 Cinux 的 `remove_dir_entry` 几乎完全一致——如果不是第一个条目就合并到前一个的 rec_len，如果是第一个就清零 inode 字段。这是 ext2 目录项格式的固有特性，所有实现都必须遵循相同的规则。

## QEMU 端到端验证

理论讲完了，现在来跑一下。首先创建 ext2 测试镜像，然后用 QEMU 启动内核（ext2.img 挂在 AHCI 端口），如果一切正常，你会在串口输出中看到类似这样的挂载日志：

```
[EXT2] Mounting ext2 filesystem on AHCI port 1
[EXT2] Superblock valid: magic=0xef53
[EXT2]   block_size=1024  inode_size=128
[EXT2]   blocks=4096  inodes=128  groups=1
[EXT2] BGDT loaded: 1 groups
[EXT2] Root inode: size=1024 mode=0x41ed
```

进入 Shell 后，可以尝试以下命令验证读写功能：

```
ls /
cat /hello.txt
```

`cat /hello.txt` 应该输出 "Hello from ext2!" 这行文字。这短短几个字代表的调用链是：Shell 解析命令 -> sys_open -> VFS 路径解析 -> ext2 lookup -> 读取根目录块 -> 找到 "hello.txt" 条目 -> 读取 inode -> 通过直接块指针定位数据块 -> AHCI DMA 读取 -> 拷贝到用户空间。写入操作同样经过了完整的路径验证——create 从分配 inode 到插入目录项，mkdir 从初始化 "."/".." 到更新父目录链接计数，unlink 从移除目录项到释放块和 inode。

## 设计反思与下一步

回头看整个 ext2 驱动，最明显的简化在于 I/O 模型——单 DMA buffer 没有缓存层，每次操作都是同步磁盘 I/O。Linux 的 page cache 和 SerenityOS 的 BlockView 都是为了解决这个问题的。但对于教学内核来说，这种"每次都和磁盘打交道"的方式有一个被低估的优势：行为完全可预测，调试时不用担心缓存一致性问题。当你发现一个 bug 时，不需要怀疑"是不是缓存没刷"——因为根本没有缓存。

另一个值得反思的设计选择是"同步写入"。Cinux 中所有的写操作（write_block、write_disk_inode、write_superblock、write_bgdt）都是同步的——调用者发出写入请求后阻塞等待 AHCI DMA 完成。这意味着文件系统操作是原子的（从调用者的角度看，要么成功要么失败，没有中间状态），但也意味着性能上限很低——一次 `create` 操作可能触发五六次同步磁盘写入（分配 inode 位图、写 inode、添加目录项可能涉及条目分裂或新块分配、更新父目录 inode、更新超级块和块组描述符）。Linux 的回写机制把"标记 dirty"和"实际写盘"分离，通过 `pdflush` 内核线程异步刷盘，大幅提高了写入吞吐量。

我们在这个 tag 中实现了大约 1800 行的 ext2 驱动代码（ext2.cpp 约 1810 行 + ext2.hpp 约 475 行 + ext2_types.hpp 约 310 行），涵盖了 Superblock 解析、BGDT 加载、inode 寻址、文件读写（含间接块和块按需分配）、目录遍历与突变（create/mkdir/unlink）、路径解析、块/inode 位图分配器。这是一个功能完整的可读可写磁盘文件系统驱动，已经能支撑 Shell 的 ls/cat 以及后续的 execve 加载 ELF。

下一个 tag 将会围绕进程加载展开——如何把 ext2 上的 ELF 可执行文件读入内存，设置进程的代码段和数据段，然后通过 execve 系统调用跳转到用户程序的入口点。那时候 ext2 的文件读写能力将真正发挥价值——Shell 不再是内置在 ramdisk 里的玩具，而是可以从磁盘加载真正的应用程序。
