# ext2 目录操作与实战验证

> 标签：ext2、目录操作、路径解析、VFS 集成、QEMU 验证
> 前置：[028_fs_ext2 第 3 篇](028-fs-ext2-3.md)

## 前言

前三篇我们搭好了 ext2 驱动的数据结构、DMA 缓冲区和文件读取路径。现在还差最后几块拼图——目录操作和路径解析。文件系统的"目录"不是什么神秘的东西，本质上就是一个"名字到 inode 号"的映射表，存在磁盘上的数据块里。但在 ext2 中，这个映射表的格式有一些有趣的设计选择——变长记录、链表式连接、4 字节对齐——这些细节在遍历和搜索目录项时都需要正确处理。

这一篇我们要讲的是目录遍历（readdir）、路径解析（lookup）以及 VFS 集成，最后用 QEMU 启动完整系统做端到端验证——这是整个 tag 028 的高潮。

## Ext2DirEntry：变长的链表式目录项

ext2 的目录数据块由一系列 `Ext2DirEntry` 结构体组成。每个结构体有一个 8 字节的固定头部（inode 号 4 字节 + rec_len 2 字节 + name_len 1 字节 + file_type 1 字节），后面紧跟变长文件名。整个条目的长度由 `rec_len` 决定，必须 4 字节对齐。所以即使文件名只有 3 字节，`rec_len` 也是 12（8 头部 + 3 名字 + 1 填充，对齐到 4）。

这种变长设计比固定长度的目录项（比如 FAT 的 32 字节条目）灵活得多——文件名最长可达 255 字节，而且通过 rec_len 合并可以高效地回收已删除条目的空间。

## 目录遍历：readdir 的实现

Cinux 的 `ext2_dir_readdir` 是一个静态函数，通过 `g_ext2_instance` 访问 Ext2 对象。它接受一个 VFS inode 和一个索引号，返回对应索引处的文件名。索引 0 和 1 被硬编码返回 "." 和 ".."——这是所有 Unix 目录都有的两个特殊条目，不需要从磁盘读取。索引 2+ 的条目需要扫描目录数据块。

扫描过程是：先计算目录占用的总块数（`dir_size / block_size`），然后对每个块调用 `read_block()` 读取数据，在块内按 `rec_len` 步进遍历每一个条目。当 `found` 计数器到达 `target`（即 `index - 2`）时就找到了目标条目，拷贝文件名到输出 buffer 并返回 1。

这种"索引号"驱动的 readdir 接口和 Linux 的 `getdents` 系统调用有微妙的区别。Linux 的 `getdents` 返回的是从指定偏移开始的一批目录项（批量返回，减少系统调用次数），而 Cinux 的接口是"给我第 N 个条目"——更简单，每次只返回一个。在 Shell 的 `ls` 命令实现中用起来很方便：从 index=0 开始递增调用 readdir，直到返回 0（没有更多条目）。

## 路径解析：从 "/" 到目标 inode

有了目录遍历的能力，路径解析就是顺理成章的事情了。`Ext2::lookup(const char* path)` 实现的是经典的 Unix 路径解析算法：从根 inode（inode 2）开始，逐级分解路径组件，每级在当前目录中查找匹配的目录项。

解析过程从跳过前导 `/` 开始，然后进入 while 循环。每次迭代提取一个路径组件（到下一个 `/` 或字符串末尾的连续非斜杠字符），调用 `lookup_in_dir()` 在当前目录中查找。`lookup_in_dir()` 本身就是一次完整的目录扫描——读取目录 inode 的所有数据块，在每个块中按 `rec_len` 步进，逐字节比对每个条目的 `name_len` 和文件名内容。找到匹配就返回 inode 号，找不到返回 0。

如果中间某一级不是目录（通过读取 inode 的 `i_mode` 检查 `EXT2_S_IFDIR` 位），解析直接失败——你不能在普通文件后面继续跟路径组件。所有组件解析完毕后，调用 `get_cached_inode(current_ino)` 返回最终的 VFS inode。

这个实现虽然简单，但有一个性能特性值得了解：每次路径解析的每一级都需要从磁盘读取目录块。对于路径 `/etc/motd`，需要读根目录块（找到 `etc`），再读 `etc` 目录块（找到 `motd`）——至少两次磁盘读取。Linux 通过 dentry cache 把"目录项到 inode 号"的映射缓存下来，第二次访问同一路径时几乎不需要磁盘 I/O。Cinux 没有这一层缓存，但教学内核的场景下路径解析的频率很低（主要发生在 Shell 执行 `ls`、`cat` 时），影响可以接受。

## VFS 集成：InodeOps 的分发机制

ext2 驱动和 VFS 的对接点在 tag 027 定义好的 `InodeOps` 结构体上。`populate_vfs_inode()` 在把磁盘 inode 转换为 VFS inode 时，根据文件类型自动绑定对应的 ops 对象——目录绑 `&ext2_dir_ops`，普通文件绑 `&ext2_file_ops`：

```cpp
uint16_t mode_type = disk.i_mode & EXT2_S_IFMT;
if (mode_type == EXT2_S_IFDIR) {
    cached.vfs_inode.type = InodeType::Directory;
    cached.vfs_inode.ops  = &ext2_dir_ops;
} else if (mode_type == EXT2_S_IFREG) {
    cached.vfs_inode.type = InodeType::Regular;
    cached.vfs_inode.ops  = &ext2_file_ops;
}
```

tag 028 的 `populate_vfs_inode` 只设置了五个字段：`ino`（inode 编号）、`size`（文件大小）、`type`（文件类型）、`ops`（InodeOps 函数指针表）和 `fs_private`（回指指针）。没有设置 mode、nlink、blocks、uid、gid 等字段——这些会在后续 tag 的 stat 功能中按需添加。

系统调用处理函数在拿到 Inode 之后，通过 `inode->ops->read(...)` 或 `inode->ops->readdir(...)` 调用对应的 ext2 静态函数——整个过程对系统调用层是透明的，它不知道也不需要知道底层是 ext2。

内核启动时的集成代码是这样的：

```cpp
static cinux::drivers::ahci::AHCI ahci;
// ... AHCI 初始化 ...

static cinux::fs::Ext2 ext2(ahci, 1);  // g_ext2_instance = this
if (!ext2.mount()) {
    cinux::lib::kprintf("[EXT2] mount failed!\n");
}

cinux::fs::vfs_mount_init();
cinux::fs::vfs_mount_add("/", &ext2);
```

注意 `AHCI` 对象从局部变量改为 `static`——因为 `Ext2` 构造函数保存了 `ahci` 的引用，如果 `ahci` 是局部变量，出了作用域后引用就悬空了。这个改动是 tag 028 引入的，很容易被忽略。

## QEMU 端到端验证

理论讲完了，现在来跑一下。首先创建 ext2 测试镜像，然后用 QEMU 启动内核（ext2.img 挂在 AHCI 端口），如果一切正常，你会在串口输出中看到类似这样的挂载日志：

```
[EXT2] Mounting ext2 filesystem on AHCI port 1
[EXT2] Superblock valid: magic=0xef53
[EXT2]   block_size=1024  inode_size=128
[EXT2]   blocks=4096  inodes=128  groups=1
[EXT2] BGDT loaded: 1 groups
[EXT2] Root inode: size=1024 mode=0x41ed
```

进入 Shell 后，可以尝试以下命令验证读取功能：

```
ls /
cat /hello.txt
```

`cat /hello.txt` 应该输出 "Hello from ext2!" 这行文字。这短短几个字代表的调用链是：Shell 解析命令 -> sys_open -> VFS 路径解析 -> ext2 lookup -> 读取根目录块 -> 找到 "hello.txt" 条目 -> 读取 inode -> 通过直接块指针定位数据块 -> AHCI DMA 读取 -> 拷贝到用户空间。tag 028 只验证了读取——写入操作（create、mkdir、unlink）将在 tag 028b 中实现和验证。

## 设计反思与下一步

回头看整个 ext2 驱动，最明显的简化在于两个方面。一是 I/O 模型——单 DMA buffer 没有缓存层，每次操作都是同步磁盘 I/O。Linux 的 page cache 和 SerenityOS 的 BlockView 都是为了解决这个问题的。但对于教学内核来说，这种"每次都和磁盘打交道"的方式有一个被低估的优势：行为完全可预测，调试时不用担心缓存一致性问题。

二是 InodeOps 的设计——静态函数指针表配合全局指针 `g_ext2_instance`，虽然限制了单实例，但实现极其简洁。tag 028b 会将 `InodeOps` 扩展为虚基类，通过引用注入支持多分区挂载。

我们在这个 tag 中实现了大约 1300 行的 ext2 驱动代码（ext2.cpp 约 757 行 + ext2.hpp 约 245 行 + ext2_types.hpp 约 310 行），涵盖了 Superblock 解析、BGDT 加载、inode 寻址、文件读取（含间接块）、目录遍历、路径解析和 VFS 集成。这是一个功能完整的只读磁盘文件系统驱动，已经能支撑 Shell 的 ls/cat 操作。

下一个 tag (028b) 将为 ext2 添加写入能力——文件创建（create）、目录创建（mkdir）、删除（unlink），以及底层的块/inode 位图分配器。届时 ext2 将从一个只读骨架进化为支持完整文件系统操作的驱动。
