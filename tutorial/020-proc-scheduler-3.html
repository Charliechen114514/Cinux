抢占调度的两个暗坑 ​

标签：调试, 时间片, IF 标志, context_switch, sti, 中断门 前置：020-2 时间片轮转：从 tick 到 context_switch

前言 ​

前两篇我们把抢占式调度器"应该怎样工作"讲清楚了——Spinlock 保护共享队列、PerCPU 保存运行时状态、tick 驱动时间片轮转、schedule 做上下文切换。现在该讲点真实的东西了：我实际上把这些代码跑起来的时候，到底发生了什么。

说实话，tag 020 的 debug 过程比 tag 019 还要"精彩"。tag 019 的三个 bug 各自独立，虽然诡异但因果关系比较清晰。tag 020 的两个暗坑则是一环扣一环：第一个坑让抢占根本没触发，我修好之后信心满满地加到 6 个线程准备看漂亮的轮转输出，结果撞上了第二个坑——只有第一个被抢占的任务能恢复中断，后续 5 个线程全部带着 IF=0 跑到结束。串口输出里的线程 A 切出去之后，B/C/D/E/F 就像商量好了似的排着队顺序跑完，仿佛调度器完全不存在。这一章我们就一个一个拆。

暗坑 1：时间片过长，抢占从未触发 ​

说实话，这个坑让我怀疑了自己整整一个下午。代码逻辑看起来完全正确——PIT 以 100Hz 产生 IRQ0、ISR stub 调用 PIT::irq0_handler()、handler 递增 tick 后调用 Scheduler::tick()、tick 计数器达到阈值后调用 schedule()。每一步都对。但串口输出就是没有交错的迹象。让我们从头开始拆。

现象 ​

先看第一个 bug。3 个线程各跑 5 次迭代，完全顺序执行，没有任何交错：

[A] thread_a iteration 0
...
[A] thread_a iteration 4
[B] thread_b iteration 0
...
[B] thread_b iteration 4
[C] thread_c iteration 0

三个线程像排好队一样一个接一个跑完，完全没有抢占的迹象。看起来像是合作式调度——但代码里根本没有 yield() 调用。定时器中断明明在跑，tick() 也被调用了，为什么没有触发切换？

根因 ​

问题出在一个非常朴素但容易被忽略的地方：时间片太长了。

PIT 配置为 100Hz（每 tick 10ms），DEFAULT_TIME_SLICE 最初设为 10，也就是说 100ms 才触发一次抢占。而忙循环 for (volatile int j = 0; j < 1000000; j++) {} 在 QEMU TCG 模式下执行极快——单次迭代不到 5ms。每个线程 5 次迭代总耗时不到 50ms，远小于 100ms 的时间片。线程在自己的时间片内就跑完了，定时器中断根本没来得及介入。

这是一个在虚拟化环境中特别容易踩的坑。物理机上 100 万次空循环可能需要 50-100ms，但在 QEMU TCG 模式下——尤其是较新版本的 QEMU 对简单指令序列有优化——10-20ms 就跑完了。你以为时间片足够短，实际上工作负载更轻。

修复 ​

修复方案是双管齐下。第一，缩短时间片：DEFAULT_TIME_SLICE 从 10 改为 2（20ms）。第二，增大忙循环：迭代次数从 100 万提升到 2000 万，确保每个线程至少需要 3-5 个时间片才能完成。

cpp
// scheduler.hpp
static constexpr int DEFAULT_TIME_SLICE = 2;  // 原来是 10

// main.cpp
for (volatile int j = 0; j < 20000000; j++) {}  // 原来是 1000000

这个参数组合（2 tick / 20ms 时间片 + 20M 次忙循环）在 QEMU TCG 和 KVM 模式下都能可靠地触发抢占。OSDev Wiki 的 Scheduling Algorithms 页面建议"A frequently chosen compromise for the quantum is between 20ms and 50ms"——20ms 正好在推荐范围的下限，对教学内核来说足够紧凑。

这个 bug 的调试过程本身值得一提。当三个线程顺序跑完时，我首先怀疑的是 tick() 根本没被调用——也许中断没开？也许 EOI 没发？于是在 tick() 入口加了一行 kprintf，重新运行后发现 tick() 确实在被调用，而且 current_slice_ 在稳定增长。接下来怀疑 schedule() 有问题——也许 pick_next() 返回了 null？又加了一行打印，发现 schedule() 根本没被调用。问题缩小到 if (current_slice_ >= DEFAULT_TIME_SLICE) 这个条件——current_slice_ 增长到 9（即 10 个 tick 后），才触发了 schedule()。这时候才恍然大悟：10 个 tick = 100ms，线程 50ms 就跑完了，当然等不到抢占。

教训 ​

在虚拟化环境中测试抢占调度时，要确保工作负载足够大或者时间片足够小，让定时器中断有机会介入。不要想当然地认为"100 万次循环应该够了"——在 TCG 模式下可能真的不够。一个更稳健的测试策略是故意把时间片设到最小（1 tick = 10ms），让抢占尽可能频繁地发生，这样更容易暴露调度器中的其他问题。同时注意，不要用 kprintf 作为工作负载来测试抢占——kprintf 涉及串口 I/O，执行时间不可预测，在不同环境下差异极大。用纯 CPU 计算的忙循环（for (volatile int j = 0; j < N; j++) {}）加上 volatile 防止优化，是更可靠的测试手段。

暗坑 2：context_switch 丢失 IF 标志 ​

第一个 bug 修好后我以为应该没问题了——时间片缩短到 20ms，忙循环增加到 2000 万次，6 个线程轮转起来应该能看到漂亮的交替输出。信心满满地重新编译运行，结果看到了一个比第一个 bug 更诡异的现象。这个 bug 从发现问题到找到根因花了我整整一个晚上，因为它的表现形式非常有迷惑性——不是"不工作"，而是"工作了一半"。

[B] done

A 跑了 2 次迭代后被成功抢占，切到了 B。但 B 从头到尾跑完了 10 次迭代，中间没有任何抢占。C、D、E、F 也一样——全部顺序跑完。只有 A 在被抢占后，等到 B-F 都跑完了才回来继续执行。这个现象非常反直觉：如果抢占成功了第一次，为什么后续的抢占全部失败？调度器选了下一个任务并执行了 context_switch，但切换过去之后，新任务似乎再也收不到定时器中断了。

这个现象说明什么？第一次抢占成功了，但后续所有的抢占都失败了。调度器选了下一个任务并执行了 context_switch，但切换过去之后，新任务再也收不到定时器中断。定时器中断被"吞"了。

根因：中断门、RFLAGS 和 callee-saved 的盲区 ​

这个 bug 的根因藏在 x86 中断处理的一个非常微妙的硬件细节里，而且和 tag 019 遗留的一个设计假设有关——context_switch 从来不需要关心 RFLAGS。这个假设在合作式调度中完全成立，但在抢占式调度中就失效了。事情要从抢占发生时的调用链说起：

IRQ0 -> ISR stub (CPU 清除 IF) -> pit_irq0_handler -> Scheduler::tick() -> schedule() -> context_switch
1

Intel SDM Vol. 3A Section 6.12.1.3 明确指出："When accessing an exception- or interrupt-handling procedure through an interrupt gate, the processor clears the IF flag to prevent other interrupts from interfering with the current interrupt handler。" 进入 ISR 的那一刻，CPU 自动把 RFLAGS 的 IF 位清零了。这是中断门的语义——不同于陷阱门（trap gate），中断门会在进入 handler 时自动关中断。

现在关键来了。context_switch.S 只保存和恢复 callee-saved 寄存器——r15、r14、r13、r12、rbp、rbx，加上 RSP 和 RIP。总共 8 个 64 位值，64 字节。RFLAGS 不在其中。这不是疏忽——在 tag 019 的合作式调度中，context_switch 总是从普通函数（yield()）中被调用的，IF 保持为 1，不存在 RFLAGS 被修改的问题。

但 tag 020 的 context_switch 现在可以从 ISR 上下文中被调用了。此时 IF=0。当它通过 jmp *56(%rsi) 跳转到新任务时，新任务继承了 IF=0 的状态。对于首次运行的新任务——ctx.rip 指向入口函数，jmp 直接跳转过去——IF 仍然是 0。这意味着该任务永远收不到定时器中断，永远不会被抢占。

那被抢占后恢复运行的老任务呢？它们的恢复路径是这样的：context_switch 的 .restore 标签恢复 callee-saved 寄存器，然后 ret 回到调用链的上层——schedule() -> tick() -> pit_irq0_handler -> ISR stub -> IRETQ。Intel SDM Vol. 3A Section 6.12.1 指出："The IRET instruction is similar to the RET instruction except that it restores the saved flags into the EFLAGS register。" IRETQ 从栈上的中断帧还原原始 RFLAGS——其中 IF=1。所以老任务不受影响。

这完美地解释了串口输出：A 被抢占后，它的恢复路径经过 IRETQ 还原了 IF=1，所以 A 后来能继续被调度。但 B、C、D、E、F 是首次运行的新任务，jmp *56(%rsi) 直接跳到入口函数，跳过了 IRETQ 的 RFLAGS 恢复——IF 保持为 0，定时器中断永远到不了。

修复：一行 sti ​

修复方法简单到让人想抽自己——在 context_switch.S 中，切换栈之后、跳转到新任务之前加一条 sti：

asm
    /* Switch to the new task's stack */
    movq 48(%rsi), %rsp              # to+48->%rsp: switch stack pointer

    /* Re-enable interrupts -- we may have entered from ISR context (IF=0).
       New tasks must start with interrupts on; resumed tasks will restore
       their original RFLAGS via IRETQ unwinding regardless. */
    sti

    /* Jump to the new task's saved instruction pointer */
    jmp *56(%rsi)                    # to+56->RIP: resume execution

为什么这条 sti 是安全的？分两种情况看。对于新任务，sti 正确地把 IF 设为 1，定时器中断正常到达，抢占正常工作。对于被抢占后恢复的老任务，恢复路径最终经过 IRETQ 会还原原始 RFLAGS——如果原来 IF=1，IRETQ 恢复 IF=1，sti 的效果被覆盖；如果原来 IF=0（理论上不应该，但假设有这种场景），IRETQ 也会恢复 IF=0。所以 sti 对老任务是无害的冗余操作。

有人可能会担心嵌套中断的风险——从 sti 到 IRETQ 的退栈路径中，中断已经打开，如果此时又一个定时器中断到达怎么办？实际上这个窗口极短——微秒级，而定时器间隔是 10ms——命中的概率可以忽略。即使真的命中了，也不会出问题：第二次 tick() 会看到当前任务的时间片还没耗尽（刚刚重置过），直接返回。

sti 为什么放在 context_switch 而不是 ISR stub 里？ ​

一种替代方案是在 ISR stub 中、调用 context_switch 之后再 sti——如果发生了切换的话。但这需要 ISR stub 知道是否发生了切换，增加了 ISR 和调度器之间的耦合。而且 ISR stub 要怎么知道"是否发生了切换"？要么让 context_switch 返回一个 bool（改变了函数签名），要么引入一个全局标志（引入了额外的共享状态）。这两种方案都不如把 sti 放在 context_switch 内部来得干净。

把 sti 放在 context_switch 内部，紧挨着栈切换之后，是语义最清晰的位置："我已经在新任务的栈上了，该让中断恢复工作了。"这种自包含的设计让 context_switch 在任何上下文（ISR、普通函数）中调用都能正确工作，调用者不需要知道 context_switch 内部是否有 sti——它只关心"切换完成后新任务的中断状态是正确的"。

一种更精细但没必要的方案 ​

另一种修复思路是把 RFLAGS 加入 CpuContext 结构体，在保存 callee-saved 时用 pushfq 保存 RFLAGS，恢复时用 popfq 还原。这种方案更"完整"——它不仅解决了 IF 问题，还能正确处理其他标志位（比如 DF、AF 等）。但对当前仅内核线程的场景来说，这些标志位的状态不是问题——内核代码不会假设某个标志位的初始值。而且多保存/恢复一个 RFLAGS（8 字节 + 两条指令）会增加 context_switch 的开销和 CpuContext 的大小——从 64 字节变成 72 字节，汇编代码中的所有偏移量都要调整。对教学内核来说，一条 sti 是最简洁的修复。

如果未来引入用户态任务，需要在 context_switch 中保存和恢复完整的状态（包括 FPU、MXCSR、debug 寄存器等），那时再考虑把 RFLAGS 加入 CpuContext 也不迟。当前的方案遵循"够用就好"的原则。

实际上，CpuContext 结构体从 tag 019 到现在只有 8 个字段、64 字节。保持这个结构体的精简是刻意的——每多加一个字段，汇编代码中的偏移量就要全部调整，static_assert 也要更新。在确认需要之前不加字段，是避免过早复杂化的好习惯。

这个 bug 的调试比第一个更有挑战性。看到串口输出时，我最初的直觉是"定时器中断停了"——B/C/D/E/F 全部跑完没被抢占，说明中断确实没到。但 A 被抢占过，说明中断机制本身是工作的。接下来怀疑是不是 context_switch 之后中断被关了——于是在 context_switch 返回后立即检查 RFLAGS 的 IF 位，果然是 0。顺着调用链回溯：IRQ0 进入时 CPU 自动清 IF（中断门语义），tick() -> schedule() -> context_switch 全程 IF=0，context_switch 的 jmp *56(%rsi) 把 IF=0 的状态带到了新任务。而新任务没有 IRETQ 的退栈路径来恢复 IF——它直接从入口函数开始执行。问题就在这里。

排查这个 bug 的关键一步是在纸上画出"新任务"和"被抢占后恢复的老任务"两条不同的恢复路径。新任务走的是 jmp -> entry function，不经过 IRETQ；老任务走的是 .restore -> ret -> schedule() -> tick() -> ISR stub -> IRETQ。两条路径对 RFLAGS 的处理完全不同——新任务继承了当前的 IF=0，老任务通过 IRETQ 恢复了原始的 IF=1。一旦画清楚这两条路径，修复方案就显而易见了：在新任务的路径上补一个 sti。这种"在纸上画执行路径"的方法对于涉及寄存器状态和栈指针的 bug 几乎是唯一可靠的方法——对着代码读很难发现"这条路径没恢复 IF"，但画出来就一目了然了。

值得一提的是，这个 bug 和 tag 019 的 0xDEADC0DE bug 有着相似的"气味"——都是因为 context_switch 用 jmp 而不是 call 跳到新任务，导致某些"隐式传递"的状态出了问题。tag 019 的问题是栈上没有返回地址（jmp 不压返回地址），tag 020 的问题是 IF 被清零（jmp 不恢复 RFLAGS）。两次都是同一个根因的不同表现：jmp 指令不做任何"附带操作"，它只改 RIP。如果调用者依赖了某些"附带状态"（栈上的返回地址、RFLAGS 中的 IF），就必须自己保证这些状态在新任务上是正确的。

回头看：从合作式到抢占式的隐藏代价 ​

这两个 bug 放在一起看，其实反映的是同一个深层问题：合作式调度和抢占式调度之间的鸿沟比表面上大得多。表面上看，tag 020 只是在 tick() 里加了几行代码、把 yield() 改成了由定时器驱动——"不就是换了个触发方式嘛"。但实际上，从中断驱动的调度开始生效的那一刻起，你的 context_switch 就不再只是从一个普通函数中被调用了——它可能从中断上下文中被调用，而中断上下文有着完全不同的 RFLAGS 状态。这个"调用上下文的改变"引入了新的正确性约束，而旧的代码（context_switch.S）是在没有这些约束的情况下设计的。

这也是为什么 Linux 的 switch_to 宏做得那么"重"——它不仅要保存 callee-saved 寄存器，还要处理 FPU 状态、debug 寄存器、IO 位图、thread-local storage 等等。每一个"附带状态"都是在一个特定的调用场景中被发现的 bug 的修复结果。Cinux 的 context_switch 目前只需要处理 callee-saved 和 IF 标志，但这只是起点。

回头看：这两个暗坑教会了我们什么 ​

这两个 bug 有一个共同特点：它们都不是"编译报错"或"立刻 crash"那种明显的错误。第一个 bug 让系统正常运行——线程跑完了该跑的逻辑，只是没有交替。如果你不仔细看输出，甚至可能觉得"线程跑完再切换不也挺正常吗"。第二个 bug 更隐蔽——只有第一个被抢占的任务受影响，后续线程都"正常"跑完了（只不过是从头到尾跑完，中间没有抢占）。两个 bug 的表现形式都像是"正常行为"，只有当你知道"应该是怎样"时才能看出不对。

内核调试的经验之一就是：在涉及中断和上下文切换的代码中，你看到的往往是间接现象而不是根因。"线程顺序跑完"不是调度器没调用 schedule()，而是时间片设置和工作负载不匹配；"后续线程没被抢占"不是定时器停了，而是 context_switch 没有正确传递中断状态。找到根因的关键是回到硬件的执行模型——每条指令执行后，CPU 的状态是什么，RFLAGS 的每一位是什么，中断是否可达。

另一个教训是关于从合作式到抢占式的迁移。合作式 context_switch 的设计不需要考虑 RFLAGS——总是在明确的调用点切换，IF 不变。但一旦从中断上下文调用同一个 context_switch，就必须保证新任务的中断状态正确。这是从 cooperative 迈向 preemptive 的经典陷阱——你的 context_switch 函数签名没变、实现逻辑没变，但调用上下文变了，隐藏的前提假设就被打破了。Linux 在 switch_to 中不需要担心这个问题，因为它在调度点有完善的 RFLAGS 管理；xv6 的 swtch 也不需要担心，因为它总是从 sched() 调用，而 sched() 运行在中断开启的 scheduler 上下文中。Cinux 的特殊情况在于 context_switch 第一次被从中断上下文中调用——而这个调用上下文引入了 IF=0 的隐含约束。

最后一个教训和测试有关。这两个 bug 说明了一件事：抢占式调度不能只靠"代码审查看起来对"来验证。你必须设计出真正能暴露问题的测试负载。第一个 bug 的教训是：测试抢占时，工作负载必须跨越多个时间片。第二个 bug 的教训是：测试线程数不能太少——只有两个线程时，第一个被抢占的线程（A）通过 IRETQ 恢复了 IF=1 然后回来继续跑，你看到的输出"看起来是交替的"，但实际上 B 在自己的时间片里带着 IF=0 跑完了。加到 6 个线程后 B-F 全部带着 IF=0 跑完，问题才变得明显。

收尾 ​

tag 020 到这里就完整了。我们从 Spinlock、PerCPU、TSS.RSP0 三块基础设施开始，搭建了抢占式调度所需的安全框架。然后实现了 tick 驱动的时间片轮转、idle 任务的兜底机制、block/unblock 为同步原语的铺路。最后在调试中踩了两个经典的暗坑——时间片过长导致抢占从未触发、context_switch 丢失 IF 标志导致后续线程中断全关——并且逐个拆解了它们的根因和修复方案。

521 行新增代码，127 行删除，19 个文件变更。从"线程必须 yield 才能切换"到"定时器中断强制切换"，这是 Cinux 调度系统的质变。回头看，tag 020 最重要的教训不是具体的代码实现，而是一个设计理念：当你的代码从一种调用上下文（普通函数）迁移到另一种调用上下文（中断处理）时，必须重新审视所有隐含的前提假设。context_switch 在 tag 019 中从不关心 RFLAGS——因为 yield() 在普通上下文中被调用，IF 始终为 1。但当中断驱动的 tick() -> schedule() -> context_switch 链路建立起来后，IF=0 成了新的常态，而旧代码对此一无所知。这种"调用上下文的改变引入了新的正确性约束"的模式在内核开发中反复出现，值得牢记。

下一个 tag 是 021_proc_sync——在抢占式调度的基础上实现 Mutex 和 Semaphore，用 block/unblock 机制让线程在锁竞争时真正地等待和唤醒。到时候抢占式调度就不再只是"定时器打断线程"这么简单了——它会成为内核同步原语的基石。
