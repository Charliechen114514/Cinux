实模式引导的设计对比与架构决策：为什么我们要自己写 Bootloader ​

标签：x86, bootloader, MBR, 实模式, 架构设计, OS 对比 前置：从 MBR 到图形模式

前言 ​

前面两章我们从零写了一个完整的实模式引导链——MBR 读磁盘加载 Stage2，Stage2 开启 A20、初始化 VESA 图形模式、保存帧缓冲信息。你可能觉得这个过程已经够折腾了，但一个自然的问题是：为什么我们要自己写 bootloader？直接用 GRUB 不就好了吗？这一篇我们跳出代码细节，从架构设计的高度来审视 Cinux 的引导方案——和 xv6、Linux、SerenityOS、ToaruOS 做一个横向对比，讨论每一个关键的设计决策是怎么做出来的，以及有哪些替代方案被我们放弃了。

为什么不用 GRUB？ ​

GRUB 是 Linux 世界最常用的 bootloader，支持 multiboot/multiboot2 协议，能直接把内核加载到保护模式或长模式，免去了实模式引导的全部麻烦。SerenityOS 就是这么做的——用 GRUB（或 Limine）跳过实模式，内核入口直接就是保护模式或长模式，省下了大量的引导代码。在 2026 年的今天，大多数新的 hobby OS 项目也倾向于选择 GRUB 或 Limine 来简化引导流程——这完全合理，毕竟实模式引导的细节对应用开发来说没有任何用处。而且 GRUB 的多启动协议还帮你把 E820 内存映射、启动设备信息等一股脑传过来了，省去了一大堆 BIOS 调用。

但 Cinux 是一个教学操作系统，我们的目标不是"用最少的代码把内核跑起来"，而是"让读者理解从按下电源键到内核运行的完整链条"。实模式引导是 x86 几十年遗产的缩影——段:偏移寻址、BIOS 中断调用、A20 地址线、VESA 图形模式——这些概念在后续的内核开发中还会反复出现（GDT 描述符和段选择子就是段机制的延续，帧缓冲信息是 GUI 驱动的基础）。跳过实模式就等于跳过了理解 x86 架构最重要的一课。笔者自己也是在写完实模式引导之后才真正理解了段寄存器的工作机制——之前只知道公式，没有手感。

当然，如果这是一个生产操作系统，笔者绝对不会自己写 MBR。GRUB 经过几十年的打磨，处理了各种 BIOS 怪癖、支持 UEFI、能读文件系统加载内核——这些工作从零开始写至少要几千行代码。教学和工程的取舍不同，Cinux 选择教学价值。

MBR vs Multiboot：引导协议的设计哲学 ​

x86 的引导方式本质上分两大阵营。第一阵营是传统的 MBR 引导：BIOS 读 512 字节的 MBR，MBR 负责加载后续的引导程序或内核。这种方式灵活但复杂——每个 OS 都要自己处理实模式的全部细节。第二阵营是 multiboot 协议（GRUB 定义）：内核镜像里包含一个特殊头（multiboot header），GRUB 识别这个头后直接把内核加载到保护模式，在寄存器里传递机器信息（内存映射、启动设备等）。

Cinux 选择了 MBR 方式。这意味着我们需要自己处理实模式的所有初始化——段寄存器规范化、DAP 磁盘读取、A20 开启、VESA 模式设置。好处是每一步都是透明的，读者能看到完整的启动过程。坏处是代码量大、容易出 bug、而且在不同硬件上可能遇到各种奇怪的兼容性问题。笔者在调试 MBR 的那几天里确实血压拉满——CS 不确定的坑、栈踩 DAP 的坑、超 512 字节的坑——但回过头来看，这些踩坑经历对理解 x86 架构的帮助远超直接用 GRUB 跳过去。

如果读者后续想在自己的项目中使用 GRUB 或 Limine，完全没问题——理解了实模式引导的细节之后，你会更深刻地理解 GRUB 帮你省掉了哪些工作。multiboot 协议本质上就是"把我们在实模式下手动做的初始化工作，交给 GRUB 来完成"——GRUB 负责切换到保护模式、传递内存映射信息、加载内核到正确的地址。知道这些之后用 GRUB 的时候心里才有底，出了问题也知道自己排查。

xv6 选择了类似的方式，但更简化——它的 bootasm.S 做 MBR 引导后直接切保护模式，不处理 VESA 和图形模式（xv6 使用文本模式）。Linux 则把两种方式结合起来：内核自身包含一段实模式 setup 代码，既支持被 GRUB 加载（通过 multiboot/x86 boot protocol），也支持被自己的实模式 setup 代码引导。

Stage2 的存在意义：为什么 MBR 不能直接进保护模式？ ​

你可能会想：MBR 已经能读磁盘了，为什么不直接在 MBR 里设置 GDT、切保护模式、然后跳到内核？技术上当然可以——xv6 的 bootasm.S 就是这么做的，MBR 加载内核后直接 lgdt + 设置 CR0.PE 然后远跳到 32 位保护模式。

但 Cinux 把引导过程拆成了 MBR 和 Stage2 两个阶段，原因是 VESA 初始化必须在实模式下通过 BIOS 调用完成。保护模式下 BIOS 中断不可用（IVT 被屏蔽了），VBE 的 INT 0x10 调用必须在实模式或者通过复杂的 unreal mode 技巧来完成。如果我们把 VESA 初始化也塞进 512 字节的 MBR，空间根本不够——光是三个 VESA 函数加上 ModeInfoBlock 的处理逻辑就需要几百字节。

所以 Cinux 的设计是 MBR 只做最核心的事情（初始化段寄存器 + 读磁盘），Stage2 负责实模式下所有"昂贵"的初始化（A20 + VESA），两者分工明确。等到 Stage2 把该在实模式下做的事情都做完了，后续的 tag 才会设置 GDT 切到保护模式。

设计对比：Cinux vs ToaruOS vs Linux vs xv6 ​

这一段值得详细展开，因为这四个项目的引导方案代表了教学操作系统到生产操作系统引导设计的光谱。

xv6：极简引导 ​

xv6 的引导是所有项目中最简单的。bootasm.S 做标准的 MBR 入口初始化（远跳转、段寄存器归零、设置栈），然后直接开启 A20、设置 GDT、切保护模式。bootmain.c 在保护模式下用 IDE 端口 I/O 读取 ELF 格式的内核，然后跳转过去。整个过程不涉及 VESA，不涉及复杂的内存布局规划。xv6 的目标是教学操作系统的核心概念（进程、文件系统、系统调用），引导过程能省则省。

值得注意的是 xv6 用 C 语言写第二阶段引导（bootmain.c），这比 Cinux 用汇编写 Stage2 要高级得多。C 语言的好处是可以用循环、指针运算、结构体来处理 ELF header，代码可读性远超汇编。代价是需要一个支持 32 位保护模式的 C 编译环境——xv6 用 -m32 编译 bootmain.c，而 Cinux 目前 bootloader 阶段全是汇编。这是一个有意的设计选择：Cinux 的前几个 tag 追求"每行代码都能追溯到硬件行为"，用汇编更透明。

Linux：工业级引导协议 ​

Linux 的引导设计是最复杂的。内核镜像（bzImage）自带一段实模式 setup 代码，定义了完整的 x86 boot protocol。外部 bootloader（GRUB、SYSLINUX 等）加载内核后跳到这段 setup 代码，setup 代码负责版本协商、E820 内存映射查询、EDD 磁盘参数获取、视频模式设置（支持 VBE、文本模式、EFI GOP），然后解压保护模式内核。整个协议定义了几十个字段的数据结构（boot_params），涵盖了启动时需要传递给内核的所有信息。

Cinux 的 VESA 初始化和帧缓冲信息保存有点类似 Linux 的 video mode setup，但规模小得多——我们只硬编码了一个模式，而 Linux 支持用户通过命令行参数选择模式。Linux 的 boot_params 结构里有 screen_info 子结构，记录了视频模式、帧缓冲地址、光标位置等信息，内核的 fbdev 子系统驱动会从这些字段初始化显示。Cinux 的 0x6400 处 16 字节帧缓冲信息本质上就是 screen_info 的极简版本。

另一个值得对比的点是 Linux 内核的实模式 setup 代码会做 E820 内存映射查询——通过 INT 0x15 AX=0xE820 获取物理内存布局，把结果写入 boot_params.e820_map。Cinux 在当前 tag 还没做这个，但后续 tag 进入保护模式和长模式后也需要获取物理内存布局。届时会走相同的 INT 0x15 E820 路径——这是 x86 实模式提供的最后一个"免费午餐"（进入保护模式后就再也不能用 BIOS 中断了）。

ToaruOS：unreal mode 的黑魔法 ​

ToaruOS 的 bootloader 是教学 OS 中最复杂的——它使用 unreal mode 技巧：先切到保护模式开启 A20 和 32 位地址线，再切回实模式但保留 32 位地址线的能力。这样就可以在实模式下使用 32 位地址访问超过 1MB 的内存，同时还能调用 BIOS 中断。ToaruOS 的 bootloader 还包含 ISO 9660 文件系统驱动和图形启动菜单。

Cinux 没有用 unreal mode——我们的 Stage2 全程在普通实模式下运行，通过 BIOS 调用完成所有初始化。这更简单、更容易理解，代价是地址空间限制在 1MB 以内（对 bootloader 来说完全够用）。unreal mode 虽然强大，但调试起来非常困难——你需要理解保护模式和实模式之间的切换细节，CPU 的行为在各种边界情况下不太直觉。对于教学项目来说，这种黑魔法级别的技巧反而会分散读者对核心概念的理解。

SerenityOS：直接跳过实模式 ​

SerenityOS 使用 GRUB/Limine 的 multiboot 协议，内核入口就是保护模式或长模式。这样做的好处是省去了大量实模式引导代码（SerenityOS 完全没有 boot/ 目录下的汇编文件），但代价是依赖外部 bootloader。SerenityOS 的选择是合理的——它是一个功能完整的桌面操作系统，引导过程不是教学重点。

值得一提的是 SerenityOS 的 multiboot 入口代码非常优雅——它用 C++（带 extern "C" 声明）写入口点，GRUB 传来的 multiboot info 结构直接作为参数传入，内核可以立刻获取内存映射、启动设备、命令行参数等信息。这和 Cinux 在实模式下辛辛苦苦通过 BIOS 中断逐个获取这些信息形成了鲜明对比——但代价是对 bootloader 的依赖。如果 GRUB 有 bug 或者不支持你的硬件，你几乎无能为力。自己写 MBR 虽然累，但每一行代码都在你的掌控之中。

从引导到内核：信息传递的艺术 ​

bootloader 到内核之间的信息传递是操作系统设计中的一个经典问题。实模式 bootloader 拥有 BIOS 提供的丰富信息——内存映射（E820）、磁盘参数（EDD）、视频模式（VBE）——但这些信息必须在离开实模式之前收集好，因为保护模式下 BIOS 调用全部失效。

Cinux 目前只传递了帧缓冲信息（16 字节，固定在 0x6400）。后续 tag 进入长模式后，内核还需要物理内存布局信息。Cinux 的方案是在实模式引导阶段通过 INT 0x15 AX=0xE820 获取内存映射，保存在另一个固定地址，然后内核在长模式下读取。Linux 的 boot_params 结构用了一整个页面（4096 字节）来传递所有信息——帧缓冲、内存映射、ACPI 表地址、APIC 信息、命令行参数——应有尽有。Cinux 的信息传递更轻量，但随着内核功能的增长，可能也需要一个类似 boot_params 的统一结构。

这个信息传递机制的设计有一个隐含的约束：bootloader 和内核必须对数据结构的布局和地址达成一致。在 Cinux 中这是通过硬编码常量实现的（0x6400 存帧缓冲信息），Linux 通过 x86 boot protocol 文档规范。无论哪种方式，接口契约一旦确定，双方都必须遵守——修改布局需要同时更新 bootloader 和内核。

这里还有一个更深层的设计问题：固定地址传递 vs 寄存器传递 vs 结构体指针传递。实模式下选项有限——没有通用寄存器可以传递大量数据（16 位寄存器最多传 2 字节），所以固定地址或结构体指针是唯二的选择。Cinux 用固定地址（0x6400），Linux 用结构体指针（boot_params 的物理地址放在 header 里约定好）。两种方式本质上相同——都是"在某处放一个数据结构，双方都知道地址"。区别在于固定地址更简单但不灵活，结构体指针可以动态分配地址但需要更复杂的协商机制。

到了长模式阶段，Cinux 的内核入口会通过 System V AMD64 ABI 的 RDI 寄存器接收 BootInfo 结构体的指针——这时寄存器传递就可行了，因为 64 位寄存器可以容纳一个完整的物理地址。从固定地址到寄存器指针的演进，本身就是从实模式到长模式的能力提升的一个缩影。

这里有一个值得思考的设计问题：bootloader 和内核之间的接口应该多"厚"？Cinux 目前的接口只有 16 字节的帧缓冲信息，非常精简。Linux 的 boot_params 有几百个字段，覆盖了启动时可能需要的所有信息。接口越厚，内核启动越方便（不用自己去查询硬件），但 bootloader 也越复杂（需要收集更多信息）。接口越薄，bootloader 更简单，但内核启动时需要自己做更多初始化工作。Cinux 选择薄接口——只传递 bootloader "顺手就能拿到"的信息（帧缓冲参数），其他的留给内核自己处理。

为什么是 VESA 而不是 VGA？ ​

Cinux 使用 VESA BIOS Extensions 设置图形模式，而不是传统的 VGA 编程。这个选择背后有一段值得展开的历史。

VGA（Video Graphics Array）是 IBM 在 1987 年随 PS/2 引入的显示标准，提供 640×480×16 色或 320×200×256 色模式。VGA 编程通过 I/O 端口（0x3C0-0x3DF）和帧缓冲（0xA0000-0xBFFFF，64KB 窗口）完成，不依赖 BIOS 中断。优点是完全可控，缺点是分辨率太低、颜色太少、编程模型复杂（需要操作多个寄存器设置时序参数）。

VESA VBE 在 VGA 之上提供了标准化的高分辨率接口。1024×768×32bpp 比 VGA 的 640×480×16bpp 不是一个量级的提升——像素数量多了 2.5 倍，颜色数量多了 2^16 倍。而且 VBE 使用线性帧缓冲，直接往一个物理地址写像素值就行，不需要像 VGA 那样操作 I/O 端口设模式。

在 bootloader 阶段选择 VESA 而不是 VGA 是因为 Cinux 的目标是一个带有 GUI 的桌面操作系统（后续 tag 会实现窗口管理器和图形应用），640×480 的分辨率完全不够用。而 VESA 的调用方式（INT 0x10 BIOS 中断）在实模式下非常自然——正好利用了我们还没有离开实模式的窗口期。一旦切换到保护模式，想设置 VESA 模式就得自己写显卡驱动了，那是一个完全不同量级的工程。

当然 VESA 也不是终点。现代操作系统在 UEFI 环境下使用 GOP（Graphics Output Protocol）设置图形模式，或者在内核启动后通过 DRM/KMS（Direct Rendering Manager / Kernel Mode Setting）驱动直接编程 GPU。VESA 是"能用"的方案，不是"最优"的方案。但对于教学操作系统来说，VESA 在实模式下通过几次 BIOS 调用就能设好图形模式，这个简洁性是无与伦比的。

有些教学 OS（比如 xv6）根本不设图形模式，全程使用 VGA 文本模式——80×25 字符的文本界面，每个字符用 2 字节表示（1 字节 ASCII + 1 字节颜色属性）。这对纯命令行的教学内核来说够用了，但 Cinux 要做 GUI，所以必须在 bootloader 阶段就把图形模式设好。这也是 Cinux 的引导比 xv6 复杂得多的一个重要原因。

硬编码 vs 动态查询：模式号的取舍 ​

Cinux 硬编码了 VESA 模式号 0x118（1024×768×32bpp）。这个选择在教学环境中是合理的——QEMU 总是支持这个模式，代码简洁明了。但在生产环境中，硬编码模式号是危险的——不同显卡支持的模式号不同，0x118 在某些硬件上可能根本不存在。

正确的做法是动态查询：先用 INT 0x10 AX=0x4F00 获取 VbeInfoBlock，从中取出 VideoModePtr（模式号列表的远指针），然后逐个查询每个模式号的 ModeInfoBlock，找到分辨率和色深满足要求的模式。Linux 的引导过程就是这么做的——用户可以通过 vga= 内核参数指定模式，如果不指定就自动选择一个合适的。

Cinux 选择硬编码是为了减少代码复杂度——VESA 三步流程已经够多了，再加上模式枚举会让 Stage2 更臃肿。后续如果需要支持更多显示环境，可以添加一个简单的模式枚举和回退逻辑：先试 0x118，失败后试 0x117（1024×768×16bpp），再失败试 0x111（640×480×16bpp）。

这种"先硬编码，后续再泛化"的开发策略在操作系统开发中很常见。先把核心路径跑通，确保功能正确，然后再处理各种边界情况和兼容性问题。过早的泛化会导致代码膨胀和复杂度失控——这在只有一个开发者（或者少数几个开发者）的项目中尤其危险。

线性帧缓冲 vs Banked 模式：VBE 的两种显示模型 ​

在讨论 VESA 模式设置时，有一个底层的设计选择值得一提：线性帧缓冲（Linear Framebuffer，LFB）和 banked 模式的区别。Cinux 使用 LFB（BX=0x4118 中 bit14=1），这是现代的方式。但理解 banked 模式的存在有助于理解 VBE 的设计历史。

banked 模式是 VBE 1.x 时代的方式。显存被分成多个 64KB 的"bank"，每次只能访问其中一个。要访问完整的显存需要通过 VBE 调用切换当前活动的 bank——就像在一本书里翻页一样。这种方式的问题是效率低（每次切换 bank 都有一次 BIOS 调用开销）且编程复杂（画一条跨越 bank 边界的线需要特殊处理）。

LFB 是 VBE 2.0 引入的方式——整个显存被映射到一段连续的物理地址空间，可以直接用线性地址访问任何像素，不需要 bank 切换。Cinux 设置模式时 BX=0x4118 中的 0x4000（bit14）就是告诉 BIOS 要用 LFB 模式。设置成功后，PhysBasePtr（ModeInfoBlock 偏移 +0x28）指向的就是整个显存的起始物理地址，直接往那里写就行。

LFB 的缺点是需要足够的物理地址空间来映射整个显存——1024×768×4 字节 = 3MB 的线性地址空间。在实模式的 1MB 地址空间里这显然放不下，但 LFB 映射的地址通常在高位物理内存（QEMU 里是 0xFD000000），不占用低 1MB 的空间。到了保护模式和长模式后通过页表映射就能正常访问了。

回顾与展望 ​

实模式引导是 x86 操作系统开发中最"底层"的部分——直接和硬件、BIOS、几百年积累的兼容性包袱打交道。我们从按下电源键那一刻开始（CPU 从 0xFFFFFFF0 取第一条指令），一路走到了 1024×768×32bpp 的图形模式和整齐的帧缓冲信息。这个过程涉及的技术概念——段:偏移寻址、BIOS 中断、DAP 磁盘操作、A20 地址线、VESA 图形模式——每一个都是 x86 架构演进的缩影。

从下一篇开始，我们要进入保护模式和长模式的世界——GDT、IDT、分页、C++ 内核、进程调度、文件系统。那些才是操作系统开发的"正片"。但实模式引导教会我们的那些原则——精打细算的空间管理、防御性编程、错误处理策略、接口设计——在整个内核开发过程中都不会过时。

实模式的局限与告别 ​

实模式有几个根本性的限制，决定了我们必须尽快离开它。第一，1MB 的地址空间上限（即使开了 A20 也只有 64KB 的 High Memory Area 可以额外访问），对现代操作系统来说远远不够。第二，没有内存保护——任何代码都能写任何地址，一个野指针就能让整个系统崩溃。第三，没有分页——没有虚拟地址，没有按需加载，没有内存映射文件。第四，16 位寄存器的限制让数据处理非常笨拙，连一个 32 位的整数运算都需要多次进位处理。

但实模式有一个无可替代的优势：BIOS 中断。INT 0x10 做屏幕输出、INT 0x13 做磁盘读取、INT 0x15 做内存查询和 A20 控制、INT 0x16 做键盘输入——这些服务是硬件厂商在 BIOS ROM 里提供的，免费且经过充分测试。一旦切换到保护模式，IVT 被屏蔽，所有这些服务都不可用了。这就是为什么 Cinux 在 Stage2 里把所有需要 BIOS 的事情（VESA 初始化、A20 开启）都在实模式下做完——保护模式下再做这些事情需要自己写驱动（直接操作 I/O 端口或 MMIO），复杂度完全不是一个量级。

进入保护模式后，我们获得的是 32 位地址空间、段级内存保护、以及未来开启分页的基础。再往后进入长模式，64 位地址空间和解锁的额外寄存器让内核开发终于进入"正常"的编程体验。但这一切的起点都是实模式——x86 的设计哲学是"向后兼容至上"，每一代新功能都是在旧模式上叠加，而不是替代。理解实模式就是理解 x86 之所以这么复杂的历史根源。

地址模型的设计决策 ​

实模式引导中最大的设计挑战是地址模型。MBR 被加载到 0x7C00，链接地址设为 0x7C00，段寄存器统一为 0——这是一种"平坦地址"模型，偏移量等于物理地址。Stage2 被 MBR 跳转时 CS=0x0800，但链接地址设为 0——这是一种"段偏移"模型，CS 提供基地址，链接偏移是段内偏移。

为什么两种地址模型不同？因为 MBR 的空间太小，只能用最直接的方式——链接地址等于运行地址。Stage2 有更多空间，用段偏移模型可以让代码更简洁（所有偏移从 0 开始）。如果 Stage2 也用平坦地址模型（链接地址 0x8000），每次修改 Stage2 加载地址时需要同时修改链接脚本和跳转指令，容易出错。

这个地址模型的选择还影响了后续的调试体验。MBR 的平坦地址模型意味着 GDB 中的地址和代码中的地址直接对应——break *0x7c00 就能断在 MBR 入口。Stage2 的段偏移模型则需要一点理解：GDB 显示的线性地址是 CS × 16 + IP，所以 Stage2 入口在 GDB 里显示为 0x8000（0x0800 × 16 + 0），但链接地址是 0——反汇编时看到的符号地址都是 0 开始的偏移。这种"两层地址"的感觉在后续进入保护模式和长模式后会更常见——虚拟地址和物理地址的映射本质上也是同一个问题。

踩坑总结：实模式引导中的那些"经典陷阱" ​

写完整个实模式引导链之后回头看，有几个坑几乎每个 bootloader 开发者都会踩到，值得单独总结一下。

第一个大坑是 CS 不确定。这不是一个理论上的问题——笔者在 Bochs 和 QEMU 上测试时真的遇到了不同的 CS 值。Bochs 上 CS=0x0000 一切正常，QEMU 的 SeaBIOS 上偶尔 CS=0x07C0，字符串输出就全废了。ljmp 那条远跳转不是"可选的最佳实践"，而是"必须写的保命代码"。

第二个大坑是 MBR 超 512 字节。最初把 mbr.S 和 common/serial.S 放在一起链接时，代码偶尔能跑偶尔崩溃，排查了半天才发现是 .text 段超了。BIOS 只加载第一个扇区，common.S 的代码根本没被读进内存，call 跳到的是未初始化的内存区域。从那以后笔者对 MBR 的空间预算变得非常敏感。

第三个大坑是段超越前缀。在 vesa_save_framebuffer_info 里，从 ModeInfoBlock（ES 段）读数据写到 FB Info（GS 段），必须用 %es: 和 %gs: 前缀。不加的话默认用 DS 段——而 Stage2 里 DS=0x0800，读到的就是完全错误的位置。这个 bug 的隐蔽之处在于：如果 DS 碰巧和 ES 指向同一个段（比如都是 0），数据碰巧是对的，所以可能在某些环境下通过测试，换一个环境就炸了。

第四个大坑是 DAP 偏移 1 的保留字节。DAP 结构的第二个字节是保留字段，规范要求必须为 0。Cinux 的代码没有显式写入这个字节——依赖的是"BIOS POST 后低地址内存为零"的假设。这个假设在 QEMU 和大多数真机上都成立，但并不是规范保证的行为。更严谨的做法是显式写 movb $0, 1(%si)。这种"依赖未定义行为"的 bug 在 C++ 里大家已经很熟悉了（UB 是万恶之源），在汇编里同样存在。

A20 开启方式的取舍 ​

A20 地址线的开启方式有四种之多，这个数量本身就说明了问题的复杂性——每种方式都是不同时代的产物，没有一种在所有硬件上都可靠。

BIOS INT 0x15 AX=0x2401 是 Cinux 使用的方式，也是最安全的。它通过 BIOS 抽象了底层硬件差异，BIOS 内部会根据主板类型选择最合适的开启方法。缺点是需要一次中断调用，在 BIOS 有 bug 的主板上可能失败。Fast A20（端口 0x92 bit 1）最快——两条 I/O 指令就搞定了，但端口 0x92 的 bit 0 连着 reset line，操作不当会触发系统重启。键盘控制器 8042 方式最复杂——需要向键盘控制器的命令端口发送一系列命令，等待响应，还要处理键盘控制器繁忙的情况。端口 0xEE 方式是某些 BIOS 特有的，不是所有硬件都支持。

Cinux 选择 BIOS 方式是因为在 QEMU 环境中它总是有效的，而且代码最简洁。如果未来需要在真机上运行，可以添加回退逻辑：先试 BIOS 方式，失败后试 Fast A20，再失败试 8042 方式。Linux 内核的 A20 开启代码就是这样做的——层层回退，直到找到一种可行的方式。这种"渐进式回退"的设计模式在操作系统开发中很常见——先试最优方案，失败后回退到兼容方案。

A20 还有一个有趣的测试方法：写一个值到 0x100000（1MB 边界），然后检查 0x00000 处是否也被修改了。如果相同，说明 A20 没开（地址回绕）；如果不同，说明 A20 已开。Linux 的 A20 测试函数就是用这个方法验证开启是否成功的。Cinux 没有做这个测试——在 QEMU 中 INT 0x15 返回成功就足够了。但在真机上，即使 BIOS 说成功了，也值得做一次验证。

实模式的未来——UEFI 和 Legacy Boot 的消亡 ​

在 2026 年的今天，实模式引导已经是一个正在消亡的技术。UEFI（Unified Extutable Firmware Interface）取代了传统 BIOS，直接从保护模式或长模式启动，完全跳过了实模式阶段。新出厂的主板几乎都默认使用 UEFI，Legacy BIOS/CSM（Compatibility Support Module）支持正在被逐渐移除。

这对教学操作系统意味着什么？实模式引导的代码在真实硬件上可能跑不起来了——UEFI 固件不加载 MBR，而是从 FAT32 分区读取 EFI 系统分区里的 .efi 可执行文件。但 QEMU 仍然支持 Legacy BIOS 启动（使用 SeaBIOS 固件），所以在虚拟环境中学习和测试实模式引导完全没问题。

从长远来看，Cinux 可能需要添加 UEFI 启动支持。但实模式引导的教学价值不会因此降低——段机制、BIOS 中断、A20、VESA 这些概念是理解 x86 架构演进的基础。即使未来 UEFI 成了唯一的启动方式，理解"过去是怎么做的"依然能帮助你理解"现在为什么这么做"。就像学 C 语言理解指针和内存管理对写高级语言程序依然有帮助一样。而且 x86 CPU 的上电行为到今天也没有变——它仍然从实模式开始执行，只是 UEFI 固件在你看不到的地方更快地完成了模式切换。

话说回来，UEFI 本身也不是什么完美的方案——它引入了大量的复杂性（GPT 分区表、EFI 系统分区、安全启动、PE/COFF 格式的 .efi 可执行文件）。很多 OS 开发者抱怨 UEFI 的复杂度比实模式引导还高，只是把一种痛苦换成了另一种痛苦。对于教学来说，传统 BIOS 启动的"简单粗暴"反而更容易理解。

另外，即使 UEFI 绕过了实模式，x86 CPU 的上电行为没有变——它仍然从实模式（或者叫" unreal mode "）开始执行，只是 UEFI 固件接管了这个过程，在你看不到的地方完成了模式切换。理解实模式就是理解硬件的真实行为，而不仅仅是固件帮你封装后的抽象。

帧缓冲信息传递：bootloader 到内核的接口设计 ​

Stage2 在 0x6400 保存了 16 字节的帧缓冲信息（物理地址、pitch、宽、高），这是 bootloader 和内核之间唯一的接口契约。后续内核初始化图形驱动时会直接从 0x6400 读取这些数据。

这个接口设计有几个值得讨论的点。第一，为什么用固定地址而不是结构体传递？因为在实模式下没有标准的调用约定（不像 System V AMD64 ABI 用 RDI 传第一个参数），固定地址是最简单可靠的方式。Linux 使用 boot_params 结构（物理地址 0x7000 开始）传递更丰富的信息，思路相同但规模更大。第二，为什么只保存 16 字节而不是完整的 ModeInfoBlock？因为 ModeInfoBlock 有 256 字节，大部分字段在内核阶段没用。只保存关键字段既节省空间又让接口更清晰。

收尾 ​

到这里，实模式引导的三个篇章全部完成了。我们不仅写了代码，还理解了每一个设计决策背后的考量——为什么自己写 bootloader 而不用 GRUB，为什么拆成 MBR 和 Stage2 两个阶段，为什么用段偏移地址模型，为什么帧缓冲信息只保存 16 字节。这些决策在后续的内核开发中还会反复出现——接口设计、地址模型、信息传递方式——都是操作系统设计的基础课题。

回顾这三个篇章走过的路：第一篇从 BIOS 把接力棒交给 MBR 开始，我们在 512 字节的限制里精打细算，实现了段寄存器初始化、精简版字符串输出、DAP 磁盘读取、远跳转到 Stage2。第二篇里 Stage2 走完了实模式的最后一段旅程——开启 A20 突破 1MB 限制、通过 VBE 三步曲设置 1024x768 图形模式、保存帧缓冲信息留给内核。这一篇跳出代码细节，从设计哲学的角度审视了整个引导方案，和 xv6、Linux、ToaruOS、SerenityOS 做了横向对比。每一章都不是孤立的——第一章的 512 字节限制决定了第二章的 MBR+Stage2 拆分，第二章的实模式 BIOS 调用决定了 VESA 必须在切换保护模式之前完成，这些约束环环相扣。

实模式引导虽然繁琐，但它教会了我们很多在后续内核开发中依然适用的原则：精打细算的空间管理（512 字节 MBR 里的每一字节都有存在理由）、防御性编程（ljmp 规范化 CS、段超越前缀处理跨段数据）、错误处理策略（panic 快速失败而非静默继续）、以及跨模块接口设计（固定地址传递帧缓冲信息）。这些原则在内核的内存管理、进程调度、驱动开发中还会反复出现。

下一篇我们要离开实模式了。设置 GDT，切换到保护模式，然后继续推进到 64 位长模式。真正的 x86 内核开发才刚刚开始——但有了实模式引导打下的基础，后续的一切都会变得更容易理解。
